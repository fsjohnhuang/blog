** ~char*~, ~const char*~, ~string~, ~wstring~ 和 ~CString~ 间的类型转换
   1. ~const char*~ 被称为 C style string
      #+begin_src c
      // 字符串字面量
      "Hello world"
      // 数组形式
      char hi[3] = {'H', 'I', '\0'};
      // 直接赋首地址
      char hi[3] = "HI";
      // 省略长度
      char hi[] = "HI";
      char hi[] = {"HI"};
      #+end_src
   2. ~string~ 和 ~wstring~ 是C++标准的字符串类型
      - ~string~ 位于 ~<string.h>~ 文件中，采用MBCS(多字节字符集)，即英语字母和半角标点符号占用一个字节，而中文和全角标点符号占两个字节；
      - ~wstring~ 采用UNICODE字符集，默认采用UTF-16的存储方式，所以所有字符都占两个字节；
   3. ~CString~  采用UNICODE字符集，默认采用UTF-16的存储方式，所以所有字符都占两个字节。并且作为MFC专用字符串类型，和MFC的各个控件有密切的关系。

   #+begin_src c++
   // string -> const char*
   string szText = "abc";
   const char* czText = szText.c_str();

   // string -> char*
   string szText = "abc";
   const int iLen = szText.length() + 1;
   char* lpChar = new char[iLen];
   strcpy_s(lpChar, iLen, szText.c_str());

   // * -> string
   const char *c = "abc";
   string szText(c);

   // const char * -> char *
   const char *lpcz = "abc";
   const int iLen = strlen(lpcz) + 1;
   char* lpChar = new char[iLen];
   strcpy_s(lpChar, iLen, lpcz);   

   // string -> wstring 需要CString作为中间件
   string st("hi");
   CString cs(st.c_str());
   wstring wst(cs);

   // wstring -> string 需要CString作为中间件
   wstring wst("hi");
   CString cs(wst.c_str());
   string st = CStringA(cs);
   #+end_src

   若不是MFC环境，那么就无法通过CString作为string和wstring之间转换的桥梁，此时需要使用C函数mbstowcs_s和wcstombs_s，并且在使用前需要通过 ~setlocale(LC_ALL, "")~ 设置本地字符集。

** 指针
   
*** 基础
    内存变量简称为变量，每定义一个变量系统会为该变量分配一块内存，并且内存是有地址的。
    #+begin_src c
    int idx = 1; // 系统为变量idx分配一块大小为4个byte的内存空间。
    &idx; // 通过取址符，获取变量idx的内存空间的第一个byte的地址。
    #+end_src
    指针变量简称为指针，用于存放其它变量的地址。
    #+begin_src c
    int *ptrIdx; // 一个存放int类型的变量的地址的指针变量
    ptrIdx = &idx; // 将int类型的变量地址赋值给指针变量
    #+end_src
    通过*操作符来操作指针变量指向的内存变量的值。
    #+begin_src c
    *ptrIdx += 1; // idx内存变量值从1更新为2。
    *&idx += 1; // idx内存变量值从2更新为3。
    #+end_src

*** 应用场景    
    1. 函数参数

	 int idx; // 定义int类型的内存变量idx，但内存的值还没被初始化，因此该值是随机无意义的。
	 scanf_s("%s %d", message, &idx); // 从标准输出获取字符串，并格式化后赋值给message和idx

    #+end_src
    空指针判断： ~ptrIdx == 0~

*** 野指针
    指针变量所指向的内存已被释放，但指针的值不会被清零。对野指针操作的结果不可预知。
    #+begin_src c
    int *ptrIdx = (int)malloc(sizeof(int));
    *ptrIdx = 1;
    free(ptrIdx);
    // 此时ptrIdx依然保存着原来所指向的内存变量的地址，但该内存变量的内存空间已经被释放，因此ptrIdx将变为野指针。
    // 更安全的做法是释放内存后，将指针赋值为0
    ptrIdx = 0;
    #+end_src

    #+begin_src c
    // message标识的内存空间中，存放的是数组第一个元素所在的内存空间的地址
    // *message则是根据message标识的内存空间中存储的另一个内存空间的地址，获取后者内存空间的数据
    const char *message = "hello";
    #+end_src

*** 数组的地址

** 结构体的内存对齐
   CPU以 *块* 划分内存空间，并以 *块* 为单位读写内存空间。
   *块* 大小称为memory granularity(内存读取粒度)，可以设置为1、2、4、8、16字节. 通过宏 ~#pragma pack(4)~ 指定内存对齐的字节数,取值为1,2,4,8,16.
   而结构体的元素是按顺序存储,但不是紧密排列的.而是每个元素必须分配 *块* 大小的倍数,以便提高CPU操作内存的速度.
   64位系统默认的 *块* 大小为4字节.

* 三大函数(拷贝构造，拷贝赋值，析构)
   若没有定义，编译器会自动补充拷贝构造、拷贝复制和析构函数。
   若类中包含指针，则必须自定义这3大函数，否则会造成浅拷贝，且在进行拷贝赋值时会造成内存泄漏(Memory Leak)

** String正确定义三大函数的示例
   #+begin_src c++
#pragma once
#include <iostream>
#include <string>
using namespace std;

class String {
public:
	// 构造函数(ctor)
	String(const char* cstr = 0) {
		if (cstr == 0) {
			m_data = new char[1];
			*m_data = '\0';
		}
		else {
			m_data = new char[strlen(cstr) + 1];
			strcpy(m_data, cstr);
		}
	}
	// 拷贝构造函数(copy ctor)
	String(const String& o) {
		m_data = new char[strlen(o.m_data) + 1];
		strcpy(m_data, o.m_data);
	}
	// 拷贝赋值函数(copy assignment operator)
	String& operator = (const String& o) {
		// 检测自我赋值
		if (this == &o) return *this;

		delete[] m_data;
		m_data = new char[strlen(o.m_data) + 1];
		strcpy(m_data, o.m_data);
		return *this;
	}

	// 析构函数(dtor)
	~String() {
		delete[] m_data;
	}
	char* get_c_str() const {
		return m_data;
	}
private:
	char* m_data;
};

ostream& operator << (ostream& os, const String& s) {
	return os << s.get_c_str();
}
   #+end_src

      struct __list_iterator {
        typedef __list_iterator<T, Ref, Ptr> self;
        typedef Ref reference;
        typedef Ptr pointer;
        typedef __list_node<T>* link_type;

        link_type node;
        reference operator* () const { return (*node).data; } // 获取节点的数据对象
        pointer operator-> () const { return &(operator*()); } // 操作节点的数据对象的成员
        bool operator== (const self& x) const { return node == x.node; }
        bool operator!= (const self& x) const { return !operator==(); }
        self& operator++ () { node = (link_type)(node->next); return *this; } // ++i
        self operator++(int) { self tmp = *this; ++*this; return tmp;} // i++
        self& operator-- () { node = (link_type)(node->prev); return *this; } // --i
        self operator--(int) { self tmp = *this; --*this; return tmp;} // i--

** 模板模板参数(template template parameter)
   用于模板实参也是模板的场景。
   1. 模板仅包含单个模板参数的场景

      template<class T, template<class T> class Container> // 类型参数Container的实参也是一个包含单个模板参数的类模板
      struct ctx {
        Container<T> el;
        ctx(Container<T>  e) : el(e) {}

* 创建对象时 ~ClassName inst~ 和 ~ClassName inst()~ 的区别

** 创建栈对象时
   1. ~ClassName inst~
      1. 当没有自定义构造函数时会使用编译器为struct或class创建的默认构造函数。
      2. 当存在自定义构造函数时编译器将将不会为struct或class创建默认无参数的构造函数，若没有自定义无参数的构造函数则会报编译失败。
   2. ~ClassName inst()~
      无论是struct或class该语法均无效。因为这个和创建临时对象的语法有冲突，创建临时对象的语法为 ~ClassName()~ 。

   即 ~ClassName inst~ 和 ~ClassName inst = ClassName()~ 是等价的。

** 创建堆对象时
   ~new ClassName~ 和 ~new ClassName()~ 都是调用无参数构造函数。
   #+begin_src c++
   char *str = new char[10]; // 申请一个长度为10的字符数组
   char *c = new char(10); // 申请一个字符空间，并初始化为10
   delete c;
   delete[] str;
   #+end_src

* 头文件和源文件的作用
  C++支持 *分别编译(separate compilation)*, 即a.cpp使用到b.cpp中定义的全局方法 ~void test()~ ，但可以分别对a.cpp和b.cpp编译为两个独立的目标文件a.o和b.o，然后运行前链接(link)两者即可。
  原理：
  1. 编译阶段，a.cpp中声明 ~void test();~ 那么编译后会生成一个符号表(symbol table)，其中就包含声明 ~void test()~ 的信息；
  2. 链接阶段，链接器遍历符号表并从其它目标文件中搜索符号表的项目对应的定义，若链接成功则生成可执行文件。

**  声明和定义
   - 声明：仅仅用于表示某形式的符号存在，当具体定义的位置未知。一个符号可被多次声明。
   - 定义：完整描述符号相关的信息，并根据其提供的信息分配内存空间。一个符号有且仅有一个定义。

** 头文件
   将所有相关的函数声明写在头文件中，然后通过 ~#include~ 将其引入到需要它的源文件中。
   头文件本身不参与编译，因此在执行编译命令时只需包含源文件即可。而头文件则必须位于 ~#include~ 指定的路径上。

   头文件的内容
   1. 变量声明 ~extern int a;~
      - 类成员数据的声明(不能赋值)
      - 类静态数据的声明，可赋值
      - 常量的定义
   2. 函数声明 ~void f();~
      - 类成员函数的声明
      - 非类成员函数的声明
      - 静态函数的定义
      - 类内联函数的定义


   可包含定义的3个情况说明：
      1. 可包含const对象和static对象的定义
	 全局的const对象和static对象默认没有extern的声明，所以引入它们后它们的作用域为当前源文件，并不会在源文件中相互影响。且可保证所有源文件的const和static对象相同。
      2. 可包含内联函数的定义
	 内联函数是编译时必须得到定义，然后在调用处展开。因此cpp规定内联函数可以被定义多次，那么写在头文件中就更合理了。
      3. 可包含struct和class的定义
	 1. 写法1：将class定义写在头文件，而成员函数的实现写在源文件。
	 2. 写法2：成员函数定义在class内，那么会被识别为内联函数，那么就可以定义多次。
	    - 注意点：若成员函数定义在class外，且位于相同的头文件中，那么会造成定义多次是错误的。

   不能包含默认命名空间声明，如 ~using namespace std;~ 不能使用，必须使用 ~std::string~ 的形式。

* 函数的调用约定
  
** 函数调用过程
   1. 把调用者的地址压栈，当函数执行完成后返回该地址继续往下执行；
   2. 把实参压栈或存储到寄存器；
   3. 跳转到函数；
   4. 把函数需要使用的寄存器压栈；
   5. 执行函数；
   6. 处理函数返回值；
   7. 恢复寄存器的值；
   8. 清理参数和处理返回值；
   9. 返回到调用者的地址。

** 函数调用约定
   是对函数调用的一个约束和规定，用于描述函数参数传递方式和调用结束后堆栈的清理方式。主要为三个方面
   1. 函数参数的传递方式（寄存器传递参数、采用哪些寄存器传递、参数压栈的顺序等）
   2. 函数调用结束后栈帧清理和栈顶指针恢复方式（由调用者处理、由函数处理）
   3. 函数修饰名的产生方式(C编译方式和C++编译方式将有所不同)

   函数签名的组成实际为： *返回值+调用约定+函数名+形参列表* 。因此声明和定义中上述4部分必须相同，否则编译失败。
   
   常见的函数调用约定为：__cdecl(C/C++默认的调用约定)、__stdcall、__fastcall
   其它不常用调用约定为：__pascal、__thiscall等

** ~__cdecl~
   是C Decalaretion的缩写，作为C/C++默认的函数调用约定。
   规则：
   1. 采用栈传递参数，参数 *从右向左依次入栈* ；
   2. 由 *调用者* 恢复栈帧和栈顶指针
   3. 以C编译方式编译时自动将函数名重命名为 ~_<函数名>~ 即 ~int add(int a, int b)~ 被重命名为 ~int _add(int a, int b)~

   注意：若要使用可变参数的函数(如printf)，那么调用约定必须使用__cdecl。

** ~__stdcall~
   是Standard Call的缩写，作为C++的标准调用约定。WINAPI、CALLBACK实际上都是采用__stdcall，即Windows大部分API函数都是采用该调用约定。
   规则：
   1. 采用栈传递参数，参数 *从右向左依次入栈* ；
   2. 由 *被调用者* 恢复栈帧和栈顶指针
      即函数执行完后，压栈的参数将一同被弹栈。若函数为可变参数，那么函数无法预知压栈的参数个数，那么就无法有效恢复现场。因此该调用约定无法支持可变参数的函数地好用。
   3. 以C编译方式编译时自动将函数名重命名为 ~_@<参数字节数><函数名>~ 即 ~int add(int a, int b)~ 被重命名为 ~int _@8add(int a, int b)~

   相较 ~__cdecl~ 的好处：仅需编译一份和函数本身绑定的恢复栈顶指针的代码，而 ~__cdecl~ 则需要在每个调用处生成恢复栈顶指针的代码。

** ~__fastcall~
   规则：
   1. 函数从左到右的第1、2个参数通过ecx和edx寄存器传递；其余参数则 *从右向左依次入栈* ；
   2. 由 *被调用者* 恢复栈帧和栈顶指针
   3. 以C编译方式编译时自动将函数名重命名为 ~@<函数名>@<参数字节数>~ 即 ~int add(int a, int b)~ 被重命名为 ~int @add@8(int a, int b)~
 
   由于使用寄存器传递参数，因此速度较 ~__stdcall~ 更快。

** ~__thiscall~
   只能作为C++类成员函数的调用约定，同时也是C++类成员函数的默认调用约定。
   规则：
   1. 采用栈传递参数，参数 *从右向左依次入栈* ；
   2. 若参数个数确定，则通过ecx寄存器将this指针传递给函数；若参数个数不确定，则等所有参数压栈后再将this指针压栈；
   3. 若参数个数确定，则由被调用者恢复栈顶指针；若参数个数不确定，则由调用者恢复栈顶指针。

** 函数修饰名
   编译期间根据函数签名对函数名称进行重命名，用于在链接阶段通过函数修饰名定位函数。
   一般场景下开发人员并不用知道和指定函数修饰名，但如下情况则需要指定：
   1. C++中为重构函数、构造函数和析构函数等指定函数修饰名；
   2. 在Java、Nodejs和汇编程序中调用C/C++函数，若函数组成的4个部分发生变化那么函数修饰名也随之变化。

   C和C++的函数修饰名生成方式均不会修改函数名的大小写；而采用 ~__pascal~ 调用约定的函数修饰名和函数名一致，但全部转换为大写。

*** C++的函数修饰名生成规则
    ~__cdecl~, ~__stdcall~ 和 ~__fastcall~ 在C++编译方式下的非成员函数的函数修饰名规则整体相同，仅参数开始标识符有所不同。
    - ~__stdcall~ 的参数开始标识符为 ~@@YG~
    - ~__cdecl~ 的参数开始标识符为 ~@@YA~
    - ~__fastall~ 的参数开始标识符为 ~@@YI~

    函数修饰名生成规则： 
    - 参数列表不为空 ~?<函数名><参数开始标识><参数表>@Z~
      - 参数表包含 *返回值类型代号* 和 *形参类型代号*
      - 示例: ~int __cdecl function1(char*, unsgined long)~ 的函数修饰名为 ~?function1@YAHPADK@Z~
    - 参数列表为空 ~?<函数名><参数开始标识><返回值类型代号>XZ~
      - 示例: ~int __cdecl function2()~ 的函数修饰名为 ~?function2@YAHXZ~

**** 类成员函数的函数修饰名
     由于存在函数重载、构造函数和析构函数和public、protected和private等访问控制符，因此类成员的函数修饰名生成规则更为复杂。

*** ~extern "C"~ 的作用
    是C++编译器提供的，用于告知编译器以C方式编译和链接。
    - 导出时用于告知编译器依照C方式编译封装接口，但函数内部依然按照C++方式编译；
    - 导入时告知编译器和链接器按照C方式查找和链接所要调用的函数。
    
    若调用者采用C++方式编译，而被调用者采用C方式编译，那么它们两者关联的函数修饰名将不同，从而导致链接时报 ~error LINK2019: 无法解析的外部符号~ 的异常。

* 动态链接库(DLL)的开发和使用
  从16位Windows起引入动态链接库(DLL)，其中包含函数和数据。是Windows平台下实现共享函数概念的一种实现方式。扩展名为 ~.dll~ 或 ~.ocx(包含ActiveX控件的库)~ 。
  动态链接库包含两个文件：引入库(.LIB)文件和DLL文件。
  - 引入库(.LIB)文件，包含被DLL导出的函数和名称和地址位置。若编译出来的文件没有 .LIB文件，那么要么缺少 .DEF文件 要么缺少 ~extern "C" __declspec(dllexport)~ 声明导出的函数。
  - DLL文件，包含函数和数据。
    
  应用程序编译时只需依赖引入库文件，即编译出来的可执行文件仅包含DLL函数的地址而不是函数代码；在链接阶段才会使用DLL文件。因此
  - 编译时只需依赖引入库文件
  - 发行时引入库文件和DLL文件要一起发行

** VC++支持3种DLL
   1. Non-MFC DLL(非MFC动态库)
      不依赖MFC类结构，直接用C语言写的DLL，导出的函数是标准的C接口，能被非MFC和MFC应用所调用。
   2. Regular DLL(常规DLL)
      依赖MFC类结构(即继承CWinApp类，但该类没有消息循环)，被导出的函数可以是C函数、C++类或C++成员函数。只要能调用类C函数的应用即可调用，如VC++,Dephi,VB,Borland C等。
   3. Extension DLL(扩展DLL)
      依赖MFC类结构(即继承CWinApp类，但该类没有消息循环)，被导出的函数可以是C函数、C++类或C++成员函数。只能由MFC应用调用。

*** VS下创建Non-MFC DLL
    1. DLL入口函数——dllmain.cpp文件详解
       1. DLL入口函数和EXE入口函数的差别
	  - EXE入口函数只会被调用1次
	  - DLL入口函数可能会被调用多次
       2. 代码
	  #+begin_src c++
	  #include "stdafx.h"

	  BOOL APIENTRY DllMain(HMODULE hModule, // 由于不同的应用程序引入DLL时分配的地址均不同，因此该入参用于保存当前应用程序指向该DLL模块的句柄(实际上，是指向_DGROUP段的一个选择符)
						  DWORD ul_reason_for_call, // 动态库被调用的原因，当在进程/线程装入或卸载动态库时，操作系统会调用入口函数(APIENTRY标识的函数)
					  LPVOID lpReserved)
	  {
              switch (ul_reason_for_call) {
	      case DLL_PROCESS_ATTACH: // 主线程执行LoadLibrary加载DLL时执行
	      case DLL_THREAD_ATTACH: // 工作线程执行LoadLibrary加载DLL时执行
	      case DLL_THREAD_DETACH: // 工作线程执行FreeLibrary卸载DLL，或工作线程退出时执行
	      case DLL_PROCESS_DETACH: // 主线程执行FreeLibrary卸载DLL，或主线程退出时执行
		break;
              }
              return TRUE;
	  }
	  #+end_src
    2. 通过 ~extern "C" __declspec(dllexport)~ 导出Non-MFC DLL
       1. Header文件——头文件为可选项
	  #+begin_src c++
	  // my.h
	  // 若当前为C++项目，则导出函数需要用 extern "C" __declspec(dllexport) 标识
	  // 若当前为C项目，则导出函数需要用 __declspec(dllexport) 标识即可
	  #ifdef __cplusplus
	  #define DLLEXPORT extern "C" __declspec(dllexport)
	  #else
	  #define DLLEXPORT __declspec(dllexport)
	  #endif
	  
	  DLLEXPORT  int add(int, int);
	  #+end_src
       2. CPP文件
       #+begin_src c++
       #include "my.h"
       // 在项目对应的.cpp文件中写导出函数，如项目名为my则对应的文件为my.cpp
       DLLEXPORT  int add(int a, int b) {
         return (a + b);
       }
       #+end_src
    3. 通过 .DEF文件 导出Non-MFC DLL
       1. CPP文件(头文件可选)
	  #+begin_src c++
	  int add(int a, int b){
	    return (a+b);
	  }
	  #+end_src
       2. .Def文件
	  #+begin_src 
	  LIBRARY
	  EXPORTS
	  add @1
	  sub @2 NONAME
	  #+end_src
	  1. EXPORTS下罗列导出表的函数信息
	  2. ~add @1~ 表示导出add函数，编号为0x1且函数导出名为add。那么映射函数地址时可以通过函数名或编号寻找函数地址

	     - ~GetProcAddress(hInstance, (char*)0x1)~

	     - ~GetProcAddress(hInstance, (char*)0x2)~

*** 导入表
    编译器会根据 ~__declspec(dllimport)~ 将相关动态链接库和所调用的函数生成导入表（记录DLL和DLL的函数）。
    那么当启动程序时，操作系统会根据导入表加载对应的动态链接库。

** 调用DLL——显式链接
   前提：仅有 *DLL文件* ，而没有 *.LIB文件*
      #+begin_src c++
      #include <Windows.h>
      // 定义函数指针
      typedef int (*ptrAdd) (int, int);

      void main(){
         ptrAdd add; // 声明函数指针变量

          HINSTNANCE hDLL = LoadLibraryA("my.dll"); // 加载动态链接库DLL到程序的虚拟内存中，路径为相对于编译后的可执行文件路径

          int result = add(1, 2);

          FreeLibrary(hDLL); // 释放动态链接库
      }
      #+end_src

**** ANSI ASCII 编码
     采用1byte存储字符编码，因此最多能表示255个字符。
     分为原始ASCII表和扩展ASCII表
     - 原始ASCII表仅使用0x0-0x7F的编码，存放英文字母、数字和常用符号。
     - 扩展ASCII表使用0x80-0xFF，存放使用频率很低的字符。

**** MBCS编码(GB2312/GB2312-80/BIG5)
     基于ASCII的扩展表(0x80-0xFF)，采用2byte存储中文字符的编码，并且每个byte的编码都使用ASCII扩展表的编码方式。如字符 "中" GB2312的MBCS编码为0xD6D0，其中0xD6和0xD0在ASCII扩展表中都分别对应一个不常用的字符，而一起则被识别为字符 "中"。
     缺点：每个国家或地区所使用的MBCS编码均不相同，如台湾则采用BIG5。因此若发送方采用GB2312对字符进行编码，接收方采用BIG5进行解码，那么看到的有可能就是乱码。

**** UNICODE编码
     UNICODE是一套包含全世界字符的编码表，占3byte且编码范围为0x0~0x10FFFF，可容纳100多万个符号。
     注意：UNICODE仅规定了字符的二进制编码，但没有规定具体的这些二进制编码的存储方式。
     具体的存储方式如下：
     - UTF-32，长度固定的存储方式，每个字符均采用32bit无符号整数存储。那么对于字母A这种UNICODE编码只是0x000040的字符，则有3byte的空间是浪费的。
       - 由于UTF-32中单个字符由多个字节组成，并且字节序列的差异将导致解码时得到不同的字符，因此需要区分字节序列为大端还是小端。
     - UTF-16，长度固定的存储方式，UNICODE默认存储方式。每个字符以16bit无符号整数的整数倍来存储，即字母A和其它UNICODE编码小于等于16bit的字符可采用16bit完成存储，而大于则需要采用32bit存储一个字符。
       - 由于UTF-16中单个字符由多个字节组成，并且字节序列的差异将导致解码时得到不同的字符，因此需要区分字节序列为大端还是小端。
       - BOM(Byte Order Mark)
	 - LE —— ~0xFF FE~
	 - BE —— ~0xFE FF~
       - 适用场景：传输和存储的字符中，大部分字符的UNICODE编码占多字节，那么采用UTF-16既不会浪费空间，解码算法简单和高效。
     - UTF-8，变长的存储方式
       - UNICODE编码和UTF-8存储映射表
	 | Unicode编码(16进制) | UTF-8字节流(二进制)        |
	 |---------------------+----------------------------|
	 | 000000 - 00007F     | 0xxxxxxx                   |
	 | 000080-0007FF       | 110xxxxx 10xxxxxx          |
	 | 000800-00FFFF       | 1110xxxx 10xxxxxx 10xxxxxx |
	 | 010000-10FFFF       | 11110xxx 10xxxxxx 10xxxxxx 10xxxxxx |
       - 虽然UTF-8中单个字符可能由多个字节组成，但由于其编码方式的原因，大端还是小端的字节序列并不会导致解码得到不同的字符。
       - 在存储某些中文字符时，UTF-8可能会占用3byte，比UTF-16的还要多1byte。
       - BOM(Byte Order Mark) —— ~0xEF BB BF~
       - 适用场景：传输和存储的字符中，大部分字符的UNICODE编码占单字节，那么采用UTF-8不会浪费空间，当然解码算法相对UTF-16而言复杂和低效。
 
**** 多字节字符 ~char~ 和 宽字符 ~wchar_t~ 的使用
     C语言提供的，C++继承过来。

     | char   | wchar_t | 作用                                             |
     |--------+---------+--------------------------------------------------|
     | print  | wprintf | 打印到控制台                                     |

**** MBCS和UNICODE间的转换
     1. MBCS -> UNICODE
         #+begin_src c++
         #include <locale.h>
         #include <Windows.h>

         setlocale(LC_ALL, "");

          // MBCS
         char czText[] = "中国";
         // 获取转换宽字符时需要的多少个16位空间
         DWORD dwSize;
         dwSize = MultiByteToWideChar(
           CP_ACP,  // 源字符串采用的字符编码，CP_ACP表示ASCII
           NULL, // dwFlags
           czText, // 源字符串指针
           -1, // 从0开始获取到哪个索引，-1为除\0外
            NULL, // 目标字符串指针
            0); // 从0开始复制到目标指针哪个索引
          
         // 转换
         wchar_t *wcText = new wchar_t[dwSize];
         if (!wcText) {
           // new申请空间失败则回收并退出
           delete[] wcText;
           return 1;
         }
         MultiByteToWideChar(CP_ACP, NULL, czText, -1, wcText, dwSize)
         #+end_src

         #include <locale.h>

      由于每个被创建的进程，父进程必定会提供一个 ~LPSTARTUPINFO~ 的结构体，一般为将不使用的值清0。如通过鼠标点击启动程序时，实际上是explorer.exe使用shell32的ShellExecute来运行程序，而ShellExecute会将StartupInfo中不使用的值清0。
      而ollydbg会将dwFlags设置为STARTF_FORCEOFFFEEDBACK，而ShellExecute不会。因此可通过StartupInfo的信息判断是否通过ollydbg启动程序。
      #+begin_src c++
      #include <Windows.h>

      BOOL isDebug(LPSTARTUPINFO lpStartupInfo) {
        STARTUPINFO si;
        GetStartupInfo(&si); // 获取当前进程启动时的StartupInfo
	
         if ((si.dwFillAttribute != 0) || (si.dwFlags & STARTF_FORCEOFFFEEDBACK) {
	 return TRUE;
         }
         return FALSE;
      }
      #+end_src
      注意：若用explorer启动后再以附加的方式debug，上述方法将无法检测出来。

**** 远程线程注入
     目标：在目标进程中运行自定义的代码。
     操作流程：
     1. 在目标进程创建线程加载自定义DLL
     2. 在目标进程中运行自定义代码

     #+begin_src c++
     // dwProcessID为目标进程ID
     // czDllPathName为自定义DLL路径
     BOOL LoadDll(DWORD dwProcessID, char *czDllPathName) {
       // 1. 获取目标进程句柄
       HANDLE hProcess = OpenProcess(PROCESS_ALL_ACCESS, FALSE, dwProcessID);
       if (hProcess == NULL) {
         return FALSE;
       }
       // 2. 计算自定义的DLL路径长度
       DWORD dwLength = strlen(czDllPathName) + 1;
       // 3. 在目标进程分配私有内存空间，返回的是虚拟内存地址
       LPVOID lpAllocAddr = VirtualAllocEx(hProcess, NULL, dwLength, MEM_COMMIT, PAGE_READWRITE); 
       if (lpAllocAddr == NULL) {
         CloseHandle(hProcess);
         return FALSE;
       }
       // 4. 拷贝自定义的DLL路径到目标进程的内存空间
       BOOL bRet = WriteProcessMemory(hProcess, lpAllocAddr, czDllPathName, dwLength, NULL);
       if (!bRet) {
         CloseHandle(hProcess);
         return FALSE;	 
       }
       // 5. 获取kernel32.dll模块的句柄，在相同的操作系统下所有进程获取的kernel32.dll的句柄都是一样的。即从全局句柄表获取。
       HMODULE hModule = GetModuleHandle("Kernel32.dll");
       if (hModule == NULL){
         CloseHandle(hProcess);
         return FALSE;	 
       }
       // 6. 获取LoadLibraryA方法的地址，在相同的操作系统下所有进程获取的kernel32.dll的所有函数的地址都是一样的。因此目标进程中通过该地址也能定位相同的方法。
       DWORD dwLoadAddr = (DWORD)GetProcAddress(hModule, "LoadLibraryA");
       if (dwLoadAddr == NULL) {
         CloseHandle(hModule);
         CloseHandle(hProcess);
         return FALSE;	 
       }
       // 7. 创建远程线程，并执行LoadLibraryA方法加载自定义DLL
       // 由于线程执行的函数签名，只需符合返回值和入参均占4个字节即可被执行，而HMOUDLE LoadLibraryA(LPSTR lpPath)完全符合该函数签名。
       HANDLE hThread = CreateRemoteThread(hProcess, NULL, 0, (LPTHREAD_START_ROUTINE)dwLoadAddr, lpAllocAddr, 0, NULL);
       if (hThread == NULL) {
         CloseHandle(hModule);         
         CloseHandle(hProcess);
         return FALSE;	 
       }
       return TRUE;
       // 然后自定义的DLL中的DllMain函数将被执行，那么只需将恶意代码写在DllMain函数中即可被自动执行。
     }
     #+end_src

***** 什么是注入？
      在第三方进程不知情或不允许的情况下将 *模块* 或 *代码* 写入对方进程空间，并设法执行的技术。
      已知的注入方式：远程线程注入、APC注入、消息钩子注入、注册表注入、导入表注入、输入法注入等。

** 进程的句柄表
*** 内核对象
    进程(EPROCESS)、线程(ETHREAD)、文件(FILE_OBJECT)、互斥体(Mutex)、事件(Event)等在内核(ring0，也称为0环)都有一个对应的结构体，那么称这些由内核管理这些结构体为内核对象。
    Win32函数 ~CloseHandle~ 可以关闭的入参就是内核对象。（进程(EPROCESS)、线程(ETHREAD)、文件(FILE_OBJECT)、互斥体(Mutex)、事件(Event)、File mapping、Job、Named pipe、Semaphore、Socket、Access Token、Communications device、Console input、Console screen buffer）
    另外可以通过查看Win32函数CreateXXX的入参，若包含安全属性 ~LPSECURITY_ATTRIBUTES~ 即该函数用于创建内核对象。

*** 为何需要句柄表？
    内核对象的地址位于内核空间中，若直接将内核对象地址交给程序使用，虽然可以直接操作内核对象，但若程序修改地址继续访问内核其它空间，那么将因内存无法访问导致系统蓝屏(如果是用户态的内存无法访问只会导致当前程序宕机——C005错误)。
    因此通过句柄表暴露内核对象给程序，而不是直接暴露内核对象的地址。

*** 句柄表结构
    1. 每个进程都有一个句柄表，用于记录当前进程相关的内核对象地址和句柄映射；
    2. 句柄表结构
       | 句柄 | 内核对象地址| 是否允许被子进程继承|
    3. 不同的进程可以共享内核对象(通过OpenProcess创建的进程可以共享其它进程的内核对象，进程间非父子关系)
       - 非父子关系的进程间内核对象共享，句柄表中的句柄值不一定相同，且句柄值仅在所属的进程中有意义。
       - 父子关系的进程间继承内核对象，句柄表的可继承的内核对象的句柄值一致，不可继承的内核对象的句柄值和内核对象地址均设置为0。
	 - 从而实现父子进程间共享内核对象和句柄。
    4. 内核对象中存在一个计数器，用于记录被多少个进程共享。
       - CloseHandle就是将指定的内核对象的计数器减1。
       - 当内核对象的计数器值变为0时，此时该内核对象就会被销毁。
	 - 线程例外
	   1. 先要将线程终止
	   2. 然后线程内核对象的计数器值为0
	   3. 这样才能回收线程内核对象
	 - 进程例外，进程下的线程未被回收，那么即使进程内核对象的计数器为0也不会被回收

*** 进程共享句柄
**** 从父进程继承句柄
     1. 在父进程中创建可被继承的内核对象——安全属性 ~SECURITY_ATTRIBUTES~
        一般设置为NULL，采用父进程的安全属性且该内核对象不允许被子进程继承即可。
        #+begin_src c++
        typedef struct _SECURITY_ATTRIBUTES {
          DWORD nLength;
          LPVOID lpSecurityDescriptor; //  安全描述符结构体，用于指明当前内核对象哪些用户可以访问、关闭等。设置为NULL即采用父进程的安全配置。
          BOOL bInheritanceHandle; // 当前内核对象是否允许被子进程继承，默认为FALSE
        } SECURITY_ATTRIBUTES, *PSECURITY_ATTRIBUTES;

        // Example
        SECURITY_ATTRIBUTES sa;
        ZeroMemory(&sa, sizeof(sa));
        sa.nLength = sizeof(sa);
        sa.bInheritanceHandle = TRUE;
        #+end_src
     2. 创建子进程时，设置子进程允许继承父进程的内核对象——将CreateProcess的 ~BOOL bInheritHandles~ 设置为 ~TRUE~
     3. 那么子进程就可以继承父进程中设置为允许被继承的内核对象
	- 继承操作为：
	  1. 复制父进程的句柄表作为子进程句柄表的基础
	  2. 不允许被继承的内核对象在句柄表的记录，句柄值和地址均设置为0

**** 非父子进程共享句柄——OpenProcess
     参考进程/OpenProcess函数说明

   而进程ID则是全局句柄表的句柄值，因此进程ID是全局有效，而进程私有句柄表的句柄则是某进程中有效。

**** 启动和关闭过程中NC消息顺序

**** 关闭窗口的方式
     完整的退出流程:
     1. 点击窗口右上角的关闭按钮, 发出WM_CLOSE消息
     2. 消息处理函数中调用DestroyWindow函数, 发出WM_DESTROY消息
     3. 消息处理函数中调用PostQuitMessage函数, 发出WM_QUIT消息
     4. GetMessage捕获到WM_QUIT,返回0那么程序则真正退出
	
     注意: 由于WM_DESTROY消息必须被窗口所属线程捕获后执行消除处理函数才能继续后续消息发送和捕获处理,因此 *线程A调用DestroyWindow无法关闭线程B的窗口*
     线程A关闭线程B的窗口的方式:
     #+begin_src c++
     HWND hwnd; // 被关闭的窗口句柄
     PostMessage(hwnd, WM_CLOSE, NULL, NULL);
     #+end_src

** 内存管理
   
*** 虚拟内存和物理内存
    
**** 32位操作系统
     每个进程启动时都会被分配4GB的虚拟内存，其中2GB为用户空间(0~7FFFFFFF，也称为线性地址)，另外2GB为内核空间(8FFFFFFF~FFFFFFFF)。
     当进程真正要使用内存时，操作系统才会真实分配物理内存空间并映射到该进程的虚拟内存上。

     Intel x86架构将物理内存分为一个个4kB的物理页。
     - 物理内存和内存条之间还有一层映射，而不是直接对应的。
     - 可供使用的物理内存 = MmNumberOfPhysicalPages * 4(KB) + 使用磁盘部分容量作为SWAP内存
     - 能够识别的物理内存
       - 32位系统最多可以识别物理内存位64GB(为什么呢？)，但由于操作系统的限制， 如XP只能识别4GB，而Windows2003服务器版本可识别4GB以上。
     
***** 虚拟内存中的用户空间(0~7FFFFFFF)
      1. 0~0x0000FFFF(共64KB)为空指针赋值区，不可被进程所用。
      2. 0x00010000~0x7FFEFFFF为用户模式区，进程可使用的虚拟内存，但需要申请使用才能有对应的物理页。
	 - 平时所说的申请内存空间，就是申请用户模式区的内存空间。
      3. 0x7FFF0000~0x7FFFFFFF(公64KB)为禁入区，不可被进程所用。

***** 访问已分配的虚拟内存的过程
      由于物理内存容量必定小于所有进程的虚拟内存总和，因此操作系统会将休眠的进程的物理页资源释放（物理页的内容将被转移到硬盘上），那么进程再次访问其虚拟内存时将发生如下流程：
      1. 已分配的虚拟内存
      2. 缺失页异常
      3. 操作系统处理异常
      4. 为虚拟内存挂物理页
      5. 恢复原数据到物理页上

***** 私有内存(Private Memory)的申请和释放
      私有内存：被单个进程独占的物理页。
      进程中绝对部分都是共享/映射内存，只有栈/堆内存和自行通过 ~VirtualAlloc/VirtualAllocEx~ 申请的采用私有内存。

****** FILE_MAP_COPY详解
       1. 断点的原理
	  1. 在代码行开头添加 ~0xCC~ (即汇编指令 ~int 3~)
	  2. 当CPU执行到该指令时会挂起
       2. kernel32.dll、user32.dll等只会加载一份到物理内存供多个进程共享使用，那么某个进程修改共享内存的内容时，会发生如下结果
	  - 若采用FILE_MAP_ALL_ACCESS，那么修改的结果会写到对应的文件中，且其它进程可读取到修改结果。
	  - 若采用FILE_MAP_COPY，那么修改时会先复制一份对应的物理页副本，修改时仅修改物理页副本，因此修改结果不会写入对应的文件中，且其它进程不会读到修改结果。

** VBScript通过CreateObject的过程详解
   参考：http://demon.tw/reverse/vbscript-internal-createobject.html

   1. CreateObject的入参为ProgID，作为COM组件的名称标识
   2. 调用Win32 API ~CLSIDFromProgIDEX~ 根据ProgID获取对应的CLSID
      1. 读取 ~HKEY_CLASSES_ROOT/<ProgID>/CLSID~ 的默认value值，若存在则直接返回，否则继续第2步。
      2. 读取 ~HKEY_CLASSES_ROOT/<ProgID>/CurVer~ 的默认value值
      3. 读取 ~HKEY_CLASSES_ROOT/<HKEY_CLASSES_ROOT/<ProgID>/CurVer的默认value值>/CLSID~ 的默认value值
   3. 通过 ~HKEY_CLASSES_ROOT/CLSID/<clsid>/InProcServer32~ 或 ~HKEY_CLASSES_ROOT/CLSID/<clsid>/LocalServer32~ 获取服务程序启动路径和命令
      - InProServer32 表示为进程内COM组件
      - LocalServer32 表示为进程外COM组件
      - 若存在 ~HKEY_CLASSES_ROOT/CLSID/<clsid>/Control~ 则说明该组件为ActiveX控件
      - 若存在 ~HKEY_CLASSES_ROOT/CLSID/<clsid>/Insertable~ 则说明该组件可以被嵌入到一个OLE文档容器中
      - 若存在 ~HKEY_CLASSES_ROOT/CLSID/<clsid>/Programmable~ 则说明该组件支持自动化，即支持IDispatch接口，所以可以被脚本语言使用。
	新版本可能会存在 ~HKEY_CLASSES_ROOT/CLSID/<clsid>/Implemented Categories~ ，获取其默认值，然后查看 ~HKEY_CLASSES_ROOT/Component Categories/<默认值>~ 的value 409的值是否为Automation Objects。存在即表示该COM组件支持自动化。
   4. 通过 ~HKEY_CLASSES_ROOT/TypeLib/<HKEY_CLASSES_ROOT/CLSID/<clsid>/TypeLib的默认value值>/1.0/0/win32~ 获得COM对象的类型库路径（获取COM组件提供的接口方法）
      PE文件的TYPELIB资源是*.idl源码文件编译后的类型库的二进制数据(可以反编译回去)，其中包含每个接口函数的参数和返回值定义。
   5. 通过CoGetClassObject获取IClassFactory接口的指针
      失败则返回“ActiveX部件不能创建对象”或“类不支持Automation操作”等错误信息。
   6. 调用IClassFactory接口的CreateInstance方法获取IUnknown接口指针
      IUnknown接口包含QueryInterface, AddRef 和 Release。
   7. 调用IUnknown接口QueryInterface方法查询该COM是否支持IDispatch接口，只有支持IDispatch接口的COM类才能用CreateObject创建对象。

** JavaScript中的 ~Math.random~
   - 特点：
     1. 由于预先生成128个随机数供后续使用，因此在使用调用该函数前已经知道后续随机数和顺序；
     2. 通过借助操作系统的能力，获取随机数生成器的seed
	- Linux下读取/dev/urandom这一由操作系统不断产生伪随机数的特殊文件的8个字节来作为seed
	- Windows下则调用rands函数，该函数由提供且随机性足够强的随机数发生器
	- 其它则以当前系统高精度的时间戳作为seed
     3. 但由于随机数生成器的seed仅为2^64个组合，因此无法提供像密码一样安全的随机数，不能直接用于安全等场景。
	- 随机数生成器的seed范围小或算法不够优质，那么在多次取值后可能会出现周期性表现。
	- 可视化周期性表现的方法：通过连续生成大量随机数，并转换为RGB等颜色值，然后打点得到图像即可看到图像出现规律的花纹。
     4. 若要用于安全性高的场景则需要使用 ~window.crypto.getRandomValue~ 函数
	内部使用1024位种子的ARC4流密码。

** 伪随机数(Pseudo Random Number)
   相同的 *随机数生成器seed* 和 *随机数生成算法* 即可 生成相同的 *随机数* 。
   1. 随机数生成分布均匀性——由 *随机数生成算法* 决定
   2. 随机数的可预测性
      - 破解思路：破解者不知道当前使用的seed，通过所生成的随机数暴力破解出seed。
	由于 *随机数生成算法* 对外提供服务，因此破解者根本不用获得源码即可实现不断调用算法实现暴力破解。
      - 提供破解门槛的因素
	- seed的位数，越多越好（1024+）。
	- 问题：若采用时间戳作为seed，即使采用ms。若破解者将破解用的服务器时间和应用系统服务器时间实现秒级同步，那么seed的组合则大幅降低到1000个。
	  因此需要对时间戳进行服务端私有处理再作为seed。

** 真随机数(True Random Number)
   真随机数需要从现实世界进行采集，其中Linux中的/dev/random就是通过设备驱动程序采集背景噪音的方式生成真随机数。
   
* 加壳
  
** 概述
   壳分为4类
   1. 压缩壳
   2. 加密壳
   3. 介于压缩壳和加密壳之间的混合壳
   4. 虚拟化

   普通二进制代码的逆向工程流程：
   1. 通过 *反汇编* 工具将 二进制代码反汇编为汇编程序；
   2. 然后可以人工分析汇编程序，或将汇编程序反编译为高级语言。
    
   *加壳的目的* 就是让反汇编、反编译变得尽量复杂，反出来的结果尽可能无意义。

** 加壳流程
   1. 逆向工程时会对二进制程序(Windows为PE，Unix/Linux为ELF，MacOS为Mach-O格式)进行反汇编，得到一个个处理单元(Block)。这些Block可能是一个函数或函数的一部分，里面包含指令和数据。
      - 破解思路：block配合编译信息和文件格式信息，就可以看到每个Block对应的函数签名。
      - 加壳方式思路：
            1. 针对Block的重要程度进行选择混淆、加密和虚拟化等安全系统不同的加壳操作，那么一个Block可能会被打碎为多个Block，并且散落到不同的位置；
	2. 编译保护：加入假分支、立即数加密、指令混淆等
      - 加壳工具：有的加壳工具会智能提示哪些Block采用哪些加壳方式。
   2. 安全壳代码
      - 将原始程序装到壳代码中，而壳代码有如下功能
	1. 启动程序时实际是运行壳代码，然后由壳代码运行原始程序；
	2. 由于原始程序已经被处理成操作系统无法识别并运行的代码，因此必须通过壳代码解析、分配Block、修复Block的重定位，最后生成一个合法的可执行程序。
