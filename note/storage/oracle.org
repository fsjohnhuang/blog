** merge
   作用：判断source_table和target_table是否满足ON中的条件，若满足则用source_table满足的记录更新target_table，否则插入target_table。
   #+begin_src sql
   merge into <target_table>
   using <source_table>
   on (<conditional expression>......)
     when matched then
         update ...
     when not matched then
         insert ...
   #+end_src
   1. 示例
      #+begin_src sql
      merge into user a
      using (select 123 user_id, 'john', 18 username from dual) b
      on (a.user_id = b.user_id and a.username = b.username)
      -- 省略这句则只插入不更新
      when matched then
        update set a.age = b.age

        -- 省略这句则只更新不插入
      when not matched then
        insert (user_id, username, age) values(123, 'john', 18);
      #+end_src
   2. 带where限制条件
      #+begin_src sql
      merge into user a
      using (select 123 user_id, 'john', 18 username from dual) b
      on (a.user_id = b.user_id and a.username = b.username)
      -- 省略这句则只插入不更新
      when matched then
        update set a.age = b.age where b.city <> '江东'

        -- 省略这句则只更新不插入
      when not matched then
        insert (user_id, username, age) values(123, 'john', 18) where b.city <> '江东';
      #+end_src
   3. 带delete功能
      在 ~when matched then~ 子句中可以包含 ~delete where b.city = '江东'~ 的子句。

** 分层/递归查询
   作用：在进行递归遍历树形结构的时候用。

   select ...

   connect by prior
   #+end_src

** over order by函数
   作用：全集合连续求和
   #+begin_src sql
   -- 1. 按department排序，并以department进行分组后执行sum(sal)；
   -- 2. 第一组 '连续求和' 的值为第一组的department中的sal值总和；
   -- 3. 第二组 '连续求和' 的值为第一组的 '连续求和' + 第二组的department中的sal值总和；
   -- 4. 第N组的 '连续求和' 都是第N-1组的 '连续求和' + 第N组的department中的sal值总和。
   select department, sum(sal) over(order by department) '连续求和' from emp;
   #+end_src
   注意：
   1. ~sum(sal) over()~ 和 ~sum(sal)~ 效果是一样的，即全集合求和。但若查询语句中出现了 ~sum(sal) over......~ 那么不能再出现写 ~sum(sal)~ ；
   2. ~sum(sal) over(order by department, name)~ 先按部门再按名称排序，然后进行全集合连续求和。

** over partition by 组合
   作用：分区求和
   #+begin_src sql
   -- 按department分组，并分别统计各组的sal值总和。
   select
     department,
     sum(sal) over(partition by department) '部门总和',
     100*round(sal/sum(sal) over(partition by department)) '占部门工资份额%'
   from emp;
   #+end_src

** over partition by order by 组合
   作用：分区连续求和
   #+begin_src sql
   -- 1. 对每组department内的记录，根据name进行组内连续求和；
   -- 2. 第N组的department的连接求和，与第N-1组的department的连续求和相互独立。
   select
     department,
     sum(sal) over(partition by department order by name) '部门连续求和',
   from emp;
   #+end_src

** row_number函数 over partition by order by 组合
   作用：分区排序
   #+begin_src sql
   -- 按department分组，在组内按sal倒序排列，然后组内从上至下生成从1开始计算的行号；
   -- 每个分组的行号都是从1开始算起。
   select department, name, row_number() over(partition by department order by sal desc) '排名' from emp;
   #+end_src

** group by rollup函数组合
   作用：基于group by分组语句的统计结果，以倒数第二个字段为分组条件再次汇总。

** grouping函数
   作用：针对group by rollup和group by cube组合。 ~grouping(<column_name>)~ 若为记录本身的值则返回0，若为统计的值则返回1。
   那么就可以为group by rollup和group by cube的结果添加列名。
   #+begin_src sql
   select m,
   (case when ((grouping(m) = 1) and (grouping(area) = 1)) then '总计'
         when ((grouping(m) = 1) and (grouping(area) = 0)) then '月份小计'
         else area end) area,
   sum(incoming)
   from emp
   group by rollup(m, area);
   #+end_src

** 锁(Lock/Enqueues)
   某个会话独占一个资源，从而阻塞其他会话操作该资源。

   Oracle的锁分类：
   1) Enqueues，队列类型的锁，通常和业务相关(DML)。
      #+begin_example sql
      -- 获取Oracle的锁类型
      select type, name from V$LOCK_TYPE;
      #+end_example
   2) Latches，系统资源方面的锁，比如内存结果、SQL解析。

   |        | Latch                  | Lock                 |
   |--------+------------------------+----------------------|
   | 队列性 | 否                     | 是                   |
   | 时长   | 很短                   | 可能很长             |
   | 层面   | 数据库资源层           | 业务应用层           |
   | 目的   | 保证数据库资源的完整性 | 保证业务操作的完整性 |

*** 锁的原则
    1) 只有被修改时，行才会被锁定
       - Oracle不存在锁升级，因此一条DML修改多少条记录，就这几条记录被加上行锁(TX)，用于阻止其它事务对相同行进行修改。
    2) 读(read)永远不会阻止写(update/delete/insert)
       - 但执行 ~select ... for update~ 会阻止写。
    3) 写(update/delete/insert)永远不会阻止读(read)
       - 当一行被修改后，Oracle会通过 *回滚段* 提供给数据的一致性读。
         - 当更新字段A，还没commit时，Oracle实际已经将新值写入字段A了。但旧值会保存在回滚段，其他会话读取时是从回滚段读取字段A的值。

*** TM锁和TX锁——对单表的锁定
    TM表锁：发生在insert,update,delete以及select for update操作时，目的是保证操作能够正常进行，并阻止其它人对表执行DDL操作。
    TX行锁/事务锁：用于正在修改的记录，阻止其它会话进行DML操作。

    TM锁5种模式间的互斥关系
    | 模式 | 手工锁定的SQL                             | 排斥的模式 | 允许的其它会话的DML                        |
    |------+-------------------------------------------+------------+--------------------------------------------|
    |    2 | lock table t in row share mode;           | 6          | select, insert, update, delete, for update |
    |    3 | lock table t in row exclusive mode;       | 4,5,6      | select, insert, update, delete, for update |
    |    4 | lock table t in share mode;               | 3,5,6      | select                                     |
    |    5 | lock table t in share row exclusive mode; | 3,4,5,6    | select                                     |
    |    6 | lock table t in exclusive mode;           | 2,3,4,5,6  | select                                     |
    手工锁定的应用
    #+begin_example sql
    lock table dept in share mode; /* 其它会话不能修改dept */

    update emp
      set sal = sal * 1.1
      where deptno in (select deptno from dept where loc='DALLAS');
    update budget
      set total = total * 1.1
      where deptno in (select deptno from dept where loc='DALLAS');

    commit; /* 释放锁 */
    #+end_example

*** RI锁——基于引用关系的锁定
    对具有主外键关系的表做DML操作时，锁会加在操作的表和相应的引用表上。
    insert主表时会在从表上加TM锁，而update和delete则不会在从表加锁。
    insert和delete从表时会在主表上加TM锁，update则不会。

    根据Oracle版本有所不同。

*** BI锁和外键索引——没有索引的外键
    更新主表的主键时，当从表的外键有索引时则通过索引查找从表记录，若没有索引则需要对从表进行全表扫描。这个过程中将一直对主表的记录加TX锁。
    因此对从表的外键需要加锁，并且不要修改主键。

**** 主从表
     主键所在的表称为主表；外键所在的表称为从表。从表外键字段的值，必须在主表中主键字段中已存在。
     #+begin_example sql
     -- 主表
     create table user (
       id number(10) primary key,
       name varchar(2) not null
     );

     -- 从表
     create table user_info (
       id number(10) primary key,
       user_id number(10) references user(id),
       sex varchar2(1) default 'F' check(sex in ('F', 'M'))
     );

     select u.name, i.sex from user u join user_info i on (u.id = i.user_id);
     #+end_example

*** 死锁
    Oracle能自动识别和释放死锁。

*** 示例——会话A和B并发insert主键值相同的记录
    #+begin_example sql
    -- 会话A和B并发执行 insert into tbl(id) values(1); 并且均不执行commit

    select sid, type, id1, id2, lmode, request, block from v$lock wheree type in ('TM', 'TX') order by 1,2;

    -- B   TM   123 0   3     0       0
    -- B   TX   121 322 6     0       1
    #+end_example

    -- 假设A被B阻塞了

*** 示例——select for update
    #+begin_example sql

    select * from tbl where id=2 for update;
    -- 在会话B执行
    update tbl set name='a' where id=2;

    -- sid type id1 id2 lmode request block

    延伸
    1) ~for update skip locked~
       对满足where子句，且尚未加锁的记录加锁。
    2) ~for update nowait~
       其它会话锁定了 ~for update nowait~ 语句将要锁定的记录时，不等待锁释放而是直接抛出"ORA-00054"
    3) ~for update wait <n>~
       其它会话锁定了 ~for update wait~ 语句将要锁定的记录时，则等待n秒后仍然未释放锁而是抛出"ORA-00054"
    4) 通过 ~of~ 仅对查询的结果集中满足特定条件的记录加锁
       - 不使用 ~of~ 则对符合where子句的所有表格的记录加锁
         #+begin_example sql
         -- 对tbl1和tbl2中id大于3的记录加锁
         select a.id, b.id where tbl1 a join tbl2 b on (a.id = b.id) where a.id > 3 for update;

         -- 对tbl1中id大于3的记录加锁
         select a.id, b.id where tbl1 a join tbl2 b on (a.id = b.id) where a.id > 3 for update of a.id;
         #+end_example

** Latch
   Latches，系统资源方面的锁，比如内存结果、SQL解析。
   保证资源的串行访问：
   1) 保护SGA的资源访问(不涉及PGA)；
      - 共享池的SQL解析、SQL重用
      - 数据缓冲池的数据访问、磁盘写入、读取到内存、修改数据块和数据段扩展等
   2) 保护内存的分配，防止内存结构的损坏。

   查看数据库拥有的latch的信息
   #+begin_example sql
   select * from v$latchname;
   #+end_example

   无法立即获取Latch后的行为：
   1) wait方式
      - spin，会话继续使用CPU(CPU空转)，达到一个间隔后再次尝试申请Latch，直到达到最大重试次数为止。
      - sleep，释放CPU并等待一段时间后，再次获取CPU资源来尝试申请Latch，直到达到最大重试次数为止。
   2) no wait方式，转向申请其它可用的Latch。

*** 相关动态视图
    1) v$latch，对每个latch的统计信息的汇总，每一条记录表示一种latch。
       #+begin_example sql
       -- 查询和Buffer Cache相关的latch
       select name, -- latch名称
              gets, -- 以Willing to wait请求模式latch的请求成功数
              misses, -- 初次尝试不成功次数
              sleeps, -- 成功获取钱sleeping次数
              immediate_gets, -- 以Immediate模式latch的请求数
              immediate_misses -- 以Immediate模式latch的请求失败数
       from v$latch where name like 'cache%';
       #+end_example
    2) v$latchholder，包含当前latch持有者的信息。
       - 然后通过pid和sid的信息，关联视图v$session,v$session_wait可定位对应的会话信息。
    3) v$latch_children
       - 在SGA中某些资源(如library cache)使用多个latch来保护，这些latch保护同一个资源则成为了子latch。
    4) AWR报告中也会包含全局范围的Latch信息

*** Shared Pool的Latch争用——绑定变量
    解析SQL时会消耗CPU和Latch争用而产生等待，因此大量动态SQL(非绑定变量)会引发大量的SQL硬解析。

*** Buffer Cache的Latch争用
    1 Latch->N Hash Bucket->N个以链表结构连接的Buffer Headers->N个Data Blocks

    读取数据块过程：
    1) 对数据块的地址进行hash(Hash the block address)
    2) 获取Latch访问Hash Bucket(Get Bucket latch)
    3) 访问Buffer Headers(Look for header)
    4) 访问Data Blocks(Found, read block)
    5) 若缓存命中失败看，则通过Shadow Process从磁盘获取并写入Buffer Cache(Not Found Read block off disk)

**** 热块问题
     多个Session并行访问由同一个Latch锁定的一组数据块中的某几个数据块，即使不是完全相同的数据块，但它们是由同一个Latch来锁定，所以就会出现访问等待。
     分类：
     1) 表数据块争用
     2) 热块索引数据热块
     3) 文件头数据块——并发修改
        ITL(事务槽)，当多个会话对同一个记录进行修改时就会争用这个事物槽。

** 优化器和执行计划
   优化器只能在数学逻辑上优化SQL，但无法按照业务要求优化SQL。高效的SQL来自于对业务的理解和SQL执行过程的理解。
   示例：
   #+begin_example sql
   -- 没有优化过的SQL，其consistant gets为14
   select t1.id, sum(t2.value) from tbl t1 join tbl t2 on (t1.id >= t2.id) group by t1.id;

   -- 通过使用Oracle的叠加式分析函数，可使consistant gets降低为7

   #+end_example

   - 缺点
     1) 无法输出相关统计信息，如逻辑读、物理读和递归调用等。
     2) 无法判断表执行了多少次。
2) 方式2
   - 方式
     #+begin_example sql
     set autotrace on;
     #+end_example
   - 优点
     能够输出相关统计信息，如逻辑读、物理读和递归调用等。
   - 缺点
     1) 必须等待SQL语句执行完，才能出结果。
     2) 无法判断表执行了多少次。
3) 方式3
   - 方式
     #+begin_example sql
     alter session set statistics_level=all;
     -- 执行要分析的语句 select * from tbl;
     select * from table(dbms_xplan.display_cursor('sql_id/hash_value', null, 'allstatslast'));
     #+end_example
   - 优点
     1) 从stats能够判断表执行了多少次。
     2) 从E-Rows得到执行计划预计返回的记录数
     3) 从A-Rows得到执行计划实际返回的记录数
     4) 从A-Time得到每一步执行的时间(HH:MM:SS.FF)，从而知道SQL在哪里耗时。
     5) 从Buffers得到真实的逻辑读或一致性读
     6) 从Reads得到真实的物理读
   - 缺点
     1) 必须等待SQL语句执行完，才能出结果。
     2) 无法输出物理读和递归调用
4) 方式4
   - 方式
     #+begin_example sql
     -- 1. 开启跟踪
     alter session set events '10046 trace name context forever,level 12';
     -- 2. 执行要分析的语句 select * from tbl;
     -- 3. 关闭跟踪
     alter session set events '10046 trace name context off';
     -- 4. 找到开启跟踪后新增的文件
     -- 5. 格式化文件
     tkprof /path/to/trc_file /path/to/target_file sys=no sort=prsela,exeela,fchela
     #+end_example
   - 优点
     1) 可以看到SQL对应的等待事件
     2) 列出函数内的SQL
     3) 能够输出相关统计信息，如逻辑读、物理读和递归调用
     4) 能够输出解析时间和执行时间
     5) 可跟踪整个程序包
   - 缺点
     1) 无法判断表执行了多少次。

*** CBO(Cost Based Optimizer，基于代价)优化器
8i之后采用的优化器，依据一套数据模型，计算数据访问和处理的成本，择最优成本为执行方案。
成本计算包含：
1) 数据访问的成本估算
   1) IO
      - 全表扫描（多数据块）
      - 索引（单数据块，多数据块）
   2) CPU
2) 数据处理的成本估算——CPU


优化器的2种工作模式：
1) first_rows(n)，以最快的速度抽取前面的记录。
   #+begin_example sql
   -- 如分页查询
   select name from (
     select rownum rn, name from (
       select name from tbl order by name
     ) where rownum <= 20
   ) where rn >= 11;
   #+end_example
2) all_rows，以最快的速度返回整个结果集，适合在OLAP应用上使用
   #+begin_example sql
   -- 全表统计
   select id, count(*) from tbl;
   #+end_example
优化器工作模式设置
1) 全局参数设置
   #+begin_example sql
   optimizer_mode ALL_ROWS;
   #+end_example
2) 会话级设置
   #+begin_example sql
   alert session set optimizer_mode=all_mode;
   #+end_example
3) SQL级Hints
   #+begin_example sql
   select /*+ all_rows */ id, count(*) from tbl;
   #+end_example

优化器的影响因子很多，可以大概知道有以下3个。

**** 优化器的影响因子——Selectivity(选择性)
1) 表格字段的选择性
   #+begin_example sql
   select column_name, num_distinct from user_tab_col_statistics where table_name='<table_name>' order by 2 desc;
   #+end_example
   num_distinct表示对应的列中包含多个唯一的值，1/num_distinct为选择率，选择率越低即选择性越高，则越适合建B-tree索引。
   选择性越低越不适合建B-tree索引。
   优化器也会根据这个数据来选择是否走相应的索引。
2) 索引的选择性
   #+begin_example sql
   select index_name, distinct_keys from user_indexes where table_name='<table_name>';
   #+end_example
   1/distinct_keys为选择率，选择性低则表示该索引越没有效果。

**** 优化器的影响因子——Cardinality
Cardinality就是在执行计划中表示每一步操作返回的记录数(不是实际返回的记录数，而是Oracle估算出来的)。
CBO通过对这个值的权重计算，决定使用哪一种方式访问数据。
10gR2前的执行计划中的 ~Card~ 就是Cardinality，而10gR2以后就是通过 ~Rows~ 表示。

**** 优化器的影响因子——Clustering Factor
#+begin_example sql
select index_name, clustering_factor from user_indexes where table_name='<table_name>';
#+end_example
多个数据会存储在一个数据块上，若索引对应的值出现在多个数据块上，那么该值的clustering factor值就越大，表示通过索引查找数据时需要在多个数据块间查询，访问代价比较大。

**** TODO 直方图
#+begin_example sql
-- 直方图统计，for all column size 1表示忽略Cardinality的值

                                   '<table_name>',
                                   cascade=>true,

#+end_example

~consistant gets~ 一致性读，数值越少性能越好。

- 表访问方式
  - table access full，全表扫描
  - index range scan，范围扫描
  - index full scan，全索引扫描
  - index full fast scan，全索引并行扫描，应用在通过索引就可以获取到结果的场景。如 ~select count(*) from t;~
  - index skip scan
  - index unique scan 唯一索引扫描
- 表关联方式
  - nested loop join，两张表逐条配对，因此要以小表作为驱动表，关联表则需要建索引。
    在内表(非驱动表)的目标列上建有索引对该表关联方式有效。而外表(驱动表)的目标列建索引，最多就是INDEX FULL SCAN或INDEX FULL FAST SCAN。
  - (sort) merge join，两张表各自先排序，然后再配对。索引仅能避免数据排序过程，作用不大。
  - hash join，将一张小表进行hash后与大表的记录进行配对，当小表没有索引时效果更好。索引对查询基本不影响。

*** RBO(Rule Based Optimizer)优化器
8i之前采用的优化器，以一套死板的规则来控制数据的访问。

** Hints
   使用范畴：尽量避免开发人员使用，而是DBA在排查和优化时使用。
   注意：
   1. hints上的table_name为from后的表名，若表名有别名则table_name必须为别名；
   2. 即使添加Hint执行计划也不保证按Hint执行，若发现没有按Hint执行则需要查看统计信息是否过时了。

*** Access Path Hints(访问路径的Hints)
1) ~/*+ full(<table_name>) */~ 全表扫描
   #+begin_example sql
   select /*+ full(user) */ * from user where id = 1;
   #+end_example
   - 多块读：由于Oralce支持多块读的特性，假如有100条记录，而每次IO读取6个数据块，那么全表扫描在只需17次即可读完。而索引的话还要200次才能读完。因此一次读取表中大部分记录时，全表扫描反而性能更高。
   - 并行读：
2) ~/*+ index(<table_name> [<index_name>]) */~ 走索引
   - 注意：
     1. 当select语句的表起了别名，那么 ~<table_name>~ 必须写为别名；
     2. 若表名或别名被修改，那么hint也要修改
   - 示例：
     #+begin_src sql
     select /*+index(ttsh id_tt_shohou)*/ * from shohou_hist ttsh where .....;
     #+end_src
3) ~/*+ no_index(<table_name>) */~ 不走索引
   #+begin_example sql
   -- 默认会走索引，但传null或空字符串都会导致走索引效率比不走还低很多
   select * from user where user_name='';

   select /×+no_index(user)*/ * from user where user_name='';
   #+end_example
4) ~/*+ index_ffs(<table_name> [<index_name>] */~ Index Fast Full Scan
   就是对索引进行多块读和并行读，因此适用于记录多数据块多的情况。记录少数据块少时，就不如index range scan。
5) ~/*+ index_ss(<table_name>) */~ index skip scan
   用于替代全表扫描的数据访问方式，对于前导重复率高的复合索引，有时index skip scan性能好些。
   - 复合索引的前导——就是复合索引的第一个字段

*** Table Join Hints(表关联的Hints)
1) ~/*+ use_nl(<table_name1>[, <table_name2> ...]) */~ Nested Loop Joins
   其中的表将作为Inner Table，若其中出现已被Oracle选择为Outer Table的表则会被自动从Inner Table参数中剔除。但可以再通过 ~ordered~ Hints指定Outer Table。
   - Outer Table(驱动表)，提供查询依据数据的表。一般以记录较少的表作为Outer Table。
   - Inner Table，被查询的表。一般将Inner Table中已建索引的字段作为连接列。
   - ~ordered~ 就是指定从from子句的左至右，左边的表为Outer Table。
     #+begin_example sql
     -- 虽然dept的记录数明显少于user，但通过ordered设置后仍然会以user作为Outer Table
     select /*+ ordered use_nl(dept)*/ * from user a join dept d on (a.dept_id=d.id);
     #+end_example
2) ~/*+ use_hash(<table_name>[, <table_name2> ...]) */~
   - 适用场景
     - 一个大表和一个小表的关联
     - 表上没有索引
     - 返回的结果集比较大
3) ~/*+ use_merge(<table_name>[, <table_name2> ...]) */~ merge join
   对两个结果集进行排序后再进行Nested Loops，性能比其他两个差。
   - 适用场景
     - 关联的结果集因业务需求已经排序。
4) ~/*+ leading((<table>[, <table_name2> ...]) */~ 设置第一个参数为Outer Table
   CBO中默认会尝试以每个表作为Outer Table来计算成本，然后以成本最低的作为最终的Outer Table。

*** 其他Hints
1) ~/*+ append */~ 以直接方式插入数据
   |--------+------+--------+------+--------+-------------------+------|
   | in use | free | in use | free | in use | <-high water mark | free |
   |--------+------+--------+------+--------+-------------------+------|
   - 传统插入方式
     会从high water mark左侧的寻找可用的磁盘空间，若空间足够则插入数据。若不足才到high water mark右侧寻找空间插入数据。
     - 缺点：
       1) 增加寻找时间
       2) 存在数据块争用的并发问题
   - 直接插入方式
     直接到high water mark右侧寻找空间插入数据，即使左侧存在足够的空间也不管。
   - 实例
     #+begin_example sql
     insert /*+ append */ into t select * from t;
     #+end_example
2) ~/*+ dynamic_sampling(<table_name> <level>)*/~ 设置动态采样的级别
   #+begin_example sql
   select /*+ dynamic_sampling(t 3) */~ count(*) from t;
   #+end_example
   - 注意：级别越高收集的信息越多，但效率越低。取值范围0~10，其中0是不使用dynamic sampling。默认值为2。
   - 适用场景：
     - 临时表；
     - 经常数据量大规模(超原数据量10%)发生变化，导致统计信息过期；
3) ~/*+ parallel([<table_name> <n>])*/~ 指定并行度（OLAP特别适用）
   #+begin_example sql
   select /*+ parallel(t 2)*/ * from t;
   #+end_example
   - 注意：参数 ~n~ 不是指执行时发起多个进程执行，而是每一步(order by, group by等)有多少个并行进程执行，也就是至少有n个进程处理。
   - 缺省参数为使用Oracle默认的并行度
4) ~/*+ driving_site(<table_name>)*/~ 指定操作发生在哪个节点上
   #+begin_example sql
   select /*+ driving_site(dept)*/ *
   from emp, dept@rsite -- emp在本地，而detp在远端
   where emp.dep_id = dept.id
   #+end_example
   默认情况下会将远端表的数据拉取到本地进行操作，但通过 ~driving_site~ 则可将本地数据推送给远端并在远端操作完成后再返回结果集。
5) ~/*+ cardinality(<table_name> <rows>) */~ 模拟结果集的cardinality，即告知Oracle某表将返回多少条记录

** 等待事件
SQL*Net message from client：Oracle自家的数据库通信协议。

要设置baseline再查看等待事件的指标才有用。

*** 事件
1) ilde wait events(空等待)
   可能是无法获取资源导致的空等待，也可能是没有要处理的操作而空等待。
   #+begin_example sql
   -- state为waiting则为空等待
   select sid, event, state from v$session;
   #+end_example
2) db file sequential read
   逐个数据块读取到SGA时发生，回滚段读(一致性读)也会发生。假如走索引时出现大量db file sequential read，那么说明索引可能建得有问题。
   - 如何解决
     1) 数据的正常读取时，不用解决；
     2) 可能走错索引；
     3) 考虑复合索引、位图索引或全文索引；
     4) 改用全表扫描+并行
     5) 升级磁盘硬件
3) log file switch(archiving needed)
4) data file scattered read
   在进行full table scan或index fast full scan时，可以以multiblock read的形式将数据块读取到SGA中。通过参数 ~db_file_multiblock_read_count~ 设置一次IO读取的数据块数量。
   - 解决方法
     1) 数据的正常读取时，不用解决；
     2) 等待事件影响性能，考虑建索引；
     3) 等待事件影响性能，考虑并行。
5) direct path read/write
   数据被直接读取到PGA内存中时发生。
   - 排序时由于内存不足，被写入磁盘的临时表空间，然后重新读取到内存时
   - 并行操作的slave进程的数据读取/写入
   - 其他的属于某个会话私有数据的读取操作
   - 解决方法
     1) 数据的正常读取时，不用解决
     2) 增大PGA区域
     3) 并行
     4) 升级磁盘硬件
6) log file sync
   用户commit/rollback时，lgwr需要将log buffer的数据写入log file上。表示提交太频繁，可以考虑批量写入。
7) buffer busy waits
   内存中对相同的数据块有多个并发请求时，将导致该等待。即热块。
   - 解决方法
     1) segment header(段头)的热块，则采用ASSM处理(10g默认的处理方式)
     2) data block的热块，则采用反向索引
     3) undo header的热块，则采用automatic undo management(默认的处理方式)
     4) undo block的热块，则增大回滚段
8) free buffer waits
   - 系统I/O成为瓶颈
   - latch争用
   - SGA太小
   - SGA太大，dbwr无法快速地把脏数据刷到磁盘上
   - 解决方法
     1) 提高I/O性能
     2) 异步I/O
     3) 增加多个dbwr进程
     4) 增大SGA
9) db file scattered read
   - 文件离散读，基本可以锁定是表扫描相关问题。

*** 通过索引访问数据的流程
 1) 从索引的root开始读取
 2) 读取索引的branch
 3) 读取索引的键值
 4) 根据键值的rowid读取数据
 假如root和branch正好在同一个数据块，则可以合并到一个consistent get，那么整个过程即执行3次consistent gets。否则为4次consistent gets。
 如果SQL语句仅访问被索引的列，那么数据库只会读取索引中的数据，而不用读取表；否则读取索引后，还要根据rowid查找表中的行。

*** 索引类型
    Oracle没有SQL Server和MySQL中的聚集索引的索引类型。

    索引中的数据是已经排序的。
    1) B-tree索引(就是Balance Tree索引)
       - 原理： 一般为3层：根(root)、分支(branch)、叶(leaf)，叶之间存在链接。
       - index range scan的过程
         1) 访问root
         2) 访问branch
         3) 访问leaf
         4) 由于leaf间存在链接，因此直接访问其它leaf即可
       - 语法
         #+begin_example sql
         -- 创建索引
         create index <index_name> on <table_name>(<column_name>);
         #+end_example
    2) Bitmap索引
       - 场景：适合OLAP中值重复率高的字段，不适合DML频繁操作(因为操作时会将整个表锁定，而不是仅锁定操作的记录)。
       - 原理：
         #+begin_example sql
         select * from user where sex='M' and age > 18;

         |            | Row1 | Row2| Row3 |
         | M          | 1    | 0   | 1    |
         | F          | 0    | 1   | 0    |
         | age > 18   | 1    | 0   | 0    |
         |M and age>18| 1    | 0   | 0    |
         将where子句的按位运算判断哪些记录符合条件。
         #+end_example
       - 语法
         #+begin_example sql
         -- 创建索引
         create bitmap index <index_name> on <table_name>(<column_name>);
         #+end_example
    3) 全文索引(Text Index)
       - 场景：适合 ~like '%关键字%'~
       - 原理：利用分词库将字段自拆解后建立一组基表和基表的索引组成，所以全文索引并不是段对象。
       - 执行计划中出现Domain Index则是使用全文索引。
       - 缺点：
         1) 占用过多的磁盘空间（全文索引的大小为原表的1.5倍+）
         2) 维护成本高（恢复重建成本高）
         3) bug多，用户少
       - 语法
         #+begin_example sql
         -- 创建索引
         create index <index_name> on <table_name>(<column_name>) indextype is ctxsys.context;

         select count(*) from user where contains(name, 'john') > 0;
         #+end_example

*** 分区表
    表建立分区后，表则变为一个逻辑名称就不再是段对象，而是分区为段对象，数据是放在分区中。
    一个分区表的各个分区可以放置在不同的表空间中，那么需要删除某分区时，只需直接删除其所属的表空间即可。
    分区除了能提高性能外，也为数据管理提供便利。

    什么场景下需要使用分区表：访问频繁+数据量大。
    每个分区表记录数最好不要超过500W条。

**** 分区裁剪
     由于Oracle在扫描时是按段扫描的，对于非分区表即从文件头开始到高水位(high water marker)扫描；而分区表则从某分区的文件头到高水位扫描，而不是所有分区都扫描。

**** 分区的分类
     Oracle 11g提供7种分区。
     查看分区情况： ~select * from user_tab_partitions;~
     查看分区数据： ~select * from <table_name> partition(<partition_name>)~


     1) List Partitioning
        - 场景：分区数量固定的
        - 操作：
          #+begin_src sql
          create table <table_name>(<column_definition>+)
          partition by list(<column_name>)
          (
            partition p1 values('value1', 'value2'),
            partition p1 values('value3')
          );
          #+end_src
     2) Range Partitioning
        - 场景：以日期时间或数值等范围作为分区条件
        - 操作
          1) 创建按数值的分区表
             #+begin_example sql
             create table <table_name>(<column_definition>+)
             partition by range(<column_name>)
             (
               partition p1 values less than (1000),
               partition p2 values less than (2000),
               partition p3 values less than (3000),
               partition pmax values less than (4000) -- 此时若插入列值大于4000,则会报无法插入的异常。
             );
             #+end_example
        - 添加分区 ~alter table <table_name> add partition <partition_name> values less than(maxvalue)~
        - 删除分区 ~alter table <table_name> drop partition <partition_name>~ ，分区中的数据也会被删除。
        - 启用可移动分区行 ~alter table <table_name> enable row movement~ ，当将列值从1500修改为2500，那么将导致记录所属分区应该发生变化。由于默认行不可移动，因此会被禁止。启动该选项则可以更新。
     3) Hash Partitioning
        - 场景：通过哈希打散数据后进行分区存储，业务需求不强。用于规避热点问题。
        - 操作
          #+begin_src sql
          create table <table_name>(<column_definition>+)
          partition by hash(<column_name>)
          (
            partition p1,
            partition p2
          );
          #+end_src
     4) Composite Partitioning(复合分区)
        将多种分区方式组合一起进行分区。
        - 复合分区的种类
          1) Oracle 10g提供2种组合分区
             1) Range-Hash
             2) Range-List
          2) Oracle 11g提供6种组合分区
             1) Range-Hash
             2) Range-List
             3) Range-Range
             4) List-Range
             5) List-Hash
             6) List-List
        - 操作
          #+begin_src sql
          -- Range-Hash
          create table <table_name>(<column_definition>+)
          partition by range(<column_name1>)
          subpartition by hash(<column_name2>)
          subpartitions 4 -- Hash分区
          -- Range分区
          (
               partition p1 values less than (1000),
               partition p2 values less than (2000),
               partition p3 values less than (maxvalue)
          );
          #+end_src
     5) Interval Partitioning(间隔分区)
        Oracle 11g提供的新分区类型，可以指定时间间隔进行自动化分区。
        - 操作
          #+begin_src sql
          -- 按日、周分区
          create table <table_name>(<column_definition>+)
          partition by range(<column_name>) interval (numtodsinterval(1, 'DAY'))
          (
            partition p1 values less than (timestamp' '2015-02-01 00:00:00') -- 指定日期前的记录存储到第一个分区，然后该日期后每一日作为一个分区。
          );
           -- 按月分区
          create table <table_name>(<column_definition>+)
          partition by range(<column_name>) interval (numtoyminterval(1, 'MONTH'))
          (
            partition p1 values less than (timestamp '2018-02-01 00:00:00') -- 会创建当月的分区
          );
           -- 按年分区
          create table <table_name>(<column_definition>+)
          partition by range(<column_name>) interval (numtoyminterval(1, 'YEAR'))
          (
            partition p1 values less than (TO_DATE('1995-01-01 00:00:00', 'SYYYY-MM-DD HH24:MI:SS')) -- 会创建当年的分区
          );
          #+end_src
     6) system分区

**** 分区索引
     1) Local Index
        索引对应某个分区，某分区的DDL操作不会导致其他分区索引无效，仅影响其自身的索引而已。
        - 操作
          #+begin_example sql
          -- 创建索引
          create index <index_name> on <table_name>(<column_name>) local;
          -- 查看索引
          select * from user_ind_partitions;
          #+end_example
        - 索引字段必须为分区字段
     2) Global Index
        索引对应表的所有分区，某分区的DDL操作会导致整个索引无效。而索引失效则会影响执行计划。
        - 操作
          #+begin_example sql
          -- 创建索引
          create index <index_name> on <table_name>(<column_name>) global;
          #+end_example
        - 索引字段必须为分区字段
     3) 前缀索引或自定义索引
        在自定义数据区间定义索引。自定义索引必须使用maxvalue。
        - 操作
          #+begin_src sql
          create index <index_name> on <table_name>(<field_name>) global
          partition by range(<field_name) {
              partition p1 values less then(1000);
              partition maxp values less then(maxvalue);
          };
          #+end_src
        - 索引字段必须为分区字段


     Local Index和Global Index在查询效率上是一样的，甚至Local Index的性能还低于Global Index。
     因为当需要扫描多个Local Index时则需要重新从索引的root向下读取branch和leaf，而Global Index则是一直在leaf上逐个读取。

*** 索引失效的场景

**** 分析单条索引
     1. 将索引设置为监控状态
        #+begin_src sql
        -- 将id_tt_test索引设置为监控状态，或刷新v$object_usage中的USED字段为NO。
        alter index id_tt_test monitoring usage;
        #+end_src
     2. 查看是否有使用索引
        #+begin_src sql
        -- INDEX_NAME 为索引名称
        -- TABLE_NAME 为所属表名称
        -- MONITORING YES表示处于监控状态
        -- USED YES表示索引已启用，刚设置为监控状态时必定为NO
        -- START_MONITORING 开始监控的时间
        -- END_MONITORING 结束监控的时间
        select * from v$object_name where name = 'id_tt_test';

        -- 执行目标分析的SQL语句，然后执行上述语句查看是否用走索引。
        #+end_src
     3. 关闭监控状态
        #+begin_src sql
        -- 将id_tt_test索引设置为非监控状态
        alter index id_tt_test nomonitoring usage;
        #+end_src

**** 失效原因
     1. 在索引列上使用函数
        - 解决方法：建立函数索引，那么查询时必须使用相同的函数和非字段入参值也必须相同。
          #+begin_src sql
          create index id_tt_test
          on tt_test(to_char(datetime_created, 'YYYYMMDD')
          tablespace test_index;

          -- 查询时则要
          select * from tt_test where to_char(datetime_created, 'YYYYMMDD') < '20210101';
          #+end_src
     2. 索引的列进行隐式的类型转换
        - 问题SQL
          #+begin_src sql
          -- col1的数据类型为VARCHAR2，因此下面的语句等价于 select * from tt_test where to_number(col1) = 5;
          select * from tt_test where col1 = 5;
          #+end_src
     3. Where子句中包含不等于操作(~<>, !=, not, not in, not exist~)
        - 解决方法：使用 ~or~ 替换，如 ~age <> 0~ 替换为 ~age > 0 or age < 0~
     4. Where子句中使用 ~is null~ 或 ~is not null~
        - 解决方法：使用函数索引
          1. where子句中通过 ~nvl(column, defaultValue) = defaultValue~ 代替 ~column is null~
          2. 在列上建立函数索引 ~create index id_tt_test on tt_test(nvl(column, defaultValue)) tablespace text_index;~
     5. 对于组合/复合索引，Where子句中没有包含引导列
        - 原因：对于 ~create index index_emp on emp(col1, col2, col3);~ 复合索引，其中 ~col1~ 为引导列，因此where子句中若不出现引导列，如 ~where col2=3 and col3=4~ 是无法使用该复合索引。
        - 9i以前的where子句中必须包含联合索引的第一个字段，否则无法使用索引。
        - 9i以后可通过index skip scan，即使where子句中不包含联合索引的第一个字段，也可以利用该复合索引。
     6. ~like '%'~ 百分号在前面
     7. 查询SQL不变，但随着系统的使用本来生效的索引，现在不走索引的原因：
        1. 随着表记录的增加，通过where子句筛选出来的记录数大于全表记录的10%~15%,CBO计算认为走索引性能低于全表查询；
           - 由于Oracle支持多块读和并行读，因此筛选的记录数大于10%~15%时，TABLE ACCESS FULL（只支持单块读）的性能会高于INDEX RANGE SCAN + TABLE ACCESS INDEX BY ROWID
        2. 统计信息失效
           - 解决方法：重新搜索统计信息
        3. 索引本身失效
           - 解决方法：在线重建索引
     8. 由于表关联方式的不同，虽然索引没有失效，但没有对执行结果产生正向影响。
     9. 是否使用系统数据字典或系统视图
        - 由于系统数据字典和系统视图没有经过分析，所以导致极差的“执行计划”；
        - 注意：不要擅自对系统数据字典进行分析，否则可能导致死锁或系统整体性能下降。
     10. 索引列值存在NULL，导致不能使用索引。
         - 如 ~select count(*) from user;~ 若索引没有NULL则会使用INDEX FAST FULL SCAN，若存在NULL则会使用TABLE ACCESS FULL。
         - 查看列值是否含NULL ~select num_nulls from user_tab_col_statistics where table_name = <table_name> and column_name = <column_name>~
     11. 索引列值的选择性不高。
         - 原因分析：
           1. 若记录数有100w条，但索引列值仅有10,20,30和40这4个值，那么Oracle默认假设数据均匀分配所以每个值均占有25w条记录，那么查询时从多个执行计划中自然会选择TABLE ACCESS FULL的那个；
           2. 但实际情况是10占90w条记录，那么对10以外的使用索引查询明显效果更佳。
         - 处理方式： 针对该列重新收集统计信息，从而改变Oracle对执行计划的选择。通过 ~exec dbms_stats.gather_table_stats~ 重新收集。
         - 查看列值选择性： ~select num_distinct from user_tab_col_statistics where table_name = <table_name> and column_name = <column_name>~
           - num_distinct的值越高表示选择性越高。
     12. 并行查询无法使用索引，必须通过Hint强制查询走索引。
         - 并行查询？？

*** 冗余索引或索引碎片的弊端
    1. 耗用大量存储空间（索引段的维护和管理）
       对基本进行DML操作时，尤其执行delete操作时会引起索引表的index_entry的逻辑删除，当一个索引块中所有index_entry都被删除后，才会将该索引块删除。因此insert和delete操作会引起索引碎片问题。
    2. 增加DML完成时间
    3. 增加恢复时间

**** 以下均是错误的认知
     1. Oracle的B-Tree随着时间的推移会变得不平衡，需要重建索引；
        - 正确认知：Oracle的B-Tree会实现自平衡，和有效利用索引碎片。
     2. 索引的集群因子(Clustering Factor)不同步，需要重建索引。
        - 正确认知：反映索引键值所对应的表中的数据排序情况
          1. 如果clustering factor的值越接近索引中叶块(leaf_block)的数目，那么表中的数据越有序；若越接近表中的行数，则表中的数据越无序。
             ~select * from user_indexes where index_name = <index_name>~
          2. 重建索引不会影响集群因子，只有通过重组表的数据才能改变集群因子。

**** TODO 什么情况下应该重建索引
     1. 索引深度超过4（最基础的索引包含root，branch，leaf，即深度为3）；
        - 查看索引深度 ~select index_name, blevel from user_indexes where index_name = <index_name>~
        - blevel(二元高度)
          blevel对把ROWID返回给用户进程时需要的IO操作起关键作用，因为blevel的每一级别都会增加一个读取块，并且这些块无法顺序读取，因此每读一块都要一个独立的IO操作。（若可以顺序读取，则可以通过一个IO操作一次性读取出来）
          删除和更新操作都会增加blevel的高度。
        - 查看深度、当前叶块数目和已删叶块数目等逻辑结构，收集统计信息后执行 ~select height, blocks, lf_rows, lf_blks, lf_rows_len, lf_blk_len, br_rows, br_blks, del_lf_rows from index_stats~
     2. 已删除的索引条目(index_entry)占现有索引条目的20%～25%；
     3. 索引或分区索引因介质故障损坏；
     4. 标记为UNUSABLE的索引需要重建；
        - 6个导致分区索引不可用(UNUSABLE)的操作（http://blog.sina.com.cn/s/blog_98a873890101lndf.html）
        - 如何查看？？
     5. 索引移动到新的表空间或需要改变某些存储参数；
     6. 通过SQL*Loader加载数据到表分区后，需要重建分区索引；
     7. 重建索引以启用键压缩；
     8. 位图索引本质不同于B-Tree，因此需要重建。

**** 优化手段——在线重建索引
     公司DBA都用这个方式重建索引。
     注意：执行rebuild时不要中断操作，否则会造成对该表的所有操作均报错，只能DBA手工处理，故障时间将很长。

     #+begin_src sql
     -- 普通表的在线重建索引
     alter index <index_name> rebuild online nologging;

     -- 分区表的在线重建索引
     alter index <index_name> rebuild partition <partition_name> online nologging;
     #+end_src

**** 优化手段——索引合并(index coalesce)
     公司DBA没有使用过这个进行索引合并。

     好处：
     1. 不需要占接近磁盘存储空间的2倍；
     2. 可以在线操作；
     3. 无需重建索引结构，而是尽快合并索引叶块，这样可避免系统开销过大。


     #+begin_src sql
     alter index <index_name> coalesce
     #+end_src

** 物化视图(Materialized View)
   普通视图不存在物理存储结构，但其基表都删除后该普通视图这一数据库对象依然会存在，只是查询时会报错而已。
   物化视图是特殊的物理表
   1. 可以通过user_tables查询出来；
   2. 是段对象(segment)，因此可以有自己的物理存储属性；
   3. 占用物理存储磁盘空间，可以从user_segment查询出来。

*** 手工刷新
    默认采用手工刷新，即 ~ON DEMAND~ 。
    仅在该物化视图需要被刷新，才进行刷新(REFRESH)。

*** 自动刷新
    ~ON COMMIT~ 即一旦任意一张基表发生事务提交COMMIT，则立即刷新。

*** 创建语句
    #+begin_src sql
    create materialized view <view_name> [options...] as select * from <table_name> .....;
    #+end_src
    options:
    - ~BUILD [IMMEDIATE | DEFERRED]~ 是否在创建视图时生成数据，默认为 ~IMMEDIATE~ ，即生成。 ~DEFERRED~ 是需要时再生成数据。
    - ~REFRESH [FAST | COMPLETE | FORCE | NEVER]~
      - FAST 为增量刷新，新增、修改、删除的才刷新，其他没有发生任何变化的就不刷新。
      - COMPLETE 为全表刷新
      - FORCE 能增量刷新则增量刷新，否则全表刷新
      - NEVER 不刷新
    - ~ON [DEMAND | COMMIT]~ 手工刷新还是实时刷新
    - ~START WITH~ 通知数据库完成从主表到本地表第一次复制的时间，结合 ~ON DEMAND~ 使用。
    - ~NEXT~ 制定刷新的时间间隔，下一次刷新的时间=上一次执行完成的时间+时间间隔，结合 ~ON DEMAND~ 使用。

**** 示例 —— 定时刷新
     #+begin_src sql
     create materialized view mv_test
     refresh  force
     on demand
     start with sysdate
     next sysdate + 1 -- 一天后刷新，若要一分钟刷新一次则要设置为 sysdate + 1 / 1440
     as select * from user;
     #+end_src

**** 示例 —— 增量刷新
     原理：通过物化视图日志记录基表的记录变化，然后在执行物化视图刷新时则会根据日志对数据进行刷新。一个物化视图日志可能对应多个物化视图。
     必须添加的操作：
     1. 对每张基表执行 ~create materialized view log on <source_table> with <rowid | primary key | object id>~ ；
        - sequence 必须配合其他3种方式创建物化视图日志，即在其它方式的基础上添加更新序列而已。
          #+begin_src sql
          create materialized view log on <source_table> with <rowid | primary key | object id> sequence(<field1, field2, ...) using new values;
          #+end_src
        - 物化视图日志名称为 ~mlog$~ + 表名中前20个字符。若出现重复则Oracle会自动在后面添加数字。
     2. ~as select~ 中查询出来的字段必须包含基表的rowid。


     采用rowid时：
     #+begin_src sql
     -- 删除：drop materialized view log on <source_table1>;
     -- 修改：alter materialized view log on <source_table1> add primary key;
     create materialized view log on <source_table1> with rowid;
     create materialized view log on <source_table2> with rowid;

     create materialized view mv_test
     refresh fast
     on demand
     start with sysdate
     next sysdate + 1 -- 一天后刷新，若要一分钟刷新一次则要设置为 sysdate + 1 / 1440
     as select a.rowid, b.rowid, ...... from <source_table1> a join <source_table2> b;
     #+end_src

*** 删除语句
    #+begin_src sql
    drop materialized view <view_name>;
    #+end_src

** 并行
   应用场景：
   1) OLAP数据仓库
   2) 整块的数据读取操作
      - FTS(Full Table Scan)
      - IFFS(Index Fast Full Scan)

   并行执行高效的要素：
   1) 待处理的数据分布均匀
   2) 数据库服务器资源充足

*** 机制
    1) 用户连接数据库
    2) 数据库启动server process处理用户请求
    3) 用户发起并行查询，则server process则变为并行协调进程(qc，即Query Coordinator)
       - QC负责分发任务和接收整合数据
       - 并行工作进程通过消息和QC进程通信
       - 每个并行工作进程负责自己分配的工作

*** 并行的执行计划
    | Operation         | Description                                                                                                                         |
    |-------------------+-------------------------------------------------------------------------------------------------------------------------------------|
    | PX BLOCK ITERATOR | the first step in a parallel pipeline. breaks up the table into chunks that are processed by each of the parallel servers involved. |
    | PX SEND           | indicate that data is being sent from one parallel process to another.                                                              |
    | PX RECEIVE        | indicate that data is being received by one parallel process from another.                                                          |
    | PX SEND QC        | this is a send operation to the parallel query coordinator process.                                                                 |
    | PX COORDINATOR    | indicate that the parallel query coordinator is receiving the data from the parallel streams and returning it to the SQL statement. |
    | RANGE             | rows are distributed based on ranges of values. this is typical when sort operations are parallelized.                              |
    | HASH              |                                                                                                                                     |
    | RANDOM            | rows are randomly assigned to parallel query slaves.                                                                                |
    | ROUND ROBIN       | rows are distributed one at a time in a circular fashion, just as you would deal cards in a game of poker.                          |

    | In-Out Value | Other Tag Name                | Description                                                                                                                           |
    |--------------+-------------------------------+---------------------------------------------------------------------------------------------------------------------------------------|
    | P->S         | PARALLEL_TO_SERIAL            | the top level of parallel query. the results are fed in parallel to  the qc.                                                          |
    | P->P         | PARALLEL_TO_PARALLEL          | the parallel processing that pass results to a second set of parallel processes.                                                      |
    | S->P         | PARALLEL_FROM_SERIAL          | indicate a serial bottleneck within a parallel statement, since it suggests that parallel processing might wait on serial processing. |
    | PCWP         | PARALLEL_COMBINED_WITH_PARENT | output of step goes to next step in same parallel process.                                                                            |
    | PCWC         | PARALLEL_COMBINED_WITH_CHILD  | input of step comes from prior step in same parallel process.                                                                         |

*** 并行度
    设置方式
    1) 表级别
       #+begin_example sql
       alter table <table_name> parallel <n>;
       #+end_example
    2) SQL级别
       #+begin_example sql
       select /*+ parallel */ * from <table_name>;
       #+end_example

    1) 并行DDL
       - ~create table ... as select~

       - ~alter table ... split partition~
       - ~alter table ... coalesce partition~
    2) 并行DML(分区表)

** 变量绑定

*** 目的
    1) 减少SQL硬解析的次数
    2) 减少系统资源的开销
    3) 减少Latch争用

*** 场景
    - 适合的应用场景——OLTP
      1) 用户并发量高
      2) 表中有主键
      3) 操作的数据量少
      4) 执行计划基本相同
      5) SQL的重复率高
    - 不适合的应用场景——OLAP(瓶颈在IO)
      1) 执行计划多变
         如查询范围和条件变化大，对应的执行计划应该有所不同。
      2) 用户少
      3) SQL解析对系统性能影响小

*** 硬解析和软解析
    - SQL硬解析，语法解析后通过CBO生成执行计划。
      1) 解析(Parsing)
         1) 语法分析(Syntax Check)
         2) 语义分析(Semantic Check)
         3) 共享池搜索执行计划(Shared Pool Check)，判断是否采用软解析
      2) CBO分析得到多个执行计划(Optimization)
      3) 获取最优执行计划(Row Source Generation)
      4) 执行(Execution)
    - SQL软解析，语法解析后直接采用之前的执行计划。
      1) 解析(Parsing)
         1) 语法分析(Syntax Check)
         2) 语义分析(Semantic Check)
         3) 共享池搜索执行计划(Shared Pool Check)，判断是否采用软解析
      2) 执行(Execution)

    #+begin_example sql
    -- 一次硬解析，后面使用时都是软解析
    select * from tbl where id=:id;

    -- 只解析一次
    begin
      for i in 1..4 loop
        execute immediate 'select * from tbl where rn=:i' using i;
      end loop;
    end;
    /
    #+end_example

*** 操作
    1) 查看SQL解析数据
       #+begin_example sql
       select sql_text, -- SQL脚本
              parse_calls, -- 解析的总次数，包含硬解析和软解析
              loads, -- 硬解析的总次数
              execution -- 执行的总次数
       from v$sql
       where sql_text like 'select count(*) from tbl%';
       #+end_example
    2) 清理shared pool
       #+begin_example sql
       alter system flush shared_pool;
       #+end_example

*** softer_soft_parse
    参数：session_cached_cursor
    对于已经关闭的cursor，可以把它的资源保留在PGA中，用于后续对cursor的继续使用。
    参数值为当前会话的游标的缓存大小
    1) 0则不缓存cursor，再次使用时需要创建cursor，并从Shared Pool中搜索SQL的解析，有则软解析无则硬解析；
    2) 大于0则可缓存相应数量的cursor，再次使用时则从缓存中搜索，若命中则直接打开cursor即可。

    v$sql_shared_cursor

*** Cursor(游标)
    Cursor是在Session Memory中有个指针指向Shared Pool的Library Cache中的某条已被解析并生成执行计划的SQL语句。

**** 父子游标
     父游标(parent cursor)：第一条运行的SQL
     子游标(child cursor)：后续的SQL

     实例：分别在两个不同的用户下执行相同的语句 ~select * from user;~ ，按理在Shared Pool的Library Cache中应该仅有一条记录，但由于是在不同的用户下执行，因此会有两条记录。
     #+begin_example sql
     select sql_id, child_number, sql_text from v$sql where sql_text = 'select * from user';
     #+end_example
     | SQL_ID | CHILD_NUMBER | SQL_TEXT           |
     |--------+--------------+--------------------|
     | abc    |            0 | select * from user |
     | abc    |            1 | select * from user |
     #+begin_example sql
     select sql_id,
            auth_check_mismatch -- 权限差异产生该子游标，Y为是，N为否
     from v$sql_shared_cursor where sql_id='abc';
     #+end_example
     | SQL_ID | AUTH_CHECK_MISMATCH |
     |--------+---------------------|
     | abc    | N                   |
     | abc    | Y                   |

     还有其它原因导致会产生多个子游标(即无法重用SQL的执行计划)，可以通过查询v$sql_shared_cursor。

**** 游标共享
     共用某条已被解析并生成执行计划的SQL语句。
     启用：
     #+begin_example sql
     -- exact 仅SQL语句完全相同时才能使用游标共享，即N个SQL就产生N个父子游标对。那么需要开发人员自行采用变量绑定来达到游标共享的使用条件
     -- force Oracle会自动将where子句的断言值替换为变量绑定，即生成N个SQL产生1个父子游标对。
     -- similar
     alter session set cursor_sharing=exact
     #+end_example

** 导致性能问题的可能原因
1) 表没有正确建立索引——导致错误的执行计划
2) 表没有及时的分析——因表提供的信息量不足，导致错误的执行计划
3) 热块——数据块的争用
   - 反向索引：本来邻近的数据块，通过建立反向索引就会分散开来。
4) 锁的阻塞——业务设计缺陷
5) SQL解析消耗大量CPU——变量绑定
6) 低效SQL——SQL自身问题
7) 数据库负载过大——架构设计问题

*** 统计消息不准，当谓词越界时造成性能问题
    谓词越界，SQL语句查询条件超出统计消息所记录的范围，导致因Oracle优化器错误地选择SQL语句的执行计划。
    注意：此时执行计划反馈的Rows、Bytes和Cost数据都是错误的。

**** 一般发生的场景
     1. 业务临时表或接口表
        - 表操作特点：先导入大量数据到临时表，然后处理完后将结果导入到业务表，最后删除临时表数据。
        - 问题原因：自动收集统计信息时，临时表数据小，但白天处理数据时已导入大量数据到临时表。导致处理数据时执行计划走错。
     2. 巨大表
        - 问题原因：若巨大表存在1000W条记录，那么要10%即100W条记录被修改过才会触发自动收集统计信息，因此很容易导致处理数据时执行计划走错。

**** 示例
     1. 表t1(id)包含1到10000个值
        #+begin_src sql
        create table t1 (col1 number);
        create index idx_t1 on t1(col1);

        begin
            for i in 1..10000 loop
                insert into t1 values(i);
            end loop;
            commit;
        end;
        /
        #+end_src
     2. 收集统计信息
        #+begin_src sql
        -- 此时执行 select * from user_tab_col_statistics where table_name='t1'; 返回空集合
        -- 而执行 select * from user_tab_statistics where table_name='t1'; 返回的记录的GLOBAL_STATS、USER_STATS均为NO
        -- 即尚未收集该表格的统计信息。
        exec dbms_stats.gather_table_stats(owner => 'MYTEST', tablename => 'T1');
        -- 此时执行 select * from user_tab_col_statistics where table_name='t1'; 返回一条关于col1的记录
        -- 而执行 select * from user_tab_statistics where table_name='t1'; 返回的记录的GLOBAL_STATS为YES，而USER_STATS为NO。

        -- 通过dbms_stats.convert_raw_value将low_value和high_value翻译为可读信息
        var number :x;
        exec dbms_stats.convert_raw_value(select low_value from user_tab_col_statistics wehre table_name='T1' and column_name='COL1', :x);
        #+end_src
     3. 插入超过原记录数10%的新记录，并且不手动重新收集统计信息（也没有到晚上自动重新收集统计信息）
        #+begin_src sql
        begin
            for i in 10001..10000000 loop
                insert into t1 values(i, to_char(i));
            end loop;
            commit;
        end;
        /

        -- 等待几分钟后查询得到YES
        select stale_stats from user_tab_statistics where table_name = 'T1';
        #+end_src
     4. 此时得到的执行计划将有可能完全错误
        - 示例：如执行计划显示性能很高，但执行如下查询却相当耗时
          #+begin_src sql
          set timing on;
          select count(col1) from t1 where col1 between 10001 and 100000;
          #+end_src
     5. 手动执行统计信息收集
        #+begin_src sql
        exec dbms_stats.gather_table_stats('TEST', 'T1');
        #+end_src
     6. 查看统计信息是否已经更新
        #+begin_src sql
        -- 更新后应该返回NO
        select stale_stats from user_tab_statistics where table_name = 'T1';
        #+end_src
     7. 此时执行计划会恢复正常。

** 性能问题定位方式
   原则：从小范围分析问题
   1) SQL层
      - 工具：执行计划
      - 影响因素：数据量会影响同一个SQL是否高效。
   2) 会话层
      - 会话层可以定位哪个用户的SQL产生性能问题
      - 工具
        - V$SESSION
        - V$SESSTAT，查看SESSION占用的资源
        - V$SESSION_WAIT，查询SESSION卡在哪个位置
          #+begin_example sql
          select sid, event from V$SESSION_WAIT where sid in (99,232);
          -- 99  enq: TX - row lock contention 表示会话ID为99的会话正在请求TX锁
          -- 232 SQL Net message from client
          #+end_example
        - V$SQL
        - V$LOCK，查看数据库中当前的所有锁
          #+begin_example sql
          -- 查看动态视图的字段
          desc V$LOCK;

          select sid,  -- 持有锁的会话ID
                 type, -- 锁的类型
                 id1, -- 当type为TM时，id1代表表对象的名称，可以通过 select * from dba_objects where object_id = 来获取该对象的信息。
                 id2, -- id1+id2是回滚段的位置，加排它锁和请求排它锁的会话的id1+id2相同。
                 lmode, -- 6 表示排它锁，4 表示数据块头修改的排它锁，3 表示共享锁，0 表示没有获取到锁。级别越高的锁限制越多。
                 request, -- 0 表示没有请求锁， 6 表示请求排它锁
                 block -- 0 表示没有阻塞其它会话，1 表示阻塞其它会话
          from V$LOCK where type in ('TM', 'TX') order by 1,2;

          -- 获取当前会话的SID
          select sid from V$MYSTAT;
          #+end_example
        - SQL_TRACE
   3) 系统层
      - 工具
        - AWR(Oracle8中为STATSPARK) 不结合业务很难分析性能问题。如业务就是负载重，而AWR只能告知负载重，不能说明是性能问题。
        - OS Tools(top, iostat)
** 统计信息

*** 优化器收集的统计信息如下
    1. Table Statistics
       - Number of Rows
       - Number of blocks
       - Average Row Length
    2. Column Statictics
       - Number of Distinct Values(NDV) in Column
       - Number of Nulls in Column
       - Data Distribution (Histogram)
    3. Index Statictics
       - Number of Leaf Blocks
       - Levels
       - Clustering Factor

* 表空间
  数据库文件：数据文件、日志文件和控制文件。
  数据库 1-n 表空间。
  表空间 1-n 数据文件。
  用户 n-n 表空间。用户和表空间没有隶属关系。

  数据库的逻辑结构：表空间(tablespace)、段(segment)，数据块(data block)和模式对象(schema object)。
  表空间(tablespace)，每个数据库创建时默认会创建SYSTEM表空间，可以对除SYSTEM外其他表空间，进行online和offline操作。
  段(segment)，表空间内存储的对象，分为数据段、索引段、回退段和临时段。段由区组成，每个区的大小不一定相同，但必须是数据块大小的整数倍。
  数据块(data block)，是数据库中最小的IO单位，也是内存数据缓冲区的单位和数据文件存储空间的单位。由参数DB_BLOCK_SIZE设置，必须设置为操作系统块大小的整数倍。
  模式对象(schema object)，表、聚簇、视图、索引序列生成器、同义词等。

  #+begin_src sql
  -- 创建临时表空间
  create temporary tablespace mydb_temp
  tempfile '/path/to/mydb_temp.dbf' -- 指定临时表空间存储的数据文件
  size 1024m
  autoextend on
  next 50m maxsize 2048m
  extent management local;

  -- 创建数据表空间
  create tablespace mydb
  datafile '/path/to/mydb.dbf' -- 指定数据表空间存储的数据文件
  size 1024m
  autoextend on
  next 50m maxsize 2048m
  extent management local;

  -- 为数据表空间添加数据文件
  alter tablespace mydb add datafile '/path/to/datafile.dbf'
  size 1024m
  autoextend on
  next 50m maxsize 2048m
  extent management local;

  -- 创建用户并指定表空间
  create user my identified by my123
  default tablespace mydb
  temporary tablespace mydb_temp;
  #+end_src

* 表

** 分类
   1. 堆表/堆组织表（Heap Organized Table）
      - 默认创建的就是堆组织表。
      - 行无序存储，访问采用全表扫描，即按命中顺序来获取数据，而不是数据插入的顺序。
      - 当增加数据时，在段中找到第一个适合数据大小的空闲空间；当删除数据后，留下的空间允许随后的insert和update重用。
      - 存储速度快，适合write IO多的场景。
      - 相关参数
        - pctfree
        - pctused
        - initrans 事务槽，默认值为2.每个并发事务都需要一个事务槽，若可能会对相同的块多个并发操作，则考虑设置大点的值。
   2. 分区表
   3. 索引组织表（IOT, Index Organized Table）
      - 行根据主键有序存储在索引段，即数据存放在插入前已经确定了（一般就是先插入的记录排在前面）。
      - 由于IOT将普通表和索引二合为一，所以经常以主键查询表数据，那么可以考虑建立索引组织表，加快表的访问速度。但插入和删除速度相对普通表要慢一些。
      - IOT的叶子节点不存储ROWID，并且叶子节点存储所有表列。
        - 若表列很多时，可以划分索引段和溢出段，加快访问速度。索引段存储除主键外其它常访问的字段，其它字段则存储在溢出段中。
      - 创建语句
        #+begin_src sql
        create table iot_timran(
          id int,
          name varchar2(50),
          sal int,
          description varchar2(1000),
          constraint pk_timrn primary key(id)
        )
        organization index -- IOT
        pctthreshold 30 -- 当行数据在叶子节点占用大小超过30%，则就爱那个index_entry分为一分为二，包含主键和using指定字段的则存储在索引段，其他存储在溢出段。
        overflow tablespace users -- 溢出段所属的表空间
        including name, sal; -- 指定除主键外哪些字段存储在索引段。

        -- select index_name, index_type, table_name from from user_indexes; IOT的索引的index_type为IOT-TOP。
        -- select segment_name, segment_type, partition_name from user_segments; 查看是否产生两个段。
        #+end_src
   4. 索引聚簇表(Cluster Table)
      - 两个相互关联的表的相关数据，存储在一个簇块中，并且多个表共用一个索引。当联合查询时，只要扫描一个数据块就可以，可以提高IO效率。
      - 不宜使用簇表的场景
        1. 表数据将被大量修改；
        2. 单表全表扫描；
        3. 频繁对表进行truncate操作，聚簇的表是不能truncate的，只能truncate簇；
        4. 若查询的结果的数据量经常超出一两个Oracle数据块。
      - 创建语句
        #+begin_src sql
        -- 创建簇段(Cluster Segment)
        create cluster cluster1(deptno number)
        PCTUSED 60
        PCTFREE 10
        -- 每个聚簇键关联1024字节的数据，假设数据块大小为8KB，那么一个数据块会存储7个聚簇键。即deptno值为10～60的数据均存在同一个数据块中。
        -- 若size过小则导致一个聚簇键无法保存一整条数据，那么会产生跨数据块的串链，那么就失去建立簇表的意义了。
        -- 若size过大则导致则每个数据块存储的聚簇键过少，浪费空间。
        SIZE 1024
        TABLESPACE USERS
        storage (
          INITIAL 128k
          NEXT 128k
          MINEXTENTS 2
          MAXEXTENTS 20
        );
        -- 基于簇，创建两个相关表
        create table student(deptno number, sname varchar2(20)) cluster cluster1(deptno);
        create table address(deptno number, detail varchar2(20)) cluster cluster1(deptno);
        -- 为簇创建索引
        -- 在建立索引前无法对表进行查询操作。因为查询操作是先从聚簇索引上拿到一个聚簇键值，在其中获取数据块地址，再读取数据。
        create index index1 on cluster cluster1;
        #+end_src
      - 查看
        #+begin_src sql
        select * from user_clusters;
        select * from user_clu_columns;
        #+end_src
      - 删除
        #+begin_src sql
        drop cluster cluster1; -- 仅仅适合删除空簇
        drop cluster cluster1 including tables; -- 删除簇和簇表
        drop cluster cluster1 including tables cascade constratints; -- 删除簇、簇表和外键月是
        #+end_src
   5. 哈希聚簇表
      - 数据存储上和索引聚簇表一样，而定位方面是采用哈希函数直接定位数据的物理位置，而不是采用索引。
      - 适用：
        1. 表大小可估算；
        2. 绝大多数查询可通过hash列进行等值查询。
      - 不适用：
        1. 全表扫描；
        2. 大量DML操作。
   6. 有序哈希聚簇表
      - 数据存储和数据定位方式和哈希聚簇表一样，但数据存储上还会按照某个指定列有序存储。
      - 适用：where子句是hash列，order by子句是某个用于有序存储的指定列。
      - 创建语句
        #+begin_src sql
        CREATE TABLE <table_name>(<column> <data_type> SORT);
        #+end_src
   7. 临时表
      - 临时表的数据放在PGA中，若超过PGA中则会将数据放在临时表空间
   8. 外部表
      - Oracle 9i后提供的功能
   9. 压缩表
   10. 嵌套表
       - 没啥用。

** 外部表
   结论：不是适用于实现文件元数据存储。

   Oracle 9i后提供的功能，通过向Oracle提供外部表的元数据，把一个操作系统文件、目录等当作一个只读的数据库表进行操作。外部表的特点：
   1. 位于文件系统中，按一定的格式分割数据；
   2. 可通过SQL语句访问外部表，而且不需要事先加载外部表到数据库中；
   3. 外部表为只读状态，无法执行DML操作，也不能建索引；
   4. 只能使用DBMS_STATS包采集外部表的统计信息。

        LOCATION('tb1.exp', 'tb2.exp') -- 指定外部表的文件
        PARALLEL 2 -- 并行方式填充数据，并行度必须和文件数量一致才能生效。
        AS -- 填充使用的原始数据

** 临时表
   1. 临时表本质是cache，事先建好表结构，存放会话相关的数据。并存在2种模式
      - on commit delete rows，基于事务的临时段，在事务提交时，就会自动删除记录；
      - on commit preserve rows, 基于会话的临时段，会话结束时相关数据会被自动清理。
   2. WITH子句实质是Oracle自动为用户创建临时表；
   3. 没有DML锁，但可以建立索引、约束、试图和触发器；
   4. 会产生少量UNDO信息，因此会产生少量REDO信息。但不会为修改临时表的操作记录生成REDO信息，因此对临时表的操作是不可恢复的。
      - UNDO信息，用于回滚到事务中的一个SAVEPOINT。如执行INSERT操作，当插入到第100条记录时因违反约束条件导致插入失败，那么必须回滚到INSERT操作之前。
      - 由于UNDO数据必须生成REDO来保护UNDO操作。
      - 而临时表一般执行INSERT和SELECT操作，SELECT操作不产生UNDO数据，因此若不执行INSERT操作则不会产生REDO数据。

* 分页查询优化
** Oracle伪列
   伪列和表列相似，但并不存储在表中
   1. 可以查询；
   2. 不可以插入、修改和删除。


   常用的伪列：
   1. ~ROWID~ 表中行的物理存储地址，可以唯一标识数据库中的一行，所以可以通过 ~ROWID~ 快速定位表中的一行。
      - 通过索引查询时，也是从索引中提取rowid，然后根据rowid定位表中的数据行。
   2. ~ROWNUM~ 是查询返回结果中行的序号，可以用于限制返回的行数。
      - 从1开始
      - 分页时的注意事项
        ~select * from emp where rownum > 2~ 或 ~select * from emp where rownum = 2~ 将报查询失败
        - 原因：
          1. rownum是针对查询结果，根据从数据文件或缓冲区读取记录的顺序生成的伪列，即先查出结果集之后再加上去的一个列；
          2. 若rownum=2，那么由于从数据文件或缓冲区读取出来的第一条记录的rownum为1，而被丢弃，rownum归0。然后再从从数据文件或缓冲区读取出来一条记录，其rownum再次被设置为1，也会被丢弃，rownum又归0。
      - 性能误解
        由于rownum针对查询结果而生成的伪列，因此对查询性能的优化并无影响，但由于缩小的返回的记录集合，因此会减少记录的网络IO。
** 优化——通过最大记录序号限制查询范围
   #+begin_src sql
   -- 不带排序
   select *
   from (select rownum rn, t.* from <table_name> t where rownum < &maxnum)
   where rn > &minnum;

   -- 带排序
   select *
   from (select rownum rn, t.*
         from (select * from <table_name> order by <column_name>) t
         where rownum < &maxnum)
   where rn > &minnum;
   #+end_src
