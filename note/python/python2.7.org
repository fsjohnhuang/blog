* Python虚拟机
1) CPython
2) PyPy，用Python实现的虚拟机
3) Jython，用Java实现的虚拟机
4) IronPtython，用C#实现的虚拟机
5) RPython，带静态类型的Python

** 虚拟机运行机制
1) 完成模块（源码或字节码）的加载和链接
2) 将源码编译为字节码，写入内存供CPU读取
3) 从内存中读取字节码并执行，结束后将字节码回写到硬盘（文件后缀为.pyc或.pyo）

后续再次执行该脚本时，虚拟机会先检查字节码文件是否和源码一致（通过修改时间），一致则直接执行字节码。

* 包机制
Python代码按照模块(module)划分，一个模块可以是一个文件或一个包含一个或多个子模块的目录。而模块同时可以视作包(package)

** 模块检索
~str, len 和 Exception~ 等存在于全局作用域，在Python中被称为builtin scope；其他模块则通过 ~import~ 语句引入。

*** import语句
1) import语句可导入如下对象
   1) 模块文件(.py文件)
   2) C或C++扩展（已编译为共享库.so或DLL文件）
   3) 包（包含多个模块）
   4) 内建模块（使用C编译并已链接到Python解释器中）
2) 运行原理
   - Python解释锡会根据 ~sys.path~ 和 环境变量 ~PYTHONPATH~ 中定义的本地目录路径，按顺序依次检索模块，而 ~sys.path~ 默认第一元素为空字符串，即表示运行目录。
   - 运行目录：执行程序时程序所在的目录，固定不变。
   - 当前工作目录（Current Working Directory）：可通过环境变量等修改，作为读写本地文件系统中文件/目录的当前目录。
   - 导入的模块会以模块名和模块对象对的形式保存到 ~sys.modules~ 中，因此对于多次导入同一模块时，默认情况下仅第一次导入生效。
   - import实际上会调用__import__导入模块。

* .pyc、.pyo和.pyd文件
.pyc为.py的字节码(byte code)文件，内部保存的是PyCodeObject的序列化信息，不是二进制代码。
.pyd为Python的动态链接库文件。

注意：
1) .pyc会保存在源码相同的目录下；
2) .pyc的内容与Python版本有关，因此Python2.4生成的.pyc不能在Python2.5上运行；
3) .pyo是优化版的.pyc文件，体积更小。

1) 方式1 ~export PYTHONDONTWRITEBYTECODE=1~
2) 方式2 通过 ~python -B 脚本.py~
3) 方式3 在导入的地方写
   #+BEGIN_EXAMPLE python
   import sys

   sys.dont_write_bytecode = True
   #+END_EXAMPLE

*** 自定义上下文管理器
上下文管理器就是遵循了上下文管理协议（Context Management Protocol）的类， 即实现了 ~__enter__~ 和 ~__exit__~ 方法的对象。

#+BEGIN_EXAMPLE python
import sys
import cx_Oracle

class OracleContext(object):
    def __init__(self, uri, schema, passwd, err={}):
        self.connstr = '%s/%s@%s' % (schema, passwd, uri)
        self.err = err
        self.err['value'] = None

    def __enter__(self,):
    """进入with语句块时会调用上下文管理器实例的__enter__函数，并将其返回值绑定到as子句后的变量上。
  
    返回值为列表或元组时，as子句为 as (a,b,c) 进行解构
    """
        self.conn = cx_Oracle.connect(
            self.connstr, encoding='utf-8', nencoding='utf-8')

        return self.conn

    def __exit__(self, ex_type, ex_value, ex_traceback):
    """退出with语句块时会掉用上下文管理器实例的__exit__函数，无论with语句块是否抛出异常。

    若with语句块中没有抛出异常，或者执行break/continue/return语句，那么以None作为参数调用context_manager.__exit__(None, None, None)
    若抛出异常则会以sys.exc_info()的作为参数调用context_manager.__exit__(*sys.exc_info())

    Args:
        ex_type: 异常类型
        ex_value: 异常实例
        ex_traceback: 异常发生时的调用堆栈信息

    Returns:
        当with语句块中抛出异常时，返回True表示忽略异常，False则重新抛出异常
    """
        if ex_type:
             self.err['value'] = str(ex_value)
        
        if self.conn:
            if ex_type:
                self.conn.rollback()
            else:
                self.conn.commit()

            self.conn.close()

        return True

# 使用
rows = []
with OracleContext('','','') as conn:
    cursor = conn.cursor()
    cursor = cursor.execute('select 1 from dual')
    rows = cursor.fetchall()

# 虚拟机会将上述语句翻译为
rows = []
context_manager = OracleContext('', '', '')
exit = type(context_manager).__exit__
conn = type(context_manager).__enter__(context_manager)
try:
    cursor = conn.cursor()
    cursor = cursor.execute('select 1 from dual')
    rows = cursor.fetchall()
except:
    if not exit(context_manager, *sys.exc_info()):
        raise
else:
    exit(context_manager, None, None, None)
#+END_EXAMPLE

*** 通过contextlib模块定义上下文管理器
contextlib模块提供一些帮助方法和修饰器去定义用于with语句块的对象。

1) contextmanager修饰符
#+BEGIN_EXAMPLE python
from contextlib import (
    contextmanager,
)
import cx_Oracle

@contextmanager
function db_transaction(connstr):
    """yield前的代码作为__enter__方法，yield后的代码作为__exit__方法。
    with语句块的异常将通过yield语句传递到生成器函数中。
    """
    conn = cx_Oracle(connstr)
    try:
        yield conn
    except:
        conn.rollback()
    else:
        conn.commit()
    finally:
        conn.close()      

rows = []
with db_transaction('') as conn:
    cursor = conn.cursor()
    cursor = cursor.execute('select 1 from dual')
    rows = cursor.fetchall()
#+END_EXAMPLE

3) closing上下文管理器
#+BEGIN_SOURCE python
class closing(object):
    def __init__(self, thing):
        self.thing = thing

    def __enter__(self,):
        return self.thing

    def __exit__(self, *exc_info):
        self.thing.close()

* 迭代器/游标
当迭代器取完数据后，再次尝试取元素时会抛出 ~StopIteration~ 异常。

** 可迭代对象
1) 列表
2) 元组
3) 字符串
4) 字典
5) 文件

** 操作
#+BEGIN_EXAMPLE python
# 迭代器
a = [1,2,3,4,5]
ia = iter(a)  # iter函数会将可迭代对象转换为迭代器

print(ia.next())  # 1
print(next(ia))   # 2
for x in ia:
    print(x)      # 3,4,5

# 生成器函数
def gen():
  yield 1
  yield 2
  yield 3
  yield 4
  yield 5

ib = gen()
print(ib.next())  # 1
print(next(ib))   # 2
for x in ib:
    print(x)      # 3,4,5

# 生成器表达式
ic = (i for i in range(1,6))
print(ic.next())  # 1
print(next(ic))   # 2
for x in ic:
    print(x)      # 3,4,5
#+END_EXAMPLE

** 自定义迭代器
迭代器就是实现了next方法的类
#+BEGIN_EXAMPLE python
class Text(object):
    def __init__(self, text):
        self.__cursor = 0
        self.__text = text

    def __iter__(self,):
        return self

    def next(self,):
        if self.__cursor == len(self.__text):
            raise StopIteration()
        else:
            i = self.__cursor
            self.__cursor += 1
            return self.__text[i]

for char in Text('abcd'):
    print(char)
#+END_EXAMPLE

from functools import wraps

def add1_missing_attrs(func):
    """定义个普通的修饰器
    """
    def inner(*args, **kwargs):
        result = func(*args, **kwargs)
        return result + 1

    return inner

def add1_leaving_attrs(func):
    """由于装饰器实际是高阶函数，因此原函数的__name__,__doc__和__module__等函数特性会被外层函数替换掉， 通过functools.wraps复制原函数特性到外层函数特性。
    """
    @wraps(func)
    def inner(*args, **kwargs):
        result = func(*args, **kwargs)
        return result + 1

    return inner    

def addx(x):
    def tmp(func):
        @wraps(func)
        def inner(*args, **kwargs):
            return x + func(*args, **kwargs)
        
        return inner
    
    return tmp

@add1_missing_attrs
def add2(n):
    return n + 1

@add1_leaving_attrs
def add3(n):
    return n + 2

@addx(4)
def add4(n):
    return n

** Lambda函数
Lambda函数又称为匿名函数，Python中的Lambda函数有以下限制：
1) 函数体中必须仅包含一个表达式，表达式的返回值作为函数的返回值；
2) 闭包/周围全局作用域(surrounding global scope)采用延迟绑定。

def bug():
    return [lambda x: x * i for i in xrange(0, 5)]

def bingo():
    return [lambda x, i=i: x * i for i in xrange(0, 5)]

bug_results = map(lambda f: f(1), bug())
bingo_results = map(lambda f: f(1), bingo())

print(bug_results)   # 回显 [4,4,4,4,4]
print(bingo_results) # 回显 [0,1,2,3,4]

class SDK(object):
    """1. API为/users/:users/status对应的SDK为SDK().users('john').status。
       2. API为/users/status对应的SDK为SDK().users.status。
    """
    params = ('users')

    def __init__(self, path=''):
        self.__path = path

    def __getattr__(self, seg):
        if params.count(seg) > 0:
            return lambda v: SDK('%s/%s/%s' % (self.__path, seg, v))
        else:
            return SDK('%s/%s' % (self.__path, seg))

    def __str__(self,):
        return self.__path

    def __repr__(self,):
        return '<SDK %s>' % self.__path

#+BEGIN_EXAMPLE python
from pprint import pprint

class ModelMetaclass(type):
    """自定义类型类。
    必须继承type，并且约定俗成名称以Metaclass结尾。
    """
    def __new__(cls, name, bases, attrs):
        """创建类型时会调用__new__方法来创建。
        
        Args:
            cls: 准备创建的类对象
            name: 类名称
            bases: 元组类型，父类列表
            attrs: 字典类型，属性和方法

        Returns: 类型
        """
        if name == 'Model':
            return type.__new__(cls, name, bases, attrs)
        
        mappings = dict()
        for k, v in attrs.iteritems():
            if isinstance(v, Field):
                mappings[k] = v
                pprint('Found mapping: %s ==> %s' % (k, v))

        for k in mappings.iterkeys():
            attrs.pop(k)

        attrs['__table__'] = name
        attrs['__mappings__'] = mappings
        return type.__new__(cls, name, bases, attrs)


class Model(dict):
    __metaclass__ = ModelMetaclass

    def save(self,):
        tpl_sql = 'insert into %s(%s) values(%s)'
 
        fields = []
        params = []
        args = []
        for k, v in self.__mappings__.iteritems():
            fields.append(v.name)
            params.append('?')
            args.append(getattr(self, k, None))

        sql = tpl_sql = % (self.__table__, ','.join(fields), ','.join(params))
        pprint('SQL: %s' % sql)
        pprint('ARGS: %s' % str(args))


class Field(object):
    def __init__(self, name):
        self.name = name

* 日志
官网： https://docs.python.org/2/library/logging.html

** 概念
日志类型：诊断日志、审计日志。
诊断日志：记录与应用操作相关的日志，如异常日志（用于线上排查问题）
审计日志：为商业分析而记录的日志，如Apache的访问记录日志（统计PV、UV和最热资源、出错率等信息）

Logger，调用的接口，自身带日志级别特性，一个Logger可对应0~N个Handler；
LogRecord，将从Logger得到的日志传递到Handler；
Handler，用于配置日志输出地点和保存时间，自身带日志级别特性，一个Handler对应一个Formatter；
Filter，用于配置输出日志的级别（debug,info,warn,error和critical）；
Formatter，用于配置日志的输出格式。
默认输出格式： ~<日志级别>:<Logger实例名称>:<日志消息内容>~ ，如 ~WARNING:root:warn message~ 。

日志级别
1) notset，最低日志级别；
2) debug，详细信息，只有开发调试时需要关注；
3) info，证明事件按预期工作，突出强调应用的运行过程；
4) warn/warning，发生可容忍的错误，但需要修复；
5) error，严重错误，但程序可以继续运行，需要马上修复；
6) critical/fatal，严重错误，程序已经不能继续运行。

logging日志记录流程
1) 判断当前Logger的日志级别是否可用，不可用则不输出日志，并流程结束；
2) 创建LogRecord，判断是否符合注册到当前Logger的Filter条件，不符合则不输出日志，并流程结束；
3) 将LogRecord交给当前Logger的Handler（若没有配置Handler则进入第四步），判断Handler的日志级别是否可用，若可用则判断是否符合注册到该Handler的Filter条件，若符合则输出日志；
4) 若当前Logger是否含父日志对象，有则将LogRecord传递给父日志对象，递归执行第一步直至到根日志对象。

若产生了LogRecord但整个流程都没有找到Handler进行判断Handler的日志级别是否可用，若可用则判断是否符合注册到该Handler的Filter条件等后续操作，那么会向sys.stdout输出提示信息。

#+BEGIN_EXAMPLE python
import os
import time
import logging
from functools import wraps

logging.basicConfig(level=logging.DEBUG)

def record_time(func):
    @wraps(func)
    def inner(*args, **kwargs):
        # time.clock()获取当前程序占用的CPU时间（time.time会包含其他程序占用的CPU时间）
        start = time.clock()
        ret = func(*args, **kwargs)
        end = time.clock()
        logger.debug('function %s running with args %s and kwargs %s cost %s secs.'
                    , func.__name__
                    , str(args)
                    , str(kwargs)
                    , end-start)

        return ret

    return inner


@record_time
def add(a, b):
    return a + b

** 多进程

*** Linux下的进程
1) *会话(Session)* ，是一组 *进程组* 的集合，当用户通过ssh或其它终端接入Linux时就会创建一个会话，其下的所有进程组默认属于该会话
   1) *进程* 创建新 *会话* 时会同时创建一个 *默认进程组* ，并且该进程将作为默认进程组的组长进程，会话ID和进程组ID与进程该进程ID一致；
   2) 组长进程不能创建其他会话。
2) *进程组(Process Group)* ，是一组 *进程* 的集合，可以接收来自同一个会话的 *信号*
   1) 每个进程必须属于一个进程组；
   2) 子进程默认从属于父进程所属的进程组；
   3) 进程组的生存期为从进程组创建到最后一个进程离开（进程终止或挪到其他进程组）；
   4) 每个进程仅能修改自身或子进程所属的进程组；
   5) 进程只能挪到同一个会话下的其它进程组；
   6) 每个进程组必须有一个组长进程，且进程组ID和组长进程ID一致；
   7) 进程组分为 *后台进程组* 和 *前台进程组*
      1) *前台进程组* 
         1) 同一个时间最多存在1个前台进程组；
         2) 在终端进行如下操作只会发送到前台进程组
            1) ~C-c~ SIGINT(kill foreground process)
            2) ~C-d~ (terminate input or exit shell)
            3) ~C-z~ SIGTSTP(suspend foreground process)
            4) ~fg~ 恢复被挂起的前台进程
            5) ~C-/~ SIGQUIT，终止前台进程组并生成core文件
      2) *后台进程组* 
         1) 可以同时存在多个；
         2) 创建方式： 命令后添加 ~&~ 即可，如 ~ping 127.0.0.1 &~ 。
      3) *守护进程(Daemon)*
         - 特殊类型的后台进程，随系统启动而启动。
   8) 同一个管道的命令属于同一个进程组
      - 在bash中执行 ~ps aux | grep 123~ 涉及3个进程
        1) bash(pid=1, pgid=1, sid=1)
        2) ps(ppid=1, pid=2, pgid=2, sid=1)，创建一个新的进程组，执行第一个命令的进程作为组长进程
        3) grep(ppid=1, pid=3, pgid=2, sid=1)
3) 进程的7个状态
   1) R可执行状态(TASK_RUNNING)
      - 意味进程进入某个CPU的可执行队列中，并不一定正在运行，而进程调度器将从各个CPU的可执行队列中选择一个进程在对应的CPU上运行；
      - 操作系统教材中描述的进程状态READY和RUNNING在Linux中统一为TASK_RUNNING状态。
   2) S可中断的睡眠状态(TASK_INTERRUPTABLE)
      - 意味进程因等待某事件发生（Socket连接、信号量等）而被挂起，此时进程被放进对应事件的等待队列中，当这些事件发生时（由外部中断触发或由其他进程触发），那么对应等待队列中的进程将被唤醒。
      - 大部分进程都处于该状态
   3) D不可中断的睡眠状态(TASK_UNINTERRUPTABLE)
      - 和S相似进程处于睡眠状态，但CPU仅响应外部硬件的中断，而不响应异步信号量的中断。也就是执行 ~kill -9~ 无法杀死处于该状态的进程。
      - TASK_UNINTERRUPTABLE的意义在于内核在执行某些操作时不能被异步信号量打断，如调用系统调用read从某个设备文件读取数据，那么流程为 read->执行设备驱动代码->与物理设备进行交互，若中途响应异步信号量中断，那么有可能造成设备状态混乱的情况。
      - 进程处于TASK_UNINTERRUPTABLE状态的过程非常短暂，因此通过ps命令很难发现处于此状态的进程。
      - 系统调用vfork被调用后当前进程会处于TASK_UNINTERRUPTABLE状态，直到子进程调用exec或exit为止。
   4) T暂停或跟踪状态(TASK_STOPPED or TASK_TRACED)
      - TASK_RUNNING/TASK_INTERRUPTABLE -> TASK_STOPPED，向进程发送SIGSTOP信号量（SIGSTOP和SIGKILL一样都是强制性的，不允许进程trap该信号量自定义处理函数来忽略默认行为）。
      - TASK_STOPPED -> TASK_RUNNING，向进程发送SIGCONT信号量。
      - TASK_TRACED意味着进程被暂停下来，等待跟踪它的进程对其进行操作。处于TASK_TRACED状态的进程不响应SIGCONT信号量，只有跟踪进程调用系统调用ptrace执行PTRACE_CONT、PTRACE_DETACH等操作，或跟踪进程退出时，它才会恢复为TASK_RUNNING状态。
   5) Z退出-僵尸状态(TASK_DEAD-EXIT_ZOMBIE)
      - 进程退出时，进程处于TASK_DEAD状态，此时进程所占用的资源将被释放，最后剩下仅包含退出码和一些统计信息的task_struct结构体的壳供父进程获取子进程的退出信息。由于保留task_struct结构体的壳，因此此时的状态为EXIT_ZOMBIE。
      - 子进程退出过程中会向父进程发送SIGCHLD信号量，通知父进程来收尸进入EXIT_DEAD状态完成退出。
      - 父进程可以通过 ~wait(wait4,waitid)~ 系列的系统调用来等待子进程退出和收尸，并获取子进程退出信息。
      - 若父进程在子进程退出前已经退出，那么父进程在退出前会将其子进程转移给同一进程组的下一个进程或init进程。
      - 僵尸进程的危害
        - 进程不再执行任务，但仍然占用PCB进行维护，浪费内存资源。
   6) X退出-进程即将被销毁(TASK_DEAD-EXIT_DEAD)
      - 当进程为多线程程序中被detach过的进程，或父进程通过clone创建的将SIGCHLD的处理函数设置为SIG_IGN的子进程，那么该进程退出时将不保留task_struct来等待父进程收尸，即不转移为EXIT_ZOMBIE状态，直接转移为EXIT_DEAD状态释放进程资源。
      - 这个状态很短暂，无法通过ps命令查看。
4) 进程状态转移
   1) 从非TASK_RUNNING转移为TASK_RUNNING
      - 由别的进程或中断处理程序执行唤醒操作。
   2) 从TASK_RUNNING转移为非TASK_RUNNING
      - 响应信号量进入TASK_STOPPED或TASK_DEAD状态
      - 主动执行系统调用进入TASK_INTERRUPTABLE状态(通过sleep函数)或TASK_DEAD状态(如exit函数)
      - 因资源不满足进入进入TASK_INTERRUPTABLE状态或TASK_UNINTERRUPTABLE状态(如select函数)
   3) 示例：向一个处于task_interruptable状态的进程发送信号量SIGKILL来结束进程
      1) 将进程状态从TASK_INTERRUPTABLE转移为TASK_RUNNING；
      2) 然后当该进程被CPU执行时响应SIGKILL，状态从TASK_RUNNING转移为TASK_DEAD。
5) PCB
   - 对于多道程序操作系统而言，内存中将存在多个进程，为管理这些进程操作系统内核为每个进程建立一个task_struct结构体保存该进程相关的信息，这个结构体就是PCB(Process Control Block)。
   - Linux下采用链式方式组织PCB。
   - PCB的组成
     1) 标识符，每个进程都有进程标识符、用户标识符和组标识符。
        1) PID是32位无符号有序正数；
        2) 新创建的进程PID是最近创建的进程PID+1；
        3) Linux下允许PID上限为32767，若超过这个数则会重新使用已闲置的PID。
     2) 进程状态，7个进程状态。
     3) 进程优先级，整数，优先级高的进程则会更早获取CPU时间。
     4) 程序计数器，用于记录将被执行的下一条指令的地址。
     5) 内存指针
        1) 程序代码和进程相关的指针；
        2) 和其他进程共享内存的指针。
     6) 上下文数据
        - 进程得到CPU时间时寄存器中的数据。当进程失去CPU时间时会将寄存器中的数据暂存到相应的PCB中，待重新获取CPU时间时再次恢复到寄存器和堆栈中。
     7) I/O状态信息
        - 显式的I/O请求
        - 分配给进程的I/O设备
        - 被进程使用的文件列表等
     8) 审计信息
        - CPU时间总和
        - 使用的时钟总和
        - 时间限制
        - 审计号等
6) 进程创建的方式
   1) fork
   2) system
   3) exec
7) 进程的运行环境
   - 进程的运行环境就是进程启动时的环境变量，而这些环境变量是由 *系统环境变量副本* 和 *父进程的环境变量副本* 组成。
   - 特点是
     1) 只允许本进程修改自身的运行环境，父进程和子进程都无权修改；
     2) 增删改运行环境不影响父进程和已创建的子进程。
   - 系统环境变量 ~PYTHONPATH~ 将决定模块查找路径
8) 进程结构
   1) 代码段
   2) 数据段
   3) 堆栈段
9) 进程间通信(IPC)
   - 由于操作系统隔离各个进程可以访问的的内存地址空间(即进程拥有各自的用户地址空间)，因此内核提供一套称为IPC的机制用于进程间交换数据。
   - 底层原理：内核开辟一段地址空间作为缓冲区，进程A将数据从A的用户地址空间复制到内核缓冲区，然后再将数据从内核缓冲区复制到进程B的用户地址空间。
   - 方式
     1) 管道（匿名管道、命名管道）
     2) 消息队列
     3) 共享内存
     4) 信号量
     5) Socket

*** 进程创建方式——system
system函数会通过启用 ~/bin/sh~ 进程执行入参指定的shell命令，shell命令执行期间信号量SIGCHID会被阻塞（即告知内核不要在发送SIGCHID信号量），而SIGINT和SIGQUIT会被忽略。
system函数内部逻辑为：
1) 通过系统调用fork创建一个子进程，失败则返回-1，正常则执行第2步；
2) 然后在子进程中调用系统调用excl执行 ~/bin/sh -c <command>~ 执行入参指定的命令，若excl执行失败则返回127，成功则返回入参命令的返回值；
3) 而父进程进入TASK_INTERRUPTABLE状态，等待子进程结束后恢复为TASK_RUNNING，并返回子进程的返回状态；
4) 若父进程在休眠状态被信号量终端时，则返回-1。

而Python中的 ~os.system~ 则是对C标准库中的 ~system~ 函数的封装。
#+BEGIN_EXAMPLE python
import os

# 返回值为0则表示命令执行成功，其他值表示执行失败。
# 若有异常则抛出OSError。
ret_code = os.system('ls')
#+END_EXAMPLE

*** 进程创建方式——exec函数族
  在Linux系统中创建进程分为两步：创建进程实例 和 加载进程映像。
而exec函数族则用于在当前进程实例上加载新的进程映像替代原有的进程映像，即将代码段换成新的程序代码，废弃原有的数据段和堆栈段，保留进程其他信息（PID/PPID/PGID/SID）。
  os.exec*函数族为对Linux下exec函数族的封装。因此Python程序通过os.exec*函数族创建新进程时，Pyton程序将马上退出，因此os.exec*函数族没有返回值。

*** IPC——管道
  管道分为 *匿名管道(限于父子进程间通信)* 和 *命名管道(任意进程间通信)* 。
  管道的原理是在内存开辟一段区域供通信双方的进程读取和写入。原理类似于两个进程通过操作同一个文件来实现通信，但比操作文件更高效。

半工管道：仅有一方可读、另一个方可写。
双工管道：双方均可读可写，通过两个半工管道实现。

**** 匿名管道
特点：
1) 匿名管道仅存活在内存中，进程退出时释放；
2) 管道为单向通信；
3) 管道内部保证同步机制，保证数据访问一致性；
4) 管道采用环形父进程向只写管道中写入的数据，将传递到子进程的只读管道中；
5) 



#+BEGIN_EXAMPLE c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <string.h>

#define INPUT 0  // 只读管道
#define OUTPUT 1 // 只写管道

void execute_subprocess(int* fds) {
    printf("in the spawned/child process...\n");
    // 由于子进程不从父进程读取信息，因此子进程关闭只读管道
    close(fds[INPUT]);
    // 子进程向只写管道写入数据，向父进程传递
    char* message = "test data";
    write(fds[OUTPUT], message, strlen(message));
    _exit(0);
}

void execute_main_process(int* fds) {
    printf("in the spawning/parent process...\n");
    // 由于父进程不向子进程传递信息，因此父进程关闭只写管道
    close(fds[OUTPUT]);
    // 父进程从只读管道读取子进程传递过来的信息
    char buf[256];
    // read将阻塞当前进程，直到只读管道中有数据可读取位置。
    int count = read(fds[INPUT], buf, sizeof(buf));
    printf("%d bytes of data received from subprocess: %s\n", count, buf);
    exit(0);
}

void main(){
    // 文件描述符，用于保存只读管道和只写管道，从而构成双工管道
    int fds[2];
    pipe(fds);

    pid_t pid;
    pid = fork();

    if (pid == -1){
        printf("Error in fork\n");
        exit(1);
    }
    else if (pid == 0){
        execute_subprocess(fds);
    }
    else {
        execute_main_process(fds);
    }
}

**** 命名管道
命名管理使用文件系统，任意两个独立的进程都可以访问它。命名管道特性如下：
1) 通过 ~ls -al~ 显示文件类型为p；
2) 大小总为0，因为其内容一直存在于内存中，而不是保存在磁盘上；

** 多线程

 CPython的线程就是C语言的pthread，并通过操作系统调度算法进行调度。
 因此thread模块是轻量级，而threading模块是对thread模块的封装。
 Queue模块是同步的、线程安全的队列类，一般和threading配合使用实现FIFO、LIFO和优先队列等。

** GIL(Global Interpreter Lock)
CPython解释器内部通过GIL实现线程安全，即任何时候仅有一个Python线程执行。
因此Python线程在多核服务器上，运行IO密集型计算性能好，但运行计算密集型则性能下降。

多线程环境下，虚拟机执行流程：
1) 设置GIL
2) 切换到某个线程并执行
3) 运行指定的字节码集合
4) 线程主动让出控制(调用time.sleep())
5) 设置线程为休眠状态
6) 解锁GIL
7) 回到步骤1

注意：当调用外部代码（如C/C++扩展函数）时会先释放GIL让其它线程运行。
