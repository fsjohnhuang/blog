** 虚拟机运行机制
1) 完成模块（源码或字节码）的加载和链接
2) 将源码编译为字节码，写入内存供CPU读取

后续再次执行该脚本时，虚拟机会先检查字节码文件是否和源码一致（通过修改时间），一致则直接执行字节码。

** 模块和包的区别
包为一个或多个模块的集合，一般压缩打包为tarball的文件形式，内含

*** setuptools(不支持Python3.x)

为PEAK(Python Enterprise Application Kit)开发的Python的基于distutils工具的增强工具，提供依赖项管理机制，更方便地创建和发布Egg包。
*distutils* 是标准库中内置的用于Python包安装和发布的工具，但功能单一业界不直接使用。

**** 打包格式
两种格式在安装时都不需要重新编译程序，并且两者本质上就是个Zip文件。
1) Egg: 后缀为.egg，由setuptools在2004年引入。
2) Wheel: 后缀为.whl，由PEP427在2012年定义，用于替代Egg的二进制包的标准格式。
   Wheel又分为三个子类型：
   1) Pure Wheels——只能兼容某个版本的Python，如py2或py3
   2) Universal Wheels——可以兼容py2和py3
   3) Extension Wheels

两者的区别[[https://packaging.python.org/discussions/wheel-vs-egg/][https://packaging.python.org/discussions/wheel-vs-egg/]]：
1) Wheel 有一个官方的 PEP427 来定义，而 Egg 没有 PEP 定义
2) Wheel 是一种分发格式，即打包格式。而 Egg 既是一种分发格式，也是一种运行时安装的格式，并且是可以被直接 import
3) Wheel 文件不会包含 .pyc 文件
4) Wheel 使用和 PEP376 兼容的 .dist-info 目录，而 Egg 使用 .egg-info 目录
5) Wheel 有着更丰富的命名规则。
6) Wheel 是有版本的。每个 Wheel 文件都包含 wheel 规范的版本和打包的实现
7) Wheel 在内部被 sysconfig path type 管理，因此转向其他格式也更容易

***** ~python setup.py sdist~ 详解
 用于构建源码分发包。
 选项：
 1) ~--formats~ 设置压缩格式
    - 默认值： ~--formats=gztar,zip~ 表示Unix下为 ~.tar.gz~ 而Windows下为 ~.zip~ 。
    - 可用格式表
      | Format | Description |
      |--------+-------------|
      | zip    | .zip        |
      | gztar  | .tar.gz     |
      | bztar  | .tar.bz     |
      | xztar  | .tar.xz     |
      | ztar   | .tar.Z      |
      | tar    | .tar        |
 2) ~--owner~ 设置分发包的所有者
 3) ~--group~ 设置分发包的所属组
 4) ~--no-defaults~ 剔除默认的打包内容规则
 5) ~--no-prune~ 剔除默认的排除内容规则
 6) ~--manifest-only~ 仅从MANIFEST.in生成MANIFEST文件，而不执行构建打包

****** 默认打包内容规则
 1) 默认打包内容：
    1) all Python source files implied by the py_modules and packages options.
    2) all C source files mentioned in the ext_modules or libraries options.
    3) scripts identified by the scripts option.
    4) anything that looks like a test script: test/test*.py (currently, the Distutils don’t do anything with test scripts except include them in source distributions, but in the future there will be a standard for testing Python module distributions)
    5) README.txt (or README), setup.py (or whatever you called your setup script), and setup.cfg.
    6) all files that matches the package_data metadata.
    7) all files that matches the data_files metadata.
 2) 默认排除内容：
    1) all files in the Distutils “build” tree (default build/)
    2) all files in directories named RCS, CVS, .svn, .hg, .git, .bzr or _darcs

****** 自定义打包内容规则
通过和setup.py同目录的MANIFEST.in可自定义打包内容规则，MANIFEST.in内每行包含一条文件筛选命令，运行结果为生成MANIFEST文件(临时文件)，MANIFEST内每行为打包的文件路径。
#+BEGIN_EXAMPLE in
include *.py *.yaml # 从项目根目录开始包含所有后缀为.py和.yaml的文件

exclude test.* # 从项目根目录开始排除所有匹配test.*的文件

recursive-include dir1 *.exe # 从目录dir1开始递归包含所有后缀为.exe的文件

recursive-exclude dir2 test.* # 从目录dir2开始递归排除所有匹配test.*的文件

prune dir3 # 排除目录dir3下的所有文件

graft dir4 # 包含目录dir4下的所有文件
#+END_EXAMPLE

****** MANIFEST生成规则
1) 若存在MANIFEST文件，并且首行没有标识该文件由MANIFEST.in生成，则直接使用；
2) 若存在MANIFEST.in文件，则将该文件和默认打包规则和排除规则合并后生成MANIFEST文件；
3) 若不存在MANIFEST.in文件，则直接根据默认打包规则和排除规则生成MANIFEST文件。

作为setuptools的一个分支后来又合并会setuptools项目中，而又因为共享命名空间，因此在导入setuptools时 ~import setuptools~ 时实际上也导入了distribute。

** ~from xxx import *~ 的陷阱
#+BEGIN_EXAMPLE python
# bar.py
from datetime import datetime

# __main__.py
import datetime
from bar import *

# 由于from bar import *会将bar.py中通过from datetime import datetime也导入到__main__.py中，
# 而bar.py导入的type类型的datetime替代了__main__.py需要的类型为module的datetime，
# 因此下列语句会报错。
print(datetime.datetime.now())
#+END_EXAMPLE
因此导入模块时拒绝 ~*~ ，最佳实践为
#+BEGIN_EXAMPLE
# 形式1
from xxx import (
  xxx,
  xxx,
)

# 形式2
import xxx
#+END_EXAMPLE

#+BEGIN_EXAMPLE

* .pyc、.pyo和.pyd文件
.pyc为.py的字节码(byte code)文件，内部保存的是PyCodeObject的序列化信息，不是二进制代码。
.pyd为Python的动态链接库文件。

注意：
1) Python2.x中.pyc会保存在源码相同的目录下；
2) Python3.x+开始.pyc会保存在同目录下的__pycache__目录下；
3) .pyc的内容与Python版本有关，因此Python2.4生成的.pyc不能在Python2.5上运行；
4) .pyo是优化版的.pyc文件，体积更小；

** 禁止生成.pyc文件
当仅有一个py文件时，并不会产生.pyc文件。

1) 方式1 ~export PYTHONDONTWRITEBYTECODE=1~
2) 方式2 通过 ~python -B 脚本.py~
3) 方式3 在导入的地方写
   #+BEGIN_EXAMPLE python
   import sys

   sys.dont_write_bytecode = True
   #+END_EXAMPLE

* 全局变量
由于Python的变量采用定义声明结合的方式，因此对在局部对全局变量赋值时会产生歧义。
因此Python采取原则是——局部优先，在没有明显表示该变量为全局变量时，将优先作为局部变量定义。
#+begin_src
# 全局变量a和b
a = 1
b = [2, 3]

# 若没有定义同名的局部变量，那么全局变量可直接读取
def read_global_directly():
    print(a)
    print(b[0])

def update_global():
    # 通过关键字global声明当前作用域变量a为全局变量
    global a
    a = 2

    # 这里没有明显表示b为全局变量，因此会定义一个新的局部变量b

*** 自定义上下文管理器
上下文管理器就是遵循了上下文管理协议（Context Management Protocol）的类， 即实现了 ~__enter__~ 和 ~__exit__~ 方法的对象。

#+BEGIN_EXAMPLE python
import sys
import cx_Oracle

class OracleContext(object):
    def __init__(self, uri, schema, passwd, err={}):
        self.connstr = '%s/%s@%s' % (schema, passwd, uri)
        self.err = err
        self.err['value'] = None

    def __enter__(self,):
    """进入with语句块时会调用上下文管理器实例的__enter__函数，并将其返回值绑定到as子句后的变量上。

    返回值为列表或元组时，as子句为 as (a,b,c) 进行解构
    """
        self.conn = cx_Oracle.connect(
            self.connstr, encoding='utf-8', nencoding='utf-8')

        return self.conn

    def __exit__(self, ex_type, ex_value, ex_traceback):
    """退出with语句块时会掉用上下文管理器实例的__exit__函数，无论with语句块是否抛出异常。

    若with语句块中没有抛出异常，或者执行break/continue/return语句，那么以None作为参数调用context_manager.__exit__(None, None, None)
    若抛出异常则会以sys.exc_info()的作为参数调用context_manager.__exit__(*sys.exc_info())

    Args:
        ex_type: 异常类型
        ex_value: 异常实例
        ex_traceback: 异常发生时的调用堆栈信息

    Returns:
        当with语句块中抛出异常时，返回True表示忽略异常，False则重新抛出异常
    """
        if ex_type:
             self.err['value'] = str(ex_value)

        if self.conn:
            if ex_type:
                self.conn.rollback()
            else:
                self.conn.commit()

            self.conn.close()

        return True

# 使用
rows = []
with OracleContext('','','') as conn:
    cursor = conn.cursor()
    cursor = cursor.execute('select 1 from dual')
    rows = cursor.fetchall()

# 虚拟机会将上述语句翻译为
rows = []
context_manager = OracleContext('', '', '')
exit = type(context_manager).__exit__
conn = type(context_manager).__enter__(context_manager)
try:
    cursor = conn.cursor()
    cursor = cursor.execute('select 1 from dual')
    rows = cursor.fetchall()
except:
    if not exit(context_manager, *sys.exc_info()):
        raise
else:
    exit(context_manager, None, None, None)
#+END_EXAMPLE

class SDK(object):
    """1. API为/users/:users/status对应的SDK为SDK().users('john').status。
       2. API为/users/status对应的SDK为SDK().users.status。

    params = ('users')

    def __init__(self, path=''):
        self.__path = path

    def __getattr__(self, seg):

        else:
            return SDK('%s/%s' % (self.__path, seg))

        return self.__path

        return '<SDK %s>' % self.__path

** ~__str__~ 和 ~__repr__~ 的区别
   两者均可实现将类到字符串的转换。

   1) ~__str__~
      1) 调用时机
         1) ~print()~
         2) ~str()~
         3) ~'{}'.format()~
      2) 作用
         返回供人们阅读的内容
   2) ~__repr__~
      1) 调用时机
         1) 在REPL中查看对象时
         2) 对于列表和字典等容器，即使调用 ~str()~, ~print()~ 等都总会调用 ~__repr__~
         3) ~repr()~
      2) 作用
         存在的目的在于调试，便于开发者使用。即直接复制其返回的内容到命令行，就可以直接执行。
      3) 注意：当找不到 ~__str__~ 则会自动调用 ~__repr__~ ，因此至少需要添加一个 ~__repr__~ 方法。
   3) ~__unicode__~
      Python2中数据类型str只能存储ASCII码，而unicode才能存储Unicode码（而Python3的数据类型str则存储Unicode码），因此为更方便地输出超出ASCII字符集的字符，Python2中提供 ~__unicode__~ 方法。
      1) 调用时机
         1) ~unicode()~
            若找不到 ~__unicode__~ 则会调用 ~__str__~ 并将其结果按当时的编码方式解码返回。
      2) Python2中的最佳实践

         class Car(object):

                 self.color = color
                 self.mileage = mileage

             def __repr__(self):
                 return '{}({!r},{!r})'.format(self.__class__.__name__, self.color, self.mileage)

             def __unicode__(self):
                 return u'a {self.color} car'.format(self=self)

** 概念
日志类型：诊断日志、审计日志。
诊断日志：记录与应用操作相关的日志，如异常日志（用于线上排查问题）
审计日志(Audit Log)：为商业分析而记录的日志，如Apache的访问记录日志（统计PV、UV和最热资源、出错率等信息）
以规定格式准确详细记录每笔操作流水，用于后续分析统计。


Logger，调用的接口，自身带日志级别特性，一个Logger可对应0~N个Handler；
LogRecord，将从Logger得到的日志传递到Handler；
Handler，用于配置日志输出地点和保存时间，自身带日志级别特性，一个Handler对应一个Formatter；
Filter，用于配置输出日志的级别（debug,info,warn,error和critical）；
Formatter，用于配置日志的输出格式。
默认输出格式： ~<日志级别>:<Logger实例名称>:<日志消息内容>~ ，如 ~WARNING:root:warn message~ 。

日志级别
1) notset，0, 最低日志级别；
2) debug，10, 详细信息，只有开发调试时需要关注；
3) info，20, 证明事件按预期工作，突出强调应用的运行过程；
4) warn/warning，30, 发生可容忍的错误，但需要修复；
5) error，40, 严重错误，但程序可以继续运行，需要马上修复；
6) critical/fatal，50, 严重错误，程序已经不能继续运行。

logging日志记录流程
1) 判断当前Logger的日志级别是否可用，不可用则不输出日志，并流程结束；
2) 创建LogRecord，判断是否符合注册到当前Logger的Filter条件，不符合则不输出日志，并流程结束；
3) 将LogRecord交给当前Logger的Handler（若没有配置Handler则进入第四步），判断Handler的日志级别是否可用，若可用则判断是否符合注册到该Handler的Filter条件，若符合则输出日志；
4) 若当前Logger是否含父日志对象，有则将LogRecord传递给父日志对象，递归执行第一步直至到根日志对象。

若产生了LogRecord但整个流程都没有找到Handler进行判断Handler的日志级别是否可用，若可用则判断是否符合注册到该Handler的Filter条件等后续操作，那么会向sys.stdout输出提示信息。

import os

def record_time(func):
    @wraps(func)
    def inner(*args, **kwargs):
        # time.clock()获取当前程序占用的CPU时间（time.time会包含其他程序占用的CPU时间）
        start = time.clock()
        ret = func(*args, **kwargs)
        end = time.clock()
        logger.debug('function %s running with args %s and kwargs %s cost %s secs.'
                    , func.__name__
                    , str(args)
                    , str(kwargs)
                    , end-start)

        return ret

    return inner


@record_time
def add(a, b):
    return a + b

  1) 采用单个byte存储，最多只能表示256种符号的字符编码集
     - ~ASCII（American national Standard Code for Information Interchange，美国国家标准信息交换码）~ 中1个byte代表95种符号的一种。只使用十六进制的20~7F的byte。
     - ~ISO Latin 1~ 或 ~ISO 8859-1~ 对 ASCII 进行了扩展。只剩下十六进制的00~1F和80~9F的byte留空。
     - ~Windows-1252~ 或 ~CP-1252~ 在ISO 8859-1的基础上增加27种字符。只剩下十六进制的00b~1F留空。
  2) 采用两个byte存储，但也仅能表示部分符号而已

  3) Unicode编码标准
     - 目标：表示所有符号的字符编码集。
     - 编码标准：每种符号对应一个code point，形如 ~U+xxxx~ 其中的x为十六机制字符。总共能表示110万个符号，现在已有11万个符号在册。
     - 传播和存储的编码方式：
       Unicode为一套揽括ASCII、GB2312等编码的编码标准，但在实际存储和传播时的编码规则。而UTF8等为通过bytes编码来表示Unicode中的每个code point，被称为encoding。
       - ~UTF-8(8-bit Unicode Transformation Format)~
         - 作为最流程的对Unicode进行传播和存储的编码方式，采用可变长的编码方式，以单字节为编码单元
           1) 英文采用1个字节表示，即ASCII
           2) 汉字采用3个字节表示
           3) 生僻字采用4-6个字节表示
         - Unicode和UTF-8转换时机和用法
           - 在文件存储和网络传播时将Unicode转换为UTF-8
           - 从文件和网络读取到内存时，将UTF-8转换为Unicode

** BOM(byte order mark，字节顺序标记)
   BOM(byte-order mark)，对于多字节编码字符集，不同的系统拥有不同的BOM（大端和小端两种字节序列）用于解释字符编码的字节序列，位于最前几个字节中。
   对于大部分编辑器而言，由于他们能识别Unicode编码，因此会自动处理掉BOM。但当程序无法自动处理BOM时，则会将BOM作为正文的一部分来进行解释，从而造成乱码。
   UTF-8为单字节编码字符集，因此BOM在UTF-8中并没有标识字节序列的作用，甚至可以省略。而UTF-8 with BOM(即utf-8-sig)中的BOM的作用变为告知编辑器/浏览器，当前文本采用UTF-8编码而已。
   | 编码                   | 十六进制    | Unicode    |
   |------------------------+-------------+------------|
   | UTF-8                  | EF BB BF    | \uefbbbf   |
   | UTF-16 (Big Endian)    | FE FF       | \ufeff     |
   | UTF-16 (Little Endian) | FF FE       | \ufffe     |
   | UTF-32 (Big Endian)    | 00 00 FE FF | \u0000feff |
   | UTF-32 (Little Endian) | FF FE 00 00 | \ufffe0000 |
   | Unicode标准压缩        | 0E FE FF    | \u0efeff   |

*** 各操作系统的规则
    - Windows 建议所有Unicode文本均应该以BOM开头，用于识别文件使用的编码和字节顺序
      - 使用Windows系统自带文本编辑器以UTF-8编码方式保存文本时，会自动在文本最开始位置插入BOM
      - 使用第三方编辑器(如editplus)以UTF-8编码方式保存文本时，不会自动在文本最开始位置插入BOM
    - Linux/UNIX 不使用BOM

*** 进程结束信号 SIGTERM 和 SIGKILL、SIGSTOP, SIGINT，SIGABRT的区别

4) SIGABRT(6)，程序执行 ~abort~ 函数发送

** 操作系统简史
1) 手工操作
   没有操作系统的概念，必须逐个用户通过打孔卡片完成输入，然后等待CPU计算处理后，再将结果输出到卡片/纸条等媒介．
2) 联机批处理
   存在批处理程序，用户将作业(程序和数据)通过输入机输入，等待作业数达到一定量后批量通过输入机传输给CPU计算处理，然后将结果输出到输出机．
   但在作业输入输出的过程中，CPU依然处于空闲状态。
3) 离线批处理
   输入机、卫星机、高速磁带和输出机组成一套与CPU独立的输入输出系统。用户将作业(程序和数据)通过输入机(多台)录入作业，作业通过卫星机连接高速磁带从而将作业存储到高速磁带上，当CPU(主机)处于空闲状态时则自动从高速磁带上读取作业执行，再将结果输出到高速磁带，然后卫星机从高速磁带读取结果输出到输出机。
   但每次CPU(主机)内存中仅存放一道作业，每当它运行期间发出IO请求后，高速的CPU便会处于等待低速IO完成的空闲状态。
4) 多道程序系统
   同时把多个程序加载到内存，并允许它们交替在CPU中运行，它们共享系统中各种硬、软件资源，当一道程序因IO请求而暂停运行时，CPU便立即转去运行另一道程序。
   多道程序系统的出现，标志这操作系统逐渐趋成熟的阶段，先后出现作业作业调度管理、处理机管理、存储器管理、外部设备管理、文件系统管理等功能。
   由于多个程序同时在计算机中运行，开始有了空间隔离的概念，只有内存空间隔离，才能让数据更加安全、稳定。除了空间隔离之外，多道技术还第一次体现了时空复用的特点，遇到IO操作就切换程序，提高CPU利用率，让计算机的工作效率更高。
   但若一个程序一直占用CPU，那么其他程序将无法获取CPU资源。
5) 多道批处理系统
   在多道程序系统上添加批处理功能，在系统运行期间不允许用户与作业发生交互。
   好处提高系统吞吐量。
6) 分时系统
   一台计算机可同时连接多个用户终端，而每个用户可在自己的终端上联机使用计算机，好像自己独占机器一样。
   分时技术：把处理机的运行时间分成很短的时间片，按时间片轮流把处理机分配给各联机作业使用。
   特点：
   1) 多路性，若干个用户同时使用一个台计算机。微观上是各用户轮流使用计算机；宏观上是各用并行工作。
   2) 交互性，用户可根据系统对请求的响应进一步向系统提出新的请求。因此分时系统又称为交互式系统。
   3) 独立性，用户之间可以独立操作，互不干扰。
   4) 及时性，系统对用户的输入能及时作出响应。
   5) 限制：分时系统在接收多个用户请求时，由于内存空间有限，因此会将未轮到的作业放入磁盘，一旦轮到则调入内存。若CPU时间片用完后，又将作业存回磁盘。
   6) 缺点：当没有发生IO操作而是因CPU时间片用完被切换掉，这样并没有提高CPU的效率，反而会降低。
7) 实时系统
   用于实时控制和实时信息处理的应用领域。实时系统能够及时响应随机发生的外部事件，并在严格的时间范围内完成对该事件的处理。
   如：飞机控制和导弹控制系统。
   特点：
   1) 及时响应，每个信息接收、分析处理和发送的过程必须在严格的时间限制内完成；
   2) 高可靠性，需采取冗余措施，双机系统前后台工作，可包括必要的保密措施等。
8) 通用操作系统
   操作系统的3种基本类型：多道批处理系统、分时系统、实时系统。
   通用操作系统：具有多种类型特征的操作系统，可以同时兼由多道批处理、分时和实时处理的功能，或其中两种的功能。

** 多进程

*** 守护进程
运行在后台并且只在需要时才唤醒的进程，称为守护进程。（如电子邮件、新闻、打印等）

*** Linux下的进程
查看进程层级关系： ~ps -ejH~ 或 ~ps axjf~

1) *会话(Session)* ，是一组 *进程组* 的集合，当用户通过ssh或其它终端接入Linux时就会创建一个会话，其下的所有进程组默认属于该会话
   1) *进程* 创建新 *会话* 时会同时创建一个 *默认进程组* ，并且该进程将作为默认进程组的组长进程，会话ID和进程组ID与进程该进程ID一致；
   2) 组长进程不能创建其他会话。
   3) 会话结束时会销毁其下的所有前台和后台进程组（由于守护进程随系统启动而启动，不属于该会话下，因此不会被销毁）
      具体是向其下的前台和后台进程发送信号量SIGINT和SIGHUP来结束进程。
      - 让进程不被结束 ~nohup <command> &~
2) *进程组(Process Group)* ，是一组 *进程* 的集合，可以接收来自同一个会话的 *信号*
   1) 每个进程必须属于一个进程组；
   2) 子进程默认从属于父进程所属的进程组；
   3) 进程组的生存期为从进程组创建到最后一个进程离开（进程终止或挪到其他进程组）；
   4) 每个进程仅能修改自身或子进程所属的进程组；
   5) 进程只能挪到同一个会话下的其它进程组；
   6) 每个进程组必须有一个组长进程，且进程组ID和组长进程ID一致；
   7) 进程组分为 *后台进程组* 和 *前台进程组*
      1) *前台进程组*
         1) 同一个时间最多存在1个前台进程组；
         2) 在终端进行如下操作只会发送到前台进程组
            1) ~C-c~ SIGINT(kill foreground process)
            2) ~C-d~ (terminate input or exit shell)
            3) ~C-z~ SIGTSTP(suspend foreground process)
            4) ~fg~ 恢复被挂起的前台进程
            5) ~C-/~ SIGQUIT，终止前台进程组并生成core文件
      2) *后台进程组*
         1) 可以同时存在多个；
         2) 创建方式： 命令后添加 ~&~ 即可，如 ~ping 127.0.0.1 &~ 。
      3) *守护进程(Daemon)*
         - 特殊类型的后台进程，随系统启动而启动。
   8) 同一个管道的命令属于同一个进程组
      - 在bash中执行 ~ps aux | grep 123~ 涉及3个进程
        1) bash(pid=1, pgid=1, sid=1)
        2) ps(ppid=1, pid=2, pgid=2, sid=1)，创建一个新的进程组，执行第一个命令的进程作为组长进程
        3) grep(ppid=1, pid=3, pgid=2, sid=1)
3) 进程的7个状态
   1) R可执行状态(TASK_RUNNING)
      - 意味进程进入某个CPU的可执行队列中，并不一定正在运行，而进程调度器将从各个CPU的可执行队列中选择一个进程在对应的CPU上运行；
      - 操作系统教材中描述的进程状态READY和RUNNING在Linux中统一为TASK_RUNNING状态。
   2) S可中断的睡眠状态(TASK_INTERRUPTABLE)
      - 意味进程因等待某事件发生（Socket连接、信号量等）而被挂起，此时进程被放进对应事件的等待队列中，当这些事件发生时（由外部中断触发或由其他进程触发），那么对应等待队列中的进程将被唤醒。
      - 大部分进程都处于该状态
   3) D不可中断的睡眠状态(TASK_UNINTERRUPTABLE)
      - 和S相似进程处于睡眠状态，但CPU仅响应外部硬件的中断，而不响应异步信号量的中断。也就是执行 ~kill -9~ 无法杀死处于该状态的进程。
      - TASK_UNINTERRUPTABLE的意义在于内核在执行某些操作时不能被异步信号量打断，如调用系统调用read从某个设备文件读取数据，那么流程为 read->执行设备驱动代码->与物理设备进行交互，若中途响应异步信号量中断，那么有可能造成设备状态混乱的情况。
      - 进程处于TASK_UNINTERRUPTABLE状态的过程非常短暂，因此通过ps命令很难发现处于此状态的进程。
      - 系统调用vfork被调用后当前进程会处于TASK_UNINTERRUPTABLE状态，直到子进程调用exec或exit为止。
   4) T暂停或跟踪状态(TASK_STOPPED or TASK_TRACED)
      - TASK_RUNNING/TASK_INTERRUPTABLE -> TASK_STOPPED，向进程发送SIGSTOP信号量（SIGSTOP和SIGKILL一样都是强制性的，不允许进程trap该信号量自定义处理函数来忽略默认行为）。
      - TASK_STOPPED -> TASK_RUNNING，向进程发送SIGCONT信号量。
      - TASK_TRACED意味着进程被暂停下来，等待跟踪它的进程对其进行操作。处于TASK_TRACED状态的进程不响应SIGCONT信号量，只有跟踪进程调用系统调用ptrace执行PTRACE_CONT、PTRACE_DETACH等操作，或跟踪进程退出时，它才会恢复为TASK_RUNNING状态。
   5) Z退出-僵尸状态(TASK_DEAD-EXIT_ZOMBIE)
      - 进程退出时，进程处于TASK_DEAD状态，此时进程所占用的资源将被释放，最后剩下仅包含退出码和一些统计信息(PID,退出状态,运行的CPU时间等)的task_struct结构体的壳供父进程获取子进程的退出信息。由于保留task_struct结构体的壳，因此此时的状态为EXIT_ZOMBIE。
      - 子进程退出过程中会向父进程发送SIGCHLD信号量，通知父进程来收尸进入EXIT_DEAD状态完成退出。
      - 父进程可以通过 ~wait(wait4,waitid)~ 系列的系统调用来等待子进程退出和收尸，并获取子进程退出信息。
      - 若父进程在子进程退出前已经退出，那么父进程在正常退出前会将其子进程转移给同一进程组的下一个进程或init进程。
        - 若子进程被归属到init进程后，init进程会定时调用 ~wait~ 系统调用，来清理其子进程的task_struct结构体的壳。
        - 若子进程被归属到同一进程组的下一个进程，那么要等待该进程调用 ~wait~ 系统调用来清理。
      - 僵尸进程的危害
        - 进程不再执行任务，但仍然占用PCB进行维护，浪费内存资源。
      - 清理僵尸进程
        - SIGKILL无效果，由于僵尸进程不能再接收任何信号量，因此向僵尸进程发送SIGKILL并不能结束进程。
        - 通过向僵尸进程的父进程发送SIGTERM即可 ~for i in `ps -ef | grep -v grep | grep defunct | awk "{print $3}"`; do kill -9 $i; done~
   6) X退出-进程即将被销毁(TASK_DEAD-EXIT_DEAD)
      - 当进程为多线程程序中被detach过的进程，或父进程通过clone创建的将SIGCHLD的处理函数设置为SIG_IGN的子进程，那么该进程退出时将不保留task_struct来等待父进程收尸，即不转移为EXIT_ZOMBIE状态，直接转移为EXIT_DEAD状态释放进程资源。
      - 这个状态很短暂，无法通过ps命令查看。
4) 进程状态转移
   1) 从非TASK_RUNNING转移为TASK_RUNNING
      - 由别的进程或中断处理程序执行唤醒操作。
   2) 从TASK_RUNNING转移为非TASK_RUNNING
      - 响应信号量进入TASK_STOPPED或TASK_DEAD状态
      - 主动执行系统调用进入TASK_INTERRUPTABLE状态(通过sleep函数)或TASK_DEAD状态(如exit函数)
      - 因资源不满足进入进入TASK_INTERRUPTABLE状态或TASK_UNINTERRUPTABLE状态(如select函数)
   3) 示例：向一个处于task_interruptable状态的进程发送信号量SIGKILL来结束进程
      1) 将进程状态从TASK_INTERRUPTABLE转移为TASK_RUNNING；
      2) 然后当该进程被CPU执行时响应SIGKILL，状态从TASK_RUNNING转移为TASK_DEAD。
5) PCB
   - 对于多道程序操作系统而言，内存中将存在多个进程，为管理这些进程操作系统内核为每个进程建立一个task_struct结构体保存该进程相关的信息，这个结构体就是PCB(Process Control Block)，多个PCB组成进程表。
   - Linux下采用链式方式组织PCB。
   - PCB的组成
     1) 标识符，每个进程都有进程标识符、用户标识符和组标识符。
        1) PID是32位无符号有序正数；
        2) 新创建的进程PID是最近创建的进程PID+1；
        3) Linux下允许PID上限为32767，若超过这个数则会重新使用已闲置的PID。
     2) 进程状态，7个进程状态。
     3) 进程优先级，整数，优先级高的进程则会更早获取CPU时间。
     4) 程序计数器，用于记录将被执行的下一条指令的地址。
     5) 内存指针
        1) 程序代码和进程相关的指针；
        2) 和其他进程共享内存的指针。
     6) 上下文数据
        - 进程得到CPU时间时寄存器中的数据。当进程失去CPU时间时会将寄存器中的数据暂存到相应的PCB中，待重新获取CPU时间时再次恢复到寄存器和堆栈中。
     7) I/O状态信息
        - 显式的I/O请求
        - 分配给进程的I/O设备
        - 被进程使用的文件列表等
     8) 审计信息
        - CPU时间总和
        - 使用的时钟总和
        - 时间限制
        - 审计号等
6) 进程创建的方式
   1) fork
   2) system
   3) exec
7) 进程的运行环境
   - 进程的运行环境就是进程启动时的环境变量，而这些环境变量是由 *系统环境变量副本* 和 *父进程的环境变量副本* 组成。
   - 特点是
     1) 只允许本进程修改自身的运行环境，父进程和子进程都无权修改；
     2) 增删改运行环境不影响父进程和已创建的子进程。
   - 系统环境变量 ~PYTHONPATH~ 将决定模块查找路径
8) 进程结构
   1) 代码段/文本区域(text region)，存放处理器执行的代码；
   2) 数据段(data region)，存放变量和进程执行期间使用的动态分配的内存；
   3) 堆栈段(stack region)，存放这活动过程调用的指令和本地变量。
9) 进程间通信(IPC, Inter-Process Communication)
   - 由于操作系统隔离各个进程可以访问的的内存地址空间(即进程拥有各自的用户地址空间)，因此内核提供一套称为IPC的机制用于进程间交换数据。
   - 底层原理：内核开辟一段地址空间作为缓冲区，进程A将数据从A的用户地址空间复制到内核缓冲区，然后再将数据从内核缓冲区复制到进程B的用户地址空间。
   - 方式
     1) 管道（匿名管道、命名管道）
     2) 消息队列
     3) 共享内存
     4) 信号量
     5) Socket

*** 进程调度算法
    1) 先来先服务调度算法(FCFS)
       有利于长作业，不利于短作业。
    2) 短作业优先调度算法(SJ/PF)
       有利于短作业，不利于长作业。
    3) 时间片轮转法(Round Robin, RR)
    4) 多级反馈队列
       优先级队列+时间片轮转，新任务最初放到优先级最高的队列，当执行过后当因时间片用完被切换下来后会放入优先级低一级的队列，那么短作业则会优先执行完，长作业则自动被识别并降级执行。另外还可以设置任务被执行的优先级，从而增加可控性。(Linux系统提供)
*** Linux进程创建方式——system
    system函数会通过启用 ~/bin/sh~ 进程执行入参指定的shell命令，shell命令执行期间信号量SIGCHID会被阻塞（即告知内核不要在发送SIGCHID信号量），而SIGINT和SIGQUIT会被忽略。
system函数内部逻辑为：
1) 通过系统调用fork创建一个子进程，失败则返回-1，正常则执行第2步；
2) 然后在子进程中调用系统调用excl执行 ~/bin/sh -c <command>~ 执行入参指定的命令，若excl执行失败则返回127，成功则返回入参命令的返回值；
3) 而父进程进入TASK_INTERRUPTABLE状态，等待子进程结束后恢复为TASK_RUNNING，并返回子进程的返回状态；
4) 若父进程在休眠状态被信号量中断时，则返回-1。

而Python中的 ~os.system~ 则是对C标准库中的 ~system~ 函数的封装。
#+BEGIN_EXAMPLE python
import os

# 返回值为0则表示命令执行成功，其他值表示执行失败。
# 当子进程是被信号中断时，返回值为负数。
# 若有异常则抛出OSError。
ret_code = os.system('ls')
#+END_EXAMPLE

*** Linux进程创建方式——exec函数族
    exec*族函数为POSIX系统的直接映射，在Unix系统中创建进程分为两步：创建进程实例 和 加载进程映像。
而exec函数族则用于在当前进程实例上加载新的进程映像替代原有的进程映像，即将代码段换成新的程序代码，废弃原有的数据段和堆栈段，保留进程其他信息（PID/PPID/PGID/SID），并且原来已打开的描述符均可继续使用，原来进程的环境变量也继续使用。
  os.exec*函数族为对Linux下exec函数族的封装。因此Python程序通过os.exec*函数族创建新进程时，Pyton程序将马上退出，因此os.exec*函数族没有返回值。
#+BEGIN_EXAMPLE python
import os

"""
v，表示第二参为tuple或list，用于输入要执行的命令和参数。
l，表示采用*args输入要执行的命令和参数。
p，表示第一参数为命令名，将从环境变量PATH中搜索；否则第一参数为命令的绝对路径。
e，表示最后一个参数为环境变量，类型为mapping或dict。
"""
os.execv('/bin/echo', ('echo', 'helloworld'))
# $ bash -c "/bin/echo $TEST"
os.execve('/bin/bash', ('bash', '-c', '/bin/echo $TEST'), {"TEST": 'test1'})
os.execvp('echo', ('echo', 'helloworld'))
os.execvpe('bash', ('bash', '-c', '/bin/echo $TEST'), {"TEST": 'test1'})

os.execl('/bin/echo', 'echo', 'helloworld')
# $ bash -c "/bin/echo $TEST"
os.execle('/bin/bash', 'bash', '-c', '/bin/echo $TEST', {"TEST": 'test1'})
os.execlp('echo', 'echo', 'helloworld')
os.execlpe('bash', 'bash', '-c', '/bin/echo $TEST', {"TEST": 'test1'})
#+END_EXAMPLE
os.exec*函数族底层都是调用os.execve。

*** 进程创建方式——fork(Unix 独有)
Unix/Linux系统下提供一个 ~fork()~ 的系统调用，该函数会复制当前进程映像创建子进程，然后分别在父进程和子进程中各返回一次（一共返回两次），其中在父进程的返回值为子进程的PID，而在子进程的返回值恒为0。

默认情况下子进程会继承父进程的所有文件描述符(file descriptor)。
*注意：当进程下存在多个线程时，那么这种方式是无法安全地创建子进程的。*

#+BEGIN_QUOTE
The child process begins, is effectively identical to the parent process. All resources of the parent are inherited by the child process. Note that safely forking a multithreaded process is problematic.

Available on Unix only.
#+END_QUOTE

os模块封装了很多系统调用，如 ~os.fork()~ 就封装了UNIX系列的系统调用 ~fork~ 因此返回值也和系统调用fork一致，只是报错时会抛出OSError异常。而Windows下没有fork调用，因此下列代码无法在Windows下运行。
#+BEGIN_EXAMPLE python
from __future__ import print_function
import os

print('Process (%s) start...' % os.getpid())
pid = os.fork()

if pid == 0:
    print('I\'m child process (%s) and my parent process is %s.' % (os.getpid(), os.getppid()))
else:
    print('I (%s) just created a child process (%s).' % (os.getpid(), pid))

# 输出：
# Process (876) start...
# I (876) just created a child process (877).
# I am child process (877) and my parent is 876.
#+END_EXAMPLE

*** 进程创建方式——spawn(Windows和Unix均支持)
    父进程会创建一个全新的子进程，子进程不会继承父进程中已打开的文件描述符和句柄，这种方式创建子进程相对fork和forkserver两种方式而言慢很多。
    Windows采用的就是这种方式，因此说Windows下进程的创建成本比Unix/Linux高。

*** 进程创建方式——forkserver(仅支持通过Unix pipes传递文件描述符的Unix支持)
    启动一个独立的Fork Server进程，后续当需要一个新的进程时，父进程则连接Server进程请求其创建新进程，而Fork Server进程本身为单线程进程因此通过调用fork()能够安全地创子进程，并且不用继承任何无用的文件描述符。
    #+BEGIN_QUOTE
    A fork server starts. From then on, whenever a new process is needed, the parent process connects to the server and requests that it fork a new process. The fork server process is single threaded so it is safe for it to use fork(). No unnessary resources are inherited.
    #+END_QUOTE

*** TODO IPC——管道
管道是所有*nix系统均提供的一种古老的IPC方式。
管道是一种半双工的通信机制，就是一端读，另一个端写。若要实现全双工通信，则要通过两个管道实现。
管道遵循FIFO操作规则，并且数据只能被读取一次，读取后管道中这段数据会被清理掉。
管道的原理是在内存开辟一段区域供通信双方的进程读取和写入。原理类似于两个进程通过操作同一个文件来实现通信，但比操作文件更高效。
分为 *匿名管道(pipe，限于父子进程间通信)* 和 *命名管道(fifo，任意进程间通信)* 。

**** 匿名管道
在shell中输入 ~ls -al | grep test~ 创建一个匿名管道让 ~ls -al~ 和 ~grep test~ 两个进程父子进程进行通信。

特点：
1) 匿名管道仅存活在内存中，父进程退出时释放；
3) 管道内部采用阻塞同步机制，保证数据访问一致性；

***** C程序
#+BEGIN_EXAMPLE c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <string.h>

#define INPUT 0  // 管道的只读端
#define OUTPUT 1 // 管道的只写端

void execute_subprocess(int* fds) {
    printf("in the spawned/child process...\n");
    // 由于子进程不从父进程读取信息，因此子进程关闭只读管道
    close(fds[INPUT]);
    // 子进程向只写管道写入数据，向父进程传递
    char* message = "test data";
    write(fds[OUTPUT], message, strlen(message));
    _exit(0);
}

void execute_main_process(int* fds) {
    printf("in the spawning/parent process...\n");
    // 由于父进程不向子进程传递信息，因此父进程关闭只写管道
    close(fds[OUTPUT]);
    // 父进程从只读管道读取子进程传递过来的信息
    char buf[256];
    // read将阻塞当前进程，直到只读管道中有数据可读取位置。
    int count = read(fds[INPUT], buf, sizeof(buf));
    printf("%d bytes of data received from subprocess: %s\n", count, buf);
    exit(0);
}

void main(){
    // 文件描述符，用于保存只读管道和只写管道，从而构成双工管道
    int fds[2];
    pipe(fds);

    pid_t pid;
    pid = fork();

    if (pid == -1){
        printf("Error in fork\n");
        exit(1);
    }
    else if (pid == 0){
        execute_subprocess(fds);
    }
    else {
        execute_main_process(fds);
    }
}

*** TODO IPC——信号量
    信号是操作系统提供的一种软件中断，采用异步方式向应用程序传递相关消息。最早由Unix操作系统引入，后来在各操作系统上都有类似的实现，但各平台的信号机制实现不尽相同。
    而Python中的signal模块并不是使用Python来实现，而是依赖具体的平台实现，即仅平台包含的信号可用而其他平台的信号则不可用。
    另外Python中对每个信号都有一个默认的信号处理程序，应用程序收到信号后若没有自定义信号处理程序，则会调用默认的信号处理程序。

信号特点
1) 操作系统的一种软件中断
2) 异步方式传递
过程
1) 终端用户按下中断键时,通过信号机制生成一个信号；
2) 应用程序接收信号后,使用自定义或默认的信号处理程序处理信息.

python的signal库直接使用操作系统定义的信号,因此不尽相同.

预设信号处理函数:
1) signal.SIG_DFL, 信号的默认处理函数
2) signal.SIG_IGN, 直接忽略信号

SIGINT, 键盘中断信号, ~C+c~ 发送该信号.

**** 注意事项
1) 信号的支持和实现在不同的操作系统上是有差异的
2) 除了SIGCHLD信号外, 信号处理函数一旦设置后就不会改变,除非显示重新设置
3) 在临界区无法临时屏蔽信号
4) 信号是一种异步通信方式,因此在长时间计算时延时较严重
5) 当程序在执行I/O操作时收到信号中断时,在执行完信号处理函数后或途中就会报异常
6) Python中的信号是由C编写的,C的信号处理函数必定有返回值,因此不用关注同步错误,如SIGFPE和SIGEGV
7) Python已为部分信号提供信号处理函数,如SIGINT就是抛出KeyboardInterupter异常
8) 当使用多线程时,必须在主线程接收信号,并且信号不能作为线程间的通信方式

     - ~full()~ 判断队列是否满
     - ~empty()~ 判断队列是否空

        conn1, conn2 = Pipe()
        # conn1 作为输入端，conn2 作为输出端

        import time
        import random
        from multiprocessing import Pipe, Process

        def func(conn1, conn2):

                    val = conn2.recv()
                    print('received: {}'.format(val))
                except EOFError:
                    # 当管道中所有输入端都close后，再在输出端调用recv则会抛出EOFError异常。
                    conn2.close()
                    print('over')

        conn1, conn2 = Pipe()
        conn2.close() # 关闭位于主进程的输出端
        Process(target=func, args=(conn1, conn2)).start()
        # 目前位置，创建了1条管道，主进程有1个输入端和1个输出端，子进程有1个输入端和1个输出端。
        for i in range(20):

** 多线程

   60年代在OS中能拥有资源和独立运行的基本单元是进程，但由于进程的创建、销毁和切换存在较大的时空开销，因此引入轻量级进程。而对称多处理机(SMP)可满足并行运行，而多进程并行开销也比较大。
   80年代出现能够独立运行的基本单位——线程。进程作为CPU调度的最小单位，每个进程中至少有一个线程，一个进程的线程可以在多个处理器中并行执行(CPython不行)。
   每个线程有独立的寄存器、堆栈和TCB，和其它线程共享进程的text region, data region和文件描述等。
   寄存器和堆栈的作用：
   - 用于指示被执行的指令序列的程序计数器、保留局部变量、状态参数和返回地址等。

   TCB(Threading Control Block)用于描述线程的动态特性，包含如下信息：
   1) 线程状态
   2) 当线程不运行时，被保存的现场资源
   3) 一组执行堆栈
   4) 存放每个线程的局部变量主存区
   5) 访问同一个进程的主存和其他资源

   多个TCB组成 *线程表* 。

   线程分为 *用户级线程(User-Level Thread)* 和 *内核级线程(Kernel-Level Thread,或称为轻量级进程)*
   - *用户级线程(User-Level Thread)*
     - 线程表位于用户空间；
     - 由用户程序实现线程的创建、销毁和调度，成本比内核级线程低；
     - 无法利用多核并行运行；
     - 调用系统调用时，将导致其所属进程被中断；
     - 若系统仅有用户级线程，那么CPU调度以进程为单位，进程内线程的调度则由用户程序控制；
     - 用户级线程的程序实体是运行在用户态下的程序。
   - *内核级线程(Kernel-Level Thread,或称为内核支持线程、轻量级进程)*
     - 线程表位于内核空间；
     - 由系统内核实现线程的创建、销毁和调度；
     - 可以利用多核并行运行；
     - 调用系统调用时，只导该线程被中断；
     - CPU调度以线程为单位；
     - 内核级线程的程序实体可以运行在任何状态下的程序。
   - 混合实现
     用户级和内核级的多路复用，每个内核级线程对应多个用户级线程。

   Windows是用内核级线程；

    CPython的线程就是C语言的pthread，并通过操作系统调度算法进行调度。
    因此thread模块是轻量级，而threading模块是对thread模块的封装。
    Queue模块是同步的、线程安全的队列类，一般和threading配合使用实现FIFO、LIFO和优先队列等。

*** Linux的NPTL
    内核2.6以前的调度单元均为进程，内核并没有真正支持线程。而LinuxThread则是通过系统调用 ~clone()~ 创建一份调用程序的拷贝，并且完全共享调用进程的地址空间（而 ~fork()~ 则不能共享调用进程的地址空间），来实现线程的内核级支持（线程实现完全为用户态，内核根本不知道线程的存在）。但很多地方没有遵循POSIX标准，如信号量处理、调度、进程间通信原语等。
    为改进LinuxThread的内核支持，IBM发起NGTP(Next Generation POSIX Threads)，而RedHat发起NPTL，后来NPTL取胜，从RHEL3起内核2.6都支持NPTL，并且成为GNU C的一部分。
    NPTL的设计
    线程在内核里仍然当作一个进程，有NPTL库调用 ~clone()~ 创建进程，并提供唤醒线程的同步原语futex等。
    NPTL是一个1个内核级线程对应1个用户级线程的线程库，即调用 ~pthread_create()~ 创建线程，底层调用 ~clone()~ 创建新进程。
    也提供m个用户级线程对应n个内核级线程。

*** GIL(Global Interpreter Lock)
    多核服务器上，CPython解释器内部通过GIL实现线程安全，让任何时候Python进程内仅有一个线程在执行。（如Java进程内的多个线程可以在多个处理器上并行运行，而CPython则不行），因此CPython不存在缓存一致性的问题。
    因此Python线程在多核服务器上，运行IO密集型计算性能好，但运行计算密集型则性能下降。

 多线程环境下，虚拟机执行流程：
 1) 设置GIL
 2) 切换到某个线程并执行
 3) 运行指定的字节码集合
 4) 线程主动让出控制(调用time.sleep())
 5) 设置线程为休眠状态
 6) 解锁GIL
 7) 回到步骤1

 注意：
 1) 当调用外部代码（如C/C++扩展函数）时GIL会被锁定，直到调用结束。
    - 由于期间没有Python字节码被运行，因此不会做线程切换
 2) 外部代码也可以主动解锁GIL。

 GIL源码在Python/ceval.c文件中。

            但不管wait参数为何值，整个程序都会等到所有任务执行完毕。
            #+END_QUOTE
         5) ~result(timeout=None)~ 获取执行结果
         6) ~add_done_callback(fn)~ 回调函数
         7) ~done()~ 判断某个任务是否完成
         8) ~cancel()~ 取消某个任务
       - 示例1——submit

** 协程(Coroutine, 又称微线程、纤程)
   协程是一种用户态的轻量级线程，由用户程序控制协程的调度，而不像线程是由内核来控制调度。
   优点：
   1) 切换开销小，由用户程序控制保存执行环境后再执行调度，而不是由内核控制保存栈、寄存器等执行环境；
   2) 单线程内实现并发效果，在IO密集型操作时提供CPU利用率；
   3) 多个协程任务内修改数据不能加锁。
   缺点：
   1) 无法利用多核并行处理能力；
   2) 一个协程任务中被阻塞，即整个线程都被阻塞；
   3) 仅适用于IO密集型操作，否则反而降低性能。

   一般采用gevent库来使用协程，内部主要使用Greenlet，它是以C扩展模块形式接入Python的轻量级协程。
   gevent方法：
   - ~gevent.spawn(task: function, *args, **kwargs) -> Greenlet~
     创建协程任务，其中 ~*args~ 和 ~**kwargs~ 为传递给task的参数
   - ~Greenlet实例.join()~ 阻塞当前线程直到协程结束
   - ~gevent.joinall((Greenlet实例1, Greenlet实例2))~ 阻塞当前线程直到指定的协程结束
   - ~Greenlet实例.value~ 获取协程的返回值

*** 基本用法
    #+BEGIN_EXAMPLE python
    # 必须位于文件其它引入之前，用于对time.sleep等IO操作进行打补丁。
    # 当在协程任务中使用time.sleep等IO操作时会自动通过协程完成任务调度，而不是一直阻塞。
    from gevent import monkey
    monkey.patch_all()

    import gevent
    import threading
    from time import sleep

    def task1(a):
        print(threading.current_thread().getName()) # 回显DummyThread-1，表示为假线程即协程
        sleep(10) # 发生IO即跳转到task2
        print(a)
        return a+1

    def task2(b):
        print(threading.current_thread().getName()) # 回显DummyThread-2，表示为假线程即协程
        sleep(4) # 发生IO即跳转到task1
        print(b)
        return b+1

    gs = []
    gs.append(gevent.spawn(task1, 1))
    gs.append(gevent.spawn(task2, 2))

    gevent.joinall(gs)

    print(gs[0].value) # 2
    print(gs[1].value) # 3
    #+END_EXAMPLE

** IO模型
   1) 阻塞IO(Blocking IO)
      |                | application      |                  | kernel            |                               |
      |----------------+------------------+------------------+-------------------+-------------------------------|
      | process        | recvfrom         | - system call -> | no datagram ready |                               |
      | blocks         |                  |                  | $               $ | wait for data                 |
      | in call        |                  |                  | datagram ready    |                               |
      |----------------+------------------+------------------+-------------------+-------------------------------|
      | to system call |                  |                  | copy datagram     |                               |
      | `recvfrom`     |                  |                  | $               $ | copy data from kernel to user |
      |                | process datagram | <- return OK -   | copy complete     |                               |
      - 说明
        当用户程序调用系统调用recvfrom，那么kernel就会去准备数据(用户程序被阻塞)，当数据准备好后就将数据从内核拷贝到用户内存(用户程序被阻塞)，然后内核返回结果后用户程序才解除阻塞。
      - Blocking IO操作
        - Socket接口的 ~accept,connect,send,recv~ 等接口
      - Python示例
        #+BEGIN_EXAMPLE python
        from socket import *

        # Server
        srv = socket(AF_INET, SOCK_STREAM)
        srv.bind(('127.0.0.1', 8080))
        srv.listen()
        while True:
            conn, addr = srv.accept() # 阻塞
            data = conn.recv(1024)   # 阻塞
            print(data.decode('utf-8'))
            conn.send('hello'.encode('utf-8')) # 阻塞

        # Client
        client = socket(AF_INET, SOCK_STREAM)
        client.connet(('127.0.0.1', 8080))
        while True:
            client.send('hi'.encode('utf-8')) # 阻塞
            data = client.recv(1024) # 阻塞
            print(data.decode('utf-8')
        #+END_EXAMPLE
   2) 非阻塞IO(Non-blocking IO)
      |                    | application      |                  | kernel            |                               |
      |--------------------+------------------+------------------+-------------------+-------------------------------|
      | process repeatedly | recvfrom         | - system call -> | no datagram ready |                               |
      | call system call   |                  | <- EWOULDBLOCK - |                   |                               |
      | `recvfrom`,        | recvfrom         | - system call -> | no datagram ready |                               |
      | waiting for        |                  | <- EWOULDBLOCK - |                   | wait for data                 |
      | an                 | recvfrom         | - system call -> | no datagram ready |                               |
      | OK return          |                  | <- EWOULDBLOCK - |                   |                               |
      | by polling         | recvfrom         | - system call -> | datagram ready    |                               |
      |--------------------+------------------+------------------+-------------------+-------------------------------|
      |                    |                  |                  | copy datagram     |                               |
      |                    |                  |                  | $               $ | copy data from kernel to user |
      |                    | process datagram | <- return OK -   | copy complete     |                               |
      - 说明
        当用户程序调用系统调用recvfrom，那么kernel就会去准备数据，若为准备好则不阻塞而是返回一个error。然后用户程序轮询执行上述操作，直到准备好数据，那么内核将数据从内核拷贝到用户内存(用户程序被阻塞)，然后返回内核结果。
      - 缺点
        1) 循环调用系统调用 ~recvfrom~ 将推高CPU占用率，因此轮询期间一般通过 ~time.sleep~ 延长轮询周期；
        2) 任务完成的响应时延较大，通过轮询方式无法及时读取数据，降低整体数据的吞吐量。
      - Python示例
        #+BEGIN_EXAMPLE python
        from socket import *

        # Server
        srv = socket(AF_INET, SOCK_STREAM)
        srv.bind(('127.0.0.1', 8080))
        srv.listen()
        srv.setblocking(Flase) # 设置socket的接口全为非阻塞
        conns = []
        del_conns = []
        while True:
            try:
                conn, addr = srv.accept() # 阻塞
                conns.append(conn)
            except BlockingIOError:
                pass

            for conn in conns:
                try:
                    data = conn.recv(1024)
                    if not data:
                        del_conns.append(conn)
                    else:
                        print(data.decode('utf-8'))
                        conn.send('hello'.encode('utf-8')) # 阻塞
                except BlockingIOError:
                    pass
                except ConnectionResetError:
                    del_conns.append(conn)

            for conn in del_conns:
                conns.remove(conn)
                conn.close()

            del_conns = []

        # Client
        client = socket(AF_INET, SOCK_STREAM)
        client.connet(('127.0.0.1', 8080))
        while True:
            client.send('hi'.encode('utf-8')) # 阻塞
            data = client.recv(1024) # 阻塞
            print(data.decode('utf-8')
        #+END_EXAMPLE
   3) 多路复用IO(IO Multiplexing，也称为事件驱动IO/Event Driven IO)
      |                                                | application       |                      | kernel            |                               |
      |------------------------------------------------+-------------------+----------------------+-------------------+-------------------------------|
      | process blocks in call to system call          | select/poll/epoll | - system call ->     | no datagram ready |                               |
      | select/poll/epoll, waiting for one of possibly |                   |                      | $               $ | wait for data                 |
      | many sockets to become readable.               |                   | <- return readable - | datagram ready    |                               |
      |------------------------------------------------+-------------------+----------------------+-------------------+-------------------------------|
      | process blocks while data                      | recvfrom          | - system call ->     | copy datagram     |                               |
      | copied into                                    |                   |                      | $           $     | copy data from kernel to user |
      | application buffer                             | process datagram  | <- return OK -       | copy complete     |                               |
      - 说明：
        和阻塞IO一样，但可以通过系统调用select、poll或epoll可以在一个线程中同时监听多个IO操作。
      - ~select,poll,epoll,kqueue~ 的区别 (https://www.cnblogs.com/George1994/p/6725972.html, https://www.cnblogs.com/Eva-J/articles/8324837.html, https://www.jianshu.com/p/6a6845464770)
        1) ~select~ 通过轮询方式监听多个IO操作，因此当监听的IO操作多时，效率较低。Windows唯一支持的多路复用IO方式，Unix支持；

             - timeout 内核等待指定描述符就绪的时间长度
             - 返回值： -1 失败，0 超时，>0 成功

        3) ~epoll~ 采用事件机制而不是轮询机制，每个IO操作的fd都绑定一个回调方法，当fd接收到数据时即调用回调方法，然后通过信号量通知用户进程数据已到。Unix支持；
           - C函数签名 ~int epoll_wait(int epfd, struct epoll_event *events, int maxevents, int timeout)~
           - 两种模式
             1) ET模式——立即处理
                当检测到描述符事件通知应用程序，应用程序立即处理。能减少epoll被重复触发的次数，效率比LT高。
                - 注意：Socket必须采用非阻塞模式，避免文件描述符在阻塞读或阻塞写时将其他文件描述符的任务饿死。
             2) LT模式——延迟处理
                当检测到描述符事件通知应用程序，应用程序不会立即处理，而是会在下次通知应用程序该事件时才处理。
           - 优点：
             1) 没有最大并发连接的限制
             2) 只有活跃可用的fd才会调用callback
             3) 通过mmap()文件映射内存的方式加速用户和内核间信息传递，减少复制开销(内核和用户空间共享一块内存)
           - 内部原理
             - 前提：高并发连接中，每次处理的活跃连接数量占比很小
             - 原理：
               1. 通过链表存储活跃的连接对象，那么在遍历时仅遍历这些对象即可。而select则遍历所有连接对象，但仅有小部分是活跃连接，因此做了大量无用功。
                  - 链表的每个节点的结构为eventpoll
                    字段：lock, mtx, wq, poll_wait, rdllist, rbr, ovflist, user
               2. 读写事件则通过红黑树存储，那么添加、删除和修改读写事件的时间复杂度都是O(1)
                  - 红黑树的每个节点的结构为epitem
                    字段：rbn, rdllink, next, ffd, nwait, pwqlist, ep, fllink, event
        4) ~kqueue~ 和 ~epoll~ 基本一样。BSD提供支持；
        5) ~/dev/poll~ Solaris提供支持。
      - 优点
        1) 单线程中监听多个IO操作，CPU等资源占用率较低。
      - 缺点
        1) 若网络IO不是很高时，性能不一定比多线程+non-blocking io高。多路复用IO主要为提高能处理的网络IO数量。
      - 监听过程
        1) 用户进程通过select/poll/epoll等将IO对象的文件描述符(fd)拷贝到内核空间
        2) 内核空间的fd接收到数据后，会通过信号量告知用户用户进程数据已到
        3) 用户进程再调用系统调用recvform，要求内核将数据从内核内存空间复制到用户进程的内存空间，同时内核内存空间的数据被清空等待接收后续fd的数据。
      - 典型应用
        1) Java NIO
        2) Nginx(epoll,poll,select)
      - 设计的架构模式——Reactor模式

        import select # select模块包含select,poll和epoll三种方式

                            r.close()
                            read_lst.remove(r)
                        else:
                            print(data.decode('utf-8'))
                    except ConnectionResetError:
                        r.close()
                        read_lst.remove(r)

   IO涉及两个阶段：
   1) 等待数据到达，然后通过设备驱动读取数据到内核空间缓冲区中(Waiting for the data to be ready)
      - 用户空间(级别号为3的虚拟内存空间保护) <---> 内核空间
      - 用户空间组成
        1) User Applications
        2) GNU C Library (glibc)
   2) 将数据从内核缓存区拷贝到用户进程的缓冲区中(Copying the data from kernel to the process)
      - 内核空间(级别号为0的虚拟内存空间保护) <---> 设备空间
      - 内核空间组成
        1) System Call Interface
        2) Process Management
        3) Virtual File System(VFS)
        4) Memory Management
        5) Network Stack
        6) Arch/Device Drivers(和Hardware Platform通信)
           内核会为每个设备驱动预留缓存区，发起IO操作后，内核会检查缓冲区是否有数据，有则复制到用户缓冲区，无则调用设备驱动读取数据。



   上述5种IO模型均是在这两个阶段有所不同来作区分的。

   POSIX(Portable Operating System Interface for Computing System)对synchronous IO和asynchronous IO的定义：
   1) A synchronous I/O operation cause the requesting process to be blocked until the I/O operation completes;
   2) An asynchronous I/O operation does not cause the requesting process to be blocked.

*** Python的多路复用IO模块selectors
    自动选择适合当前操作系统平台的多路复用IO。
    #+BEGIN_EXAMPLE python
    from socket import *
    import selectors

    sel = selectors.DefaultSelector()

    srv = socket(AF_INET, SOCK_STREAM)
    srv.setsockopt(SOL_SOCKET, SO_REUSEADDR, 1)
    srv.bind(('127.0.0.1', 8080))
    srv.listen()
    srv.setblocking(False)

    def accept(srv):
        conn, addr = srv.accept()
        events = selectors.EVENT_READ|selectors.EVENT_WRITE # 监听可读和可写事件
        sel.register(conn, events, service_connection) # 将fd添加到监听列表

    def service_connection(conn, mask):
        if mask & selectors.EVENT_READ:
            # ready to read.
            try:
                data = conn.recv(1024)
                if not data:
                    sel.unregister(conn)
                    conn.close()
                else:
                    print(data.decode('utf-8'))
            except ConnectionResetError:
                sel.unregister(conn)
                conn.close()

        if mask & selectors.EVENT_WRITE:
           # ready to write.
           msg = 'hello'
           sent = 0
           while sent < len(msg):
               sent += conn.send(msg)

    sel.register(srv, selectors.EVENT_READ, accept) # 将fd添加到监听列表

    while True:
        """
        阻塞当前线程并监听所有fd。

        Args:
            **kwargs:
                timeout(int): 阻塞超时，默认为None表示一直阻塞直到有响应

        Returns:
            list(tuple(key: SelectorKey, events)):
                key is a SelectorKey namedtuple that contains a fileobj attribute which representing socket object, and a data attribute which as parameter passed by function register as the last one.
                events is an event mask of the operations that are ready (i.e. selectors.EVENT_READ, selectors.EVENT_WRITE)

        """
        events = sel.select(timeout=None)
        for obj, mask in events:
            callback = obj.data  # 获取注册时fd对应的回调函数
            callback(obj.fileobj # 获取fd自身
                     , mask)
    #+END_EXAMPLE

** TTL
   TTL(Time to Live)：指定数据包经过的路由器的最大值，若路由器接收到数据包的TTL为0则不会转发，而是直接丢掉。数据包没经过一个路由器那么TTL值为减1。
   经过的路由器越多，速度越慢。
   可以通过程序修改TTL值。

** 2MSL问题
   MSL，一个数据包在网络上存活的最长时间，一般为1~2分钟(LwIP为1分钟，Windows为2分钟)。
   2MSL即为2倍的MSL，作为TCP4次挥手中客户端从TIME_WAIT转换CLOSED的等待时间
   1) 若在这段时间中收到服务端重新发送过来的 ~FIN~ 请求，那么证明服务端没有收到客户端响应的 ~ACK~ 。那么会再次响应 ~ACK~ 给服务端，并等待2MSL。
   2) 若在这段时间中没有收到服务端重新发送过来的 ~FIN~ 请求，则证明服务端收到客户端响应的 ~ACK~ 。那么客户端就可以转换为CLOSED状态。

   *注意：在从TIME_WAIT到CLOSED转换期间，IP+端口将被锁定不能被重新绑定*
   1) 由于客户端不存在IP+端口绑定的需求，因此不受影响；
   2) 若为服务端先调用close系统调用，那么TIME_WAIT到CLOSED的状态转换发生在服务端，因此若在此期间绑定相同的IP+端口，则会报错。
      - Python中通过如下设置允许在此期间绑定相同的IP+端口
        #+BEGIN_EXAMPLE python
        from socket import *

        srv = socket(AF_INET, SOCK_STREAM)
        srv.bind((HOST, PORT))
        srv.listen()
        sk.setsockopt(SOL_SOCKET, SO_REUSEADDR, 1)
        #+END_EXAMPLE

** 两种基本数据传输方式(Data Transport)
   1) TCP(Transmission Control Protocol)
      - 特点
        1) Base connection
        2) Byte stream oriented
        3) Reliable, packets dropped in the network are detected and retransmitted by the sender.(no need to worry about packet loss)
        4) In-order data devilery, data is read by your application in the order it was written by the sender.
      - 注意：
        1) 不能发送空消息，若发送空消息需要服务端和客户端用户程序作特殊处理，防止两端阻塞。
      - 示例
        #+BEGIN_EXAMPLE python
        from socket import *

        # server
        HOST = '127.0.0.1'
        PORT = 1080
        BACKLOG = 5
        with socket(AF_INET, SOCK_STREAM) as srv:
            srv.bind((HOST, PORT))
            srv.listen(BACKLOG)

            while True:
                conn, addr = srv.accept()
                with conn:
                    data = conn.recv(1024)
                    if not data:
                        break
                    print(data.decode('utf-8'))
                    conn.sendall(b'OK')

        # client

** TCP的粘包问题
   原因：
   1) 由于TCP是面向字节流，所以没有明确的数据边界；
   2) 当发送端缓冲区的长度大于网卡的MTU时，TCP会将数据拆分为多个数据包发送出去。这样对于大数据量而言，也无法通过包作为消息边界；
   3) Nagle算法
      将多次间隔较小且数据量小的数据，合并为一个大的数据块再进行封包。这样接收端就无法通过包来作为消息边界。

   基于上述原因，必须由用户程序自行解决粘包问题。

** 字节顺序(byte order/byte endianness)问题
   有些CPU采用big-endiance，而有些CPU则采用little-endiance，因此涉及多字节数据时(如4byte的整型)，则需要考虑编码解码的字节顺序。
   1) big-endiance/MSB(Most Significant Bit) 高位优先，低字节数据存放在内存高地址处，高字节数据存放在内存低地址处。
   2) little-endiance/LSB(Lest Significant Bit) 低位优先，低字节数据存放在内存低地址处，高字节数据存放在内存高地址处。

   UTF-8采用8bit编码，因此不存在字节顺序问题。即使存在则在客户端和服务端两边约定字节顺序，并以约定的字节顺序编码和解码数据即可。
   通过 ~sys.byteorder~ 获取本机的字节顺序。
   *TCP/IP指定采用big-endiance的byte order作为network order，用于底层网络协议栈，如IP地址和端口号等。因此若要通过网络传输数据，则需要将数据转换为big-endiance*
   Python的socket模块提供以下方法，用于将整型从network order转换为本机的byte order.
   #+BEGIN_EXAMPLE python
   """
   Convert 32-bit positive integers from network to host byte order.
   """
   socket.ntohl(x)
   """
   Convert 16-bit positive integers from network to host byte order.
   """
   socket.ntohs(x)

   """
   Convert 32-bit positive integers from host to network byte order.
   """
   socket.htonl(x)
   """
   Convert 16-bit positive integers from host to network byte order.
   """
   socket.htons(x)
   #+END_EXAMPLE

** UDP广播
   传输层中TCP无法实现广播，仅能通过UDP实现广播。

*** 原理
    1) 目标IP地址中的主机地址为全为二进制1；
    2) 将数据包发送给网络设备(集线器(hub)、交换机(switch))；
    3) 网络设备收到广播的数据包，然后向当前网段中所有主机分别转发该数据包；
    4) 主机接收到数据包，在网络层拆包发现IP为广播地址，然后将数据包往传输层传递；
    5) 传输层拆包发现PORT于主机上某服务对应则将数据包继续往应用层传递，否则则丢弃数据包。

    注意：
    1) 源主机仅向网络发送一份数据包，不是向每台主机发送一份，而是由网络设备实现数据包复制和转发。因为网络设备就是设计用于转发数据包，所以数据包复制和转发的效率远远高于源主机；
    2) 广播只能确保同一网段上的所有主机在网络层必定会接收数据包。

*** Python示例
    #+BEGIN_EXAMPLE python
    from socket import *

    DEST = ('<broadcast>', 7788) # 通过占位符 <broadcast> 实现自动替换为程序运行时所在网段的广播地址
    client = socket(AF_INET, SOCK_DGRAM)
    client.setsockops(SOL_SOCKET, SO_BROADCAST, 1)
    client.sendto(b'hi', DEST)
    #+END_EXAMPLE

** 集线器(hub)、交换机(switch)和路由器(router,也称为网关(gateway))
   1) 集线器(hub)
      1) 连接同一网段中的主机
      2) 接收数据包后，将通过广播的形式转发数据包到其他主机中。
   2) 交换机(switch)
      1) 连接同一网段中的主机
      2) 接收数据包后
         1) 若没有转发记录，则通过广播的方式转发请求；
         2) 若有转发记录，则通过单播的方式转发请求。
   3) 路由器(router,也称为网关(gateway))
      1) 连接多个网段
      2) 路由器的转发规则配置
         1) 通过 RIP/RIP2 协议自动实现转发
         2) 通过配置静态路由规则
            - Network 配置目标主机IP地址 或 目标主机的网络地址
            - Mask    配置目标主机的子网掩码
            - Next Hop 配置下一跳路由器的IP地址
              1) 如路由器A分别有192.168.1.1和192.168.2.1两个端口，路由器B分别有192.168.2.2和192.168.3.1两个端口，路由器C分别有192.168.3.2和192.168.4.1两个端口
              2) 现在从192.168.1.2向192.168.4.2发送数据
              3) 那么在路由器A中需要设置
                 - Network 192.168.4.0
                 - Mask 255.255.255.0
                 - Next Hop 192.168.2.2
              4) 那么在路由器B中需要设置
                 - Network 192.168.4.0
                 - Mask 255.255.255.0
                 - Next Hop 192.168.3.2
      3) 第一次从192.168.1.2向192.168.4.2发送数据包的流程
         1) 由于主机192.168.1.2不知道主机192.168.4.2的MAC地址，因此会向192.168.1.0网段的所有主机发送ARP包；
         2) 网关192.168.1.1接收到目标IP为192.168.4.2的ARP包后，查找是否能匹配路由转发规则，能则将网关自己的MAC地址返回给192.168.1.2；
         3) 主机192.168.1.2接收到网关的响应后，发送如下数据包
            - 源IP：192.168.1.2
            - 目标IP：192.168.4.2
            - 源MAC：192.168.1.2的MAC地址
            - 目标MAC：192.168.1.1的MAC地址
         4) 网关192.168.1.1接收数据包后，拆包并重新打包再转发
            - 源IP：192.168.1.2
            - 目标IP：192.168.4.2
            - 源MAC：192.168.2.1的MAC地址
            - 目标MAC：192.168.2.2的MAC地址
         5) 网关192.168.2.2接收数据包后，拆包查看是否能匹配路由转发规则，能就重新打包转发
            - 源IP：192.168.1.2
            - 目标IP：192.168.4.2
            - 源MAC：192.168.3.1的MAC地址
            - 目标MAC：192.168.3.2的MAC地址
         5) 网关192.168.3.2接收数据包后，拆包查看是否能匹配路由转发规则，能就重新打包转发。(若网关不知道192.168.4.2的MAC地址，那么就会丢弃该数据包，并向192.168.4.0网段的主机发送ARP包来获取192.168.4.2的MAC地址，然后等待下次请求再转发。)
            - 源IP：192.168.1.2
            - 目标IP：192.168.4.2
            - 源MAC：192.168.4.1的MAC地址
            - 目标MAC：192.168.4.2的MAC地址

** MAC地址
   MAC地址为物理网卡的全球唯一的编号，由 3 bytes 的厂商编号 和 3 bytes 的该厂商生产的网卡编号组成。
   MAC地址，标记实际转发数据时的设备物理地址，在实际跨网段转发数据时数据包中的源MAC和目标MAC地址将会发生变化。
   IP地址，标记逻辑地址，在实际转发数据时数据包中的源IP和目标IP地址将不会发生变化。
   Network Mask，和IP地址一起确定网络号。

** 网络攻击
   1) TCP半连接攻击（也称为SYN Flood）
      是一种典型的DoS(Denial of Service)攻击。客户端仅发送SYN，当接收到服务端的SYN-ACK后不发送ACK来完成TCP三次握手，导致待建立连接的请求队列满后拒绝访问。
   2) DNS劫持
      需要直接劫持DNS服务器
   3) DNS欺骗
      获取域名IP映射记录的方式有两种
      1) 客户端向DNS服务器请求获取映射记录
      2) DNS服务器主动向客户端发送映射记录
         因此黑客可通过主动发送映射记录的方式，让客户端记录伪造的域名IP映射记录。
   4) ARP攻击
      1) 攻击者将自己的MAC地址发送给目标主机，并告知目标主机其他主机的MAC地址是攻击者的MAC地址；
      2) 后续目标主机与其他主机通信时则会将请求发送给攻击者的主机；
      3) 攻击者记录请求后，再将请求转发给原来的接收主机以防被发现。

        @param {tuple} from_addr
        @param {list<tuple>} to_addrs
        """

import os
import hashlib

def md5sum(dir):
    md5s = []
    size = 8014
    for curr_dir, _, filenames in os.walk(dir):
        for filename in filenames:
            filepath = os.path.join(curr_dir, filename)
            md5 = hashlib.md5()
            with open(filepath, 'rb') as f:
                chunck = f.read(size)
                while chunck:
                    md5.update(chunck)
                    chunck = f.read(size)

            md5s.append((filepath, md5.hexdigest()))

    return md5s

* 命名规范
1) 变量名要有描述性，不能太宽泛
   #+begin_example python
   # BAD
   day, host, card, temp
   # GOOD
   day_of_week, hosts_to_reboot, expired_cards
   #+end_example
2) 变量名要让人猜出类型
   - bool 约定俗成如下为bool类型
     1) 以is_开头 ~is_superuser~
     2) 以has_开头 ~has_error~
     3) 以allow_开头 ~allow_vip~
     4) 以use_开头 ~use_msgpack~
     5) debug
   - int/float
     1) 释义为数字的所有单词 ~age,port,radius~
     2) 以_id结尾 ~user_id,host_id~
     3) 以length/count开头或结尾 ~length_of_username,max_length,users_count~
   - str
     1) 以str_开头 ~str_age~
   - list/tuple/dict
     1) 通过 document string 标注类型
   - 自定义类型
     采用匈牙利命名法
     - 示例 ~pl_students~ 表示变量的类型为 ~List<Person>~
3) 保证名称最多由3个单词组成
4) 不要使用带否定含义的名称
   - 示例 用 ~is_special~ 代替 ~is_not_normal~
5) 保持命名一致性
   - 示例 在某方法中采用image作为图片变量名，那么在其他方法中也要采用image作为图片变量名
6) 若后续代码中不会引用该变量，那就不要定义
   #+begin_example python
   def demo_boring():
       a = 1
       b = 2
       c = {
         'a': a,
         'b': b,
       }
       return c

   # 应该修改为
   def demo():
       return {
         'a': a,
         'b': b,
       }
   #+end_example

class Query(object):
    prev_node = None
    payload = None

    def __init__(self, prev_node,):
        if prev_node not is None:
            self.prev_node = prev_node

    def __eval(self,):
        nodes = []
        curr = self
        while curr.prev_node is not None:
            nodes.append(curr)
            curr = curr.prev_node
        nodes.append(curr)
        ret = None
        for node in nodes.reverse():
            action, args = node
            ret = Query["__%s" % action](ret, *args)

        return ret

    def find(self, slctr,):
        self.payload = ("find", (slctr,))
        return Query(self)

        we = self.__eval()

** 历史

** 特点
1) 运行时替代原有方法、属性等
2) 在不修改第三方源码的情况下增强功能

** Python的实现原理
Python的命名空间开放且通过 ~dict~ 来实现，3种命名空间：
1) locals，定义在函数内部的变量、函数和类型。通过 ~locals()~ 获取
2) globals，定义在模块内部的变量、函数和类型。通过 ~globals()~ 获取
3) builtin

*** 模块引入(module import)和符号检索(name lookup)机制
1) 执行 ~from <module> import <member>~ 或 ~import <module>~ ；
2) 检查 ~dict~ 类型的 ~sys.modules~ 中是否已缓存该模块，若没有则引入并保存到 ~sys.modules~ 中，若存在则直接引用；
3) 将该模块从 ~sys.modules~ 加入到命名空间 ~globals~ 这个 ~dict~ 中；
4) 执行 ~<member>.<method>~ 或 ~<module>.<member>~ 则会从命名空间 ~globals~ 中检索。
