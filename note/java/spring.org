* Spring MVC 4

** JavaEE分层模型
   1) 表现层(View)
      底层通过称为DTO(Data Transfer Object, 数据传输对象)的JavaBean向表现层传输状态信息。
      - Servlet和JSP，JSP将被Web服务器编译为Servlet，然后Web服务器运行Servlet对外提供服务。Servlet3.x新增如下特性
        - 异步请求
        - 注解
        - 增强的Servlet API
        - 非阻塞IO
      - FreeMarker
      - Velocity
   2) 控制器层(Controller)
   3) 服务逻辑层(Service)
   4) 数据访问服务层(DAO,Data Access Object)
      在MyBatis中称为Mapper层
      - Hibernate，全自动ORM
      - MyBatis，半自动ORM
   5) 领域对象层(Domain Object)
      由一系列POJO(Plain Old Java Object, 普通的传统的Java对象)组成。

** Spring MVC 4
   通过继承HttpServlet的org.springframework.web.servlet.DispatcherServlet作为前端控制器来拦截用户请求并分派请求到具体执行业务控制的Controller，并处理请求。视图技术包含JSP、Velocity、Tiles、iText和POI等，功能分工上包含控制器、验证器、命令对象、模型对象、处理程序映射视图解析器等。
   Spring核心容器依赖common-logging的JAR包。

*** MVC模型
    1) Model1
       View和Controller层均通过JSP实现，将控制逻辑和表现逻辑混杂在一起，代码重用性、可扩展性和维护性极低，但由于架构简单开发速度快。
    2) Model2
       JSP作为View、Servlet作为Controller、JavaBean作为Model，提高代码重用、可扩展和维护性。

     在dispatcher-servlet.xml通过Bean组件配置handler和URL路径映射
     #+BEGIN_EXAMPLE xml
     <!-- /WEB-INF/dispatcher-servlet.xml -->
     <!-- 配置Handler和URL路径映射 -->

     <bean class="org.springframework.web.servlet.handler.BeanNameUrlHandlerMapping"/>
     <!-- 启用继承HandlerAdapter的SimpleControllerHandlerAdapter实例来调用Handler的handleRequest方法 -->
     <bean class="org.springframework.web.servlet.mvc.SimpleControllerHandlerAdapter"/>
     <!-- 配置View Resolver -->
     <bean class="org.springframework.web.servlet.view.InternalResourceViewResolver“>
       <property name="prefix">
         <value>/WEB-INF/views/</value>
       </property>
       <property name="suffix">
         <value>.jsp</value>
       </property>
     </bean>

     <!--
         /WEB-INF/views下的文件默认不能通过URL直接访问，必须通过Controller进行映射。

     <mvc:view-controller path="/success"/> <!-- 访问/success时会自动映射到/WEB-INF/views/success.jsp -->

     // src/org/myproject/controller/HelloController.java
     package org.myproject.controller;
     import javax.servlet.http.HttpServletRequest;
     import javax.servlet.http.HttpServletResponse;
     import org.apache.commons.logging.Log;
     import org.apache.commons.logging.LogFactory;
     import org.springframework.web.servlet.ModelAndView;

     public class HelloController implements Controller {
         private static final Log logger = LogFactory.getLog(HelloController.class);

         // 返回一个包含视图名称或视图名称和模型的ModelAndView对象
         @Override
         public ModelAndView handleRequest(HttpServletRequest request, HttpServletResponse response){
             logger.info("handleRequest invoked");
             ModelAndView mv = new ModelAndView();
             // 添加模型数据，可为任意的POJO对象

         }
     }

*** 注解
    1) ~@Controller~
       全限定类名org.springframework.stereotype.Controller，其标识的类实例将作为作为Controller。其标识为 ~@RequestMapping~ 的公开实例方法将作为具体的handler。
       在dispatcher-servlet.xml中通过配置 ~<context:component-scan base-package=""/>~ 启动包扫描功能，自动注册带 ~@Controller,@Service,@repository,@Component~ 等注解的类成为Spring Bean。
       #+BEGIN_EXAMPLE java
       import org.springframework.stereotype.Controller;
       import org.springframework.web.bind.annotation.RequestMapping;
       import org.springframework.ui.Model;

       @Controller
       public class HelloController {
           @RequestMapping("/hello")
           public String index(Model m){
               m.addAttribute("message", "test") // jsp视图中通过${message}、${request.message}或${requestScope.message}获取该值
               return "helloworld" // 返回逻辑视图名称，Spring会自动通过RequestToViewNameTranslator获取物理的ViewName(如/WEB-INF/views/helloworld.jsp)
           }
       }
       #+END_EXAMPLE

    7) Model、ModelMap和ModelAndView
       Spring MVC 在调用处理方法前会创建一个隐含的作为模型数据存储容器的模型实例，若Handler相关方法入参类型为Model或ModelMap时就会将隐含的模型实例引用传递给这些参数。
       #+BEGIN_EXAMPLE java
       @Controller
       public class HelloController {
           // @ModelAttribute 修饰的方法会在实际的handle执行前先执行，并且可以接收请求的参数
           // 可用于将从请求体获取的数据加工存放到Model或ModelMap实例中供后续的Handle使用
           @ModelAttribute
           public void createUserModel(String loginname, String password, Model model, ModelMap mm, ModelAndView mv){
               User user = User.builder()
                               .loginname(loginname)
                               .password(password);
               model.addAttribute("user", user);
               mm.addAttribute("user", user);
               mv.addObject("user", user);
           }

           @RequestMapping("/hello")
           public ModelAndView login(Model model, ModelMap mm, ModelAndView mv){
               User user1 = (User) model.asMap().get("user");
               User user2 = (User) mm.get("user");
               User user3 = (User) mv.getModel().get("user");
               mv.setViewName("viewname");

               return mv;
           }
       }
       #+END_EXAMPLE

            }

            }
            #+END_EXAMPLE
       2) ~@PathVariable~
          全限定类名org.springframework.web.bind.annotation.PathVariable，用于获取请求URL中的动态参数。仅含String类型的value参数，若省略则默认绑定同名参数。
          - 示例
            #+BEGIN_EXAMPLE java
            @RequestMapping(value="/url_include_dynamic_variable/{userId}")
            public void index(@PathVariable String userId){
              // TODO
            }
            #+END_EXAMPLE
       3) ~@RequestHeader~
          全限定类名org.springframework.web.bind.annotation.RequestHeader，用于将指定的请求头字段映射到Handle形参。
          - 参数
            | 属性         | 类型    | 说明                 |
            |--------------+---------+----------------------|
            | name         | String  | 指定请求头字段的名称 |
            | value        | String  | 指定请求头字段的名称 |
            | requried     | boolean | 是否必须绑定         |
            | defaultValue | String  | 默认值               |
          - 示例
            #+BEGIN_EXAMPLE java
            @RequestMapping("/path/to/resource")
            public void index(@RequestHeader("Accept") String[] accepts){
              // TODO
            }
            #+END_EXAMPLE
       4) ~@CookieValue~
          全限定类名org.springframework.web.bind.annotation.CookieValue，用于将指定的请求头字段Cookies或Cookies2中的键值对绑定到Handle形参。
          - 参数
            | 属性         | 类型    | 说明         |
            |--------------+---------+--------------|
            | name         | String  | 指定键       |
            | value        | String  | 指定键       |
            | requried     | boolean | 是否必须绑定 |
            | defaultValue | String  | 默认值       |
          - 示例
            #+BEGIN_EXAMPLE java
            @RequestMapping("/path/to/resource")
            public void index(@CookieValue("JSESSIONID") String sessionId){
              // TODO
            }
            #+END_EXAMPLE
       5) ~@SessionAttributes~
          全限定类名org.springframework.web.bind.annotation.SessionAttributes，只能修饰类。用于指定Model实例中哪些属性将转存到HttpSession对象中。
          - 参数
            | 属性  | 类型       | 说明                                                 |
            |-------+------------+------------------------------------------------------|
            | names | String[]   | Model中属性的名称，即存储到HttpSession中的属性的名称 |
            | value | String[]   | 属性names的别名                                      |
            | types | Class<?>[] | 存入HttpSession中的属性的类型                        |
          - 示例
            #+BEGIN_EXAMPLE java
            @Controller
            @SessionAttributes(value={"user","dept"}, types={User.class, Dept.class})
            public class HelloController{
                @RequestMapping("/hello")
                public String hello(Model model){
                    model.addAttribute("user", User.builder().username("john"));
                    model.addAttribute("dept", Dept.builder().name("IT"));
                    // 在jsp视图中可通过${sessionScope.user}从HttpSession中获取User实例
                    return "hello";
                }
            }
            #+END_EXAMPLE
       6) ~@ModelAttribute~
          全限定类名org.springframework.web.bind.annotation.ModelAttribute，将请求参数绑定到Model实例。被修饰的方法会在每个相同Controller中被@RequestMapping修饰的方法执行前被执行。
          - 参数
            | 属性  | 类型   | 说明                        |
            |-------+--------+-----------------------------|
            | value | String | 指定绑定到Model中的属性名称 |
          - 示例
            #+BEGIN_EXAMPLE java
            @ModelAttribute("name")
            public String getName(@RequestParam("username") String username){
                // 返回值将作为Model实例属性name的值
                return username;
            }

            @ModelAttribute
            public User getUser(){
                // 若没有指定属性名称，且返回值为某实例，那么会自动对类名作小写转换作为Model的属性名称
                return User.builder().name("john");
            }

            @ModelAttribute
            public void getUserManually(Model m){
                m.setAttribute("user", User.builder().name("john"));
            }

            @RequestMapping("/index")
            @ModelAttribute("message")
            public String index(){
                 // 当@RequestMapping和@ModelAttribute同时修饰一个方法时，该方法返回值将作为Model实例属性值，而视图名称即保持与RequestMapping的属性value值一致
                 return "helloworld";
            }

            @RequestMapping("/user")

*** 信息转换
    将请求体转换为对象，将对象转换为响应体。Spring3.0+新增接口HttpMessageConverter<T>，和一堆具体的实现类来处理信息转换。

    默认装配的实例有：
    - StringHttpMessageConverter
      - 将请求信息转换为String，可以读取所有MIME(*/*)的请求信息，可通过设置supportMediaTypes属性指定MIME。
      - 响应信息的MIME为text/plain
    - ByteArrayHttpMessageConverter
      - 将请求信息转换为byte[]，可以读取所有MIME(*/*)的请求信息，可通过设置supportMediaTypes属性指定MIME。
      - 响应信息的MIME为application/octet-stream
    - SourceHttpMessageConverter
      - 将请求信息转换为javax.xml.transform.[dom.DOMSource|sax.SAXSource|stream.StreamSource]，可以读取的MIME包含text/xml和application/xml的请求信息。
      - 响应信息的MIME为text/xml或application/xml
    - XmlAwareFormHttpMessageConverter
      继承FormHttpMessageConverter，当表单属性是XML数据，则用该转换器
      - FormHttpMessageConverter
        - 将请求信息转换org.springframework.util.MultiValueMap<String, ?>类型，支持读取MIME为application/x-www-form-urlencoded，不支持multipart/form-data。
        - 响应信息的MIME为application/x-www-form-urlencoded或multipart/form-data

*** 国际化

    基于Java国际化通过提供不同国家/语言环境的消息资源，然后通过ResourceBundle加载指定Locale对应的资源文件，再取得该资源文件中指定key对应的消息。
    Java国际化利用 ~java.util.ResourceBundle~

    Spring MVC 两种方式输出国际化消息：
    1) 通过 Spring MVC 标签库在视图上输出国际化消息
    2) 通过 ~org.springframework.web.servlet.support.RequestContext~ 的 ~getMessage()~ 方法在 Controller 的 Handle中输出国际化消息

    支持3种语言环境存储解析器：
    1) AcceptHeaderLocaleResolver，默认值
    2) SessionLocaleResolver
    3) CookieLocaleResolver

     #+BEGIN_EXAMPLE java
     /* MultipartFile实例包含如下方法
        byte[] getBytes() - 返回文件数据
        String getContentType() - 返回MIME类型，如image/jpeg等
        InputStream getInputStream() - 返回文件流
        String getName() - 返回表单中文件组件的属性name值
        String getOriginalFilename() - 返回上传的文件原名
        long getSize() - 返回文件的字节大小
        boolean isEmpty() - 是否有上传文件
        void transferTo(File dest) - 将上传文件保存到一个目标文件中

     @PostMapping("/upload")
     public String upload(HttpServletRequest request, @RequestParam("file") MultipartFile file) throws Exception {
         if (!file.isEmpty()){
             // 上传路径
             String path = request.getServletContext().getRealPath("/images/");
             // 上传文件名
             String filename = file.getOriginalFilename();
             File filepath = new File(path, filename);

             // 若父路径不存在则创建
             if (!filepath.getParentFile().exists()) {
                  filepath.getParentFile().mkdirs();
             }

             // 将上传文件保存到目标路径中
             file.transferTo(new File(path + File.separator + filename));
             return "success";
         }
         return "error";
     }

*** 拦截器
    通过实现接口HandlerInterceptor或抽象类HandlerInterceptorAdapter的三个方法来完成
    - ~boolean preHandle(HttpServletRequest request, HttpServeltResponse response, Object handle)~
      该方法在请求处理之前依据它声明顺序依次被调用(链式调用)，若前一个方法返回 ~false~ 则后续Interceptor和Controller都不会再执行。
    - ~void postHandle(HttpServeltRequest request, HttpServeltResponse response, Object handle, ModelAndView mv)~
      该方法在请求处理之后渲染之前依据它声明顺序倒序依次被调用(链式调用)，因此可以在这个方法中对Controller处理后返回的ModelAndView实例进行操作。
    - ~void afterCompletion(HttpServeltRequest request, HttpServeltResponse response, Object handle, Exception exception)~
      该方法在DispatcherServlet渲染对应的视图后执行，主要用于资源清理。

    示例：
    #+BEGIN_EXAMPLE xml
    <!-- dispatcher-servlet.xml -->
    <mvc:interceptors>
        <mvc:interceptor>
            <!-- 拦截所有请求 -->
            <mvc:mapping path="/*"/>
            <!-- 使用Bean定义一个Interceptor -->
            <bean class="org.john.interceptor.AuthorizationInterceptor"/>
        </mvc:interceptor>
    </mvc:interceptors>
    #+END_EXAMPLE

    #+BEGIN_EXAMPLE java
    public class AuthorizationInterceptor implements HttpInterceptor {
        private static final String[] IGNORE_URI = {"/loginForm", "/login"};

        @Override
        public boolean preHandle(HttpServletRequest request, HttpServeltResponse response, Object handle) {
            boolean flag = false;

            String servletPath = request.getServletPath();
            for (String uri : IGNORE_URI){
                flag = IGNORE_URI.contains(uri);
                if (flag){
                    break;
                }
            }

            if (!flag){
                // 获取用户凭证
                Usre user = (User) request.getSession().getAttribute("user");
                if (user == null){
                    request.setAttribute("message", "请先登录");
                    request.getRequestDispatcher("loginForm").forward(request, response);
                }
                else{
                    flag = true;
                }
            }

            return flag;
        }
    }
    #+END_EXAMPLE

*** Bean的3种实例化方式
    1) 构造方法
       Spring容器直接通过调用实现类的构造方法实例化对象。
       #+BEGIN_EXAMPLE java
       // Bean对应的实现类
       public class UserInfo{}

       // 使用Spring容器创建Bean实例
       ApplicationContext context = ClassPathXmlApplicationContext(contextConfigLocation);
       UserInfo userInfo = (UserInfo) context.getBean("useInfo");
       #+END_EXAMPLE
       #+BEGIN_EXAMPLE xml
       <bean id="userInfo" class="UserInfo"/>
       #+END_EXAMPLE
    2) 静态工厂
       Spring容器调用静态工厂的静态方法获取目标实现类的对象。
       #+BEGIN_EXAMPLE java
       // Bean对应的实现类
       public class UserInfo{}

       public static class StaticUserInfoFactory{
           public static UserInfo createBean(){
               return new UserInfo();
           }
       }

       // 使用Spring容器创建Bean实例
       ApplicationContext context = ClassPathXmlApplicationContext(contextConfigLocation);
       UserInfo userInfo = (UserInfo) context.getBean("userInfo");
       #+END_EXAMPLE
       #+BEGIN_EXAMPLE xml
       <bean id="userInfo" class="StaticUserInfoFactory" factory-method="createBean"/>
       #+END_EXAMPLE
    3) 实例工厂
       Spring容器先实例化实例工厂，然后再通过实例工厂创建目标实现类的对象。
       #+BEGIN_EXAMPLE java
       // Bean对应的实现类
       public class UserInfo{}

       public class UserInfoFactory{
           public UserInfo createBean(){
               return new UserInfo();
           }
       }
       // 使用Spring容器创建Bean实例
       ApplicationContext context = ClassPathXmlApplicationContext(contextConfigLocation);
       UserInfo userInfo = (UserInfo) context.getBean("userInfo");
       #+END_EXAMPLE
       #+BEGIN_EXAMPLE xml
       <bean id="userInfoFactory" class="UserInfoFactory"/>
       <bean id="userInfo" class="UserInfo" factory-bean="userinfofactory" factory-method="createBean"/>
       #+END_EXAMPLE

*** Bean的作用域
    1) singleton，默认值，Spring容器中最多只有一个实例，无论有多少个Bean引用它，始终指向同一个实例。用于无会话状态的Bean(Dao组件、Service组件)
       - 设置方式
         1) Java Config方式
            #+begin_example java
            @Scope("singleton")
            @Component
            public class BeanClass{}
            #+end_example
         2) XML方式
            #+begin_example xml
            <bean id="beanId" class="io.fsjohnhuang.BeanClass" scope="singleton"/>
            #+end_example
       - 注意：
         - 若Bean仅有含参构造函数,那么该函数的入参必须可以通过Spring容器注入,否则要用 ~@Autowired(required=false)~ 标识不能自动装配的参数.
         - 默认情况下在Spring容器初始化时就会实例化作用域为singleton的Bean，并将实例存放到内部的Map中，后续从Map中直接获取；
         - 可配置为lazy模式实例化
           1) xml方式
              - 所有Bean统一设置
                #+begin_example xml
                <beans default-lazy-init="true>
                  <bean ......></bean>
                  <bean ......></bean>
                </beans>
                #+end_example
              - 单独设置
                #+begin_example xml
                <beans>
                  <bean id="user" class="io.fsjohnhuang.User" lazy-init="true"></bean>
                </beans>
                #+end_example
           2) 注解方式
              - 所有Bean统一设置
                #+begin_example java
                @Lazy
                @Configuration
                public class AppConfig{}
                #+end_example
              - 单独设置
                #+begin_example java
                @Configuration
                public class AppConfig{
                    @Lazy
                    @Bean
                    public User user(){
                        return new User();
                    }
                }
                #+end_example
              - 对于组件自动扫描的，则要通过定义和使用处都调用Lazy注解才行，否则不起作用。
                - Bean定义处
                  #+begin_example java
                  @Lazy
                  @Component
                  public class Employ {}
                  #+end_example
                - Bean使用处
                  #+begin_example java
                  public class App {
                      @Lazy
                      @Autowired
                      Employ employ;
                  }
                  #+end_example
    2) prototype, 每次通过Spring容器获取时，Spring容器都会创建一个全新的实例, Spring启动时不会自动创建Bean实例。用于需要保持会话状态的Bean(Struct2的Action类)
       - 设置方式
         1) Java Config方式
            #+begin_example java
            @Scope("prototype")
            @Component
            public class BeanClass{}
            #+end_example
         2) XML方式
            #+begin_example xml
            <bean id="beanId" class="io.fsjohnhuang.BeanClass" scope="prototype"/>
            #+end_example
    3) request, 同一个HTTP请求中，每次通过Spring容器获取时将返回同一个实例；不在同一个HTTP请求中时则返回全新的实例。即实例仅在当前HTTP Request内有效。
       - 设置方式
         1) Java Config方式
            #+begin_example java
            @Scope("request") // 或 @RequestScope
            @Component
            public class BeanClass{}
            #+end_example
         2) XML方式
            #+begin_example xml
            <bean id="beanId" class="io.fsjohnhuang.BeanClass" scope="request"/>
            #+end_example
    4) session, 对于同一个HTTP Session，每次通过Spring容器获取时将返回同一个实例。
       - 设置方式
         1) Java Config方式
            #+begin_example java
            @Scope("session") // 或 @SessionScope
            @Component
            public class BeanClass{}
            #+end_example
         2) XML方式
            #+begin_example xml
            <bean id="beanId" class="io.fsjohnhuang.BeanClass" scope="session"/>
            #+end_example
    5) globalSession, 在一个全局的HTTP Session中，将返回同一个实例。仅对portlet上下文有效。(portlet现在已不怎么使用了)
    6) application, 为每个ServletContext对象创建一个实例，仅Web相关的ApplicationContext有效。
       - 设置方式
         1) Java Config方式
            #+begin_example java
            @Scope("application") // 或 @ApplicationScope
            @Component
            public class BeanClass{}
            #+end_example
         2) XML方式
            #+begin_example xml
            <bean id="beanId" class="io.fsjohnhuang.BeanClass" scope="application"/>
            #+end_example
    7) websocket, 为每个websocket对象创建一个势力，仅Web相关的ApplicationContext有效。
       - 设置方式
         1) Java Config方式
            #+begin_example java
            @Scope("websocket")

            public class BeanClass{}
            #+end_example
         2) XML方式
            #+begin_example xml
            <bean id="beanId" class="io.fsjohnhuang.BeanClass" scope="websocket"/>
            #+end_example

*** Bean的生命周期
    Spring容器只能跟踪scope为singleton和prototype的Bean的生命周期。注意:prototype的Bean不会调用销毁相关方法.
    | 序号 | 流程操作                                                                                                                             |
    |------+--------------------------------------------------------------------------------------------------------------------------------------|
    |    1 | 通过构造方法实例化Bean                                                                                                               |
    |    2 | 通过依赖注入设置属性值                                                                                                               |
    |    3 | 若实现BeanNameAware接口，则调用setBeanName方法                                                                                       |
    |    4 | 若实现BeanFactoryAware接口，则调用setBeanFactory方法                                                                                 |
    |    5 | 若实现ApplicationContextAware接口，则调用setApplicationContext方法                                                                   |
    |    6 | Pre-initialization, 调用BeanPostProcessor的postProcessBeforeInitialization方法                                                       |
    |    7 | 调用InitializingBean的afterPropertiesSet方法                                                                                         |
    |    8 | 若设置init-method属性，则调用其值指定的方法                                                                                          |
    |    9 | Post-initialization, 调用BeanPostProcesser的postProcessAfterInitialization方法。此时Bean实例为可用状态。                             |
    |   10 | 对于scope为singleton的Bean，则将准备就绪的Bean实例存放在Spring缓冲池中并返回给调用者。若scope为prototype则将Bean实例直接返回给调用者 |
    |   11 | 对于scope为singleton的Bean，则调用Disposable的destroy方法                                                                            |
    |   12 | 对于scope为singleton的Bean，若设置destroy-method则调用其值指定的方法                                                                 |

*** Bean的加载顺序

**** 关于加载顺序的误区
     1. @Configuration中的@Bean按从上往下注册是错误的
     2. @Order能控制加载顺序是错误的。@Order的正确用途如下
	1. 控制AOP的类(@Aspect标注的类)的加载顺序
	2. 控制ApplicationListener实现类的加载顺序
	3. 控制CommandLineRunner实现类的加载顺序
	4. 控制bean集合中元素的顺序(4.0+才支持)
	   #+begin_src
	   @Bean 
	   @Order(2)
	   public IBean aBean(){return new ABean();}
	   @Bean 
	   @Order(1)
	   public IBean bBean(){return new BBean();}

	   @Bean
	   public A a(List<IBean> beans){
	     beans.get(0); // 必定为bBean
	     beans.get(1); // 必定为aBean
	   }
	   #+end_src

**** 控制加载顺序
     1. ~@DependsOn~, 表示当前bean依赖另一个bean，因此要先加载并初始化另一个bean后再初始化当前bean
	- 适用场景
	  1. 直接或间接标注在带有@Component的类上
	  2. 直接或间接标注在带有@Bean的方法上
	  3. 若使用xml配置则改为 ~<bean depends-on="..."/>~

** Spring IoC/DI
   Spring IoC通过反射实习。
   IoC和DI只是从不同的角度描述同一个事物。
   - IoC(Inversion of Control, 控制反转)
     - 实例由Spring容器创建，而不是由实例的使用者来创建；
     - 程序间的关系由Spring容器来负责，而不是由调用者来直接控制。
   - DI(Dependency Injection, 依赖注入)
     - 由Spring容器将被依赖对象赋值给调用者的成员变量。

*** 基于XML的DI实现方式
    1) Setter Injection
       - 前提条件
         1) Bean类必须提供一个没有参数的构造方法
         2) Bean类必须为属性提供setter方法(通过 ~<property>~ 元素为每个属性注入值)
       - 示例
         #+BEGIN_EXAMPLE java
         public class UserInfo{}
         public class UserServiceImpl{
           private UserInfo userInfo;
           public void setUserInfo(userInfo){
               this.userInfo = userInfo;
           }
         }
         #+END_EXAMPLE
         通过Spring配置文件设置注入关系
         #+BEGIN_EXAMPLE xml
         <!-- applicationContext.xml -->
         <bean id="userInfoBean" class="UserInfo">
         <bean id="userDao" class="UserServiceImpl">
             <!-- 设置注入方式，name为调用者的成员变量名称，ref为被注入的Bean组件id -->
             <property name="userInfo" ref="userInfoBean"></property>
         </bean>
         #+END_EXAMPLE
    2) Constructor Injection
       - 前提条件
         1) Bean类必须提供有参数的构造方法(通过 ~<constructor-arg>~ 元素为构造方法的每个参数注入值)
       - 示例
         #+BEGIN_EXAMPLE java
         public class UserInfo{}
         public class UserServiceImpl{
           private UserInfo userInfo;
           private String name;
           public UserServiceImpl(UserInfo userInfo, String name){
               this.userInfo = userInfo;
               this.name = name;
           }
         }
         #+END_EXAMPLE
         通过Spring配置文件设置注入关系
         #+BEGIN_EXAMPLE xml
         <!-- applicationContext.xml -->
         <bean id="userInfoBean" class="UserInfo">
         <bean id="userDao" class="UserServiceImpl">
             <!-- 设置注入方式，name为调用者的成员变量名称，ref为被注入的Bean组件id -->
             <constructor-arg index="0" ref="userInfoBean"></constructor-arg>
             <constructor-arg index="1" value="john"></constructor-arg>
         </bean>
         #+END_EXAMPLE
    3) Interface Injection
       作为注入其他服务的接口，如JNDI数据源注入——由Tomcat或其他服务器中提供具体的JNDI数据源。
       - 示例
         #+BEGIN_EXAMPLE xml
         <bean id="dataSource" class="org.springframework.jndi.JndiObjectFactoryBean">
           <property name="jndiName">
             <value>java:comp/env/jdbc/mybatis</value>
           </property>
         </bean>
         #+END_EXAMPLE

**** Component和Bean的区别
~@Bean~ 仅能应用于Method, 用于显示标记该方法将返回Bean；

** Spring AOP

*** 术语
    - Aspect(切面), 封装用于横向插入系统功能(如日志、事任等)的类。
    - Joinpoint(链接点), 程序运行过程中的函数/方法调用。
    - Pointcut(切入点), 用于匹配Joinpoint的断言或表达式。
    - Advice(通知/增强处理), 在Pointcut中执行的具体的Aspect下的方法。
    - Target Object(目标对象), 被通知或被增强的对象，若采用动态AOP实现，那么Target Object即为一个被代理对象。
    - Proxy(代理), 将Advice应用到Target Object后，被动态创建出来的对象。
    - Weaving(织入), 将Advice应用到Target Object时生成Proxy的过程。
    - Introduction(引入), 为现有类型声明额外的方法和属性。

*** 代理类型
1) JDK动态代理
   通过调用 ~java.lang.reflect.Proxy~ 的方法 ~newProxyInstance~ 来创建代理对象实现的。
   - 限制：
     1) Target Object 必须实现业务接口才能使用JDK动态代理。
   - 示例
     #+BEGIN_EXAMPLE java
     // 定义代理类
     import java.lang.reflect.Proxy;
     import java.lang.reflect.Method;
     import java.lang.reflect.InvocationHandler;
     public class JdkProxy implements InvocationHandler{
         // 目标对象
         private UserDao userDao;
          // 创建代理类
         public UserDao createProxy(UserDao userDao){
             this.userDao = userDao;
             // 1. 获取类加载器
             ClassLoader classLoader = JdkProxy.class.getClassLoader();
             // 2. 获取目标对象实现的所有接口
             Class<?> interfaces = userDao.getClass().getInterfaces();
             // 3. 生成代理
             Object proxy = Proxy.newProxyInstance(classLoader, interfaces, this);
             return (UserDao) proxy;
         }
          /* 调用代理类的任何方法时实质上都是调用该方法，然后我们在该方法中通过反射调用目标对象的方法
          * @param Object proxy - 代理对象
          * @param Method method - 调用的方法信息
          * @param Object[] args - 入参
          * @returns Object
          */
         @Override
         public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
             // 前置增强
             System.out.println("");
              // 调用目标对象的方法
             Object ret = method.invoke(this.userDao, args);
              // 后置增强
             System.out.println("");
             return ret;
         }
     }
     #+END_EXAMPLE
2) CGLIB预编译生成代理
   CGLIB(Code Generation Library)是一个采用底层字节码技术的代码生成包，对指定的目标类生成一个子类，从而对子类实现增强。

         public Object createProxy(UserDao userDao){
             // 创建动态类对象
             Enhancer enhancer = new Enhancer();
             enhancer.setSuperClass(userDao.getClass());
             enhanser.setCallback(this);
             return enhanser.create();
         }
          /*
          * @param Object proxy - 动态类对象
          * @param Method method - 所调用的方法信息
          * @param Object[] args - 入参
          * @param MethodProxy methodProxy - 动态类对象的方法对象，用于执行父类的方法
          */
         public Object intercept(Object proxy, Method method, Object[] args, MethodProxy methodProxy) throws Throwable {
             // 前置增强
             System.out.println("");
             // 调用目标对象的方法
             Object ret = methodProxy.invokeSuper(proxy, args);
             // 后置增强
             System.out.println("");
             return ret;
         }
     }

** TODO 事务处理

   Java EE提供JTA事务API和JTS事务服务规范。

*** 概要
事务特点:
1) 原子性, 事务内的一系列操作要么都成功,要么都失败.
2) 一致性, 一旦事务结束,系统必须保证业务模型处于一致的状态.
3) 隔离性, 多个事务并行执行时,应将它们对状态的操作隔离起来.
4) 持久性, 一旦事务结束,状态结果将被持久化.

**** 传播行为
当事物方法被另一个事务方法调用时,必须指定事务应该如何传播(在现有事务中运行,或创建新事务运行)
Spring定义7种事务传播行为
| 值                        | 含义                                                                                           |
|---------------------------+------------------------------------------------------------------------------------------------|
| PROPAGATION_REQUIRED      | 若存在活动事务,则在当前事务运行；否则创建新事务运行                                            |
| PROPAGATION_SUPPORTS      | 若存在活动事务,则在当前事务运行；否则直接运行                                                  |
| PROPAGATION_MANDATORY     | 若存在活动事务,则在当前事务运行；否则抛出异常                                                  |
| PROPAGATION_REQUIRES_NEW  | 无论如何均创建新事务运行,若存在活动事务则将当前事务挂起                                        |
| PROPAGATION_NOT_SUPPORTED | 若存在活动事务则挂起当前事务,再运行                                                            |
| PROPAGATION_NEVER         | 若存在活动事务则挂起当前事务,则直接抛出异常                                                    |
| PROPAGATION_NEST          | 若存在活动事物在创建嵌套事务(独立于当前事务执行提交和回滚)；否则创建新事务运行(逻辑由厂商决定) |

通过枚举类 ~Propagation~ 定义，其中 ~Propagation.REQUIRED~, ~Propagation.REQUIRES_NEW~ 和 ~Propagation.NEST~ 最为常用。
使用示例
1. ~Propagation.REQUIRED~
   #+begin_src java
   @Service
   public class UsersService {
       @Autowired
       UserService userService;
       @Autowired
       UserMapper dao;

       @Transactional(propagation=Propagation.REQUIRED)
       public void saveUsers(List<User> users) {
         for (User user : users) {
           dao.insert(user);
           userService.save(user);
         }
       }
   }

   @Service
   public class UserService {
       @Transactional(propagation=Propagation.REQUIRED)
       public void saveUser(User user) {
         throw new RuntimeException("");
       }
   }
   #+end_src
   - 过程
     1. 调用saveUsers时会创建一个事务上下文
     2. 调用saveUser时，由于调用点已经存在一个事务上下文，因此saveUser处直接加入已有的事务上下文即可
     3. saveUser中抛出运行时异常，导致其所在的事务上下文发生回滚操作
     4. dao.insert的操作也被回滚
2. ~Propagation.REQUIRES_NEW~
   #+begin_src java
   @Service
   public class UsersService {
       @Autowired
       UserService userService;
       @Autowired
       UserMapper dao;

       @Transactional(propagation=Propagation.REQUIRED)
       public void saveUsers(List<User> users) {
         for (User user : users) {
           dao.insert(user);
           userService.save(user);
         }
       }
   }

   @Service
   public class UserService {
       @Transactional(propagation=Propagation.REQUIRES_NEW)
       public void saveUser(User user) {
         if (user == null) {
             throw new RuntimeException("");
         }
       }
   }
   #+end_src
   - 过程
     1. 调用saveUsers时会创建一个事务上下文
     2. 调用saveUser时，虽然调用点已经存在一个事务上下文，但每次调用该方法时都会创建一个新的事务上下文，并且saveUser将加入这个新的事务上下文
     3. 当saveUser执行第N次时，因user为null而抛出运行时异常，那么将会导致第N次调用所属事务上下文的回滚
     4. 前N-1次的saveUser执行内容将不被回滚
     5. dao.insert操作也不会被回滚
3. ~Propagation.NEST~
   #+begin_src java
   @Service
   public class UsersService {
       @Autowired
       UserService userService;
       @Autowired
       UserMapper dao;

       @Transactional(propagation=Propagation.REQUIRED)
       public void saveUsers(List<User> users) {
         for (User user : users) {
           dao.insert(user);
           userService.save(user);
         }
       }
   }

   @Service
   public class UserService {
       @Transactional(propagation=Propagation.NEST)
       public void saveUser(User user) {
         if (user == null) {
             throw new RuntimeException("");
         }
       }
   }
   #+end_src
   - 过程
     1. 调用saveUsers时会创建一个事务上下文
     2. 调用saveUser时，调用点已经存在一个事务上下文，因此每次调用saveUser时则在当前事务上下文上添加保存点(Transaction Savepoint)
     3. 当saveUser执行第N次时，因user为null而抛出运行时异常，那么将会事务将回滚到上一个保存点
     4. 前N-1次的saveUser执行内容将不被回滚
     5. dao.insert操作也不会被回滚
   - 注意： 必须资源管理器(如数据库)支持保存点技术时才会启用Propagation.NEST配置，否则将自动降级为Propagation.REQUIRES_NEW
   - Propagation.REQUIRES_NEW 和 Propagation.NEST的区别： Propagation.REQUIRES_NEW可以设置与调用方所在事务上下文不同的隔离级别，而Propagation.NEST只能保持和调用方所在事务上下文一致的隔离级别。
     
**** 隔离级别
| 级别                          | 说明                                                                                                                |
|-------------------------------+---------------------------------------------------------------------------------------------------------------------|
| 脏读(dirty read)              | 事务A读取了另个事务B改写但尚未提交的数据.若事务B回滚,那么事务A读取的数据将为无效数据.                               |
| 不可重复读(unrepeatable read) | 事务A在存续期间多次执行相同的查询操作,但获取的数据不尽相同.原因是事务B在此期间修改了数据并提交事务.                 |
| 幻读(phantom read)            | 事务A在存续期间多次执行相同的查询记录数量操作,但获取的数据不尽相同.原因是事务B在此期间新增或删除数据记录并提交事务. |

Spring定义5种事务隔离级别
| 值                         | 含义                                                      |
|----------------------------+-----------------------------------------------------------|
| ISOLATION_DEFAULT          | 使用数据源默认的隔离级别                                  |
| ISOLATION_READ_UNCOMMITTED | 最低的隔离级别,会出现脏读,不可重复读和幻读                |
| ISOLATION_READ_COMMITTED   | 仅允许读取已提交的数据,会出现不可重复读和幻读             |
| ISOLATION_REPEATABLE_READ  | 会出现幻读                                                |
| ISOLATION_SERIALIZABLE     | 最好的隔离级别,完全服从ACID的隔离级别,性能最低的隔离级别.不会出现脏读,不可重复读和幻读问题 |

*** @Transactional自调用失效问题
**** 失效场景一及原因
     #+begin_src java
     @Service
     public class UserService {
       @Autowired
       UserMapper dao;

       @Transactional(propagation=Propagation.REQUIRED)
       public void saveUsers(List<User> users) {
           for (User user : users) {
               saveUser(user);
           }
       }

       @Transactional(propagation=Propagation.REQUIRES_NEW)
       public void saveUser(User user) {
         if (user == null) {
           throw new RuntimeException("");
         }
         dao.insert(user);
       }
     }
     #+end_src
     - 预期结果
       每次调用saveUser方法时都会创建一个新的事务上下文，并且saveUser方法隶属于这个新的事务上下文。
     - 实际结果
       每次调用saveUser方法时均不会创建事务上下文，甚至不会执行于 ~@Transactional~ 配置相关的任何操作。
     - 原因分析
       由于Spring的 ~@Transactional~ 是通过AOP织入事务管理的相关代码，而saveUsers调用saveUser属于类内部方法自调用，因此根本调用已被AOP织入事务管理代码的saveUser方法。

**** 失效场景儿及原因
     @Transactional既可以修饰接口也可以修饰类。
     失效场景：修饰接口时AOP实现方式必须采用JDK动态代理，若采用GCLIB动态代理则失效。
     原因：由于GCLIB动态代理时直接代理目标类，不涉及目标类所继承的接口，那么修改接口的@Transactional自然就无法读取并根据其值织入事务相关代码

**** JTA(Java Transaction Manager)
XA: 作为资源管理器和事务管理器的接口规范,由Tuxedo提出并由X/Open组织维护.
XA规范定义:
1) TransactionManager: 通过TransactionManager可以管理多个ResourceManager
2) XAResource: 数据源封装的接口
3) 两段式提交: 多数据源事务提交的机制, 即整个业务流程完成时进行第一阶段提交, 然后在第二阶段提交前会检查之前是否抛出异常或第一阶段是否有提交, 若为否则对所有事务进行回滚.

JTA则为XA在Java上的实现.
1) TransactionManager: 包含常用方法(begin,commmit,rollback)
2) XAResource: 通过session进行事务管理
3) XID: 每个事务分配特定的XID

缺点:
1) 事务时间较长,锁数据时间较长
2) 低性能,低吞吐量

*** 通过实现SmartLifecycle接口感知容器的变化
    继承关系：SmartLifecycle extends Lifecycle, Phased
    | 接口           | 方法                    | 描述                                                                                                  |
    |----------------+-------------------------+-------------------------------------------------------------------------------------------------------|
    | Lifecycle      | void start()            | 当Spring容器上下文被刷新(所有bean初始化完毕后)时，若为SmartLifecycle实例则判断isAutoStartup方法是否返回true，若为false则不立即调用而是等待显式调用；若为Lifecycle实例则直接调用。注意执行完需要设置isRunning=true |
    | Lifecycle      | void stop()             | 容器关闭时，Spring容器先检查isRunning值，若为true并发现当前对象实现SmartLifecycle则调用stop(Runnable)，若实现Lifecycle则调用stop()。注意执行完需要设置isRunning=false  |
    | Liefcycle      | boolean isRunning()     | 返回当前实例的状态，返回isRunning字段值。                                                                    |
    | Phased         | int getPhase()          | 返回值决定start和stop方法在众多Lifecycle实现类的执行顺序，start顺序为从小到大，而stop顺序则刚好相反。默认值为0    |
    | SmartLifecycle | boolean isAutoStartup() | 执行start方法前先调用该方法，若返回false则不执行start方法                                             |
    | SmartLifecycle | void stop(Runnable)     | 容器关闭时，Spring容器发现当前对象实现SmartLifecycle则调用stop(Runnable)，若实现Lifecycle则调用stop()。注意必须调用Runnable.run方法，否则Spring容器关闭流程会被阻塞直到超时，超时默认为30秒 |

**** Spring容器启动与SmartLifecycle的关系 
     AbastractApplicationContext的refresh方法中，在所有bean实例化和初始化操作完毕后会调用finishRefresh方法。
	#+begin_src java
	protected void finishRefresh() {
          /* LifecycleProcessor是所有Lifecycle实例的管家，包含对Lifecycle的各种操作。
             若Spring容器中已经存在LifecycleProcessor实例则将其赋值给applicationContext的成员变量lifecycleProcessor；
	     否则注册一个类型为DefaultLifecycleProcessor的bean，并将其赋值给applicationContext的成员变量lifecycleProcessor。
           */
	  initLifecycleProcessor();

          /* DefaultLifecycleProcessor的onRefresh方法逻辑为
	     1. 筛选出isAutoStartup方法返回true的SmartLifecycle实例
             2. 根据getPhase的返回值将SmartLifecycle实例分组(LifecycleGroup)。对于只实现Lifecycle的实例，DefaultLifecycleProcessor的onReresh方法将排除它们。
             3. 根据getPhase返回值，从小到大调用LifecycleGroup的start方法
             4. LifecycleGroup的start方法内部会先遍历旗下的Lifecycle实例
             5. 若当前Lifecycle实例存在依赖项目，则递归执行依赖项start方法
             6. 最后若当前Lifecycle实例isRunning返回false，若当前为SmartLifecycle则isAutoStartup必须返回true才执行start方法
           */
          getLifecycleProcessor().onRefresh();
	}
	#+end_src

**** Spring容器关闭与SmartLifecycle的关系
     AbstractApplicationContext的doClose方法中，在Spring容器关闭(调用applicationContext.close方法或JVM关闭)时会
     1. 发布容器关闭事件 ~publicEvent(new ContextClosedEvent(this))~
     2. 调用lifecycleProcessor的onClose方法
	1. 根据getPhase的返回值将Lifecycle实例分组，仅实现Lifecycle的getPhase恒返回0
	2. 根据getPhase返回值，从大到小调用LifecycleGroup的stop方法
	3. LifecycleGroup的stop方法内部会先遍历旗下的Lifecycle实例
        4. 若当前Lifecycle实例存在依赖项目，则递归执行依赖项stop方法
	   - 对于SmartLifecycle实例的stop方法是在新线程中执行，因此多个实例可以并行执行stop方法。
	5. 通过CountDownLatch实例控制当所有Lifecycle均执行完stop方法后或超时后，再执行Spring容器关闭操作中的后续操作
     3. 销毁所有Scope为Sigleton的Bean

但可以独立打包为WAR包，再部署到Tomcat等Web容器。

1) 修改pom.xml
   #+begin_example xml
   <packaging>war</packaging>
   <build>
       <plugins>
           <plugin>
               <groupId>org.apache.maven.plugins</groupId>
               <artifactId>maven-war-plugin</artifactId>
               <configration>
                   <warName>war包名</warName>
               </configration>
           </plugin>
       </plugins>
   </build>
   #+end_example
2) 添加src/main/webapp/WEB-INF/web.xml
   将 ~Tomcat项目目录/webapps/ROOT/WEB-INF/web.xml~ 复制到 ~项目目录/src/main/webapp/WEB-INF/web.xml~
3) 修改Spring Boot启动类
   #+begin_example java
   @SpringBootApplication // 启动Spring Boot程序
   @@ComponentScan("io.fsjohnhuang")
   public class StartSpringBootMain extends SpringBootServletInitializer {
       @Override
       protected SpringApplicationBuilder configure(SpringApplicationBuilder builder) {
           return builder.sources(StartSpringBootMain.class);
       }

       public static void main(String[] args) throws Exception {
           SpringApplication.run(StartSpringBootMain.class, args)
       }
   }
   #+end_example
4) 项目编译后将war包复制到 ~Tomcat项目目录/webapps~ 下

** 热部署
   通过spring-boot-devtools可以在修改类或配置文件的时候自动重新加载Spring Boot应用。
   在pom.xml中添加
   #+begin_src xml
   <dependency>
     <groupId>org.springframework.boot</groupId>
     <artifactId>spring-boot-devtools</artifactId>
     <optional>true</optional>
   </dependency>
   #+end_src
   原理：启动LiveReload Server监控Spring Boot应用文件变化，并且仅重新加载变化的类。因此热部署速度非常快。

        </dependency>

        <!-- 下面为自定义Starter的依赖项 -->
      </dependencies>
      #+end_src
   2. 如何编写自动配置
      #+begin_src
      @Configuration
      @ConditionOnWebApplication
      @AutoConfigureOrder(Ordered.HIGHEST_PRECEDENCE + 10) // 设置自动配置类的执行顺序
      @AutoConfigureAfter({}) // 设置在哪些配置生效后执行
      @EnableConfiguationProperties({}) // 启用ConfigurationProperties引入外部配置
      public class MyConfig {
        // @Bean
      }

      @ConfigurationProperties(prefix="my.properties")
      public class MyProperties {
      }
      #+end_src
   3. 启动时加载自动配置类（借鉴了SPI机制）
      将标记@Configuration的自动配置类的全限定类名，放在classpath的META-INF/spring.factories文件中的 ~org.springframework.boot.autoconfigure.EnableAutoConfiguration~ 中。
      - 注意：内部Configuration执行完成才会执行外部Configuration。
      - 内部原理浅析：
	使用starter的项目需要通过 ~@EnableAutoConfiguration~ 注解告知Spring容器通过 ~org.springframework.core.io.support.StringFactoriesLoader~ 将 ~META-INF/spring.factories~ 中key为 ~EnableAutoConfiguration~ 的所有value加载出来，然后可以进行常规的 ~@Configuration~ 配置类解析。
	而 ~@SpringBootApplication~ 中包含 ~@EnableAutoConfiguration~ 。
   4. Starter开发模式
      1. xxx-starter项目为一个空JAR文件，仅提供辅助性依赖管理，这些依赖可能用于自动装配或者其他类库，并引用xxx-starter-autoconfigurer项目。
      2. xxx-starter-autoconfigurer项目内包含自动配置的内容

	   <artifactId>spring-boot-starter</artifactId>

	 - resources/META-INF/spring.factories(利用SPI机制，完成组件的自动装配功能)

***** 属性讲解
      Group：用于定义ConfigurationProperties的描述信息
      /Groups/ are higher-level items used to group other properties, without specifying a value itself.

      Property：用于定义Group描述的ConfigurationProperties类中属性的描述信息
      /Properties/ are configuration items for which we can specify a value. These properties are set in .properties or .yml files and can have extra information, like default values and validations.

      Hint：用于定义具体属性的可选择值以及可选择值的描述信息
      /Hints/ are additional information to help the user set the property value. For example, if we have a set of allowed value for a property. we can provide a description of what each of them does. The IDE will provide auto-completion help for these hints.

      Group属性
      | Name         | Type   | Description                                                                                                                                                                                 |
      |--------------+--------+---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
      | name         | String | Required. The property name of this group.  Set the value as same as the prefix property of ~@ConfigurationProperties~ which this group described.                                          |
      | type         | String | Optional. The full qualified class name of class which is annotated by ~@ConfigurationProperties~, or the full qualified class name of the return value of the method annotated by ~@Bean~. |
      | description  | String | Optional. The brief introdution for the group. Recommend a brief description as the first paragraph, and the last sentence is end with full stop.                                           |
      | sourceType   | String | Optional. The full qualified class name of the class which contains the method anntotated by ~@Bean~ whose return value is described by this group.                                         |
      | sourceMethod | String | Optional. The full method signature(includes square and parameter types) whose return value is described by this group.

      Property属性
      | Name         | Type       | Description                                                                                                                                                           |
      |--------------+------------+-----------------------------------------------------------------------------------------------------------------------------------------------------------------------|
      | name         | String     | Required. The full name of the property in lower case separated by hyphen(e.g. server.servlet-path).                                                                  |
      | type         | String     | Optional. The full qualified class name of this property. Use the wrapper class name rather than the primitive type(e.g. replace ~boolean~ with ~java.lang.Boolean~ ) |
      | description  | String     | Optional.                                                                                                                                                             |
      | sourceType   | String     | Optional. Set to the full qualified class name of the class annotated by ~@ConfigurationProperties~ if this property is belong to a class.                            |
      | defaultValue | Object     | Optional. The default value of this property. Empty array if the ~type~ attribute is a kind of ~Array~.                                                               |
      | deprecation  | Deprecated | Optional.  Indicate this property is deprecated when this attribute exists.                                                                                           |
      
      Deprecated属性
      | Name         | Type       | Description                                                                                                                                                           |
      |--------------+------------+-----------------------------------------------------------------------------------------------------------------------------------------------------------------------|
      | level        | String     | Optional. ~WARN~ as default value which can be used as usual. ~ERROR~ is out of control.                                                                              |
      | reason       | String     | Optional. The brief reason for deprecation.                                                                                                                           |
      | replacement  | String     | Optional. The replacement for the deprecated property.                                                                                                                |

      Hint属性
      | Name      | Type            | Description |
      |-----------+-----------------+-------------|
      | name      | String          | Required. The full name of the property in lower case separated by hyphen. Indicate the keys and values of a map by suffixed by ~keys~ and ~values~. (e.g. ~system.context.keys~, ~system.contexgt.values~ |
      | values    | ValueHint[]     | Required. A list of ValueHints to describe valid values. |
      | providers | ValueProvider[] | Optional.            |
      ValueHint属性
      | Name        | Type   | Description                                                                            |
      |-------------+--------+----------------------------------------------------------------------------------------|
      | value       | Object | Required. The valid value for specified property. Set an array if the type is ~Array~. |
      | description | String | Optional.                                                                               |
      ValueProvider属性
      | Name       | Type        | Description                                                                                     |
      |------------+-------------+-------------------------------------------------------------------------------------------------|
      | name       | String      | The full qualified class name of the provider which provide extra information of this property. |
      | parameters | JSON object | Optional.                                                                                       |

      The full qualified class name for an inner class likes ~com.xxx.test.C$Server~.

****** Provider of a property
       1. Class Reference Provider. Set the list of implements of the specified interface as the set of allowed values to this property.
	  #+begin_src js
	  {
	       "name": "my.property",
                    "providers": [{
		    "name": "class-reference",
                              "parameters": {
                                  "target": "com.my.IStrategy" // The allowed list of values is consist of the implements of com.y.IStrategy interface.
                              }
                    }] 
	  }
	  #+end_src
       2. Handle As Provider. Let IDE treat the String property as a Resource without creating an explicit dependency.
	  #+begin_src js
	  {
	       "name": "my.property",
                    "providers": [{
		    "name": "handle-as",
                              "parameters": {
                                  "target": "org.springframework.core.io.Resource" // classpath:, file:, http: an so on are can be used in this property.
                              }
                    }] 
	  }
	  #+end_src

       https://docs.spring.io/spring-boot/docs/1.3.1.RELEASE/reference/html/configuration-metadata.html#_value_provider

* MicroService
** 介绍
  源自Martin Fowler的Microservices博文(http://martinfowler.com/articles/microservices.html)，作为一种系统架构的设计风格。
  特点：
  1) 以进程为边界，划分系统业务能力和实现能力重用
  2) 主要服务间以基于HTTP的RESTful API进行通信
  3) 各服务独立开发、测试、部署、运维，和进行性能容量评估和优化

  9大特性：
  1) 服务组件化
  2) 按业务组织团队
     从过往按技术层面的团队划分方式(DBA、运维、后端、前端和设计师团队)，重新以业务线来重新划分和组织宽栈或全栈团队。
  3) 做“产品”的态度
  4) 智能端点和哑管道
     - 通信方式
       1) 请求-响应同步阻塞模式，基于HTTP的RESTful API 或 轻量级的消息发送协议
       2) 观察者异步模式，通过轻量级消息总线(如RabbitMQ)提供异步消息传递
       3) 对于性能要求较高的场景，可以使用二进制的消息发送协议(如protobuf)
  5) 去中心化治理
     每个服务应业务特点进行技术选型、开发模式、运维方式等决策。
  6) 去中心化管理数据
     每个服务应业务特点使用不同类型的数据库和独立的数据库
     - 问题
       1) 数据一致性
          - 最终一致性，即数据在最后的处理状态一致即可。中间过程中发现错误，则通过补偿机制来处理从而保证最终一致性。
       2) 分布式事务
          - 强调各服务间进行“无事务”的调用
  7) 基础设施自动化
     - 自动化测试
     - 自动化部署
  8) 容错设计
     微服务会存在个别服务不可用，而其它服务可用的情况，因此快速检测出故障源并尽可能地自动恢复服务是设计时需要考虑的。
     - 方法：每个服务中实现监控和日志记录，记录服务状态、断路器状态、吞吐量、网络延迟等关键数据的仪表盘。
  9) 演进式设计
     1) 初期：以单体系统的方式设计和实施；
     2) 中期：将经常变动或有一定时效的内容进行微服务处理，逐渐将原来的单体系统中多变的部分拆分出来。

  问题：
  1) 运维复杂性提高，由于运维对象数量将成几何级别地上升，因此需要自动化运维支撑
  2) 接口一致性，严格的 单体服务接口版本 和 整体服务接口版本(各服务接口的依赖) 的管理
  3) 分布式的复杂度，存在网络延迟、分布式事务和异步消息等


  服务治理：Dubbo（阿里）, DubboX（当当基于Dubbo），Eureka（Netflix）和Consul（Apache）等
  分布式配置管理：Disconf（百度），Archaius（Netflix），QConf（360），Spring Cloud Config，Diamond（淘宝）等
  批量任务：Elastic-Job（当当），Azkaban（ LinkedIn），Spring Cloud Task
  服务跟踪：Hydra（京东），Zipkin（Twitter），Spring Cloud Sleuth等

** Spring Cloud

*** 基本介绍
    基于Spring Boot实现的一套微服务架构解决方案，涉及配置管理、服务治理、断路器、智能路由、微代理、控制总线、全局锁、决策竞选、分布式会话和集群状态管理等。
    Spring Cloud的子项目
    1) Spring Cloud Config
       配置管理工具，支持
       1) 使用Git存储配置内容
       2) 客户端配置信息刷新
       3) 加密/解密配置内容
    2) Spring Cloud Netflix
       核心组件，对多个Netflix OSS开源套件进行整合
       1) Eureka，服务治理组件，含服务注册中心、服务注册和发行机制
       2) Hystrix，容错管理组件，实现断路器模式
       3) Ribbon，客户端负载均衡的服务调用组件
       4) Feign，基于Ribbon和Hystrix的声明式服务调用组件
       5) Zuul，网关组件，提供智能路由、访问过滤等功能
       6) Archaius，外部化配置组件
    3) Spring Cloud Bus
       事件、消息总线，用于传播集群中的状态变化或事件，比如动态刷新配置等
    4) Spring Cloud Cluster
       针对ZooKeeper、Redis、Hazelcast、Consul的选举算法和通用状态模式的实现
    5) Spring Cloud Cloudfoundry
       与Pivotal Cloudfoundry的整合支持
    6) Spring Cloud Consul
       服务发现和配置管理工具
    7) Spring Cloud Stream
       通过Redis,Rabbit或Kafka实现的消费微服务，可以通过声明式模型来发送和接收消息
    8) Spring Cloud AWS
       整合Amazon Web Service
    9) Spring Cloud Security
       安全工具包，提供在Zuul中对OAuth2客户端请求的中继器
    10) Spring Cloud Sleuth
        分布式跟踪实现，完美整合Zipkin
    11) Spring Cloud ZooKeeper
        基于ZooKeeper的服务发现和配置管理组件
    12) Spring Clould Starters
        基于Spring Boot风格项目的基础以来模块
    13) Spring Cloud CLI
        用于在Groovy中快速创建Spring Cloud应用的Spring Boot CLI插件

*** 版本说明
    版本采用命名方式而不是版本号(x.y.z)方式，具体为Angel.SR6, Camden.M1 和 Camde.BUILD-SNAPSHOT等。其中SR6表示Service Release的第6版，表示发布内容积累到临界点或一个严重的bug被修复。
    采用命名方式的原因是，Spring Cloud包含一组子项目，而各子项目的版本推进是独立的，因此采用命名方式来多个子项目的一系列兼容版本更有效果。

*** Spring Boot
    通过大量的自动化配置等方式简化Spring项目的配置过程，并提供一系列Starter POMs的定义，只需在pom.xml中加载对应的Start POM即可引入复杂Spring相关的依赖关系。
    spring-boot-starter-web是全栈Web开发模块，包含嵌入式Tomcat、Spring MVC
    spring-boot-starter-test是测试模块，包含JUnit、Hamcrest、Mockito
    spring-boot-starter-jdbc 或 spring-boot-starter-jpa 是数据库模块
    Spring Boot开发的应用打包为jar，内含Tomcat或Jetty等容器，因此省去打包为war，然后在服务器上部署Tomcat的过程。

*** Spring Cloud Eureka
    服务治理作为微服务架构中最核心和基础的模块，提供服务注册、服务发现、服务端负载均衡、客户端负载均衡功能。
    1) 服务注册
       每个服务单元启动时自动向注册中心登记其所提供的服务(主机号、IP、端口、版本号、通信协议等)
    2) 服务发现
       服务间的调用将不再通过指定具体的服务实例地址来实现，而是通过服务名称发起请求调用实现。因此调用方在发起调用前需要从注册中心获取服务实例列表，然后根据轮询规则从列表中取出一个具体的实例地址发起调用。

    Eureka包含Java服务端和客户端，提供基于RESTful API的服务治理机制，而其它语言平台也有对应的客户端(.NET的Steeltoe，Nodejs的eureka-js-client等)
    Eureka服务端支持高可用集群配置，集群分片间通过异步模式相互复制各自的状态，因此在任意时刻每个分片实例关于所有服务的状态会有细微的差别。
    Eureka客户端功能
    1) 服务注册
    2) 周期性地向注册中心发送心跳来更新它的服务租约
    3) 服务发现，查询、缓存和周期性刷新各服务状态

**** 架构说明
1) 服务提供者
   - 服务注册
     服务启动时默认会将服务自身元数据发送给注册中心
   - 服务同步
     注册中心集群中的节点会相互异步同步其上的服务信息
   - 服务续约(Renew)
     服务提供者在注册后，会维护一个心跳来告知Eureka Server该服务可用，防止从服务列表中剔除。
     - 在服务提供者的application.properties中设置
       #+begin_example bash
       eureka.instance.lease-renewal-interval-in-seconds=30 # 定义服务续约任务的调用时间间隔，默认30秒
       eureka.instance.lease-expiration-duration-in-seconds=90 # 定义服务失效的时间，默认90秒。即90秒内没有向Eureka Server发送心跳请求，那么Eureka Server就可以剔除该服务实例
       #+end_example
   - 服务下线
     当服务实例正常关闭时，会向Eureka Server发送服务下线通知，然后Eureka Server会将该服务实例状态设置为下线(Down)，并通知整个注册中心集群。
2) 服务消费者
   - 服务发现
     服务启动时会向Eureka Server获取服务列表，且默认情况下该服务列表会缓存起来，每隔30秒再从Eureka Server重新获取。
     #+begin_example bash
     eureka.client.fetch-registry=false # 禁止从Eureka Server获取服务列表，默认为true
     eureka.client.registry-fetch-interval-seconds=30 # 重新从Eureka Server获取服务列表的时间间隔
     #+end_example
   - 服务调用
     Eureka中有Region和Zone的概念，一个Region包含多个Zone；而每个服务实例必须且仅能注册到一个Zone。当服务调用时，会优先调用同一个Zone中的服务提供方。
     - 通过设置region和zone来配合物理部署架构，实现就近调用、多地容灾。
     - 设置服务所属的region和zone
       #+begin_example bash
       eureka.client.region=default # 设置所属region
       eureka.client.availability-zones=defaultZone # 设置所属zone，多个zone间通过逗号分隔
       #+end_example
3) 注册中心
   - 服务剔除
     每隔60秒(默认)将当前服务清单中超时(默认90秒)没有续约的服务剔除出去。
   - 自我保护
     Eureka Server统计心跳失败的比例在15分钟内低于85%(开发环境很容易达到，生产环境通常为网络不稳定导致)时，会保护并认为当前服务列表均可用。从而导致服务消费者可能向实际不可用的服务实例发起调用。
     - 控制台提示
       #+begin_quote

       renewals are lesser than threshold and hence the instances are not being expired just to be safe.
       #+end_quote
     - 禁用设置application.properties
       #+begin_example bash

       #+end_example

**** 重试机制
CAP原理：
ZooKeeper强调CP(一致性和可靠性)。
Spring Cloud Eureka强调AP(可用性和可靠性)，因此当服注册中心的网络发生故障，导致所大部分服务实例无法维持续约心跳，启用保护机制保留所有节点，从而实现即使其中有部分故障节点，但大部分服务依然正常。
Spring Cloud集成Spring Retry增强RestTemplate的重试能力。
#+begin_example bash
# 开启重试机制，默认是关闭的
spring.cloud.loadbalancer.retry.enabled=true

# 断路器的超时时间需要大于Ribbon的超时时间，不然不会处罚重试
hystrix.command.default.execution.isolation.thread.timeoutInMilliseconds=10000
# 请求连接的超时时间
hello-service.ribbon.ConnectTimeout=250
# 请求处理的超时时间
hello-service.ribbon.ReadTimeout=1000
# 对所有操作请求都进行重试
hello-service.ribbon.OkToRetryOnAllOperations=true
# 切换实例的重试次数
hello-service.ribbon.MaxAutoRetriesNextServer=2
# 对当前实例的重试次数
hello-service.ribbon.MaxAutoetries=1

# 场景：请求失败->向当前实例再请求一次，失败->向下一个实例A请求一次，失败->向下一个实例B请求一次，失败->整体请求失败
#+end_example
*** Spring Cloud Hystrix
由于服务间依赖通过网络远程调用的方式执行，可能会由网络问题或部分服务自身问题导致调用延迟或故障，若此时调用方的请求不断增加并积压在依赖方，进而导致各服务瘫痪。
例如：订单服务需要从库存服务获取数据，但库存服务处理缓慢导致订单服务多个线程被挂起，在高并发场景下，将导致后续订单服务的请求被拒绝，进而服务不可用。

断路器(Circuit Breaker)，向调用方返回一个错误响应，而不是长时间的等待。从而避免故障在分布式系统中蔓延。

Spring Cloud Hystrix是基于Netflix Hystrix开发啊，具备服务降级、服务熔断、线程和信号隔离、请求缓存、请求合并以及服务监控等强大功能。

命令模式适用于记录日志、撤销操作、队列请求等。
HystrixCommand 用在依赖的服务发那会单个操作结果的场景
- ~R execute()~ 同步执行，从依赖的服务返回一个单一的结果对象，或是在发生错误时抛出异常
- ~Future<R> queue()~ 异步执行，返回一个Future对象，其中包含服务执行结束时要返回的单一结果对象
HystrixObservableCommand 用在依赖的服务返回多个操作结果的场景
- ~Observable<R> observe()~ 返回Observable对象，代表操作的多个结果，是一个Hot Observable
- ~Observable<R> toObservable()~ 返回Observable对象，代表操作的多个结果，是一个Cold Observable

Hot Observable 不论“事件源”是否有“订阅者”，都会在创建后对事件进行发布。因此Hot Observable的每个“订阅者”都有可能从“事件源”的中途开始，仅能看到整个操作的局部过程。

Cold Observable 若没有“订阅者”则不会发布事件，而是等待直到有“订阅者”。因此Cold Observable的订阅者能保证从一开始看到整个操作的全过程。

**** Hystrix工作流程
1) 创建HystrixCommand或HystrixObservableCommand对象
2) 执行命令
3) 结果是否被缓存
   若当前命令的请求缓存功能被开启并且命中，那么会立即返回Observable对象。
4) 断路器是否打开
   1) 断路器打开时，Hystrix不会执行命令，而是转到fallback的处理逻辑(对应第8步)
   2) 断路器关闭时，Hystrix检查是否有可用资源来执行命令(对应第5步)
5) 线程池/请求队列/信号量是否占满
   若与命令相关的线程池和请求队列或信号量已被占满，Hystrix会转到fallback处理逻辑。
   - 注意：Hystrix为每个依赖服务创建独立的专有线程池。
6) 执行HystrixObservableCommand.constructor()或HystrixCommand.run()
   - 执行抛异常时会进行fallback处理逻辑
7) 计算断路器的健康度
   Hystrix会将成功、失败、拒绝和超时等信息报告给断路器，断路器会维护这组数据并根据它们来决定是否打开断路器。恢复期间断路器处于开启状态，对应的依赖服务的请求将被熔断/短路。恢复期满再次根据这组数据判断是否关闭断路器。
8) fallback处理/服务降级
   提供一个通用的响应结果，可以从缓存或根据静态逻辑生成。若要从网络获取数据，那么该网络请求也要进行HystrixCommand或HystrixObservableCommand包装构成联级降级策略。
9) 返回成功响应

   }

**** 请求过滤器
实现签名校验、鉴权服务等。

1) 定义过滤器
#+begin_example java
public class AccessFilter extends ZuulFilter {
    private static Logger log = LoggerFactory.getLogger(AccessFilter.class);

    // 决定过滤器在请求的哪个生命周期阶段中执行
    // pre 请求被路由前执行
    @Override
    public String filterType(){
        return "pre";
    }

    // 过滤器执行顺序。当请求在一个生命周期中存在多个过滤器，则会根据该顺序来执行
    @Override
    public int filterOrder(){
        return 0;
    }

    // 是否启用该过滤器
    @Override
    public boolean shouldFilter(){
        return true;
    }

    // 过滤器的具体逻辑
    @Override
    public Object run(){
        RequestContext ctx = RequestContext.getCurrentContext();
        HttpServletRequest request = ctx.getRequest();
        log.info("send {} request to {}",
                 request.getMethod(),
                 request.getRequestURL().toString());

        Object accessToken = request.getParameter("accessToken");
        if (accessToken == null) {
            log.warn("access token is empty");
            ctx.setSendZuulResponse(false); // 让Zuul过滤该请求，不进行路由。
            ctx.setResponseStatusCode(401);
            return null;
        }
        log.info("access token ok");
        return null;
    }
}
#+end_example
2) 使用过滤器
   #+begin_example java
   @EnableZuulProxy
   @SpringCloudApplication
   public class Application {
       public static void main(String[] args) {
          SpringApplication.run(Application.class, args);
       }

       @Bean
       public AccessFilter accessFilter(){
           return new AccessFilter();
       }
   }
   #+end_example

**** 路由详解
1) 传统路由
   通过具体指定每个路由表达式与服务实例的映射关系来实现API网关对外部请求的路由
   1) 单实例配置
      #+begin_example bash
      zulul.routes.<route-id>.path=/api-a/**
      zuul.routes.<route-id>.url=http://localhost:8080/
      #+end_example
   2) 多实例配置
      通过Ribbon实现
      #+begin_example bash
      zulul.routes.<route-id>.path=/api-a/**
      zuul.routes.<route-id>.serviceId=hello-service
      ribbon.erueka.enabled=false
      hello-service.ribbon.listOfServers=http://localhost:8080/,http://localhost:8081/
      #+end_example
2) 服务路由配置
   通过Zuul和Eureka集成整合后，除使用path和serviceId方式配置外，还有如下更简单的方式
   1) 以服务名为URL的path前缀，路由到相同服务名的服务
   #+begin_example bash
   zuul.routes.<service-name>=/<service-name>/**
   #+end_example
   1) Zuul配合Eureka，默认自动为每个服务在Zuul中配置 ~zuul.routes.<service-name>=/<service-name>/**~
      - 不未部分服务创建路由规则 ~zuul.ignored-services=<service-name-regex>~, 表达式匹配成功的服务名称将不被路由。
      - 禁止自动创建路由规则 ~zuul.ignored-services=*~
3) 多版本共存服务的路由配置
   - 场景：服务uerserivce线上运行这v1和v2两个版本，服务名为userservice-v1, userservice-v2。
   - 通用做法：以版本代号作为路由前缀定义路由规则
     - userservice-v1 路由规则应为 ~/v1/userservice/~
     - userservice-v2 路由规则应为 ~/v2/userservice/~
   - 自定义Zuul服务与路由映射关系
     #+begin_example java
     @Bean
     public PatternServiceRouteMapper serviceRouteMapper(){
         return new PatternServiceRouteMapper(
             "(?<name>^.+)-(?<version>v.+$)", // 匹配服务名
             "${version}/${name}"             // 路由路径前缀
         );
     }
     #+end_example
4) 路径匹配
   采用Ant风格
   | 通配符 | 说明                             |
   |--------+----------------------------------|
   | ?      | 匹配任意单个字符                 |
   | *      | 匹配任意数量的字符               |
   | **     | 匹配任意数量的字符，支持多级目录 |
5) 匹配顺序
   采取线性遍历的方式，在请求路径获取到第一个匹配的路由规则之后就会返回并结束匹配过程。
   - properties文件中的路由配置的读取顺序不一定和保存顺序一致
   - yaml文件中的路由配置的读取顺序与保存顺序一致
     #+begin_example yaml
     zuul:
       routes:
         user-service-ext:
           path: /user-service/ext/**
           serviceId: use-service-ext
         user-service:
           path: /user-service/**
           serviceId: use-service
     #+end_example
6) 忽略表达式
   采用Ant风格表达式，针对所有路由匹配忽略规则，如 ~zuul.ignored-patterns=/**/hello/**/~ 。能在控制台或日志中看到没有匹配路由的输出信息。
   #+begin_quote
   o.s.c.n.z.f.pre.PreDecorationFilter : No route found for uri: /api-a/hello
   #+end_quote
7) 路由前缀
   - 统一路由规则前缀： ~zuul.prefix=/<prefix>~
   - 禁止移除代理前缀(默认开启)
     - 统一移除： ~zuul.strip-prefix=false~
     - 单独移除： ~zuul.routes.<route-id>.strip-prefix=false~
8) 本地跳转(Forward)
   #+begin_example bash
   # 请求URL路径为/api-b/hello会转发到/local/hello
   zuul.routes.api-a.path=/api-b/**
   zuul.routes.api-a.url=forward:/local
   #+end_example
9) Cookie与头信息
   默认情况下会过滤掉HTTP请求头信息中的敏感信息，防止被传递到下游的外部服务器。
   - 默认值： ~zuul.sensitiveHeaders=Cookie,Set-Cookie,Authorization~
     - 而 ~zuul.ignoredHeaders~ 则是设置网关不过滤，但下游服务器再转发请求时则会自动被过滤的请求头信息。
   - 不过滤(存在安全隐患)： ~zuul.sensitiveHeaders=~
   - 对指定路由开启自定义敏感头： ~zuul.routes.<router-id>.customSensitiveHeaders=true~
   - 将指定路由的敏感头设置为空： ~zuul.routes.<router-id>.sensitiveHeader=~
10) 重定向问题
    - 问题原因：Spring Security或Shiro等登录完成后会将请求头的Host字段作为响应头字段Location的值，因此登录后会发现访问的是具体Web应用实例的地址。
    - 解决办法：spring-cloud-netflix-core-1.2.x的Zuul引入配置项 ~zuul.addHostHeader=true~ 开启，将Host配置为客户端最初请求的服务断地址
11) Ribbon与Hystrix
    采用path,serviceId配置时，默认会启用Ribbon和Hystrix功能
    - ~hystrix.command.default.execution.isolation.thread.timeoutInMillisecons~ 执行超时，Hystrix会抛出TIMEOUT。
    - ~ribbon.ConnecTimeout~ 创建请求连接超时时间
    - ~ribbon.ReadTimeout~ 创建读取超时时间
    - 当路由转发超时且超时设置小于Hystrix的 ~hystrix.command.default.execution.isolation.thread.timeoutInMillisecons~ 时，会触发重试机制。可通过以下配置关闭重试机制
      #+begin_example bash
      zuul.retryable=false # 全局禁用
      zuul.routes.<route-id>.retryable=false # 针对路由禁用
      #+end_example

* Thymeleaf
https://www.thymeleaf.org/doc/tutorials/3.0/usingthymeleaf.html#creating-and-configuring-the-template-engine

** 优点和缺点
1. 由于是通过在HTML标签上附加额外属性实现动态数据绑定，因此浏览器直接解析thymeleaf模板时会忽略thymeleaf的附加属性，从而以普通HTML文件的方式渲染页面；
2. 可直接套用JSTL、OGNL表达式。

** 语法
#+begin_example html
<!DOCTYPE html>
<html xmlns:th="http://www.thymeleaf.org">
<head>
    <meta http-equiv="Content-Type" content="text/html;charset=UTF-8"/>
</head>
<body>
    <!--/*@thymesVar id="name" type="java.lang.String"*/-->
    <p th:text="'hi, ' + ${name} + '!'">Default Message</p>
</body>
#+end_example
1. 获取变量值 ~${}~ (用法和OGNL一致)
2. with块的取值 ~*{}~
   #+begin_example html
   <ul th:object="${session.user}">
       <li th:text="*{name}">Name</li> <!-- 等价于 ${session.user.name} -->
       <li th:text="*{description}">Description</li> <!-- 等价于 ${session.user.description} -->
   </ul>
   #+end_example
3. 链接表达式 ~@{}~ (配合 ~th:href~, ~th:src~ 使用)
   #+begin_example html
   <!-- /order/detail?orderId=123 -->
   <a href="details.html" th:href="@{/order/detail(orderId=${o.id})}">view</a>
   #+end_example
4. 字符串拼接
   #+begin_example html
   <div th:text="|Welcome to our application, ${user.name}!|">Default Message</div>
   #+end_example
5. 条件判断
   1) if 只有条件成立才会显示元素
      #+begin_example html
      <a th:text="${session.user ? 'Logout' : 'Login'}">Login</a>
      <a th:if="${session.user == null}">Login</a>
      <a th:unless="${session.user != null}">Login</a>
      #+end_example
   2) switch
      #+begin_example html
      <div th:switch="${user.role}">
          <p th:case="'admin'">Admin</p>
          <p th:case="#{global.role}">Yes</p>
          <p th:case="*">Default Value</p>
      </div>
      #+end_example
6. 循环 ~th:each~
   #+begin_example html
   <tr th:each="emp, meta:${emps}">
       <td th:text="${emp.name}">Name</td>
       <td th:text="${emp.description}">Description</td>
   </tr>
   #+end_example
   - 可以遍历如下类型
     - Iterable
     - Enumeration
     - Iterator
     - Map
     - Array
   - meta包含循环属性
     - int index，从0开始的下标
     - int count，元素个数
     - int size，元素个数
     - Object current，当前元素
     - boolean even
     - boolean odd
     - boolean first
     - boolean last
7. JS inline
   #+begin_example html
   <script th:inline="javascript">
       const user = /*[[${user}]]*/ {};
       const age = /*[[${age}]]*/ 20;
       console.log(user);
       console.log(age);
   </script>
   #+end_example
