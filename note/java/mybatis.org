
JPA(Java Persistent API)，作为Java EE 5官方标准的一种ORM规范, 可以是EJB3规范的一部分,并不是ORM产品，具体通用性。规范中对数据库结构提供较完整的封装.
如Hibernat那样从POJO到数据库表的映射关系，通过规范/框架提供的存储逻辑自动生成对应的SQL并自动调用JDBC接口来执行。
Hibernate是JPA的一个实现,而且其功能是JPA的超集.

PO(Persistent Object,持久化对象)，通过该对象可对数据库执行增、删、改的操作，以面向对象的方式操作数据库。

JDBC(Java Data Base Connectivity)是由SUN公司提出的一系列规范接口，具体实现由数据库厂商实现，是一种典型的桥接模式。

DriverManager，操作一次数据库都重新建立一个数据库链接。
DataSource，提供数据库连接池优化对数据库的操作。实现javax.sql.DataSource接口，因此可以为任意第三方数据源，也可以通过JNDI从容器中获取的数据源。

JavaBeans
为符合特定规范的Java类：
1) 公有类
2) 私有字段
3) 公有getter和setter
4) 无参构造方法


MySQL的bigint对应Java的Long。

  - 程序入口
    #+BEGIN_EXAMPLE java
    package org.john;

    public static class App {
        public static void main(String[] args) {
            InputStream configStream = Resources.getResourceAsStream("mybatis-config.xml");
            SqlSessionFactory sessionFactory = new SqlSessionFactoryBuilder().build(configStream);
            SqlSession session = sessionFactory.openSession();

            User user = new User();
            user.setName("john");

            /* 通过SQL映射文件中的namespace+id的方式操作数据库
               缺点:
                 1. 编译器无法在编译时检查字符串指定的namespace+id的合法性;
                 2. SqlSession的insert/update/delete等方法入参均为Object，无法在编译时限制入参类型;
                 3. 非面向对象的编程风格。
             */
            try{
              session.insert("org.john.mapper.UserMapper.save", user);
              session.commit();
            } finally {
              session.close();
            }

            /* 通过Mapper接口的方式操作数据库
               SqlSession.getMapper方法会根据Mapper接口生成和返回代理类org.apache.ibatis.binding.MapperProxy，然后在代理类通过方法签名获取MapperMethod来执行操作
             */
           try{
             UserMapper userMapper = session.getMapper(org.john.mapper.UserMapper.class);
             userMapper.save(user);
             System.out.println(userMapper.class); // 返回的是Proxy
             session.commit();
           } finally {
             session.close();
           }
        }
    }
    #+END_EXAMPLE

* 核心对象

- ~SqlSessionFactoryBuilder~, 读取MyBatis的XML配置文件或预制定的Configuration对象初始化SqlSessionFactory。
  - 读取MyBatis的XML配置文件初始化SqlSessionFactory实例
    1) 通过 ~Resources.getResourceAsStream("mybatis-config.xml")~ 读取XML配置文件并将其构建为输入流
    2) 将输入流传给 ~new SqlSessionFactoryBuilder().build~, 内部构建XMLConfigBuilder实例并通过其方法parse，解析XML配置文件内容并返回 ~org.apache.ibatis.session.Configuration~ 实例
    3) SqlSessionFactoryBuilder根据Configuration实例创建一个DefaultSessionFactory实例，并返回给调用者使用。
- ~SqlSessionFactory~, 为单个数据库映射关系经过编译后的内存镜像。
  - 通过方法 ~new SqlSessionFactoryBuilder().build~ 从XML配置文件或一个预先定制的Configuration实例构建出来。
  - 线性安全。
  - 建议使用单例模式，在应用运行期间常存且避免重复创建。
  #+BEGIN_EXAMPLE java
  package org.john;
  import org.apache.ibatis.io.Resources;
  import org.apache.ibatis.session.SqlSession;
  import org.apache.ibatis.session.SqlSessionFactory;
  import org.apache.ibatis.session.SqlSessionFactoryBuilder;

  public class SqlSessionTemplate {
    private static SqlSessionFactory sqlSessionFactory = null;
    static {
      try{
        InputStream inputStream = Resources.getResourceAsStream("mybatis-config.xml");
        sqlSessionFactory = new SqlSessionFactoryBuilder().build(inputStream);
      } catch (Exception e) {
        e.printStackTrace();
      }
    }

    public static SqlSession getSqlSession(){
      return sqlSessionFactory.openSession();
    }
  }

  public class App {
      public static void main(String[] args) {
      SqlSession session = SqlSessionTemplate.getSqlSession();
    }
  }
  #+END_EXAMPLE

- ~SqlSession~, 持久化操作对象。
  和 ~Connection~ 对象一样为非线程安全对象，因此每次使用时必须重新创建一个。
  - 常用方法
    - ~int insert(String statement)~, statement为在mapper.xml中元素 ~<insert>~ 的id值
    - ~int insert(String statement, Obejct parameter)~, statement为在mapper.xml中元素 ~<insert>~ 的id值
    - ~int update(String statement)~, statement为在mapper.xml中元素 ~<update>~ 的id值
    - ~int update(String statement, Object parameter)~, statement为在mapper.xml中元素 ~<update>~ 的id值
    - ~int delete(String statement)~, statement为在mapper.xml中元素 ~<delete>~ 的id值
    - ~int delete(String statement, Object parameter)~, statement为在mapper.xml中元素 ~<delete>~ 的id值
    - ~<T> T selectOne(String statement)~, statement为在mapper.xml中元素 ~<select>~ 的id值
      注意：若存在多条数据库记录，则抛出异常
    - ~<T> T selectOne(String statement, Object parameter)~, statement为在mapper.xml中元素 ~<select>~ 的id值
      注意：若存在多条数据库记录，则抛出异常
    - ~<E> List<E> selectList(String statement)~, statement为在mapper.xml中元素 ~<select>~ 的id值
    - ~<E> List<E> selectList(String statement, Object parameter)~, statement为在mapper.xml中元素 ~<select>~ 的id值
    - ~<E> List<E> selectList(String statement, Object parameter, RowBounds rowBounds)~, statement为在mapper.xml中元素 ~<select>~ 的id值
      - rowBounds用于分页，属性offset为查询的当前页数，属性limit为显示多少条数据。
    - ~<K,V> Map<K,V> selectMap(String statement, String mapKey)~, statement为在mapper.xml中元素 ~<select>~ 的id值
      - mapKey用于指定表中的一个列名。
    - ~<K,V> Map<K,V> selectMap(String statement, Object parameter, String mapKey)~, statement为在mapper.xml中元素 ~<select>~ 的id值
    - ~<K,V> Map<K,V> selectMap(String statement, Object parameter, String mapKey, RowBounds rowBounds)~, statement为在mapper.xml中元素 ~<select>~ 的id值
    - ~void select(String statement, ResultHandler handler)~, statement为在mapper.xml中元素 ~<select>~ 的id值
      - handler为用于处理查询返回的复杂结果集，通常用于多表查询。
    - ~void select(String statement, Object parameter, ResultHandler handler)~, statement为在mapper.xml中元素 ~<select>~ 的id值
    - ~void select(String statement, Object parameter, RowBounds rowBounds, ResultHandler handler)~, statement为在mapper.xml中元素 ~<select>~ 的id值
    - ~void commit()~ 提交事务，内部实质是 ~this.commit(false)~
    - ~void commit(boolean force)~ 提交事务
      - 若sqlSession的autoCommit为true，即由资源管理器(对于JDBCTransactionMananger而言，则为数据库)自行决定事务提交时刻。此时force设置为true，则表示马上执行事务提交。
      - 若sqlSession的autoCommit为false(默认值)，那么不管force设置为什么，只要执行过CUD操作则会马上执行事务提交。
    - ~void rollback()~ 回滚到上次的提交点，内部实质是 ~this.rollback(false)~
    - ~void rollback(boolean force)~
      - 若sqlSession的autoCommit为true，即由资源管理器(对于JDBCTransactionMananger而言，则为数据库)自行决定事务回滚时刻。此时force设置为true，则表示马上执行事务回滚。
      - 若sqlSession的autoCommit为false(默认值)，那么不管force设置为什么，只要执行过CUD操作则会马上执行事务回滚。
    - ~void close()~
    - ~Connection getConnection()~
    - ~<T> T getMapper(Class<T> type)~, 返回mapper接口的代理对象，该对象关联了SqlSession对象，可以通过该对象直接调用方法操作数据库，参数type是Mapper的接口类型。MyBatis建议通过mapper对象访问MyBatis。
    - ~List<BatchResult> flushStatements()~ 内部调用Statement实例的executeBatch方法，而不是调用Statement实例的flushStatements方法。
      - Statement实例的flushStatements方法会将多条SQL语句推送到数据库并执行，然后当前会话可以看到执行结果，但其它会话无法看到执行结果。
      - Statement实例的executeBatch方法会将多条SQL语句推送到数据库并执行，然后当前会话和其它会话都可以看到执行结果。

** ~SqlSession#commit~ 和 ~SqlSession#close~ 的区别？
   1. ~SqlSession#commit~ 在执行 ~SqlSession#close~ 前可以执行多次，对于JDBCTransactionManager而言即创建多个提交点；
   2. ~SqlSession#close~ 执行时会对未提交的操作执行回滚，然后关闭数据库链接和清空本地缓存等信息。

* MyBatis配置文件

  <?xml version="1.0" encoding="UTF-8"?>
  <!DOCTYPE configuration
    PUBLIC "-//mybatis.org//DTD Config 3.0//EN"
    "http://mybatis.org/dtd/mybatis-3-config.dtd">
  <configuration>
      <!-- 设置内联属性或引入外部配置，并在当前文档中动态替换。
           1. 设置内联属性
           <properties>
             <property name="prop_name" value="prop_value"/>
           </properties>
           2. 引入外部配置的资源定位方式
           resource: 通过类路径引入，如org/john/test.xml。类路径就是编译后以lib或target为父目录的路径
           url: 通过网络或本地路径引入，http://www.test.com/test.xml或file:///c:\test.xml

           加载顺序
           1. 读取property子元素的值
           2. 根据properies元素的resource或url读取属性文件，并覆盖同名属性值
           3. 读取作为方法参数传递的属性值，并覆盖同名属性值


           示例：
           1. 通过方法传递属性值
              InputStream cfgStream = Resources.getResourceAsStream("mybatis-config.xml")
              InputStream propsStream = Resources.getResourceAsStream("db.properties");
              Properties props = new Properties();
              props.load(propsStream);
              SqlSessionFactory factory = new SqlSessionFactoryBuilder().build(cfgStream, props);

           2. 外部db.properties的内容为
                 username=root
                 password=root
              配置文件中其他部分动态替换为
              <property name="username" value="${username}"/>
              <property name="password" value="${password}"/>
        -->
      <properties resource="db.properties"/>
      <!-- 改变MyBatis运行时行为 -->
      <settings>
          <!--开启映射器的缓存-->
          <setting name="cacheEnabled" value="true"/>
          <!--开启关联对象延迟加载，在特定关联关系中可通过设置fetchType属性来覆盖该属性值，默认值为false。
              立即加载的好处是多条SQL一次性发送。
              延迟加载的实现原理是通过动态代理。
            -->
          <setting name="lazyLoadingEnabled" value="true"/>
          <!-- 为true时（默认值），若调用触发某个延迟加载的属性时，会自动把所有延迟加载的属性都一并加载。
               因此要启用延迟加载，那么lazyLoadingEnabled需要设置为true，并且aggressiveLazyLoading为false-->
          <setting name="aggressiveLazyLoading" value="false"/>
          <!--允许单一语句返回多结果集-->
          <setting name="multipleResultSetsEnabled" value="true"/>
          <!--使用列标签代替列名，具体表现依赖数据库驱动-->
          <setting name="useColumnLabel" value="true"/>
          <!--允许JDBC支持自动生成主键-->
          <setting name="useGeneratedKeys" value="false"/>
          <!--设置MyBatis如何自动映射列到字段
              NONE, 取消自动映射
              PARTIAL, 只自动映射没有定义嵌套结果映射的结果集（默认值），要求列名和javaBean属性名一致
              FULL, 自动映射任意复杂的结果集
            -->
          <setting name="autoMappingBehavior" value="PARTIAL"/>
          <!--自动映射时发现未知数据表列的时应如何处理
              NONE, 不处理（默认值）
              WARNING, 输出警告
              FAILING, 映射失败，抛出SqlSessionException
            -->
          <setting name="autoMappingUnknownColumnBehavior" value="WARNING"/>
          <!--配置默认的执行器
              SIMPLE, 普通的执行器（默认值）
              REUSE, 执行器会重用预处理语句(prepared statements)
              BATCH, 执行器将重用语句并执行批量更新
            -->
          <setting name="defaultExecutorType" value="SIMPLE"/>
          <!--超时时间，决定驱动等待数据库响应的秒数-->
          <setting name="defaultStatementTimeout" value="25"/>
          <!--默认返回的结果集的大小-->
          <setting name="defaultFetchSize" value="100"/>
          <!--允许在嵌套语句中使用分页，默认值为false-->
          <setting name="safeRowBoundsEnabled" value="false"/>
          <!--开启自动驼峰命名规则，默认值为false
              即数据库表字段为A_COL，自动对应POJO成员变量aCol
            -->
          <setting name="mapUnderscoreToCamelCase" value="false"/>
          <!--MyBatis3.3+利用本地缓存机制(local cache)防止循环引用(circular references)和加速重复嵌套查询。
              SESSION, 缓存一个会话中执行的所有查询（默认值）,即启用一级缓存
              STATEMENT, 相同SqlSession的不同调用将不会共享数据，即禁用一级缓存
            -->
          <setting name="localCacheScope" value="SESSION"/>
          <!--当没有为参数提供特定的JDBC类型时，为空值(null)指定原生JDBC的类型。
              NULL
              VARCHAR
              OTHER（默认值）,MySQL能识别这个值，但Oracle不能识别因此会导致在Oracle插入空值时会报错，即使在表字段可空
            -->
          <setting name="jdbcTypeForNull" value="OTHER"/>
          <!-- 指定哪个对象的方法触发一次延迟加载，属性值设置为方法名的list集合。
               默认值 equals,clone,hashCode,toString。
               设置为空字符串，则避免无意间触发加载行为。如debug断点调试时，会另起一个线程调用实体的hashCode和toString方法。
          -->
          <setting name="lazyLoadTriggerMethods" value="equals,clone,hashCode,toString"/>
          <!--指定MyBatis生成SQL语句的日志名称的前缀，默认为空
              MyBatis生成的SQL语句,SQL参数和结果条数将以DEBUG输出,
              而返回行列值以TRACE级别输出,
              SQL执行失败时会以ERROR输出SQL和出错原因.

	      默认值：JDK_LOGGING

          <setting name="logImpl" value="LOG4J2"/>

              CGLIB(3.3以前的默认值)
              JAVASSIT(3.3以后的默认值)

          <setting name="proxyFactory" value="CGLIB"/>

          <setting name="callSettersOnNulls" value="false"/>
          <!-- JDK1.8后才能使用，用于启用SQL映射文件入参名称与Mapper接口方法型参名称一致 -->
          <setting name="useActualParamName" value="true"/>
      </settings>

      <typeAliases>
          <!-- 为类型org.john.domain.User设置别名user
               1) 若仅设置type值，则别名为以首字母小写的非限定类型名称
               2) 别名不区分大小写，因此一般用小写

          <typeAlias alias="user" type="org.john.domain.User"/>

               1) 通过@Alias("alias_name")标识的类型，则以注解值作为别名

          在预处理语句(PreparedStatement)中设置一个参数 和 从结果集中取出一个值时，都会通过typeHandler将取出的值从Java类型转换为数据库类型，或从数据库类型转换为Java类型。
          日期和时间的处理在JDK1.8以前一直是个头痛的问题。通常会使用JSR310规范领导者Stephen Colebourne创建的Joda-Time来操作。JDK1.8已经实现全部JSR310规范。
          MyBatis3.4以前需要使用JSR310规范操作日期时间，那么就要手工引入mybatis-typehandlers-jsr310包的typeHandlers。
          MyBatis3.4后已经内置并自动引入JSR310的typeHandler了。
        -->
      <typeHandlers>
          <!--
            1. 通过jdbcType和javaType作为匹配标识，注册自定义/第三方类型转换器。
               若SQL映射文件中result节点的jdbcType、javaType和typeHandler节点的匹配时，则自动采用handler属性执行的类型转换器。
               若SQL映射文件中变量绑定如 #{roleName javaType=string, jdbcType=VARCHAR}，则自动采用handler属性执行的类型转换器。
            2. 若SQL映射文件中result节点指定typeHandler属性则直接将其作为类型转换器。
               若SQL映射文件中变量绑定如 #{roleName ,typeHandler=com.john.typeHandler.MyStringTypeHandler}，则直接将其作为类型转换器。
            -->
          <typeHandler jdbcType="VARCHAR" javaType="string" handler="com.john.typeHandler.MyStringTypeHandler"/>

          <!-- 也可通过包扫描注册，那么jdbcType和javaType就采用注解MappedTypes和MappedJdbcTypes的值 -->
          <package name="com.john.typeHandler"/>
      </typeHandlers>

           能够在以下4大核心对象的方法前后进行拦截
           Executor(update,query,flushStatements,commit,rollback,getTransaction,close,isClosed)
           ParameterHandler(getParameterObject,setParameters)

           在创建结果对象实例时，会使用对象工厂去构建POJO。

      <environments default="dev">
          <environment id="dev">

* MyBatis和数据库间的日期类型映射

** 4种数据库中的日期类型
   | 类型      | 说明         |
   |-----------+--------------|
   | date      | 年月日       |
   | datetime  | 年月日时分秒 |
   | timestamp | 年月日时分秒 |
   | time      | 时分秒       |
   而 ~datetime~ 和 ~timestamp~ 在数据库中的存储结构不同，其中 ~timestamp~ 更节省存储空间。

** 4种Java日期类型
   | Java类型           | 父类             | 精度             | 示例                         |
   |--------------------+------------------+------------------+------------------------------|
   | java.util.Date     | java.lang.Object | 年月日时分秒     | Sun Jul 17 09:26:07 CST 2016 |
   | java.sql.Date      | java.util.Date   | 年月日           | 2016-07-17                   |
   | java.sql.Timestamp | java.util.Date   | 年月日时分秒纳秒 | 2016-07-17 09:26:07.342      |
   | java.sql.Time      | java.util.Date   | 时分秒           | 09:26:07                     |
   其中 ~CST~ 为China Standard Time的缩写，即表示为东八区北京时间。
   注意：其实 ~java.sql.Date~, ~java.sql.Timestamp~ 和 ~java.sql.Time~ 中都保存年月日时分秒的信息，且将 ~java.util.Date~ 实例转换为以上类型时年月日时分秒均完整保留，只是读取写入和转换时会根据类自身的特点省去或将某部分的值设置为特殊值。

** Java日期类型和Oracle日期类型间的转换
   1. ~java.util.Date~, ~java.sql.Timestamp~
      - to ~date~ 丢失时分秒
      - from ~date~ 时分秒以00填充
      - to ~datetime~ 无损转换
      - from ~datetime~ 无损转换
      - to ~timestamp~ 无损转换
      - from ~timestamp~ 无损转换
      - to ~time~ 丢失年月日
      - from ~time~ 年月日固定为1970-01-01
   2. ~java.sql.Date~
      - to ~date~ 无损转换
      - from ~date~ 无损转换
      - to ~datetime~ 时分秒以00填充
      - from ~datetime~ 时分秒以00填充
      - to ~timestamp~ 时分秒以00填充
      - from ~timestamp~ 时分秒以00填充
      - to ~time~ 转换失败
      - from ~time~ 转换失败
   3. ~java.sql.Time~
      - to/from ~date~, ~datetime~ 和 ~timestamp~ 都报错
      - to/from ~time~ 无损转换

** MyBatis实体类日期字段默认采用 ~java.util.Date~ 类型
   由于 ~java.util.Date~ 包含年月日时分秒信息，因此可以与数据库4种日期类型字段进行无损转换。
   1. MyBatis会根据数据库字段类型自动执行转换；
   2. 可通过设置jdbcType，增加一层数据转换规则。并且jdbcType的类型可以与数据表字段的类型不一致。
      - 如jdbcType设置为Date，而数据库表字段类型为Datetime，那么转换过程中就会丢失时分秒信息。

* 事务管理
** 事务4个特性
  1) 原子性(Atomicity), 事务作为应用中不可再分割的最小逻辑执行单位。
  2) 一致性(Consistency), 数据库从一种一致性状态，变到另一种一致性状态。一致性通过原子性来保证。
  3) 隔离性(Isolation), 并发执行的事务之间互不干扰，即事务操作过程中的中间状态无法被其他事务访问。
  4) 持续性(Durability/Persistence), 事务一旦提交，其对数据的任何草走都将记录到永久存簇器中。

  事务管理实际是依赖底层资源服务(如MySQL等)提供实现，而JdbcTranscation则是对数据库事务的封装而已，而MyBatis、Spring的的事务管理则是对基于JdbcTranscation等对事务管理提供更高阶便利的API。

** 第一类丢失更新
   | 时间轴 | 事务A           | 事务B           | 备注              |
   |--------+-----------------+-----------------+-------------------|
   | T1     | 读取余额为200元 |                 | 初始化余额为200元 |
   | T2     |                 | 读取余额为200元 |                   |
   | T3     |                 | 更新余额为300元 |                   |
   | T4     | 修改余额为400元 | 提交            | 余额更新为300元   |
   | T5     | 发生回滚        |                 | 余额恢复回200元   |
   上述事务B的提交结果因事务A的回滚而被擦除，产生数据不一致的场景被称为第一类丢失更新。
   现在的数据库均已解决第一类丢失更新，而隔离级别则是用于处理第二类丢失更新。

** 第二类丢失更新
   1. *脏读* 一个事务能读取另外一个事务未提交的数据
      | 时间轴 | 事务A           | 事务B           | 备注            |
      |--------+-----------------+-----------------+-----------------|
      | T1     | 读取余额200元   |                 | 初始化余额200元 |
      | T2     | 修改余额为100元 | 读取余额为100元 |                 |
      | T3     | 回滚            | 修改为0元       |                 |
      | T4     |                 | 提交            | 余额为0元       |
   2. *不可重复读* 一个事务还没结束，其之前读取过的数据已经被修改
      | 时间轴 | 事务A         | 事务B           | 备注                   |
      |--------+---------------+-----------------+------------------------|
      | T1     | 读取余额200元 |                 | 初始化余额200元        |
      | T2     | 余额减少100元 | 读取余额为200元 |                        |
      | T3     | 提交          |                 |                        |
      | T4     |               | 余额减少101元   | 余额不足，执行扣减失败 |
   3. *幻读* 一个事务还没结束，其读取的记录数被删减或增加
      | 时间轴 | 事务A          | 事务B    | 备注 |
      |--------+----------------+----------+------|
      | T1     | 统计记录数为40 |          |      |
      | T2     |                | 插入记录 |      |
      | T3     |                | 提交     |      |
      | T4     | 统计记录数为41 |          |      |
      | T5     | 提交           |          |      |

** 事务的隔离级别
   | 事务隔离级别     | 脏读 | 不可重复读 | 幻读 | 解释                                                                                                                |
   |------------------+------+------------+------+---------------------------------------------------------------------------------------------------------------------|
   | READ UNCOMMITTED | Y    | Y          | Y    | 允许事务读取其它事务未提交的变更。存在脏读、不可重复读和幻读的问题                                                  |
   | READ COMMITTED   | N    | Y          | Y    | 只允许事务读取其它事务已经提交的变更。存在不可重复读和幻读的问题                                                    |
   | REPEATABLE READ  | N    | N          | Y    | 在事务持续期间，通过禁止其它事务对指定字段提交变更(CUD)，从而确保该事务可以从该字段重复读取相同的值。存在幻读的问题 |
   | SERIALIZABLE     | N    | N          | N    | 在事务持续期间，通过禁止其它事务对指定表提交变更(CUD)，从而确保该事务可以从该表重复读取相同的行数。                 |

   Oracle支持 ~READ COMMITED~ (默认) 和 ~SERIALIZABLE~ 2种事务级别
   MySQL支持全部，默认为 ~REPEATABLE READ~ 。事务设置命令为
   - READ UNCOMMITTED, ~set session transaction isolation level read uncommitted~
   - READ COMMITTED, ~set session transaction isolation level read committed~
   - REPEATABLE READ, ~set session transaction isolation level repeatable read~
   - SERIALIZABLE, ~set session transaction isolation level serializable~

** 创建事务实例的过程
1) 根据mybatis-config.xml中 ~<transactionManager>~ 的属性type值创建JdbcTransactionFactory或ManagedTransactionFactory实例
   - 底层源码
     #+BEGIN_EXAMPLE java
     // <transactionManager>节点解析方法
     private TransactionFactory transactionManagedElement(XNode context) throws Exception {
       /* 前面已经通过下面语句注册别名JDBC和MANAGED对应的工厂类
          typeAliasRegistry.registerAlias("JDBC", JdbcTransactionFactory.class);
          typeAliasRegistry.registerAlias("MANAGED", ManagedTransactionFactory.class);
        */
       if (context != null){
         String type = context.getStringAttribute("type");
         Properties props = context.getChildrenAsProperties();
         TransactionFactory factory = (TransactionFactory) resolveClass(type).newInstance();
         factory.setProperties(props);
         return factory;
       }
       throw new BuilderException("Enviroment declaration requries a TransactionFactory.");
     }
     #+END_EXAMPLE
2) 然后通过TransactionFactory实例，以Connection实例或DataSource实例、隔离级别、自动提交作为入参创建Transaction实例。
3) JdbcTransaction和ManagedTransaction的区别
   1) JdbcTranscation实际使用的是java.sql.Connection提供的事务操作，依赖数据库本身的支持。
   2) ManagedTransaction则将事务管理交由应用容器(Weblogic等)管理，自身不提供commit和rollback操作，而是由容器统一管理。

* 数据缓存机制
  分为一级和二级缓存：
  1) 一级缓存为SqlSession级别的缓存。
  2) 二级缓存为namespace级别的缓存，多个SqlSession共享的。

** 一级缓存(SqlSession级别)
   - 默认开启
   - 缓存数据保存在SqlSession实例内部的Map实例中
   - 缓存逻辑
     1) 当在同一个SqlSession实例执行相同的SQL语句时，第一次执行后会将数据以对象id为键保存到内部的Map中，后续查询则直接从缓存中读取；
     2) 若SqlSession实例执行DML(insert,update,delete)并提交到数据，那么一级缓存将被自动清空。
   - 缓存失效的4种情况
     1) 查询不在同一个SqlSession中执行
     2) 查询在同一个SqlSession中执行，但查询条件不同
     3) 查询在同一个SqlSession中执行，多次查询操作之间插入DML操作
        insert/update/delete节点提供一个flushCache属性，用于设置该操作是否会清除一级和二级缓存，默认值为true。而select节点也提供flushCache属性，不过默认值为false。
     4) 手动清除一级缓存
        #+BEGIN_EXAMPLE java
        SqlSession sqlSession = SqlSessionFactory.openSession();
        try{
          sqlSession.selectOne("xxx");
          sqlSession.clearCache(); // 手动清理一级缓存
          sqlSession.selectOne("xxx");
        }
        finally {
          sqlSession.close();
        }
        #+END_EXAMPLE

** 二级缓存(mapper级别)
   - 默认关闭
     - 在mybatis-config.xml中通过 ~<setting name="cacheEnabled" value="true"/>~ 开启二级缓存
     - 在使用二级缓存的mapper.xml中通过 ~<cache eviction="LRU" flushInterval="60000" size="512" readOnly="true"/>~ 开启
       - flushInterval, 刷新间隔毫秒数。默认值为unset，即不刷新。
       - size, 缓存数目。默认值1024
       - readOnly, 默认值为false。只读的缓存会给调用者返回相同的实例，因为这些对象不能被修改。可读写的缓存会返回缓存对象的拷贝。
       - eviction, 回收策略，默认值为LRU
         - LRU, 最近最少用策略
         - FIFO, 先进先出策略
         - SOFT, 软引用策略，移除基于垃圾回收器状态和软引用规则的对象
         - WEAK, 弱引用策略，更积极地移除基于垃圾回收器状态和弱引用规则的对象
   - *注意* 被二级缓存的对象及其父类必须实现java.io.Serializable接口，因为二级缓存数据存储介质不一定在内存，可能在硬盘或远程服务器。
   - ~select~ 节点提供useCache属性用于在当前namespace已启用二级缓存的前提下，单独启用或禁用二级缓存，默认为true。
   - 缓存数据保存在Mapper内部的HashMap实例中
   - 缓存逻辑
     不同的SqlSession实例多次执行相同的mapper的namespace下的sql语句，且传递相同的参数，导致最终生成相同的sql语句，那么第一执行后将结果缓存起来，后续查询均从缓存读取。
   - 缓存失效
     各个namespace的二级缓存失效是独立的，互不干扰。
     1) 多次查询操作之间插入DML操作
        insert/update/delete节点提供一个flushCache属性，用于设置该操作是否会清除一级和二级缓存，默认值为true。而select节点也提供flushCache属性，不过默认值为false。
     2) 由于符合eviction条件被清理

*** SqlSession、SqlSessionManager和SqlSessionTemplate的区别？

**** SqlSession
     SqlSession的默认实现类为DefautlSqlSession，为非线性安全，因此不能直接以单例模式来使用。
     因此
     1. MyBatis中使用SqlSession的方式为，每次都在方法内从SqlSessionFactory实例创建全新的SqlSession实例；
     2. 使用完毕必须手动调用 ~SqlSession#close()~ 释放链接资源。

**** SqlSessionManager
     1. 实现了SqlSession和SqlSessionFactory接口，对DefaultSqlSessionFactory进行封装，对DefaultSqlSession进行管理；
     2. 通过相同线程使用同一个SqlSession实例，减少每个方法内通过SqlSessionFactory重新创建SqlSession实例的性能损耗；
     3. 使用完毕必须手动调用 ~close()~ 释放链接资源。


     使用示例
     1. 配置文件
        #+begin_src java
        @Bean
        SqlSessionManager sqlSessionManager(SqlSessionFactory factory) {
            return SqlSessionManager.newInstance(factory);
        }
        #+end_src
     2. Service层
        #+begin_src java
        @Autowire
        SqlSessionManager ssm;

        public void doWithSSM(List<User> users) {
            // 若当前线程没有对应的SqlSession实例则从DefaultSqlSessionFactory创建一个
            if (!ssm.isManagedSessionStarted()) {
                ssm.startManagedSession();
            }

            // 两个方法使用同一个SqlSession实例操作数据库
            try {
                do1(users);
                do2(users);
            }
            finally {
                ssm.close(); // 必须手动释放链接资源
            }
        }

        public void doWithoutSSM1() {
            // do1和do2独立创建SqlSession实例操作数据库
            do1(users);
            do2(users);
        }

        public void doWithoutSSM2() {
            // 显式从SqlSessionManager创建SqlSession实例，即将其充当SqlSessionFactory使用。
            SqlSession sqlSession = ssm.openSession();
            // ......
        }

        public void do1(List<User> users) {
            for (User user : users) {
                ssm.save("xx.xx.saveUser", user);
            }

            // 若不是使用线程绑定的SqlSession实例操作数据库，在每次执行操作后，SqlSessionManager实例会自动提交/回滚并释放链接资源，若此时再调用close方法则会报错。
            if (ssm.isManagedSessionStarted()) {
                ssm.commit();
            }
        }

        public void do2(List<User> users) {
            for (User user : users) {
                ssm.save("xx.xx.saveUser", user);
            }

            // 若不是使用线程绑定的SqlSession实例操作数据库，在每次执行操作后，SqlSessionManager实例会自动提交/回滚并释放链接资源，若此时再调用close方法则会报错。
            if (ssm.isManagedSessionStarted()) {
                ssm.commit();
            }
        }
        #+end_src

**** SqlSessionTemplate
     SqlSessionTemplate是线性安全的，通过TransactionSynchronizationManager中的ThreadLocal保存线程对应的SqlSession实例，因此SqlSessionTemplate实例可被多个DAO共享使用。
     相对SqlSessionManager的差异：
     1. SqlSessionManager由使用者决定是否线程内共用SqlSession实例；而SqlSessionTemplate则由Spring事务管理器决定是否共用SqlSession实例，事务内部共用SqlSession实例，非事务不共用。同一个事务，内部存在新开线程则不共用SqlSession实例。
     2. SqlSessionTemplate额外提供线程专用的SqlSession实例的引用计数。
     3. SqlSessionTemplate位于 ~org.mybatis.spring~ 下用于集成Spring，因此集成Spring的项目中都是采用SqlSessionTemplate而不是SqlSessionManager。
     4. SqlSessionTemplate不用手动调用 ~close~ 释放链接资源。

**** 为什么SqlSessionTemplate不用手动调用 ~close~ 释放链接资源？

***** 通过 ~@Transactional~ 或 XML配置启用Spring事务的场景
      ~SqlSessionHolder extends ResourceHolderSupport~ 对SqlSession进行封装，提供引用计数、事务帮助方法（如超时处理）。即数据库归属资源管理器管理。
      ~SqlSessionHolder TransactionSynchronizationManager.getResource(SqlSessionFactory)~ 获取线程局部资源。

****** TransactionSynchronizationManager 和 TransactionSynchronization
       1. TransactionSynchronizationManager是位于 ~org.springframework.transaction.support~ 下的抽个类，内部字段均为类静态字段，且为线程局部变量，用于保存当前事务名称、状态和相关的资源实例。因此整个应用操作的是相同的TransactionSynchronizationManager的相同字段；
       2. TransactionSynchronizationManager 和 TransactionSynchronization构成Spring事务变化的订阅发布模式，用于与外界如MyBatis的事务进行联动；
       3. TransactionSynchronizationManager 是注册中心，而TransactionSynchronization则用于事件订阅和事件处理函数绑定；
       4. TransactionSynchronizationAdapter是TransactionSynchronization的实现类，提供各回调方法的空空实现，那么业务实现类仅需继承TransactionSynchronizationAdapter并实现部分需要的回调方法即可；
          1. resume
          2. suspend
          3. flush
          4. beforeCommit
          5. beforeCompletion
          6. afterCommit
          7. afterCompletion
       5. TransactionSynchronizationUtils内部的triggerAfterCommit等方法会遍历TransactionSynchronizationManager.synchronizations字段，并调用相应的回调方法；
          - 注意：Spring内部(如DataSourceTransactionManager的父类AbstractPlatformTransactionManager)会使用TransactionSynchronizationUtils触发TransactionSynchronizationManager相关的事件。
       6. 通过 ~@Transactional~ 或 XML配置都会创建TransactinSynchronization实例并通过 ~TransactionSynchronizationManager.registerSynchronization~ 方法保存到TransactionSynchronizationManager.synchronizations字段中。

****** 不用手动调用 ~close~ 释放链接资源的原因
       1. 调用SqlSessionTemplate的insert等方法时，实际是调用代理类SqlSessionInterceptor的invoke方法；
       2. SqlSessionInterceptor的invoke方法内部会调用SqlSessionUtils.getSqlSession方法获取DefaultSqlSession实例；
       3. 若当前线程中没有DefaultSqlSession实例则重新创建一个新的，并执行如下动作
          1. 该SqlSession实例缓存到TransactionSynchronizationManager的resources中，并为封装SqlSession的SqlSessionHolder实例的引用计数器加1；
          2. 创建TransactionSynchronization的子类SqlSessionUtils.SqlSessionSynchronization，并注册到TransactionSynchronizationManager.synchronizations。
       4. 然后SqlSessionInterceptor的invoke方法则利用DefaultSqlSession实例执行数据库操作；
       5. SqlSessionInterceptor的invoke方法最后会调用 ~SqlSessionUtils.closeSqlSession~ 方法关闭SqlSession实例。
          注意：这里并不是直接调用 ~sqlSession.close()~ 来关闭链接，而是将封装SqlSession的SqlSessionHolder实例的引用计数器减1；
       6. 当Spring提交事务(AbstractPlatformTransactionManager.processCommit，会调用TransactionSynchronizationUtils.triggerAfterCommit方法调用SqlSessionUtils.SqlSessionSynchronization的afterCommit回调方法。
          注意：当封装SqlSession的SqlSessionHolder实例的引用计数器为0时，就会调用SqlSession实例的close方法释放链接资源。

***** 没有启用Spring事务的场景
       1. 调用SqlSessionTemplate的insert等方法时，实际是调用代理类SqlSessionInterceptor的invoke方法；
       2. SqlSessionInterceptor的invoke方法内部会调用SqlSessionUtils.getSqlSession方法获取DefaultSqlSession实例；
       3. 若当前线程中没有DefaultSqlSession实例则重新创建一个新的，但由于没有启用Spring事务，因此不注册到TransactionSynchronizationManager中；
       4. 然后SqlSessionInterceptor的invoke方法则利用DefaultSqlSession实例执行数据库操作；
       5. SqlSessionInterceptor的invoke方法最后会调用 ~SqlSessionUtils.closeSqlSession~ 方法关闭SqlSession实例。
          注意：由于不启用Spring事务，因此是直接调用 ~sqlSession.close()~ 来关闭链接；

      若不启用Spring事务，则每次调用DefaultSqlSession的操作后就会执行DefaultSqlSession.close方法，那么和使用Executor.SIMPLE是一样了。

* 4大核心对象
Mapper执行过程就是通过4大核心对象来完成数据库操作和结果返回：
1) Executor(update,query,flushStatements,commit,rollback,getTransaction,close,isClosed)
   由它来调度StatementHandler,ParameterHandler和ResultSetHandler等来执行对应的SQL。SqlSession(DefaultSqlSession)的commit、rollback、flushStatements和close等都是调用Executor实例的对应方法。
2) StatementHandler(prepare,parameterize,batch,update,query)
   用通过它使用数据库的Statement(PreparedStatement)执行操作，是4大核心对象的核心。
3) ParameterHandler(getParameterObject,setParameters)
   用于SQL对参数的处理。
4) ResultSetHandler(handleResultSets,handleOutputParameters)
   用于进行最后数据集(ResultSet)的封装返回处理

** TODO Executor
内置3种执行器
1) SIMPLE, 简易执行器，默认的执行器. 每执行一次sql,就开启一个Statement对象,用完立即关闭Statement对象.
2) REUSE, 重用预处理语句的执行器. 执行sql时,以sql为可以查找Statement对象,不存在则创建,用完不关闭Statement对象并存放到map中,供下次使用.
   - 对于复杂且占位符多的SQL，其创建一个PrepareStatement实例的成本较高，因此REUSE比SIMPLE的性能要高；
   - 和很多数据库连接池库中的PSCache概念相同，但REUSE仅针对同一个会话重用PreparedStatement实例，因此提升效果有限。
3) BATCH, 批量处理的执行器.将DML语句添加到批处理中(addBatch),等待统一执行executeBatch.即将多个Statement对象加入到addBatch,然后统一执行executeBatch方法.
   - BATCH的PreparedStatement实例并没有得到重用
   - BATCH的事务不能自动提交，必须调用commit方法

    private SqlSession openSessionFromDataSource(ExecutorType execType, TransactionIsolationLevel level, boolean autoCommit) {
        Transaction tx = null;

        DefaultSqlSession session;
        try {
            Environment environment = this.configuration.getEnvironment();
            TransactionFactory transactionFactory = this.getTransactionFactoryFromEnvironment(environment);
            tx = transactionFactory.newTransaction(environment.getDataSource(), level, autoCommit);

            Executor executor = this.configuration.newExecutor(tx, execType);
            session = new DefaultSqlSession(this.configuration, executor, autoCommit);

        } catch (Exception e) {
            this.closeTransaction(tx);
            throw ExceptionFactory.wrapException(......);
        } finally {
            ErrorContext.instance.reset();
        }

        return session;
    }

    public Executor newExecutor(Transaction transaction, ExecutorType executorType) {
      executorType = executorType == null ? defaultExecutorType : executorType;
      executorType = executorType == null ? ExecutorType.SIMPLE : executortype;
      Executor executor;
      if (ExecutorType.BATCH == executorType){
        executor = new BatchExecutor(this, transaction);
      }
      else if (ExecutorType.REUSE == executorType){
        executor = new ReuseExecutor(this, transaction);
      }

      }

      // 启动二级缓存时
      if (cacheEnabled){

      }
      // 插件链，为执行器生成一层层动态代理对象，来附加上插件逻辑。
      executor = (Executor) interceptorChain.pluginAll(executor);

  public <E> List<E> doQuery(MappedStatement ms, Object parameter, RowBounds rowBounds, ResultHandler resultHandler, BoundSql boundSql)

    switch (mappedStatement.getStatementType()){
      case STATEMENT:
        delegate = new SimpleStatementHandler(executor, mappedStatement, parameterObject, rowBounds, resultHandler, boundSql);
        break;
      case PREPARED:
        delegate = new PreparedStatementHandler(executor, mappedStatement, parameterObject, rowBounds, resultHandler, boundSql);
        break;
      case CALLABLE:
        delegate = new CallableStatementHandler(executor, mappedStatement, parameterObject, rowBounds, resultHandler, boundSql);
        break;
      default:
        throw new ExecutorException("Unknown statement type: " + mappedStatement.getStatementType());
    }
    // .......
  }
}

public abstract class BaseStatementHandler implements StatementHandler{
  @Override
  public Statement prepare(Connection connection) throws SQLException {
    ErrorContext.instance().sql(boundSql.getSql());
    Statement stmt = null;
    try{
      // instantiateStatement会通过boundSql.getSql()获取SQL语句然后进行预编译
      stmt = instantiateStatement(connection);
      setStatementTimeout(statement);
      setFetchSize(statement);
      return stmt;
    }
    catch (SQLException e){
      closeStatement(stmt);
      throw e;
    }
    catch (Exception e){
      closeStatement(stmt);
      throw new ExecutorException("Error preparing statement. Cause: " + e, e);
    }
  }

  @Override
  public void parameterize(Statement statement) throws SQLException {
    parameterHandler.setParameters((PreparedStatement) statement);
  }

  @Override
  public <E> List<E> query(Statement statement, ResultHandler resultHandler) throws SQLException {
    PreparedStatement ps = (PreparedStatement) statement;
    ps.execute();
    return resultSetHandler.<E>handlerResultSets(ps);
  }
}

#+BEGIN_EXAMPLE java
public class Plugin implements InvocationHandler {
  public static Object wrap(Object target, Interceptor interceptor){
    Map<Class<?>, Set<Method>> singatureMap = getSignatureMap(interceptor);
    Class<?> type = target.getClass();
    Class<?>[] interfaces = getAllInterfaces(type, singatureMap);
    if (interfaces.length > 0){
      return Proxy.newProxyInstance(
        type.getClassLoader(),
        interfaces,
        new Plugin(target, interceptor, singatureMap)
      );
    }

    return target;
  }

  public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
    try{
      Set<Method> methods = singatureMap.get(method.getDeclaringClass());
      if (methods != null && methods.contains(method)){
        return interceptor.intercept(new Invocation(target, method, args));
      }
      return method.invoke(target, args);
    }
    catch (Exception e){
      throw ExcetionUtil.unwrapThrowable(e);
    }
  }
}

** QueryWrapper和UpdateWrapper的区别
   *类关系：*
   LambdaQueryWrapper -> AbstractLambdaWrapper -> AbstractWrapper -> Wrapper
   LambdaUpdateWrapper -> AbstractLambdaWrapper -> AbstractWrapper -> Wrapper
   QueryWrapper -> AbstractWrapper -> Wrapper
   UpdateWrapper -> AbstractWrapper -> Wrapper

   *类说明：*
   Wrapper：条件构造抽象类
   AbstractWrapper：用于查询条件封装，生成SQL的Where条件
   AbstractLambdaWrapper：Lambda语法使用Wrapper统一处理解析Lambda获取Column
   QueryLambdaWrapper：Lambda语法使用的查询Wrapper
   UpdateLambdaWrapper：Lambda语法使用的更新Wrapper
   QueryWrapper：Entity对象封装的查询Wrapper
   UpdateWrapper：Entity对象封装的更新Wrapper

   *内部特殊方法说明：*
   - ~apply~ 用于拼接SQL，可用于调用数据库函数保证不存在SQL注入风险使用动态入参。
     #+begin_src java
     // 会生成SQL date_format(dateColumn, '%Y-%m-%d') = '2008-08-08'
     apply("date_format(dateColumn, '%Y-%m-%d') = {0}", "2008-08-08")
     #+end_src
   - ~last~ 无视优化规则直接拼接SQL到最后，仅最后一次调用有效，前面的调用都将被忽略。存在SQL注入风险。
     #+begin_src java
     last("limit 1")
     #+end_src
   - ~or~ 拼接OR，默认使用AND连接。
     #+begin_src java
     eq("id", 1).eq("name", "john") // 生成 id=1 and name = 'john'
     eq("id", 1).or().eq("name", "john") // 生成 id=1 or name = 'john'
     #+end_src
   - ~and~ AND嵌套
     #+begin_src java
     and(i -> i.eq("id", 1).eq("name", "john")) // 生成 and (id=1 and name = 'john')
     #+end_src
   - ~nested~ 不带AND或OR的嵌套
     #+begin_src java
     nested(i -> i.eq("id", 1).eq("name", "john")) // 生成 (id=1 and name = 'john')
     #+end_src
   - ~having~ 提供不存在SQL注入风险的动态入参特性
     #+begin_src java
     having("sum(age) > {0}", 1)
     #+end_src
   - ~inSql~
     #+begin_src java
     inSql("select id from users") // 生成 in (select id from users)
     #+end_src
   - ~exists~
     #+begin_src java
     exists("select id from users where id='1'") // 生成 exists (select id from users where id = '1')
     #+end_src

*** 针对null值的插入、更新和查询策略
    插入(insertStrategy), 更新(insertStrategy) 和查询(whereStrategy) 取值范围：
    1. FieldStrategy.DEFAULT 若配置在application.yml文件中则表示NOT_NULL，若配置在注解中则表示继承application.yml中的配置；
    2. FieldStrategy.IGNORED 不处理，插入和更新时将数据表字段设置为null，查询时采用 ~IS NULL~ 查询子句；
    3. FieldStrategy.NEVER
    4. FieldStrategy.NOT_EMPTY 若为NULL或空字符串则忽略而不影响数据表操作
    5. FieldStrategy.NOT_NULL 若为NULL则忽略而不影响数据表操作

    application.yml的配置格式为：
    #+begin_src yaml
    mybatis-plus:
        global-config:
            db-config:
                insert-strategy: IGNORED
                update-strategy: IGNORED
                where-strategy: IGNORED
    #+end_src

*** 根据ID执行批处理
    通过继承ServiceImpl并使用其中的
    - ~boolean saveBatch(Collection<T> entityList, int batchSize=1000)~
      - 批处理中调用 ~baseMapper.insert~ 插入数据
    - ~boolean saveOrUpdateBatch(Collection<T> entityList, int batchSize=1000)~
      - 批处理中调用 ~baseMapper.selectById~ 根据ID逐个PO实例判断是更新还是插入，对于存在的PO则调用 ~baseMapper.updateById~ 否则调用 ~baseMapper.insert~
    - ~boolean updateBatchById(Collection<T> entityList, int batchSize=1000)~
      - 批处理中调用 ~baseMapper.updateById~ 更新数据

                return this.executeBatch(users, (sqlSession, user) -> {

                    mapper.update(user, new QueryWrapper<User>().eq("name", user.getName()));

                return this.executeBatch(users, (sqlSession, user) -> {
                    UserMapper mapper = sqlSession.getMapper(UserMapper.class);
                    if (StringUtils.checkValNull(user.getName()) || Objects.isNull(this.baseMapper.selectOne(new QueryWrapper<User>().eq("name", user.getName())))) {

                    }
                    else {

                    }
                });

            @Autowire
            SqlSessionFactory sqlSessionFactory;

            public boolean updateUserBatch2(List<User> users) {
                SqlSession sqlSession = null;
                try {
                    sqlSession = sqlSessionFactory.openSession(ExecutorType.BATCH);
                    UserMapper userMapper = sqlSession.getMapper(UserMapper.class);
                    for (User user : users) {
                        userMapper.update(user, new QueryWrapper<User>().eq("name", user.getName()));

                    sqlSession.commit();
                }
                finally {
                    if (sqlSession != null) sqlSession.close(); // 必须手动关闭连接，否则会导致连接资源耗尽。
                }
            }

    XA standard(X/Open Distributed Transaction Processing(DTP)) which is brought by X/Open is the earliest distributed transaction.

    AP(Application Program)
    - AP uses resources from a set of RMs.
    - AP defines transaction boundaries through the TX interface.

    - use XA protocol for bidirectional streaming between RMs and TM.

**** TODO Seata
     Developed by Alibaba Group and Ant Finance.
     Models:
     1. TM(Transaction Manager)
        - needs to be deployed with user application in the form of jar packages.
        - Establish long connections with TC and keep RPC throughout the transaction.
        - TM is the *initiator* of global transaction, which is in charge of begin and commit/rollback of global transaction.
     2. RM(Resource Manager)s
        - needs to be deployed with user application in the form of jar packages.
        - Establish long connections with TC and keep RPC throughout the transaction.
        - RMs are the participants of global transaction, which are in charge of reporting the execution results of branch transaction.
     3. TC(Transaction Coordinator)
        - TC is an independent service that needs to be deployed separately.
        - TC executes commit/rollback of branch transaction.


     A typical lifecycle of Seata managed distributed transaction:
     1. TM asks TC to begin a new global transaction. TC generates a XID representing the global transaction.
     2. XID is propagated through micro-services' invoke chain.
     3. RM register local transaction as a branch of the corresponding global transaction of XID to TC.
     4. TM asks TC for commit or rollback the corresponding global transaction of XID.
     5. TC drives all branch transactions under the corresponding global transaction of XID to finish branch commit or rollback.

** 绑定表(Binding Table)
   如user_info和user_detail两个表均以user_id水平分片，那么当执行关联查询(correlating query)时，ShardingJDBC会自动将这两表设置为绑定表关系。
   那么执行联表查询 ~select a.*, b.* from user_info a join user_detail b on(a.user_id = b.user_id) where a.user_id=12~ 根据user_id执行分片表达式锁定实际的SQL语句为 ~select a.*, b.* from user_info_1 a join user_detail_1 b on(a.user_id = b.user_id) where a.user_id=12~ ；
   而不是要 ~from user_info_1 a join user_detail_1 b on(a.user_id = b.user_id)~ 和 ~from user_info_1 a join user_detail_2 b on(a.user_id = b.user_id)~ 这样执行多个联表查询语句。

      - inline expression 可用的配置项目
        1. Data Node Configuration

   4. hint 对应HintShardingStrategy。通过Hint而非逻辑SQL中的分片键执行SQL路由。

   5. none 不分片策略，对应NoneShardingStrategy，不分片的策略。

*** ShardingSphere内置的2种Built-In Key Generator
    1. UUID
       use ~UUID.randomUUID()~ to generate the distributed key.
    2. SNOWFLAKE(publish by Twitter)
       default snowflake algorithm generates 64bit long integral data. Snowflake algorithm can ensure sequences of different processes do not repeat and those of the same process are ordered.
       - 组成
         1. sign bit(1bit), reserved sign bit, constantly to be zero.
         2. timestamp bit(41bit), Apache ShardingSphere snowflake algorithm starts from November 1st, 2016. And can be used until 2086.
            #+begin_src java
            Math.pow(2, 41) / (365 * 24 * 60 * 60 * 1000L); // approximately equal to 69.73 years.
            #+end_src
         3. work ID bit(10bit), Set by Java process through properties. If applied in distributed deployment, each work ID should be different.
         4. sequence number bit(12bit), use to generate different IDs in a millisecond. If the number generated in that millisecond exceeds 4096 (2 to the power of 12), the generator will wait till the next millisecond to continue.

      3. 迪卡尔路由

** SQL执行
   Sharding-JDBC通过一套自动化的执行引擎，实现自动化的平衡资源控制和执行效率。其提供如下两种模式的自适应切换：
   1. 内存限制模式
      Sharding-JDBC不对一次逻辑操作所耗费的数据库连接数量做限制，而是由内存可用容量来限制一次操作最大并发的数据库连接数，通过并发处理达成执行效率最大化。
      - 适用场景：适用于OLAP，通过放宽对数据库连接的限制提升系统吞吐量。
      - 示例：
        如果实际SQL需要对某数据库实例中的200张表进行操作，若内存容量足够时则为每张表的操作创建一个新的数据库连接。
   2. 连接限制模式
      Sharding-JDBC严格控制对一次逻辑操作所耗费的数据库连接数量。
      - 适用场景：适用于OLTP，保证系在线系统数据库资源能够被更多的应用所使用。
      - 示例：
        1. 如果实际SQL需要对某数据库实例中的200张表进行操作，则通过一个数据库连接串行对200张表进行操作；
        2. 如果实际SQL需要对10个数据库实例中的200张表进行操作，则为每个数据库实例并发创建10个数据库连接，但对于单个数据库实例而言依然是通过一个数据库连接串行对其中的200张表进行操作。

** TODO 分页查询
*** Sharding-JDBC提供的分页查询方案
    - 改写跨表或跨库的逻辑分页查询SQL
      #+begin_src sql
      select * from user order by id desc limit 1000, 10;
      -- 改写为
      select * from user1 order by id desc limit 0, 1010;
      select * from user2 order by id desc limit 0, 1010;
      #+end_src
    - 不改写单表的逻辑分页查询SQL
      #+begin_src sql
      select * from user order by id desc limit 1000, 10;
      #+end_src


    面对多条实际SQL的场景，Sharding-JDBC并不是一次性读取所有SQL的返回值到应用程序的内存中，而是采用流式归并获取数据结果集，即Sharding-JDBC会通过结果集的next方法将无需取出的记录全部跳过，防止造成内存溢出。
    但由于LIMIT的分页方式无法通过索引查询数据，所以越往后性能越差。(Oracle的rownum方式是一样的)
    所以建议以连续生成的ID作为分页条件（但实际业务很难实现）
    #+begin_src sql
    select * from user where id > 1000 and id < 1010 order by id desc;
    #+end_src

  基于BoneCP进行改进和优化的高性能JDBC连接池。

** 配置
   所有关于时间的配置项的单位都采用毫秒。

   #+begin_src yaml
   spring:
     datasource:
       hikari:
         connection-timeout: 30000 # 从连接池获取连接的最大等待时长，超时抛出SQLException。默认为30秒。
         minimum-idle: 5 # 最小连接数
         maximum-pool-size: 15 # 最大连接数
         idle-timeout: 600000 # 连接最长保持idle状态的时长，除非连接池的连接数已达到minimum-idle的下限，否则超时则触发retired(即该连接将被释放)。默认为10分钟。
         max-lifetime: 1800000 # 连接的最长生命周期，超时则释放。默认为30分钟。建议设置比数据库连接超时时长少60秒，参考MySQL的wait_timeout参数(show variable like '%timeout%';)
         auto-commit: true # 自动提交
         pool-name: HikariCP # 连接池名称，会自动添加序号作为后缀
         connection-test-query: SELECT 1 # 若JDBCDriver支持JDBC4,那么会自动使用isValid方法检查连接是否可用，否则会使用该SQL语句验证连接是否可用。
   #+end_src
