* 持续集成作用
1) 自动监控代码质量和代码覆盖率度量
2) 降低技术债务和减少维护成本
3) 作为一种沟通工具，清晰地发布和展现总体开发工作的当前状态
4) 简化和加速交付过程

持续集成促使大家保证优先修复有问题的构建

* Devops阶段
1) 无构建服务器
   开发人员通过Ant/make等工具在各自开发环境构建，并且不采取定期提交代码变更而是计划发版前集中提交代码到中央代码仓库。
2) 夜间构建
   架设构建服务器，团队成员每个工作日至少提交一次代码到中央代码仓库，但可靠可重复的单元测试步骤可有可无，并且不关心构建服务器的状态(问题)，即使通过邮件或消息通知机制通知了开发人员。
3) 夜间构建加自动化测试
   自动化测试/集成测试作为必要步骤加入到持续集成流程中，通过邮件/消息通知/即时通信等工具即时告知开发人员构建服务器的状态，并且团队成员可以轻易地看到是哪些代码变化触发特定的构建。
4) 加入度量指标
   新增自动化代码质量和代码覆盖率检查，用于一定程度上评估代码仓库质量和测试质量。自动生成API文档，对测试质量下降到一定阀值时提出报警。通过“构建展示器”实时向团队成员展示构建状态。
5) 更认真地对待测试
   持续集成将可靠可重复的测试和构建过程紧密结合，促进了测试驱动开发的开发模式，当测试通过后代码会自动部署到一个应用服务器上进行更全面的端到端测试和性能测试。
6) 自动化验收测试和自动话部署
   采用行为驱动开发和验收测试驱动开发的方式作为交流和文档工具以及文档测试工具，最终用非开发人员可理解的业务术语来发布测试结果报告。并通过早期自动化测试清晰展示出各功能状态（已完成、待完成、开发中）。当QA团队测试通过后，手动发起部署流程。
7) 持续部署
   采用自动化部署技术将变更部署到生产环境。

* 安装配置Jenkins

** 安装
*** 通过Java Web Start方式
   Java Web Start（JNLP，Java Network Launch Protocol）是一种通过URL链接运行Java桌面应用的发布技术，当应用发生更新时只需服务器版本，那么用户在下次通过Java Web Start运行应用时就会运行最新版的应用。
   运行过程：
   1) 下载JNLP文件
   2) Java Web Start下载Java应用并运行应用

*** 通过执行WAR文件
Jenkins.war内置轻量级Winstone Servlet引擎来运行服务。

#+BEGIN_EXAMPLE bash
# --httpPort=1300 ，启用HTTP协议
# --httpsPort=1300 ，启用HTTPS协议
# --httpListenAddress=0.0.0.0 ，设置监听的IP范围，默认是监听所有
# --ajp3Port=8010 ，如果Jenkins运行在另一个应用服务器上（Tomcat、Jetty或GlassFish等），则要覆盖ajp3Port
# --prefix，默认运行在8080端口且没有上下文路径，通过prefix可设置Jenkins运行的上下文路径
# --daemon，在Linux系统下作为一个守护进程运行
# --logfile=/var/log/jenkins.log，默认把日志文件写到当前目录下，可通过这个设置
# --controlPort=8001，若采用内置的Winstone服务器则可通过Winstone服务启用或关闭Jenkins，这时就要设置这个属性
#   重启：java -cp ${JENKINS_HOME}/war/winstone.jar winstone.tools.WinstoneControl reload: --host=localhost --port=8001
#   完全关闭：java -cp ${JENKINS_HOME}/war/winstone.jar winstone.tools.WinstoneControl shutdown: --host=localhost --port=8001
nohup java -jar jenkins.war --httpPort=1300 --httpListenAddress=0.0.0.0 --controlPort=8001 > /var/log/jenkins.log 2>&1 &
#+END_EXAMPLE

** 配置

*** 创建Jenkins服务专用用户
便于监控Jenkins构建所使用的系统资源和排查问题。
#+BEGIN_EXAMPLE bash
# 创建jenkins用户和用户组
useradd --create-home jenkins
#+END_EXAMPLE
然后使用jenkins登录系统后启用Jenkins，那么就会在 ~$HOME/.jenkins~ 生成Jenkins主目录。

*** Jenkins主目录
Jenkins配置、构建作业、构建产物、插件信息等都保存在Jenkins主目录，因此也占大量磁盘空间。
Jenkins备份实质就是备份该目录即可。
Jenkins主目录默认位于 ~$HOME/.jenkins~ ，但可以修改设置
#+BEGIN_EXAMPLE /etc/bashrc
export JENKINS_BASE=/usr/local/jenkins
export JENKINS_HOME=/apps/jenksin-data
#+END_EXAMPLE

#+BEGIN_EXAMPLE bash
# 运行Jenkins
nohup java -jar ${JENKINS_BASE}\jenkins.war --httpPort=1300 &
#+END_EXAMPLE

*** 内存考虑
除Jenkins自身程序所需的内存外，每个构建都会创建独立的Java子进程它们也需要构建用的内存空间。
可通过设置环境变量调整内存上限。
#+BEGIN_EXAMPLE bash
# 应用到Jenkins进程中
export JAVA_OPTS=-Djava.awt.headless=true -Xmx512m -DJENKINS_HOME=$HOME/.jenkins
# 应用到Maven构建作业所创建新的JVM进程中
export MAVEN_OPTS="-Xmx512m -XX:MaxPermSize=256m"
# 应用到Ant构建作业所创建新的JVM进程中
export ANT_OPTS="-Xmx512m -XX:MaxPermSize=256m"
#+END_EXAMPLE

** 验证方式——Security Realm
1) Jenkins’ own user database（Jenkins内置用户数据库）
   - 适合小团队，原来没有身份验证系统的团队
   - 提供注册功能(sign up)
2) LDAP
   - 用于集成LDAP的身份验证功能
3) Delegate to servlet container
   - 当Jenkins托管到某应用服务上如Tomcat，而Tomcat又允许集成关系型数据库、JNDI、JAAS或一个XML配置文件进行用户身份认证。那么就可以采用这一方式。
4) Unix user/group database
   - 当Jenkins部署在Linux系统上，那么就可以使用系统用户和用户组作身份验证（使用到Pluggable Authentication Modules，也可集成NIS）
5) CAS集成
   - 插件CAS
6) 其他
   1) 通过插件Script Security Realm，自行编写验证程序、分组程序集成到Jenkins即可。

** 授权方式——Authorization
1) Matrix-based Security
   - Overall，系统基础权限
     - Administer，允许操作Manage Jenkins界面
     - Read，只允许进入Jenkins首页。若没有这个权限，即使有Job下的其他权限也无法进入Jenkins系统。
   - Slave，从节点权限
     - Configure，创建、配置从节点
     - Delete，删除从节点
   - Job，作业权限
     - Create
     - Delete
     - Configure
     - Read，查看构建结果、构建成品等信息
     - Build，运行构建
     - Workspace，查看工作空间内容
   - Run，构建历史
     - Delete
     - Update，为构建历史新增描述
   - View，视图管理
     - Create
     - Delete
     - Configure
   - SCM，版本管理权限
     - Tag，为某次构建打上新的Tag
2) Project-based Matrix Authorization Strategy
   - 基于Matrix-based Security增加针对项目细分的权限控制
   - 分为 *系统级权限* 和 *项目级权限*
   - 系统级权限和Matrix-based Security相同
   - 项目级权限为基于系统级权限上新增权限（不能删减权限），因此系统级权限必须为最小化授权
3) Role-based Strategy
   - 安装插件Role-based Authorization Strategy
   - 通过Manage Jenkins/Manage and Assign Roles创建组和为用户添加组，并按组设置权限
     - 在Manage Jenkins/Manage and Assign Roles/Manange Roles下创建的Project roles，在Manage Jenkins/Manage and Assign Roles/Assign Roles下作为Item roles，而不是Global Roles使用。
4) Ownership-based Security
   - 安装插件Ownership Plugin

* Jenkins时间表
采用UNIX cron语法。
~MINUTE HOUR DOM MONTH DOW~
+ MINUTE： 一小时内多少分钟（0-59）
+ HOUR： 一天内多少小时（0-23小时）
+ DOM：一个月内多少天（1-31）
+ MONTH：每月（1-12）
+ DOW：星期几（0-7），其中0和7都表示周日

** 特殊值
'*'：可用来指定所有有效的值
'M-N'：可以用来指定一个范围,比如“1-5”
'M-N/X'或'*/X'：可用于在指定范围内跳跃一个X的值，比如在MINUTE字段中"*/15"表示"0,15,30,45"，"1-6/2"表示"1,3,5"
'A,B,...,Z'：可以用来指定多个值，比如“0,30”或“1,3,5”

任何空白行和以#开头的都为注释

#+BEGIN_EXAMPLE
15分钟构建一次：H/15 * * * *
每天8点构建一次：H 8 * * *
每天8点~17点，两小时构建一次：H 8-17/2 * * *
周一到周五，8点~17点，两小时构建一次：H 8-17/2 * * 1-5
每月1号、15号各构建一次，除12月：* * 1,15 1-11 *

* 构建
为跨操作系统，要将变量命名为全大写。

** Multi-configuration project
可简单理解为自动地运行所有可能的参数组合的一个参数化构建作业，非常适合在多个环境（浏览器、数据库、操作系统等）使用一个单独的构建作业完成测试。
例如：
1) 通过Slaves Aixs定义选择分别安装了Windows、Linux和Mac操作系统的三个从从节点
2) 通过User-defined Aixs定义Chrome、IE9和IE10三种浏览器标识
那么该作业会将上述两个Aixs的入参作迪卡尔积作为入参执行构建作业，按上述参数应该执行9次构建作业。

默认情况下为并发执行，但可以通过Run each configuration sequentially设置为逐个执行。
另外可以通过Combination Filter来设置Aixs组合值过滤，仅过滤后的参数能触发构建，采用groovy语法。

** 构建作业流程控制
构建作业间可以存在带触发条件的先后流程顺序，和并行执行顺序。

1) 依赖关系图
   - 插件：Dependency Graph View(依赖graphviz画图，因此服务器要先安装)
   - 通过在每一层/目录下Jenkins左侧仪表盘上的Dependency Graph可查看当前层级各作业的流程关系
2) 并行等待
   - 插件：Join
   - 场景：作业A会触发并行运行的作业B和作业C，并当作业B和作业C均运行成功后运行作业D
   - 使用方式
     1) 在Build区域中设置Trigger/call builds on other projects
     2) 在Post-build Actions区域设置Join Trigger
        1) 以无参数方式触发另一个构建作业执行，通过Projects to build once, after all downstream projects have finished这里仅用于设置；
        2) 以带参数方式触发另一个构建作业执行，通过Run post-build actions at join设置。
3) 加锁
   - 插件：Locks and Latches
   - 场景：多个并发运行的作业存在资源竞争时，可通过该插件添加悲观锁实现线性安全。
   - 使用方式
     1) 在Manage Jenkins/System Configuration的Locks下添加锁名称；
     2) 然后在作业配置的Build Environment中选择需要获取的锁即可。
4) 复制构建产物
   - 插件：Copy Artifact

   - 插件：Promoted Builds
     - 在每个一个阶段的第一个构建作业中设置Promote builds when选项，设置通过特定条件从上一阶段进阶到下一个阶段。

         - PROMOTED_NUMBER
         - PROMOTED_ID

** 从节点安装和配置
1) 在主节点上通过SSH启动从节点代理
   好处：主节点自动启动或重启从节点代理
   - 配置方式
     1) 主从节点间建立SSH KEY信任关系
     2) 在Manage Nodes/New Node下Launch method选项选择 *Launch agent via execution of command on master*
     3) 下载agent.jar到从节点
     4) 在Launch command中输入 ~ssh <user>@<ip> java -jar ~/path/to/agent.jar~
2) 在从节点上通过Java Web Start手工启动从节点代理
   缺点：从节点代理不能被Jenkins自动启动或重启，但可以通过配置为Windows服务随机启动
   - 主节点配置
     - 在Manage Jenkins/Configure Global Security/Agents
       1) 将TCP port for JNLP agents设置为Random（若主节点防火墙设置仅开放某些端口，则设置Fixed端口号）
       2) 通过Agent Protocols添加通信使用的JNLP协议
   - 从节点有3种启用方式
     1) 在从节点访问Jenkins系统，进入Manage Jenkins/Manage Nodes，进入相应节点后点击 *Launch agent from browser* 前的按钮即会下载执行JNLP。
        - 注意：由于JNLP文件要通过可执行程序javaws关联，因此环境变量PATH必须配置javaws的路径
        - 缺点：由于浏览器升级换代，因此这种方式不一定有效
     2) 在从节点CMD执行 ~javaws http://${JENKINS_URL}/computer/${NODE_NAME}/slave-agent.jnlp~
        - 注意：由于JNLP文件要通过可执行程序javaws关联，因此环境变量PATH必须配置javaws的路径
     3) 在从节点CMD执行 ~java -jar agent.jar -jnlpUrl http://${JENKINS_URL}/computer/${NODE_NAME}/slave-agent.jnlp -secret ${SECRET} -workDir "d:\jenkins_slave"~
        - 注意
          1) 要先下载agent.jar到从节点
          2) 通过[[https://nssm.cc/download][NSSM]]将上述命令安装为Windows Service
3) 以Windows远程服务（是通过WMI+DCOM，不是Windows服务）的方式安装启用从节点代理
   问题很多、配置很烦不建议使用。

*注意：*  当Slave中的环境变量发生变化，需重启Slave服务才能生效。

* 版本号管理
在项目的README中注明项目遵循哪个版本管理规范。

** 语义化版本(Semantic versioning or SemVer convention)
利用字符串形式的版本号去表达其指向的代码的具体含义。

1) Release版本号结构： ~Major.Minor.Patch~
   1) ~Major~ 主版本号，表示发布重大变更或向后不兼容变更(breaking changes or incompatible changes)
      - 当主板本号为0时，表示当前处于开发版本，功能/接口等均处于剧烈变化当中。
   2) ~Minor~ 次版本号，表示发布新特性或向后兼容变更(new features and backwards compatible)
      - initial development phase
        - 项目首个修订版本为 ~0.1.0~
        - 后续每次发布时通过升级Minor实现版本修订
   3) ~Patch~ 补丁，表示向后兼容问题修复(Backwards compatible bug fixs only)
2) Pre-release版本号结构： ~Major.Minor.Patch-PreReleaseVersion~
   - 规则：
     1) PreReleaseVersion由 ~[0-9a-zA-Z-]~ 组成；
     2) 通过句号 ~.~ 分隔多个子组件。
   - 常用标识
     1) ~alpha~ Alpha release，表示开发中和带试验性质的发布(work-in-progress or experimentation)
     2) ~beta~ Beta release，表示即将正式发布的特性已基本完成，但存在某些已知的问题待修复(feature complete for the next planned release, but may contain known bugs)
     3) ~rc~ Release candidate，表示即将作为正式发布的版本，除非发现重大问题(a release that's potentially final unless significant bugs emerge)
   - 示例： ~1.0.0-alpha~, ~1.0.1-alpha.1~, ~1.0.1-0.3.7~
   - 对于不支持pre-release版本号的dot notation的版本系统([[https://semver.org/spec/v2.0.0.html][Semantic Versioning v2.0.0]])，可以采用如 ~1.0.1-alpha01~ 的方式表示 ~1.0.1-alpha.1~
3) 带元数据的版本号结构： ~Major.Minor.Patch-PreReleaseVersion+Metadata~
   通过元数据附加发布日期、需求号和sha编码等信息。
   - 规则：
     1) 元数据由 ~[0-9a-zA-Z-]~ 组成；
     2) 多个元数据间通过句号 ~.~ 分隔；
     3) 元数据不影响版本号的优先顺序。
   - 示例： ~1.0.0+exp.sha.123~, ~1.0.1-alpha.2+201803211324~, ~1.2.1-rc+201803211312.REC321~

*** 优先级(precedence)
规则：
1) 依次从Major、Minor、Patch和Pre-release版本号比较；
2) Pre-release版本号中
   1) 非数字组件的优先于数字组件；
   2) 组件数越多的优先于组件数少的。
3) Release版本号优先于Pre-release版本号。

示例： ~1.0.0-alpha < 1.0.0-alpha.1 < 1.0.0-alpha.beta < 1.0.0-beta < 1.0.0-beta.2 < 1.0.0-beta.11 < 1.0.0-rc.1 < 1.0.0~

*** ClusterFS
    GlusterFS，是一个开源的分布式文件系统，具有强大的横向扩展能力，通过扩展能够支持数PB存储容量和处理数千客户端。GlusterFS借助TCP/IP或InfiniBand RDMA网络将物理分布的存储资源聚集在一起，使用单一全局命名空间来管理数据。GlusterFS的Volume有多种模式，复制模式可以保证数据的高可靠性，条带模式可以提高数据的存取速度，分布模式可以提供横向扩容支持，几种模式可以组合使用实现优势互补。

        gluster volume info # Show information of volume
        gluster volume start jenkins_public # Start distribute volume
        #+end_example

**** Declarative Pipeline
Pipeline Plugin2.5后支持，也是Open Blue Ocean所支持的类型。
可以内嵌Scripted Pipeline。
https://jenkins.io/zh/doc/book/pipeline/syntax/

- ~pipeline~ block(required), is a block where all other directives and blocks are declared.
- ~agent~ directive(required), instruct Jenkins to allocate an executor and workspace for the Pipeline. parameters as below:
  - any, The Pipeline will run in any available node. ~agent any~
  - none, The Pipeline will not run in any agent at top-level, but each stage should be defined with its own agent. ~agent none~
  - label, The Pipeline will run in any available node whose name mentioned. ~agent {lable 'my label name for node'}~
  - node, Like label but with more options. ~agent {node 'my label name'}~
  - docker, The Pipeline will run in docker environment. ~agent {docker {image 'python:0.1' label 'my label name' arg '--port 3000:80'}}~
- ~triggers~ Typical build trigger, defined at ~pipeline~ section.
  - ~cron~ Build job will run in regular interval.
    #+begin_example
    triggers {
      cron('H * * * *')
    }
    #+end_example
  - ~pollSCM~ Check for the changes in SCM in regular interval to run the build job
    #+begin_example
    triggers {
      pollSCM("H * * * *")
    }
    #+end_example
  - ~upstream~ Upstreaming the other build job
    #+begin_example
    triggers {
      upstream(upstreamProjects: 'job1,job2', threshold: hudson.model.Result.SUCCESS)
    }
    #+end_example
- ~stages~ directive(required), contains one or more ~stage~ tag.
- ~stage~ (rquired), contains ~steps~, ~agent~ section and other directives.
- ~steps~ directive(required), a set of commands to complete the continuous delivery process.
- ~environment~ (optional), defined at ~pipeline~ or ~stage~ section.
  - helper method
    1) ~credentials(<type>)~, <type> as below
       - ~secret-text~, string based
       - ~secret-file~, location of file
       - ~username-and-password~, specified as "username:password"
       - ~ssh-with-private-key~, location of the ssh key file and optionally "username:password"
  - sample
    #+begin_example groovy
    pipeline {
        agent none
        environment {
            JENKINS_PATH = sh(script: 'pwd', returnStdout: true)
            field = func()
            val = 'Hi'
        }
    }

    def func() {
        return 'Yep'
    }
    #+end_example
- ~input~ (optional), defined at ~stage~ section.
  get input by prompting users. Options as below
  - ~message~ The message to be show to the user.
  - ~id~ The default value.
  - ~ok~ The "OK" button
  - ~submitter~ The comma-seperated value list of users who are allowed to enter the input.
  - ~submitterParameter~ Add submitter's name into the environment variable.
  - ~parameters~ The list of parameters that need to be provided by submitter.
- ~options~ (optional), defined at ~pipeline~ or ~stage~ section.
  - ~buildDiscarder~
    #+begin_example
    options {
      buildDiscarder(logRotator(daysToKeepStr: '7'))
      buildDiscarder(logRotator(numToKeepStr: '10'))
    }
    #+end_example
  - ~preserveStashes~ To preserve the stash from the completed build process
    #+begin_example
    options {
      preserveStashes(buildCount: 5)
    }
    #+end_example
  - ~checkoutToSubdirectory~ Checkout to the particular subdirectory in workspace from the SCM.
    #+begin_example
    options {
      checkoutToSubdirectory("foldername")
    }
    #+end_example
  - ~newContainerPerStage~ must be defined at ~pipeline~ section. Allow the process run in the different Container each time when agent is running on docker.
    #+begin_example
    options {
      newContainerPerStage()
    }
    #+end_example
  - ~quietPeriod~ The pause period or sleep time
    #+begin_example
    options {
      quietPeriod(30)
    }
    #+end_example
  - ~retry~
    #+begin_example
    options {
      retry(2)
    }
    #+end_example
  - ~skipDefautCheckout~ Skip the defaut SCM checkout process in the agent
    #+begin_example
    options {
      skipDefaultCheckout()
    }
    #+end_example
  - ~skipStagesAfterUnstable~ Skip the mentioned stage after it comes to UNSTABLE state.
    #+begin_example
    options {
      skipStagesAfterUnstable()
    }
    #+end_example
  - ~timeout~ Must be defined at ~pipeline~ section.
    #+begin_example
    options {
      timeout(time: 30, unit: "MINUTES")
    }
    #+end_example
  - ~disableConcurrentBuilds~ By default, Jenkins allow to execute the same Pipeline concurrently.
    #+begin_example
    options {
      disableConcurrentBuilds()
    }
    #+end_example
  - ~timestamps~
    #+begin_example
    options {
      timestamps()
    }
    #+end_example
- ~when~ Defined at ~stage~ section
  - ~branch~ execute the stage only when the branch in SCM is matching.
    #+begin_example
    when {
      branch "master"
    }
    #+end_example
  - ~buildingTag~ execute the stage only when the build is building the tag.
    #+begin_example
    when {
      buildingTag()
    }
    #+end_example
  - ~changelog~ execute the stage only when the changelog is having specified pattern.
    #+begin_example
    when {
      changelog ".*^\\[SERVICE2\\] .+$"
    }
    #+end_example
  - ~changeset~ execute the stage only when the SCM changeset is having specified pattern.
    #+begin_example
    when {
      changeset "**/*.js"
    }
    #+end_example
  - ~changeRequest~ changeRequest is like the pull request in GitHub.
    #+begin_example
    when {
      changeRequest target: "master"
    }
    #+end_example
  - ~anyOf~
    #+begin_example
    when {
      anyOf {
        branch "master"
        branch "production"
      }
    }
    #+end_example
  - ~allOf~
    #+begin_example
    when {
      allOf {
        branch "master"
        branch "production"
      }
    }
    #+end_example
  - ~environment~ execute the stage only when the specified environment variable is matching.
    #+begin_example
    when {
      environment name: "DEPLOY_TO", value: "production"
    }
    #+end_example
  - ~equals~ execute only when expected value and actual value is matching.
    #+begin_example
    when {
      equals expected: 2, actual: currentBuild.number
    }
    #+end_example
  - ~expression~ condition is created with groovy expression and expects the Boolean true to pass.
    #+begin_example
    when {
      expression {
        return params.DEBUG_BUILD
      }
    }
    #+end_example
  - ~tag~ execute only when the tag name pattern matches the stage tag.
    #+begin_example
    when {
      tag "release-*"
    }
    #+end_example
  - ~not~ is like "NOT" in the "IF" condition
    #+begin_example
    when {
      not {
        branch "master"
      }
    }
    #+end_example
  - ~triggeredBy~ execute only when it is triggered by the specified build.
    #+begin_example
    when {
      triggeredBy "SCMTrigger"
    }
    #+end_example
  - ~post~ Execute some command after the ~stage~ or ~steps~ as per the condition.
    - ~always~ Execute no matter what current completion status is.
    - ~changed~  Execute when current completion status is different with previous one.
    - ~fixed~ Execute when current completion status is success but the previous one is either failure or unstable.
    - ~regression~ Execute when current completion status is one of failure, abort or unstable but the previous one is success.
    - ~aborted~ Execute when current completion status is aborted.
    - ~failure~ Execute when current completion status is failure.
    - ~success~ Execute when current completion status is success.
    - ~unstable~ Execute when current completion status is unstable.
    - ~unsuccessful~ Execute when current completion status is unsuccessful.
    - ~cleanup~ Execute workspace cleanup no matter what current completion status is.
  - ~tools~ Install and configure the tools in Pipeline, support following tools
    - ~maven~
    - ~jdk~
    - ~gradle~
  - ~parameters~ Define the trigger parameters to Pipeline. Refer by ~${params.yyy}~
    - ~string~
      #+begin_example
      parameters {
        string(name: 'yyy', defaultValue: 'xxx', description: 'sample')
      }
      #+end_example
    - ~text~ Multi line text datatype
      #+begin_example
      parameters {
        text(name: 'yyy', defaultValue: 'xxx', description: 'sample')
      }
      #+end_example
    - ~booleanparam~
      #+begin_example
      parameters {
        booleanparam(name: 'yyy', defaultValue: true, description: 'sample')
      }
      #+end_example
    - ~choice~ a multiple choice parameter
      #+begin_example
      parameters {
        choice(name: 'yyy', choices: ['a', 'b', 'c'], description: 'sample')
      }
      #+end_example
    - ~password~
      #+begin_example
      parameters {
        password(name: 'yyy', , defaultValue: true, description: 'sample')
      }
      #+end_example
- ~parallel~ will allow run the stages concurrently.
  #+begin_example
  stage("Parallel Stages"){
    failFast true // allow to stop the build
    parallel {
      stage("1"){
      }
      stage("2"){
      }
    }
  }
  #+end_example

pipeline {
  agent any

  environment {
    key1 = 'value1'
    key2 = 'value2'
  }

  }

  tools {

    }
  }

    }

}

         #+begin_example yaml
         jenkins-shared-library/
           - Jenkinksfiles/ # Place all Jenkinsfile.groovy here.
           - vars/ # All the files with a .groovy extension under vars/ all are global functions and variables. The file name is the function name, so file must be named in camelCased.
                   # The files with a .txt extension are the documentations in markdown format of the shared library code. The help file can be viewed from <your-jenkins-url>/pipeline-syntax/global page.
             - gitCheckout.groovy
           - src/ # It's a regular Groovy source directory(Notice: extension name must be .groovy). It's added to the classpath during every script compilation.
             - org/test/build
               - HelloWorld.groovy
           - resource/ # All the non-groovy files required for your pipelines can be managed in this folder.
                       # Load the resource by `libraryResource` function. ~def request = libraryResource 'org/test/dummydata.json'~
             - org/test/
               - dummydata.json
         #+end_example
      2) Library —— HelloWorld.groovy
         #+begin_example groovy
         package org.test.build

         // implement Serializable interface to support saving the state if the pipeline is stopped or resumed.
         class HelloWorld implements Serializable {
           private String _solutionPath

           HelloWorld(String solutionPath){
             _solutionPath = solutionPath
           }

           void build() {
             println _solutionPath
           }
         }
         #+end_example

               package io.fsjohnhuang

               class Utilities {

                 def steps

                   this.steps = steps
                 }

                   script.sh "{script.tool 'Maven'}/bin/mvn -s ${script.env.HOME}/jenkins.xml -o ${args}"
                 }
               }
               #+end_example
             - 使用

           #!groovy
           # vars/gitCheckout.groovy
           import org.test.build.HelloWorld

           def call(Map stageParams) {

             // 函数中可以直接调用steps的方法
             checkout([
                 $class: 'GitSCM',
                 branches: [[name: stageParams.branch]],

         1) 全局共享库(Global Pipeline Libraries)
            - 作用域范围：全局可引用，且运行在安全沙盒之外，因此可以运行 Java/Groovy/Jenkins内部API/Jenkins插件/第三方库 的任何方法。

              #+end_example
            - 引用第三方Java库 ~@Grab~
            - 操作
              在 Manage Jenkins/Configure System 的 Global Pipeline Libraries 设置 Jenkins Shared Library 的SCM地址。
              - Name，为在Pipeline Script中引入Jenkins Shared Library的名称
              - Default version，为默认版本。若留空那么 ~@Library(<library-name>)~ 中的 ~<library-name>~ 必须包含版本号 ~@<version>~
         2) Folder-level Shared Libraries
            - 作用域范围：当前目录及子目录下的Pipeline，共享库被认为不安全，因此将运行在沙盒中。
         3) Automatic Shared Libraries
            其他插件可以增加定义共享库的方法。例如插件github-branch-source提供一个GitHub组织目录项，这个目录项允许脚本不用任何附加配置就可以使用不受信任的库，例如github.com/someorg/somerepo。

         - ~@Library(<library-name>) import org.test.build.HelloWorld~ 引入单个shared library下的类库(定义在src下文件的类)

    discover information about the currently execution Pipeline.
    - sample:
      - ~currentBuild.getBuildCauses()~ returns ~List<hudson.model.Cause$UserIdCause>~
        - fields of ~hudson.model.Cause$UserIdCause~
          1) ~String shortDescription~ .e.g Started by user 黄8958
          2) ~String userId~ .e.g huangza
          3) ~String userName~ .e.g 黄8958
      - ~currentBuild.result~
      - ~currentBuild.displayName~
      - ~currentBuild.rawBuild~ returns instance of ~hudson.model.Run~
        - REF https://javadoc.jenkins-ci.org/hudson/model/Run.html
        - Constraints
          1) only for trusted libraries or administrator-approved scripts outside the sandbox.
          2) the instance of ~hudson.model.Run~ is not be ~java.io.Serializable~, so only access it inside a method marked @NonCPS.
        - Get user who aborted a build
          #+begin_example groovy
          /**
          * Get user's informantion who abort this build.
          * @return user's id and display name
          */
          @NonCPS
          def getAbortUser() {
              def causee = [:]
              def actions = currentBuild.rawBuild.getActions(jenkins.model.InterruptedBuildAction)
              for (action in actions) {
                  def causes = action.getCauses()

                  for (cause in causes) {
                      def user = cause.getUser()
                      causee.id = user.getId()
                      causee.displayName = user.getDisplayName()
                  }
              }

              return causee
          }
          #+end_example
        - Get build log
          #+begin_example groovy
          @NonCPS
          def getBuildLog() {
              return currentBuild.rawBuild.log
          }
          #+end_example

**** ~@NonCPS~ 的作用
     Jenkinsfile和SharedLibrary脚本中使用的对象都应该可序列化对象(实现 ~java.io.Serializable~ 接口)，因为 Jenkins 必须能够序列化脚本中的对象，以便可以暂停流程并存储在磁盘上。
     当在方法中调用不可序列化对象时，为告知 Jenkins 该方法必须一次性完整执行而不能中途暂停，我们需要在方法上添加注解 ~@NonCPS~ 。
     被 ~@NonCPS~ 标记的方法
     1) 方法入参和返回值必须为序列化对象；
     2) 方法内的局部变量可为不可序列化对象；
     3) 方法内部不能调用非 ~@NonCPS~ 方法 和 Pipeline steps；

     https://github.com/jenkinsci/workflow-cps-plugin/blob/master/README.md
     #+begin_quote
     Unlike a regular Groovy program run from a command line, the complete state of Pipeline build's program is saved to disk every time an asynchronous operation is performed, which includes most Pipeline steps. Jenkins may be restarted while a build is running, and will resume running the program where it left off. This is not intended to be efficient, and so should be limited to high-level 'glue' code directly related to Jenkins features; your project's own build logic should be run from external programs on a build node, in a ~sh~ or ~bat~ step.
     #+end_quote

**** QA
1) What is "Durability level: MAX_SURVIVABILITY"
2) sh执行失败
   - 原因：sh执行脚本时不会默认继承 ~/etc/profile~ 和 =~/.profile= 等变量。
   - 解决办法：
     1) 办法1：
        1) 在Node Properties/Environment variables中新增需要的环境变量
        2) 通过 ~withEnv~ 控制在存在环境变量时才执行某些指令
           #+begin_example groovy
           steps {
             withEnv(['MY_ENV=abbc']) {
               echo 'hi'
             }
           }
           #+end_example
     2) 办法2：在需要执行sh的脚本前，先执行 ~sh 'source /etc/profile && source ~/.profile'~
3) sh中的nohup和&无效
   - 原因：由于Jenkins中的sh用于阻塞执行时长有限的指令，而nohup和&用于非阻塞执行长时间执行的命令，因此nohup和&不生效
   - 解决办法：在Jenkins2.46中可通过设置 JENKINS_NODE_COOKIE 来实现
     #+begin_example groovy
     steps {
       sh 'JENKINS_NODE_COOKIE=dontkillme nohup python3 app.py > /var/log/app.py.log 2>&1 &'
     }
     #+end_example
4) 从GitLab下载源码，并且采用triggers/pollSCM时，无论是否有更新都会因为polling log总返回found changes，而触发任务执行。
   - 原因：这是Jenkins Pipeline的问题，若采用Freestyle Project则polling log返回结果是正常的。
   - 解决办法：
     若采用git或svn则通过hooks触发Jenkins任务执行，若采用gitlab则采用web hooks出发Jenkins任务执行。
     - GitLab + Jenkins Pipeline + GitLab Hooks Plugin
       - Jenkinsfile部分
         1) 移除Jenkinsfile中的 ~triggers~ 块
         2) 通过Jenkins Job Builder脚本设置trigger(https://docs.openstack.org/infra/jenkins-job-builder/triggers.html)
            #+begin_example yaml
            triggers:
              - gitlab:
                  trigger-push: true
                  trigger-merge-request: false
                  branch-filter-type: RegexBasedFilter
                  target-branch-regex: '.*sit.*' # 仅分支名称正则匹配的推送才会触发该任务执行
                  secret-token: '123' # 必须设置Token，否则会报403错误
            #+end_example
       - 项目仓库部分
         1) 登录项目所在的GitLab仓库
         2) 在Settings/Integrations中
            - URL输入Jenkins任务的触发URL，形如： ~${JENKINS_URL}project/${JOB_NAME}~
            - Secret Token输入 ~secret-token~ 的值
            - Trigger 下勾选GitLab发起上述请求的触发器

**** 脚本编写
     1) Job
        用于定义具体的任务
        #+begin_example yaml
        - job:
            name: job-name
            display-name: display-name
            project-type: pipeline
            defaults: global
            description: ''
            disabled: false
            concurrent: true
        #+end_example
     2) Job Template
        用于定义任务模板，然后通过project节点实例化任务。其中变量通过 ~{name}~ 引用，若要输入花括号则输入 ~{{~ 或 ~}}~ 。
        #+begin_example yaml
        - job-template:
            name: '{name}-unit-tests' # project节点会通过 {name}-unit-tests 引用模板，然后以变量name的实际值替换并实例化任务
            display-name: display-name
            project-type: pipeline
            defaults: global
            description: ''
            disabled: false
            concurrent: true
        #+end_example
     3) Job Group
        用于将任务模板归类，Job Group的name仅为分组名称供project引用而已，并不会作为name变量值传递给旗下的Job Template。
        #+begin_example yaml
        - job-template:
            name: '{name}-unit-tests'
            display-name: '{display-name}'
        - job-template:
            name: '{name}-perf-tests'
            display-name: '{display-name}'
        - job-group:
            name: '{name}-tests'
            jobs:
              - '{name}-unit-tests':
                  display-name: a
              - '{name}-pef-tests':
                  display-name: b
        - project:
            name: project-name
            jobs:
            - '{name-tests}'
        # 实例化出project-name-unit-tests和project-name-perf-tests两个任务
        #+end_example
     4) Project
        用于调用Job Template或Job Group实例化任务
        #+begin_example yaml
        - job:
            name: '{name}-tests-{branch}'
            display-name: '{display-name}'

        - project:
            name: project-name
            branch:
              - dev:
                  display-name: 'dev-name'
              - master:
                  display-name: 'master-name'
            jobs:
              - '{name}-tests-{branch}'

        # 实例化出 project-name-tests-dev 和 project-name-tests-master两个任务
        #+end_example
     5) Macro
        用于定义代码片段
        - 定义
          #+begin_example yaml
          - builder:
              name: git-prep
              builders:
                - shell: "/slave_scripts/git-prep.sh"

          - builder:
              name: docs
              builders:
                - shell: "/slave_scripts/run-docs.sh"

          - publisher:
              name: console-log
              publishers:
                - scp:
                    site: 'scp-server'
                    files:
                      - target: 'logs/$JOB_NAME/$BUILD_NUMBER'
                        copy-console: true
                        copy-after-failure: true
          #+end_example
        - 使用
          #+begin_example yaml
          - job:
              name: a
              builder:
                - git-prep
          #+end_example
     6) Defaults
        用于定义全局默认变量值
        #+begin_example yaml
        - defaults:
            name: global # name属性必须为global
            arch: 'i386'
        #+end_example
