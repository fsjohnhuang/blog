*等幂性(idempotent/idempotency)* ，某动作执行结果为某事物已达到某目标状态，在执行该动作前先检查事物是否已达到某状态，若已达到某状态则不执行，若未达到则执行。从而达到不引发副作用的前提下，不论动作执行多少次目的最终状态不变。
*收敛性* ，某动作可多次执行，每次执行都让状态更趋向于目标状态，并最终达到目标状态。由Mark Burgess在CFEngine中提出的。

* 架构
Ansible编排引擎组成：
1) Ansible Core，Ansible自身核心模块；
2) Host Inventory，主机库，用于定义可管控的主机列表；
3) Plugins
   完成模块功能的补充。
   1) Filter Plugins
   2) Callback Plugins
      - 邮件
      - 日志
4) Connection Plugins，连接插件，有local、SSH(默认，底层通过paramiko实现)和ZeroMQ(message bus，大规模时使用这个会提高执行速度)
   CLI中通过 ~-c~ 设置connection类型
5) Modules，任务模块，分为自带的Core Modules和自定义的Custom Modules；
6) Playbook，剧本。

* 工作流程
1) 读取配置信息
2) 抓取全量机器、分组列表和关联变量（可从多个静态文件、目录读取，或通过脚本读取从其他来源读取）
3) 使用host-pattern过滤机器列表
4) 根据参数确定执行模块和配置（从modules目录动态读取模块信息，自定义模块则放在modules目录下）
5) Runner执行返回
   1) Connection阶段，通过Connection Plugin连接控制节点
   2) Action阶段，通过Lookup plugin获取Action所需的变量和文件等资源
      1) 若要向管理节点传送文件，内部将使用SCP、SFTP或Rsync(基于SSH)实现
      2) 使用jinja2解释模板时，会调用filters目录下的Filter plugins（自定义Filter则放filters目录下）
      3) 在执行各阶段会调用callbacks目录下的Callback plugins执行钩子函数调用
6) 输出、结束

** 动态Inventory
    从CMDB系统或Zabbix监控系统拉取所有主机信息，然后使用Ansible进行管理。
    只需将配置项inventory指向一个可执行脚本即可，而脚本仅需支持两个选项
    - ~--list~ 或 ~-l~ ，显示所有主机以及主机组的信息（JSON格式）
    - ~--host <host>~ 或 ~-H <host>~ ，返回这个台主机的所有信息(包括认证信息、主机变量等)也是JSON格式
    
    示例
    #+BEGIN_EXAMPLE python
    #!/usr/bin/env python
    # -*- coding: utf-8 -*-
    import sys
    import argparse
    import json
    
    def list():
        r = {}
        h = ['172.17.16.10' + str(i) for i in xrange(0,2)]
        hosts = {'hosts': h}
        r['docker'] = hosts

        return json.dump(r, indent=4)

    def host(name):
        r = {'ansible_ssh_pass': '123'}
        cpis = dict(r.items)

        return json.dump(cpis)

    if __name__ == '__main__':
        parser = argparse.ArgumentParser()
        parser.add_argument('-l', '--list', help='hosts list', action='store_true')
        parser.add_argument('-H', '--host', help='hosts vars')
        args = vars(parser.parse_args())

        if args['list']:
            print list()
        elif args['host']:
            print hosts(args['host'])
        else:
            parser.print_help()
    #+END_EXAMPLE

* 自定义模块
  可以使任何语言开发自定义模块。
  注意：
  1) 核心模块是存放在Ansible代码目录的/modules/core目录下（如 ~/usr/local/lib/python2.7/dist-packages/ansible/modules/core/~）
  2) Extra模块存放路径通过ansible.cfg配置项library设置（默认为/usr/share/ansible），或在调用命令 ~ansbile~ 或 ~ansible-playbook~ 时通过参数 ~-M~ 设置路径亦可
  3) 模块最后的输出会被Ansible使用json.load格式化
  4) 当模块中使用了基础模块 ~ansible.module_utils.basic~ ，那么该模块只能在被控节点上运行且不能与管理节点有任何联系
     copy和synchronize等核心模块实际是存放在unner/action_plugins中，且使用基础模块，因此可以与管理节点有关联
  5) 返回JSON的键中避免使用Ansible的关键字，Ansible很可能会覆盖这些关键字的值
     - invocation

** Check Mode(Dry Run)
   当执行 ~ansible --check~ 时将以check mode运行Playbook，即不执行实际的变更动作，而且对于支持check mode的模块，还会报告该模块将作怎样的变更。
   注意：
   1) 由于check mode只是模拟执行，因此若后一个模块A以前面的模块执行结果作为入参，那么模块A的变更预测将无参考意义
   2) 若在自定义模块中实现 ~supports_check_mode~ ，那么即表示该模块稳定

   #+BEGIN_EXAMPLE yaml
   # 强制该任务运行在check mode，即使命令行不添加参数--check
   - shell: echo 1
     check_mode: yes

   # 强制该任务不运行在check mode，即使命令行添加参数--check
   - shell: echo 2
     check_mode: no

   # 通过变量ansbile_check_mode作为条件
   - shell: echo 3
     when: not ansible_check_mode

   - shell: echo 4
     ignore_errors: "{{ansible_check_mode}}"

   # 忽略该任务当命令行添加参数--diff
   - template:
       src: xml.conf.j2
       dest: /etc/xml.conf
       owner: root
       group: root
       mode: "0600"
     diff: no
   #+END_EXAMPLE

** 自定义echo模块
   #+BEGIN_EXAMPLE python
   # !/usr/bin/env python
   # ./library/info.py
   # 用于ansible-doc
   DOCUMENTATION = """
   ---
   module: echo
   short_description: This module acts as echo
   description:
     - This module acts as echo
   version_added: "1.1"
   options:
     str1:
       description:
         - string 1
       require: true
       default: null
     str2:
       description:
         - string 2
       require: false
       default: null
   author: fsjohnhuang
   requirements: [ "python >= 2.7.0"]
   """
   # 用于ansible-doc
   EXAMPLES = """
   - echo: str1="123" str2="333"
   """

   import sys
   # 引入ansible模块工具获取入参
   from ansible.module_utils.basic import (
       AnsibleModule,
   )
   
   module = AnsibleModule(
       argument_spec = dict(
           str1 = dict(default=None, type="str", required=True),
           str2 = dict(default=None, type="str"),
           choices = dict(default="present", choices=["present", "absent"]),
           bools = dict(type="bool"),
           name1 = dict(aliases=["name2", "name2"]), # name1、name2和name3都是指向同一个入参
       ),
       supports_check_mode = True # 开启模块的check支持，表示该模块已经经过严格的测试，当执行ansible-playbook --check时列入检查范围，否则会被跳过检查。
   )
   
   str1 = module.params["str1"]
   str2 = module.params["str2"]

   if module.check_mode:
       print("in check mode")

   if str1 is None:
      module.fail_json(msg="module echo need one argument at least")
      # 返回内容为
      # {"failed": true, "msg": "module echo need one argument at least", "rc":1}
   else:
      result = dict(echo=str1 + str2 if str2 is not None else "", changed=False, failed=False, rc=0)
      module.exit_json(**result)
      # 返回内容为
      # {"changed": false, "echo": "123333"}
   #+END_EXAMPLE

* Playbook 
  Playbook采用[[https://yaml.org/spec/1.2/spec.html#Syntax][YAML]]进行编写。
  组织规则：
  1) 一个Playbook可以包含1~N个Play，每个Play是一个完整的部署任务。每个Play定义对某组管理节点执行操作；
  2) 一个Play可以包含1~N个Task，每个Task完成一件事；
  3) 一个Play可以包含1~N个Role，每个Role是对某组Task所需的内容（handlers/templates/等）进行组织。
     
** Playbook语法示例
   #+BEGIN_EXAMPLE yaml
   --- # 首行表示该文件为YAML文件
   # 文档字符串，作者、版本、联系方式、遵循的证书规则、功能描述

   - hosts: linux:centos # 管理节点，遵循Ad-Hoc中所有模式
     gather_facts: yes # 默认第一个任务必定是收集Facts变量，但若后续任务根本不使用Facts变量，那么可以手动禁止
     become: yes  # 是否提升权限，这个可以设置在Play中，也可以设置为Task中
     become_method: sudo # 通过sudo提升权限，这个可以设置在Play中，也可以设置为Task中
     max_fail_percentage: 30 # 设置失败百分之多少时最后结果才被认为是失败
     serial: 15 # 并发对多少台被控节点作操作，可以使用如20%的百分比
     # 设置变量
     vars:
         nginx_port: 80
     vars_files:
         - "one.yml"
         - "two.yml"
     vars_prompt:
         - name: "password vaes"
           prompt: "Enter password"
           default: "secret"
           private: yes # 在命令行输入的任何字符均不可见
           confirm: yes # 需要输入两次进行确认
           encrypt: "md5_crypt"
           salt: 1234
           salt_size: 8
     # 运行Playbook前将运行的任务
     pre_tasks:
         - name: pre_tasks
           shell: hostname
     # 引入角色
     roles:
         - role: docker
           version: 1.5.0
           when: ansible_system == 'Linux' 
     # 运行Playbook任务
     tasks:
         - name: Install Nginx Package
           tags: [nginx, install] # 设置标签，可通过ansible-playbook -t <tag> 单独运行指定标签的任务，或通过ansible-playbook --skip-tags=<tag> 不执行指定标签的任务
           yum: name=nginx state=present
         - name: Copy Nginx.conf
           tags: [nginx, config]
           template: 
               src: ./nginx.conf.j2
               dest: /etc/nginx/nginx.conf
               owner: root
               group: root
               mode: 0644
               validate: 'nginx -t -c %s'
           # 触发handler
           notify:
               - Restart Nginx Service
        
         # 对于shell和command模块而而言，它们根据所执行命令的return code决定任务是否执行成功
         # 对于不关心return code的命令而言，可采用如下两种方式保证任务执行成功
         - name: run this command and ignore the result
           shell: /usr/bin/somecommand || /bin/true

         - name: run this command and ignore the result
           shell: /usr/bin/somecommand
           ignore_errors: yes

         # 引入其他Playbook
         - include: other_task.yml
           version: 1.1
           dest: /tmp/dest/
           when: ansible_all_ipv4_addresses == '128.1.3.12'
     # 运行Playbook后将运行的任务
     post_tasks:
         - name: post_tasks
           shell: hostname
     # 当某任务执行结果为changed时触发，并且延迟到所有任务执行结束后才执行的动作
     handlers:
         - name: Restart Nginx Service
           service: name=nginx state=restarted
         - include: other_handlers.yml
   #+END_EXAMPLE
   1) 检查playbook语法 ~ansible-playbook nginx.yaml --syntax-check~
   2) 检查操作的被控节点 ~ansible-playbook nginx.yaml --list-hosts~
   3) 查看将要执行的任务 ~ansible-playbook nginx.yaml --list-tasks~
   4) 预演操作结果 ~ansible-playbook nginx.yaml --check~
   5) 以交互式逐个任务执行 ~ansible-playbook nginx.yaml --step~
   6) 仅对部分被控节点执行 ~ansible-playbook nginx.yaml --limit 10.22.12.33~
   7) 一次性执行 ~ansible-playbook nginx.yaml -f 5~
   8) 执行时回显详细日志 ~ansible-playbook nginx.yaml -vvv~

** 执行顺序
   1) 变量加载
   2) Facts收集
   3) pre_tasks执行
   4) pre_tasks通知的handlers执行
   5) roles执行
   6) tasks执行
   7) handlers执行
   8) post_tasks执行
   9) post_tasks通知的handlers执行

** Handlers
   作为Ansible Playbook的Event机制。

*** 执行特点
    1) 当任务执行结果属性changed为changed时，才会触发notify中定义的handlers
    2) handler在整个Play执行完后才执行，并且同一个Play中多次触发同一个handler时，仅执行一次
       #+BEGIN_EXAMPLE yaml
       # task1,2,3执行后才会执行handlers
       - hosts: test
         name: configurate httpd
         tasks:
            - name: 1. ensure apache is a the latest version
              yum: name=httpd state=latest

            - name: 2. write the apache config file
              template: 
                  src: /srv/httpd.conf.j2
                  dest: /etc/httpd.conf
              notify:
                  - restart apache

            - name: 3. ensure apache is running
              service: name=httpd state=started

         handlers:
             - name: restart apache
               service: name=httpd state=restarted
       #+END_EXAMPLE
    3) handler按定义顺序执行，而不是按在任务中调用顺序执行
    4) handler内可以再触发其他handlers
       #+BEGIN_EXAMPLE yaml
       handlers:
           - name: restart apache
             service: name=httpd state=restarted
             notify: restart memcached

           - name: restart memcached
             service: name=memcached state=restarted
       #+END_EXAMPLE
    5) 若要在Playbook中间运行handlers，则要用meta模块 ~meta: flush_handlers~
    6) 若Playbook中途失败，那么不会执行已触发的handlers

*** 应用场景
    1) 重启服务
       #+BEGIN_EXAMPLE yaml
       ---
       # Nginx setup on Windows
       - hosts: nginx
         gather_facts: no
         tasks:
             - name: install nginx
               win_chocolatey:
                   name: nginx
                   state: present
             - name: install nssm
               win_chocolatey:
                   name: nssm
                   state: present
             - name: copy nginx.conf
               win_template:
                    src: nginx.conf.j2
                    dest: d:\path\to\nginx\conf\nginx.conf
             - name: install nginx as windows service
               win_nssm:
                    name: nginx
                    application: d:\path\to\nginx\nginx.exe
                    app_parameters_free_from: -c d:\path\to\nginx\conf\nginx.conf
                    stdout_file: d:\var\log\nginx.stdout
                    stderr_file: d:\var\log\nginx.stderr
                    start_mode: auto
                    state: started # 通过win_nssm安装服务时虽然已经设置state为started，但服务仍然未运行的，要通过handlers启动
                    notify:
                        - start nginx
         handlers:
             - name: start nginx
               win_service:
                   name: nginx
                   start_mode: auto
                   state: started
       #+END_EXAMPLE
    2) 重启系统

*** 优先级
    | 优先级 | 变量类型                                                     |
    |--------+--------------------------------------------------------------|
    | 高     | extra variables                                              |
    |        | task varialbes                                               |
    |        | block variables                                              |
    |        | role variables and include variables                         |
    |        | play vars_files                                              |
    |        | play vars_prompt                                             |
    |        | play variables                                               |
    |        | set_facts，即inventory主机清单中内置变量(ansible_ssh_user等) |
    |        | registerd variables                                          |
    |        | host facts                                                   |
    |        | playbook host_vars                                           |
    |        | playbook group_vars                                          |

*** 内置变量

    ~inventory_hostname_shortname~ 若inventory_hostname返回server1.test.com，那么inventory_hostname_shortname则返回server1
    ~hostvars~ 获取某台主机的facts信息，通过 ~hostvars[inventory_hostname]~ 获取当前执行任务的facts信息
    ~group_names~ 获取当前执行任务的主机所属的主机组的主机名列表
    ~groups~ 所有主机组的字典，如 ~groups.web~ 获取 ~[web]~ 这组主机组的所有主机facts信息
    #+BEGIN_EXAMPLE ini
    [web]
    server1
    server2
    #+END_EXAMPLE
    #+BEGIN_EXAMPLE jinja2
    upstream backend {
        {% for host in groups.web %}
        server {{host.inventory_hostname}} {{host.ansible_default_ipv4.address}}:80;
        {% endfor %}
    }
    #+END_EXAMPLE
    ~inventory_dir~ inventory清单所在目录
    ~inventory_file~ inventory清单
    ~play_hosts~ 返回当前playbook运行主机信息

*** Facts变量
    为远程目标主机的IP/OS/以太网设备/MAC地址等系统/硬件信息.
    默认情况下每次执行play时会先执行收集Facts变量,如同先执行 ~ansible all -m setup~ ,并将返回结果保存到内存中,直到playbook结束.
    返回的JSON以属性 ~ansible_facts~ 为开头。

**** 返回值意思
     ansible_fqdn 127.0.0.1的域名
    
**** 两种方式优化
     1) 在playbook的每个play中添加 ~gather_facts: no~, 禁止收集Facts变量
     2) Ansible1.8+可缓存Facts变量

     # 启用缓存
     # smart, 默认收集Facts,但有缓存时则不收集
     # implicit, 不收集Facts
     # explicit, 强制收集Facts
     gathering = smart

     # 缓存到Redis
     fact_caching_timeout = 86400
     fact_caching = redis
     fact_caching_connection = 127.0.0.1:6379

     # 缓存到本地文件
     # fact_caching = jsonfile
     # fact_caching_connection = /tmp/mycachedir
     #+END_EXAMPLE

     *QA*: 若缓存到Redis后报
     #+BEGIN_QUOTE
     TypeError: the JSON object must be str, not 'bytes'
     #+END_QUOTE
     则将ansible/plugins/cache/redis.py中90行
     #+BEGIN_SRC python
     self._cache[key] = json.loads(value)
     # 修改为
     self._cache[key] = json.loads(value.decode('utf-8'))
     #+END_SRC

** 条件判断
   通过 ~when~ 语句实现if语句，其中的逻辑运算由Jinja2提供。
   #+BEGIN_EXAMPLE yaml
   ---
   # 测试字符串
   vars: 
      url: abc
      a: [1,2,3]
      b: [2,3]
      path: path/to
   tasks:
      - debug: msg="1"
        when: url | match(".*c") # 完全匹配
      - debug: msg="2"
        when: url | search("c1+") # 部分匹配

   # 版本比较
      - debug: msg="3"
        when: ansible_distribution_version | version_compare('12.04', operator='lt', strict=True)

   # 集合测试
      - debug: msg="4"
        when: a | issuperset(b)
      - debug: msg="5"
        when: b | issubset(a)

   # 文件路径
      - debug: msg="file"
        when: path | is_file
      - debug: msg="directory"
        when: path | is_dir
      - debug: msg="symbol link"
        when: path | is_link
      - debug: msg="exists"
        when: b | exists

   # 测试任务执行结果
      - shell: /usr/bin/foo
        register: result
        ignore_errors: yes

      - debug: msg="failed"
        when: result | failed

      - debug: msg="changed"
        when: result | changed

      - debug: msg="succeeded in Ansible >=2.1"
        when: result | succeeded 

      - debug: msg="succeeded"
        when: result | success 

      - debug: msg="skipped"
        when: result | skipped 
   #+END_EXAMPLE

** 循环结构

*** 标准Loops
    被循环的值为Python List类型
    #+BEGIN_EXAMPLE yaml
    ---
    - name: print
      shell: "echo {{item}}"
      with_items:
          - item1
          - item2

    - name: print property
      shell: "echo {{item.name}} {{item.age}}"
      with_items:
          - {name: "a", age: 1}
          - {name: "b", age: 2}
    #+END_EXAMPLE

*** 嵌套Loops
    实现一对多或多对多的合并。
    #+BEGIN_EXAMPLE yaml
    ---
    - shell: "echo {{item[0]}}--{{item[1]}}"
      with_nested:
        - ["A"]
        - ["a", "b"]
    # 回显 A--a、A--b
    #+END_EXAMPLE

*** 散列Loops
    被循环的值为Python Dict类型
    #+BEGIN_EXAMPLE yaml
    ---
    vars:
        users:
            john:
                name: john
                age: 18
            mary:
                name: mary
                age: 20
    tasks:   
        - shell: "echo {{item.key}}--{{item.value.age}}"
          with_dict: user
    # 回显 john--18、mary--20
    #+END_EXAMPLE
    
*** 文件匹配Loops
    #+BEGIN_EXAMPLE yaml
    ---
    - shell: "echo {{item}}"
      with_fileglob:
          - /root/*.yaml
    #+END_EXAMPLE

*** 随机Loops
    从列表中随机选中一个值执行
    #+BEGIN_EXAMPLE yaml
    ---
    - shell: "echo {{item.key}}--{{item.value.age}}"
      with_random_choice: 
          - {name: "a", age: 1}
          - {name: "b", age: 2}      
    #+END_EXAMPLE

*** 条件判断Loops
    多次间隔执行某项操作，若执行成功则该任务成功，若超过循环次数依然失败则该任务被认为执行失败
    #+BEGIN_EXAMPLE yaml
    ---
    # 每5秒执行一次 cat /root/Ansible，然后检查回显是否以Master开头
    # 若尝试5次后都失败则该任务失败，否则为成功
    - shell: cat /root/Ansible
      register: result
      until: result.stdout.startswith("Master")
      retries: 5
      delay: 5
    #+END_EXAMPLE

*** 文件优先匹配Loops
    和文件匹配Loops相似，但当第一个匹配后就跳出循环
    #+BEGIN_EXAMPLE yaml
    ---
    - shell: "echo {{item}}"
      with_first_found:
          - /root/a.yaml
          - /root/b.yaml
    #+END_EXAMPLE

*** register Loops
    当在循环中使用register保存执行结果时，那么register中则保存则任务执行结果集合，需要通过Jinja2的循环来读取
    #+BEGIN_EXAMPLE yaml
    - name: print
      shell: "echo {{item}}"
      with_itemes:
          - item1
          - item2
      register: ret

   - debug: msg="{% for i in ret.results%} {{i.stdout}} {% endfor %}"
    #+END_EXAMPLE

** 块结构
   Block块将多个action组装成块，可根据不同条件执行一组action。
   #+BEGIN_EXAMPLE yaml
   --- 
   - hosts: test
     tasks:
         - block:
             - yum: name={{ item }} state=installed
               with_items:
                   - nginx
                   - memcached
             - template: src=templates/src.j2 dest=/etc/foo.conf
             - service: name={{ service-name }} state=started enabled=True

           vars:
             service-name: boo
             
           when: ansible_distribution == 'CentOS'
           rescue:
               - debug: msg="I caught an error"
               - command: /bin/false
               - debug: msg="I also never execute :-("
           always:
               - debug: msg="this always executes"
   #+END_EXAMPLE

** 标签
   通过 ~tags~ 为 *任务/角色/include语句* 打上标签，可实现按需执行部分任务的功能。
   #+BEGIN_EXAMPLE yaml
   - shell: echo helloworld
     tags:
       - helloworld
       - first_tag
   - shell: echo "second tag"
     tags:
       - second_tag
   #+END_EXAMPLE
   #+BEGIN_EXAMPLE bash
   # 仅执行tags包含first_tag的任务
   ansible-playbook script.yaml --tags first_tag
   # 不执行tags包含helloworld的任务
   ansible-playbook script.yaml --skip-tags helloworld
   #+END_EXAMPLE

*** 特殊标签-always
    任务属性 ~tags~ 打上 *always* 后，除非通过CLI选项 ~--skip-tags always~ 明确排除，否则其对应的任务必定会被执行。

*** 特殊标签-tagged
    设置在CLI选项 ~--tags tagged~ ，即只执行打上标签的任务。
    设置在CLI选项 ~--skip-tags tagged~ ，不执行打上标签的任务。

*** 特殊标签-untagged
    设置在CLI选项 ~--tags untagged~ ，即只执行没有打上标签的任务。
    设置在CLI选项 ~--skip-tags untagged~ ，不执行没有打上标签的任务。

*** 特殊标签-all
    设置在CLI选项 ~--tags all~ ，即执行所有任务。
    设置在CLI选项 ~--skip-tags all~ ，跳过所有任务。

** 读取动态变量
   通过lookups插件实现，并且读取操作均在被控节点上执行。
   
*** lookups file
    通过Python的codecs.open打开文件，并将文件内容作为结果返回给变量。
    #+BEGIN_EXAMPLE yaml
    ---
    vars:
        content: "{{ lookup('file', '/etc/sysconfig/network') }}"
    tasks:
        - debug: msg="{% for i in content.split("\n") %} {{ i }} {% endfor %}"
    #+END_EXAMPLE

*** lookups password
    对入参进行加密处理。
    #+BEGIN_EXAMPLE yaml
    ---
    vars:
        content: "{{ lookup('password', 'network') }}"
    tasks:
        - debug: msg="{{content}}"
    #+END_EXAMPLE

*** lookups pipe
    通过Python的subprocess.Popen执行命令将stdout作为结果返回给变量
    #+BEGIN_EXAMPLE yaml
    ---
    vars:
        content: "{{ lookup('pipe', 'date +%Y-%m-%d') }}"
    tasks:
        - debug: msg="{{content}}"
    #+END_EXAMPLE

*** lookups redis_kv
    从Redis中读取变量，因此被控节点需要安装Redis Python库。
    #+BEGIN_EXAMPLE yaml
    ---
    vars:
        content: "{{ lookup('redis_kv', 'redis://localhost:6379,ansible') }}"
    tasks:
        - debug: msg="{{content}}"
    #+END_EXAMPLE

*** lookups template
    和lookups file相似，不过在读取内容时会先用被控节点的facts渲染Jinj2模板后在作为结果返回给变量
    #+BEGIN_EXAMPLE jinja2
    # /etc/ansible/lookups.j2
    worker_processes {{ ansible_processer_cores }}
    IPaddress {{ ansible_eth0.ipv4.address }}
    #+END_EXAMPLE
    #+BEGIN_EXAMPLE yaml
    ---
    vars:
        content: "{{ lookup('template', '/etc/ansible/lookups.j2') }}"
    tasks:
        - debug: msg="{{content}}"
    #+END_EXAMPLE

*** lookups env
    从被控节点中读取环境变量
    #+BEGIN_EXAMPLE yaml
    ---
    vars:
      content: "{{ lookup('env', 'HOME') }}"  
    #+END_EXAMPLE

** 错误处理和状态转换
   #+BEGIN_EXAMPLE yaml
   # 忽略错误
   - name: this will not be counted as a failure
     command: /bin/false
     ignore_errors: yes

   # 自定义错误判定条件
   - name: this command prints FAILED when is fails
     command: /usr/bin/xxx -x -y -z
     register: command_result
     failed_when: "'FAILED' in command_result.stderr"

   # 自定义changed判断条件
   - name: validate nginx conf
     shell: "/usr/local/nginx/sbin/nginx -t"
     register: command_result
     changed_when: "command_result.stdout.find('successful')"
   #+END_EXAMPLE

** 进阶操作

*** delegate_to
    任务委派：定义任务将交给非当前play的hosts指定的目标主机执行．但facts依然为当前host的。
    1) 示例1
       #+BEGIN_EXAMPLE yaml
       ---
       - hosts: 10.1.1.1 10.1.1.2
         tasks:
           - name: Ping
             ping:

           - name: Ping 10.1.2.3
             ping:
             delegate_to: 10.1.2.3 # 由于hosts有2台目标主机，那么执行2次 ping 10.1.2.3

            - name: Create file local
              shell: 'touch /temp/test.txt'
              delegate_to: localhost
              run_once: yes
       #+END_EXAMPLE

    2) 示例2-向监控服务器添加配置信息

** 角色-依赖管理
   假如角色A和角色B执行前需要先执行角色C，那么可以在角色A和角色B的meta/main.yml中注明如何执行角色C。
   #+BEGIN_EXAMPLE yaml
   ---
   # role_a/meta/main.yml
   # role_b/meta/main.yml
   dependencies:
       - role: common
         vars:
           name: "NameInDb"
   #+END_EXAMPLE
   那么在顶层Playbook中
   #+BEGIN_EXAMPLE yaml
   ---
   # 由于角色A和角色B都以相同的入参依赖角色C，因此Ansible会合并依赖，并先执行角色C，然后一次执行角色A和角色B。

   - hosts: all
     roles:
        - role_a
        - role_b
   #+END_EXAMPLE
   *注意：若依赖的角色相同，但入参不同，则认为依赖不同的角色，不会合并依赖。*

* QA
1) 部署代码时中文乱码
   解决方法：在Ansible配置文件的defaults section下修改为
   #+BEGIN_EXAMPLE
   # module_lang = C
   # module_set_locale = True
   module_lang = en_US.UTF-8
   module_set_locale = True
   #+END_EXAMPLE
2) SSH连接失败
   原因：ansible底层采用sshpass进行ssh连接，默认情况只能连接已验证过host key的主机。（即必须已经通过SSH成功连接过）
   解决方法：在Ansible配置文件的defaults section下修改为
   #+BEGIN_EXAMPLE
   host_key_checking=False
   #+END_EXAMPLE
