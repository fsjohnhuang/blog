* 特点
1) 单个服务器支持5W并发量
2) CPU、内存等资源消耗低
3) 支持HTTP/IMAP/POP3/SMTP
4) 采用异步非阻塞方式处理请求
5) 多个连接可以对应一个进程
6) 工作进程数和CPU核数相同
7) 支持热部署、热加载和在线升级
8) 模块化开发(非DSO，加入新模块要重新编译整个Nginx)
   1) http和location这些核心模块则定义在 http://nginx.org/en/docs/http/ngx_http_core_module.html 上
9) 支持FastCGI(LNMP)和uWSGI(Python)等协议

* 反向代理(reverse proxy)
~proxy_pass protocal://ip|domain:port[uri]~
- 作用：根据URI将请求转发到不同功能的服务器(集群)
- 上下文：location, if in location, limit_except
- 示例
  1) unix domain socket ~proxy_pass unix://tmp/backend.socket:/uri/;~
  2) http 和 https ~proxy_pass <http|https>://(ip:port[uri] | upstream_name);~
     #+BEGIN_EXAMPLE bash
     server_name test.com;

     # 当客户端访问 http://test.com/forum/index.html
     # 会被转发到 http://196.168.56.191:8080/bbs/index.html
     location /forum/ {
       proxy_pass http://196.168.56.191:8080/bbs/;
     }

     # 当使用 ~,~*,^~,= 等进行模式匹配或@名字时，proxy_pass后的url只能设置为 <protocol>://<ip>:<port> 的形式
     # location后的URI部分将被直接传递至上游服务器，而不能为其指定转换为另一个URI。
     # 当客户端访问 http://test.com/forum/index.html
     # 会被转发到 http://196.168.56.191:8080/forum/index.html
     location ~* ^/forum/ {
       proxy_pass http://196.168.56.191:8080;
       proxy_pass_request_body off; # 不传递请求体给上游服务
       proxy_set_header Content-Length ""; # 不传递请求体给上游服务
       proxy_set_header X-Original-URI $request_uri;
     }

     # 当使用URL重定向，那么Nginx将使用重定向后的URI处理请求，而不再考虑上游服务器上定义的URI
     # 请求被转发到http://196.168.56.191:8080/index.php?page=$1
     location / {
       rewrite /(.*)$ /index.php?page=$1 break;
       proxy_pass http://196.168.56.191:8080/index;
     }
     #+END_EXAMPLE

** HTTP反向代理流程
   1. 处理connect阶段：proxy_pass指令；若开启cache且命中则执行步骤3，否则执行步骤2
   2. 未开启cache或击穿cache：
      1. 根据指令生成发往上游的HTTP头部及包体(由于一般上游服务的并发量并没有Nginx强大，因此先准备好数据再建立和上游服务的连接，从而缩短连接时间来降低
      2. 此时Nginx仅接收到HTTP头部的信息，然后根据 ~proxy_request_buffering~ 的值执行后续步骤，默认on。
         - on 缓冲HTTP请求包体到磁盘，等待HTTP请求包体完整后再执行后续步骤
           - 优点：缩短和上游服务连接的打开时间，从而降低对上游服务负载的影响。
           - 缺点：增大上游服务处理的迟延。
         - off 马上执行后续步骤，并且边读取HTTP请求包体边转发到上游服务
           - 优点：提高上游服务处理的时效性
           - 缺点：延长上游服务连接的打开时间。
      3. 根据负载均衡策略选择上游服务器
      4. 根据参数连接上有服务器
      5. 发送请求到上游服务
      6. 接收上游服务返回的响应头部
      7. 处理响应头部
      8. 根据 ~proxy_buffering~ 执行后续步骤，默认为on
         - on 通过临时文件缓冲响应包体(若Nginx和上游服务器均位于内网，那么它们间的网速非常快)
         - off 边接收响应包体，边发送给下游客户端(速度受限于Nginx和下游客户端的网速)
      9. 进入第3步骤
   3. 发送响应头部给下游客户端
      1. 发送响应包体给下游客户端
      2. 若打开cache则将包体加入缓存
      3. 关闭或复用连接

** 缓冲(buffer)
   1) 缓冲(buffer)和缓存(cache)的区别
      - 缓冲(buffer)
        - 作用：将数量多规模小的IO操作整理成数量少规模大的IO操作。
        - 特点：
          1) 一般为写IO因此也称为write-buffer，但读IO也可使用buffer
          2) 一般为顺序读写
      - 缓存/快取(cache)
        - 作用：当系统两端处理速度不匹配时的一种折中的方案，如CPU和Memory间分为多级cache实现CPU快速读取指令和数据
        - 特点：
          1) 一般为读操作因此也称为read-cache，但写操作也可使用cache
          2) 必须支持随机读写操作
   2) Nginx中针对代理(proxy)的缓冲设置
      - 注意：
        1) 缓冲(buffer)针对的是从应用服务返回的响应消息，是接收后尽快返回给客户端，还是先暂存在缓冲区然后当buffer被写满或应用服务返回的响应消息被读取完毕(EOF)后才开始向客户端传输。
        2) 下列指定的数值均为针对每个http response的上限，而不是对于整个buffer区指定的上限。
      - ~proxy_request_buffering on | off~
        - 作用：接收下游客户端请求的包体，是接收完再和上游服务建立连接发送，还是先建立连接边接收边发送
        - 上下文：http,server,location
        - on的应用场景
          1. 下游客户端网络较慢
          2. 上游服务并发处理能力较弱
          3. 需要高吞吐量的场景，利用Nginx的高性能特点
        - off的应用场景
          1. 需要更及时的响应
          2. 降低Nginx读写磁盘的消耗
          3. 一旦开始发送内容，proxy_next_upstream功能失效
      - ~client_body_buffer_size 8k | 16k~
        - 作用：若存在请求包体，则按情况分配缓冲区。
          1. 若接收请求头部时，已经接收完全部的包体，则不用分配缓冲区内存
          2. 若剩余待接收的包体长度小于等于client_body_buffer_size的值，则按实际需要分配缓冲区内存
          3. 若剩余待接收的包体长度大于client_body_buffer_size的值，则按client_body_buffer_size分配缓冲区内存。
             然后按照client_body_buffer_size的大小接收请求包体，若 ~proxy_request_buffering~ 为 on 则执行 步骤1， 否则执行 步骤2
             1. 若还没接收完，则将缓冲区内存的数据写入临时文件，并释放缓冲区内存空间继续按照client_body_buffer_size的大小接收请求包体，直到接收完毕。
             2. 及时将缓冲区内存的数据发送给上游服务器。
        - 上下文：http,server,location
      - ~client_body_in_single_buffer off | on~
        - 作用：将请求包体全部存放在缓冲区内存中。
        - 上下文：http,server,location
      - ~client_max_body_size 1m~
        - 作用：检查Content-Length表示的请求包体长度，若超过则返回413错误。
        - 上下文：http,server,location
      - ~client_body_temp_path path [level1 [level2 [level3]]]~
        - 作用：client_body_buffer_size 缓冲区对应的临时文件路径
        - 默认值: ~client_body_temp_path client_body_temp~ 及在Nginx目录下的client_body_temp目录
        - 由于目录下存放太多文件会严重影响磁盘IO性能，因此需要通过level1,level2,level3定义多级子目录路径
        - 上下文：http,server,location
      - ~client_body_in_file_only off | on | clean~
        - 作用：包体必须存放在文件中
          - on 即使响应结束也不会删除存放请求包体的临时文件
          - off 当包体长度小于client_body_buffer_size时则不写入临时文件，若写入则响应结束删除存放请求包体的临时文件
          - clean 包体一定要写入临时文件，响应结束删除存放请求包体的临时文件
        - 上下文：http,server,location
      - ~client_body_timeout 60s~
        - 作用：读取请求包体超时，返回408错误
        - 上下文：http,server,location        
      - ~proxy_buffering on | off~
        - 作用：缓存从应用服务返回的内容，然后再统一返回给客户端。返回的内容会暂存到由 ~proxy_buffers~ 和 ~proxy_buffer_size~ 指定的内存缓冲区中，若内存缓冲区已满则写入磁盘上。
        - 默认值： ~proxy_buffering on;~
        - 上下文：http,server,location
        - off的适用场景：
          1. 对于采用长轮训的应用而言，必须关闭缓冲区。
          2. 响应大文件。
        - 上游服务启用/禁用proxy_buffering的方式
          - 响应头部 X-Accel-Buffering 设置为yes或no
        ~proxy_buffers number size~
        - 作用：设置缓冲区数量和每个缓冲区的大小
        - 上下文：http,server,location
        - 示例：
          #+begin_example bash
          proxy_buffers 4 8k;
          #+end_example
        ~proxy_buffer_size 4k|8k~
        - 作用：设置仅用于缓冲应用服务响应头的缓冲区容量，因此其值一般比 ~proxy_buffers~ 小。
        - 上下文：http,server,location
        - 注意：
          1) 即使 ~proxy_buffering off~ 这个缓冲区也会被开启
          2) 当 ~proxy_buffering off~ 那么Nginx会尽快转发应用服务的响应消息给客户端，期间的缓冲区最大值为 ~proxy_buffer_size~ 的值
          3) 若上游服务的响应头部的长度大于 ~proxy_buffer_size~ 的值，那么Nginx会直接结束流程，而且error.log会记录upstream sent too big header。
        ~proxy_busy_buffers_size 8k|16k~
        - 作用：缓冲区被写入后，直到缓冲区中的数据被完整地传输完，这个缓冲区一直处于busy状态，即不能对该缓冲区作任何操作。因此 ~proxy_busy_buffers_size~ 用于控制同时传输数据给客户端的缓冲区最大容量。
        - 上下文：http,server,location
        ~proxy_temp_path path [level1 [level2 [level3]]]~
        - 作用：设置临时文件的目录
        - 默认值: ~proxy_temp_path proxy_temp~
        - 上下文：http,server,location        
        ~proxy_temp_file_write_size 8k|16k~
        - 作用：每次写入临时文件的大小
        - 上下文：http,server,location
        ~proxy_max_temp_file_size 1024m~
        - 作用：设置缓冲区耗尽时，写入磁盘的文件大小上限。
        - 上下文：http,server,location
        ~proxy_limit_rate 0~
        - 作用：设置读取上游服务响应的速度，0为无上限。
        - 上下文：http,server,location

** 代理重定向
   ~proxy_redirect default|off|regex replacement;~ 和
   - 作用：当上游服务器返回的响应是重定向或刷新请求（如HTTP响应码是301或者302）时，proxy_redirect可以重设HTTP头部的location或refresh字段。
   - off 即不修改Location和Refresh的值
   - default 则根据server_name和location来设置
     #+begin_example bash
     listen 80;
     server_name myproxy.com;
     
     # 下面两种设置是等价的
     location /one/ {
         proxy_pass http://upstream:port/john/;
         proxy_redirect default;
     }
      location /one/ {
         proxy_pass http://upstream:port/john/;
         proxy_redirect http://upstream:port/john/ http://myproxy.com:80/one/;
     }

     # 那么当服务端返回的Location为http://upstream:port/john/hi.html时，反向代理返回给客户端的Location就是http://myproxy.com:80/one/hi.html了。
     #+end_example
   - redirect replacement
     #+begin_example bash
     # 使用反向代理服务的server_name和port
     proxy_redirect http://uptstream:port/john/ /;
     # http转换为https
     proxy_redirect ~^http://([^:]+)(:\d+)?(.*)$ https://$1$2$3;
     #+end_example
   - 默认值： ~proxy_redirect default;~
   - 上下文：http,server,location

** 修改发送给上游服务的内容
   - ~proxy_set_header header value;~
     - 作用：设置发送给应用服务的请求头，若value为空字符串则整个Header都不会向上游服务发送
     - 默认自动添加：
       #+begin_src
       proxy_set_header Host $proxy_host;
       proxy_set_header Connection close;
       #+end_src
     - 上下文：http,server,location
     - 示例
       #+begin_example bash
       location / {
           proxy_pass https://fsjohnhuang.io;
           proxy_redirect default;
           proxy_set_header Host $host;
           proxy_set_header X-Real-IP $remote_addr;
           proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
       }
       #+end_example
   - ~proxy_hide_header header~
     - 作用：默认情况下代理接收的请求头将被复制到发送给应用服务的请求头上，通过该指令可以移除指定的请求头
     - 上下文：http,server,location
   - ~proxy_pass_request_headers on | off~
     - 作用：是否直接下游服务的请求头部发送给上游服务
     - 上下文：http,server,location
   - ~proxy_set_body value~
     - 作用：自定义发送给上游服务的请求包体
     - 上下文：http,server,location
   - ~proxy_pass_request_body on | off~
     - 作用：是否直接下游服务的请求包体发送给上游服务
     - 上下文：http,server,location
   - ~proxy_method method~
     - 上下文：http,server,location
   - ~proxy_http_version 1.0 | 1.1~
     - 上下文：http,server,location

** 处理上游服务返回的响应头部
   - ~proxy_ignore_headers field...~
     - 作用：某些响应头部可以改变Nginx的行为，适用proxy_ignore_headers可以禁止它们生效
     - 可以禁止的头部
       - ~X-Accel-Redirect~ 由上游服务指定在Nginx内部重定向，控制请求的执行
       - ~X-Accel-Limit-Rate~ 由上游设置发往客户端的速度限制，等同proxy_limit_rate指令
       - ~X-Accel-Buffering~ 由上游控制是否缓存上游的响应
       - ~X-Accel-Charset~ 由上游控制Content-Type中的Charset
       - 缓存相关
         - ~X-Accel-Expires~ 设置响应在Nginx的缓存时间，单位秒；@开头表示一天某时刻
         - ~Expires~ 控制Nginx缓存时间，优先级低于X-Accel-Expires。同时可以控制浏览器的缓存时间
         - ~Cache-Control~ 控制Nginx缓存时间，优先级低于X-Accel-Expires。同时可以控制浏览器的缓存时间
         - ~Set-Cookie~ 响应出现Set-Cookie则Nginx不缓存，可通过proxy_ignore_headers启用Nginx缓存
         - ~Vary~ 响应中出现 Vary: * 则Nginx不缓存，可通过proxy_ignore_headers启用Nginx缓存
     - 上下文：http, server, location
   - ~proxy_hide_header field~
     - 作用：设置某些响应头部不转发给下游客户端
     - 默认不转发的头部：
       - ~Date~ 由ngx_http_header_filter_module过滤模块写入，值为nginx发送响应头部的时间
       - ~Server~ 由ngx_http_header_filter_module过滤模块写入，值为nginx版本号
       - ~X-Pad~ 通常是Apache为避免浏览器BUG生成的头部
       - ~X-Accel-*~ 用于控制Nginx行为的响应头部
     - 上下文：http, server, location
   - ~proxy_pass_header field~
     - 作用：转发被proxy_hide_header禁止转发的上游响应头部
     - 上下文：http, server, location
   - ~proxy_cookie_domain (off | domain replacement)~
     - 作用：将匹配domain的Set-Cookie头部(可能是多个)中domain的值替换为replacement的值
     - 上下文：http, server, location
   - ~proxy_cookie_path (off | path replacement)~
     - 作用：将匹配path的Set-Cookie头部(可能是多个)中path的值替换为replacement的值
     - 上下文：http, server, location

** 容灾和重复处理
   ngx_http_proxy_module中的
   - ~proxy_next_upstream~ 指令
     - 语法： ~proxy_next_upstream error | timeout | invalid_header | http_500 | http_502 | http_503 | http_504 | http_403 | http_404 | http_429 | non_idempotent | off ...;~
       - error 和应用服务建立连接、发送请求或接收响应时出现错误
       - timeout 和应用服务建立连接、发送请求或接收响应时出现超时
       - invalid_header 应用服务返回空响应或非法响应
       - http_500 应用服务返回的状态码为500
       - http_502 应用服务返回的状态码为502
       - http_503 应用服务返回的状态码为503
       - http_504 应用服务返回的状态码为504
       - http_403 应用服务返回的状态码为403
       - http_404 应用服务返回的状态码为404
       - http_429 应用服务返回的状态码为429
       - non_idempotent 默认情况下请求方式为POST,LOCK,PATCH的不会发送到下一台应用服务，启用此项目则允许重复此类请求
       - off 禁止将请求传递给下一个应用服务
     - 作用：当代理和应用服务通信过程中出现指定情况时，访问会继续被分配到下一台应用服务上处理，从而实现容灾处理。
     - 默认值： ~proxy_next_upstream error timeout~
     - 上下文：http,server,location
     - 示例
       #+begin_example bash
       proxy_next_upstream error timeout invalid_header http_500 http_502 http_503 http_504 http_404;
       #+end_example
   - ~proxy_next_upstream_tries times;~
     - 作用：设置重试次数，0表示不限制。
     - 默认值： ~proxy_next_upstream_tries 0;~

  注意：当 ~proxy_request_buffering off~ 时，已经建立连接并发送数据，那么中途失败也不会触发重置机制。

   - 默认值： ~slice 0~
   - 上下文：http,server,lcation

* 负载均衡
ISO第7层的负载均衡方案

** AKF扩展立方体
   - X Axis
     - 水平/横向扩展
     - 算法：Round-Robin、Least-Connected
     - 成本：最低
     - 缺点：无法解决数据量大的问题，因此水平扩展后每台服务器所操作的数据量并没有发生变化。
   - Y Axis
     - 垂直扩展
     - 算法：按功能将单体应用重构为系统群或微服务，并基于URI通过location转发请求。
     - 成本：最高
   - Z Axis
     - 算法：基于用户IP或其他信息转发请求到特定上游服务。

** GSLB和SLB
   GSLB为全局负载均衡，以省甚至国家为范围实现负载均衡。
   SLB则调度节点和逻辑节点位于同一组部署单位上。Nginx实现的就是属于SLB。

** upstream定义负载均衡的后端服务组
   - 语法
     #+begin_example bash
     upstream <name> {
       server <ip|domain name|unix style uri> [option...];
       ...
     }
     #+end_example
   - option
     - down 表示当前server暂时不参与负载均衡
     - backup 表示当前server为备用服务器，当其它所有非备用服务器均down或忙时才将请求转发到这台服务器
       可实现主备模式。
     - max_fails 设置在fail_timeout时间段内，允许请求失败的次数。若达到则会在fail_timeout秒内请求不会转发到该台上游服务。
     - fail_timeout 设置经过max_fails失败后，服务暂停的时间。如10s, 5m, 6d等。默认为10s
     - max_conns 设置单worker进程最大的并发连接数，默认为0即没有限制
     - weight 服务访问的权重，默认为1
   - 上下文：http

** 内置均衡策略
#+BEGIN_EXAMPLE bash
http {
  # 轮询(RR,Round-Robin)
  # 后端宕机会自动剔除
  upstream balance1{
    server 123.0.0.1 max_fails=1 fail_timeout=10s;
    server 123.0.0.2;
  }

  # 加权轮询
  upstream balance2{
    server 123.0.0.1 max_fails=1 fail_timeout=10s weight=2; # 权重2/3
    server 123.0.0.2 weight=1; # 权重1/3
    server 123.0.0.3; # 没有赋予权重则表示该服务器不参与负载均衡
  }

  # 源地址散列
  # 以客户端的IP作为hash算法的关键字，映射到特定的上游服务器中。
  # 对IPv4地址使用前3个字节作为关键字，对IPv6则使用完整的地址。
  # 可以基于realip模块修改用于执行算法的IP地址。
  upstream balance3{
    ip_hash;
    # max_fails 连续请求失败n次后，后续请求不转发到该后端
    # fail_timeout max_fails条件符合后，多少秒不转发请求到后端
    server 123.0.0.1 max_fails=1 fail_timeout=10s;
    server 123.0.0.2;
    server 123.0.0.3 backup; # 当其他非backup后端down或忙时则将请求转发到backup后端服务
  }

  # fair，按后端服务器的响应时间来分配请求，响应时间短的优先分配。但由于实际网络环境复杂，因此要慎用。
  # least_conn，分发到最少连接数的后端服务

  # 基于任意关键字实现Hash算法的负载均衡
  # 语法：hash key [consistent]; 其中key可以是字符串、变量或两者结合，如 user_$arg_username
  # 上下文：upstream
  # 版本要求：version 1.7.2
  upstream balance4 {
     hash $request_uri;
     server a;
     server b;
     server c;
   }

  # least conn 优先选择连接最少的上游服务器的负载均衡
  # 若上游服务器的连接数一样，则使用round-robin算法。
  upstream balance5 {
     least_conn;
     server a;
     server b;
  }
  
  # 上述的负载均衡都是针对单个Worker进程进行
  # 通过zone name [size]可以针对所有Worker进程进行
  upstream balance5 {
    least_conn;
    # 分配出共享内存，让所有Nginx Worker进程都可以读写。
    # 设置了zone name [size]的upstream的负载均衡策略数据、运行时每个上游服务器状态数据都会存放到共享内存上。
    zone balance5 10m;
  }

  server {
    listen 80;
    server_name www.john.com;

    location /abc.go {
      proxy_pass http://balance1;

      proxy_redirect default;
      proxy_set_header Host $host:$server_port; # 设置转发到后端服务的请求头主机名为客户端请求头的主机名，而不是转发的主机名
      proxy_set_header X-Real-IP $remote_addr;
      proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for; # 向后端服务传输实际客户端IP
      proxy_connect_timeout 60s; # Nginx连接到后端服务的连接超时时间
      proxy_read_timeout 10m; # Nginx接收后端服务响应超时时间

      # proxy_next_upstream error | timeout | invalid | invalid_header | http_500 | http_502 | http_503 | http_504 | http_403 |  http_404 | http_429 | non_idempotnet | off
      # non_idempotent 默认情况对于POST、LOCK和PATCH方式的非幂等请求不会执行重试机制，若要重试则要添加该项目（Nginx 1.9.13后生效）
      # 上游服务出现指定问题时，转发请求到同组的下一个服务
      # 默认值：error timeout
      # 上下文：http, server, location
      proxy_next_upstream error timeout http_500 non_idempotent;

      # 设置重试操作的超时时间。0表示不限制，即直到重试次数达到proxy_next_upstream_retries为止。
      # 默认值为0
      # 上下文：http, server, location
      proxy_next_upstream_timeout 0;

      # 在proxy_next_upstream_timeout时间访问内可以执行的重试次数，若超时或超出重试次数则结束本轮请求转发操作。
      # 0表示不限制。由于该次数是包含第一次转发，因此1表示转发失败后不再重试。
      # 默认值为0
      # 上下文：http, server, location
      proxy_next_upstream_tries 0;

      # 返回后端ip和响应状态码
      add_header backendIP $upstream_addr;
      add_header backendCode $upstream_status;
    }
  }
}
#+END_EXAMPLE

*** nginx-sticky
基于Cookie来实现会话保持均衡策略，相对ip_hash优点如下
1) 可设置会话保持时间
2) 若前端有其他代理服务，则仍然可以实现负载均衡

缺点：
1) 不支持Cookie生命周期续期(Tenginx的session_sticky则支持空闲超时和强制超时等更灵活的策略)

注意：
1) 不能和ip_hash一起使用
2) 客户端第一次并发多个请求时，以最后一个响应的cookie作为转发凭证

**** 配置
#+BEGIN_EXAMPLE
upstream backend_with_sticky {
    # name Cookie键名，默认为route
　　# expires 设置cookie的生存期，默认不设置，浏览器关闭即失效，需要是大于1秒的值
    # domain $host
    # path /
　　# secure 设置启用安全的cookie，需要HTTPS支持
    # httponly 允许cookie不通过JS泄漏，没用过
    # hash index|md5|sha1，值加密算法
    # no_fallbackup　如果设置了这个，cookie对应的服务器宕机了，那么将会返回502（bad gateway 或者 proxy error），建议不启用
    sticky name=route expires=1h domain=.test.com path=/ no_fallbackup hash=index;

    server backend1.example.com;
    server backend2.example.com;
}

# Tenginx的session_sticky策略
upstream backend_with_session-sticky {
   # mode设置cookie的模式:
　 # insert: 在回复中本模块通过Set-Cookie头直接插入相应名称的cookie。
　 # prefix: 不会生成新的cookie，但会在响应的cookie值前面加上特定的前缀，当浏览器带着这个有特定标识的cookie再次请求时，模块在传给后端服务前先删除加入的前缀，后端服务拿到的还是原来的cookie值，这些动作对后端透明。如：”Cookie: NAME=SRV~VALUE”。
　 # rewrite: 使用服务端标识覆盖后端设置的用于session sticky的cookie。如果后端服务在响应头中没有设置该cookie，则认为该请求不需要进行session sticky，使用这种模式，后端服务可以控制哪些请求需要sesstion sticky，哪些请求不需要。
   # option设置用于session sticky的cookie的选项，可设置成indirect或direct。indirect不会将session sticky的cookie传送给后端服务，该cookie对后端应用完全透明。direct则与indirect相反。
   # maxidle设置session cookie的最长空闲的超时时间
   # maxlife设置session cookie的最长生存期
   # maxage是cookie的生存期。不设置时，浏览器或App关闭后就失效。下次启动时，又会随机分配后端服务器。所以如果希望该客户端的请求长期落在同一台后端服务器上，可以设置maxage。
   # hash不论是明文还是hash值，都有固定的数目。因为hash是server的标识，所以有多少个server，就有等同数量的hash值。
    session_sticky [cookie=name] [domain=your_domain] [path=your_path] [maxage=time] [mode=insert|rewrite|prefix] [option=indirect] [maxidle=time] [maxlife=time] [fallback=on|off] [hash=plain|md5]

    server backend1.example.com;
    server backend2.example.com;
}

** Upstream服务宕机

检测机制：
1) Nginx将请求转发到后端服务时，才发起后端服务检测；
2) 由于 ~proxy_next_upstream~ 的默认值为 ~error timeout~ ，因此若当前请求发现后端服务异常后，会向集群中的下一个服务器发起请求，从而实现容灾；
3) 若后端服务有问题，则再次将请求转发到另一个健康的后端服务；

*** IP Hash
    在宕机或扩容时，hash算法会引发大量路由变更，可能会导致缓存大范围失效。
    因此使用 ~ip_hash~ 算法实现负载均衡，当某台上游服务器宕机后，并不会自动移除该台服务器，而是通过 ~proxy_next_upstream~ 配置将请求转发到下一个健康的上游服务器，并在fail_timeout时间内不转发请求到该台服务器。
    但修改配置(移除或设置为down)并reload配置后，会导致大量路由变更。

    而通过 ~hash $remote_addr consistent;~ 启用一致性Hash算法，则可以减少因宕机或扩容造成路由变更的范围和数量。
    因此建议使用 ~hash $remote_addr consistent~ 代替 ~ip_hash~ 实现IP Hash算法。

** QA
1) 熔断机制

* 代理缓存(proxy cache)
  Nginx的proxy cache由proxy loader和proxy manager组成，缓存在磁盘上，而varnish则缓存在内存中(squid则出现比varnish更早)。
  HTTPD可缓存到磁盘或内存中。
  现在Nginx已经提供模块连接到Memcached。
  根据职责单一原理，反向代理应该专门处理请求转发，而静态内容(图片、文本等)应该放在专门的缓存服务器。然后配置反向代理根据请求URL转发请求到缓存服务器或应用服务器。
  但对于并发量不大的网点，可以将缓存服务集成到反向代理中，从而降低运维复杂度。

  注意：Nginx缓存中的键和缓存对象元数据存储为内存中，而值存储在磁盘上。

** 配置
   - ~proxy_cache_path~
     - 作用：设置缓存所在路径等信息
     - 语法
       #+begin_example bash
       proxy_cache_path path [levels=levels] [use_temp_path=on|off] keys_zone=name:size [inactive=time] [max_size=size]
       [manager_files=number] [manager_sleep=time] [manager_threshold=time]
       [loader_files=number] [loader_sleep=time] [loader_threshold=time]
       [purger=on|off] [purger_files=number] [purger_sleep=time] [purger_threshold=time];
       #+end_example
     - path 缓存数据存放的目录路径
     - levels 定义缓存目录的子目录级别数量和各级别目录的文件名由几个字符组成，最多3级子目录，每级子目录文件名多长2个字符。如1:2表示2级子目录，1级子目录的文件名称由1个字符，2级子目录的文件名称由2个字符组成。
     - keys_zone.name 键空间(位于共享内存中)的名称，一个键空间内包含多个键
     - keys_zone.size 键空间的大小，1MB能存放8000个键
     - max_size 缓存数据存放的目录的大小上限，若超过该上限Nginx会根据LRU规则清理缓存数据。
     - inactive 当缓存在指定时间内没有再被访问过，则被清理掉
     - use_temp_path 设置另一个目录用于存放临时文件，一般设置为关闭，防止不必要的数据拷贝。
     - manager_files，cache manager进程在1次淘汰过程中，淘汰的最大文件数，默认100
     - manager_sleep，执行1次淘汰循环后cache manager进程的休眠时间，默认200ms
     - manager_threshold，执行1次淘汰循环的最大耗时，默认50ms
     - loader_files，cache loader进程载入磁盘中缓存文件到共享内存，每次最多处理的文件数，默认100
     - loader_sleep，执行1次加载缓存文件到共享内存后，cache loader进程休眠时间，默认200ms
     - loader_threshold，执行1次加载的最大耗时，默认50ms
     - 上下文：http
     - 示例
       #+begin_example bash
       http {
         proxy_cache_path /opt/app/cache levels=1:2 keys_zone=nginx_cache:10m max_size=1g inactive=60m use_temp_path=off;
       }
       #+end_example
   - ~proxy_cache <zone.name> | off~;
     - 作用：启用代理缓存
     - 默认值： ~proxy_cache off;~
     - 上下文：http,server,location
     - 示例：
       #+begin_src
       location / {
         proxy_cache nginx_cache;
       }

       # 若存在多个磁盘，可以将缓存分布到多个磁盘，从而提高读写速度
       proxy_cache_path /path/to/hdd1 levels=1:2 keys_zone=my_cache_hdd1:10m max_size=10g inactive=60m use_temp_path=off;
       proxy_cache_path /path/to/hdd2 levels=1:2 keys_zone=my_cache_hdd2:10m max_size=10g inactive=60m use_temp_path=off;
       split_clients $request_uri $my_cache {
         50% "my_cache_hdd1";
         50% "my_cache_hdd2";
       }

       server {
          ...
          location / {
            proxy_cache $my_cache;
            proxy_pass http://my_upstream;
          }
       }
       #+end_src
   - ~proxy_cache_valid [code...] time;~
     - code 为返回状态码
     - 作用：配置缓存过期时间
     - 上下文：http,server,location
     - 示例
       #+begin_example bash
       location / {
         proxy_cache nginx_cache; # 要先启用缓存，下面的设置才生效
         proxy_cache_valid 200 304 12h; #  对于response status code 为 200或304的则缓存12小时
         proxy_cache_valid any 10m; # 其它情况则缓存10分钟
       }
       #+end_example
       - 通过上游服务的响应头部控制缓存时长
         - ~X-Accel-Expires~ 单位秒
           - 0表示禁止Nginx缓存此资源
           - @开头设置缓存到最近下一个时刻
         - 含 ~Set-Cookie~ 则禁止Nginx缓存
         - 含 ~Vary: *~ 则禁止Nginx缓存
           - ~Vary: *~ 表示所有请求都被视为唯一并且非缓存的，效果和使用 ~Cache-Control: private~ 一致。
           - ~Vary: <header name>, <header name>, ...~ 逗号分隔一系列HTTP头部名称，用于确定缓存是否可用。
   - ~proxy_cache_key string~
     - 作用：设置缓存的维度
     - 默认值： ~proxy_cache_key $scheme$proxy_host$request_uri~
     - 上下文：http,server,location
   - ~proxy_no_cache string...~
     - 作用：指定哪些条件下不缓存，当string为空时即为缓存，若其中某个string非空或为0则不缓存。
     - 上下文：http,server,location
     - 示例
       #+begin_example bash
       if ($request_uri ~ ^/(login|register|logon)) {
         set $cookie_nocache 1;
       }

       location / {
         proxy_no_cache $cookie_nocache;
       }
       #+end_example
   - ~proxy_cache_use_stale~
     - 作用：当应用服务器因宕机或繁忙等原因无法响应请求时，Nginx返回已过期的缓存内容给客户端。
     - 默认值：off
     - 上下文：http,server,location
     - 示例
       #+begin_src
       proxy_cache_use_stale error timeout updating http_500 http_502 http_503 http_504;
       #+end_src
     - error 当与上游建立链接、发送请求读取响应头部等情况出错时，使用缓存
     - timeout 当与上游建立链接、发送请求读取响应头部等情况超时，使用缓存
     - http_* 当上游返回这些响应状态码时，使用缓存
     - updating 表示缓存已过期但正在被别的Nginx Worker进程更新，但先返回了过期内容
       - stale-while-revalidate 缓存内容过期后，定义一段时间且在期间内updating设置有效，否则转发请求到上游服务
         - 例如： ~Cache-Control: max-age=600, stale-while-revalidate=30~
       - stale-if-error 缓存内容过期后，定义一段时间且在期间内上游服务出错则使用缓存，否则转发请求到上游服务。stale-while-revalidate包含stale-if-error。
         - 例如： ~Cache-Control: max-age=600, stale-if-error=30~         
   - ~proxy_cache_background_update off | on~
     - 作用：当proxy_cache_use_stale允许使用过期响应时，将同步生成一个仔请求，访问上游服务获取最新缓存
     - 上下文：http,server,location
   - ~proxy_cache_revalidate off|on~
     - 作用：若缓存过期，那么当请求该资源时Nginx会向后端服务发起带If-Modified-Since或If-none-match的请求，若后端返回304，那么缓存自动续期并返回把缓存返回给客户端。
       默认为off，即即使Expires或Cache-Control过期，但Nginx缓存没有过期，则直接返回缓存内容，而不会先检查后端服务资源是否已更新过。
     - 当服务单返回304，那么$upstream_cache_status为REVALIDATED
     - 上下文：http,server,location
   - ~proxy_cache_min_uses n~
     - 作用：设置当同一链接被请求达到多少次后才缓存起来，默认为1次。
     - 上下文：http,server,location
   - ~proxy_cache_lock off|on~
     - 作用：防止缓存击穿，即缓存MISS后仅第一个请求会被转发到后端服务，其它请求均被阻塞直到缓存生效，返回缓存内容。
     - 上下文：http,server,location
   - ~proxy_cache_lock_timeout 5s~
     - 作用：Nginx 1.7.8 新增，在第一个请求在指定时间内缓存还没更新，那么后续其它请求则全部会发送到服务端。
     - 上下文：http,server,location
   - ~proxy_cache_lock_age 5s~
     - 作用：Nginx 1.7.8 新增，在前一个请求在指定时间内缓存还没更新，那么后续请求则逐一按这规律发送到服务端。
     - 上下文：http,server,location
   - ~proxy_cache_bypass~
     - 作用：绕过缓存，直接转发请求到后端服务。
     - 上下文：location
     - $upstream_cache_status为BYPASS

** 调试
   在调试阶段于响应头中添加代理缓存命中信息，可有效查看配置是否成功。变量值如下
   1. MISS
   2. EXPIRED - expired, requesst was passed to backend
   3. UPDATING - expired, stale response was used dued to proxy/fastcgi_cache_use_stale updating
   4. STALE - expired, stale response was used dued to proxy/fastcgi_cache_use_stale updating
   5. HIT
   6. REVALIDATED - Nginx验证了陈旧的内容依然有效
   7. BYPASS - 响应是从上游服务获取的
   #+begin_example bash
   add_header Nginx-Cache "$upstream_cache_status from $server_addr";
   #+end_example

** 注意
   1. 当确定要使用缓存时，若上游服务返回的是200或206(部分返回)且已缓存过该资源，则更新Nginx该资源缓存的etag和last_modified值。

  nginx -s quit

  nginx -s reopen

  # 优雅地关闭Nginx master进程

  #+end_example

** reload命令的执行过程
   1. Master进程校验配置语法是否正确。
   2. Master进程根据配置文件，打开新的监听端口。在Linux中子进程会继承父继承打开的文件描述符，因此Master打开新的监听端口，所有Worker子进程都能访问到该端口。
   3. Master进程用新的配置文件启动新的Worker子进程。
   4. Master进程向老的Worker进程法案送QUIT信号。
   5. 老Worker进程关闭监听的连接(即使为Keep-Alive连接也会被关闭)，处理完当前连接后再结束进程。

   问题：若老Worker进程因长连接导致无法短时间内退出，那么就会影响原有的个别服务无法启用新配置。
   解决方法：新版本的Nginx提供worker_shutdown_timeout配置，在启动新Worker子进程后多少秒后强制关闭所有旧Worker进程。

** worker优雅的关闭详解
   - 什么是优雅的关闭
     Nginx的worker能识别当前连接是否空闲，若为空闲则将连接关闭。
   - 优雅的关闭的应用条件
     1. 优雅的关闭仅针对HTTP服务
        步骤：
        1. 设置定时器worker_shutdown_timeout
        2. 关闭监听文件描述符，保证不再接收新的连接请求
        3. 关闭空闲连接
        4. 在循环中等待全部连接关闭
           由于Nginx不会主动关闭连接，而是通过循环检查连接是否可以关闭，若可以即马上关闭。若超过worker_shutdown_timeout的时间，那么即使连接繁忙也有会被强制关闭。
           - 注意：
             1. 由于采用循环的方式关闭所有连接，因此退出进程的耗时可能会超过worker_shutdown_timeout设定的时间。
             2. 有可能存在部分Worker进程优雅关闭，而部分Worker进程的连接是被强制关闭的。
        5. 退出进程
     2. 对于WebSocket由于Nginx不解释其通信的帧，因此无法优雅关闭WebSocket的连接
     3. 对于TCP、UDP，由于Nginx无法知道具体传输多少数据报文才算结束，因此也无法优雅关闭TCP、UDP的连接

* Nginx的热部署/在线升级
  Nginx版本升级，或新增模块时使用。

  1. 编译目标版本的Nginx源文件，得到二进制可执行文件
  2. 备份原版本的Nginx二进制可执行文件 ~cp /path/to/nginx{,.old}~
  3. 用目标版本的Nginx二进制可执行文件覆盖原版本的
  4. 通知正在运行的Nginx主进程将要进行热部署
     - 原理：旧Nginx进程和新的Nginx进程都在运行，但新的请求都由新的Nginx来处理，旧Nginx进程仅处理已接收到的请求。
     - 步骤
       1. 运行新Nginx进程，并保留旧Nginx进程和移交请求处理权 ~kill -USR2 <old nginx master process pid>~
          - 问题：若原Nginx启动时是通过环境变量(~PATH~)的话(即不是通过绝对路径启动)，那么会报如下错误
            #+begin_quote
             execve() failed while executing new binary process “nginx” (2: No such file or directory)
            #+end_quote
          - 解决方法：必须采用绝对路径启动Nginx。
          - pid文件的变化：旧Nginnx的nginx.pid会移动到nginx.pid.oldbin，而nginx.pid则为新Nginx的master进程使用。
       2. 通知旧Nginx进程优雅地关闭worker进程，执行后旧Nginx的master进程依然在 ~kill -WINCH <old nginx master process pid>~
          注意：保留旧Nginx的master进程，那么当新Nginx有问题时可以回滚。
          - 回滚注意事项：若配置文件发生变化，则需要先回滚配置文件。
          - 回滚方式
            1. 方式一
               1. 旧主进程重新配置文件并拉起worker进程
                  #+begin_src bash
                  kill -HUP $(cat /path/to/nginx.pid.oldbin)
                  #+end_src
               2. 优雅地停止新Nginx的所有进程
                  #+begin_src bash
                  kill -QUIT $(cat /path/to/nginx.pid)
                  #+end_src
               3. 恢复旧Nginx二进制可执行文件
                  #+begin_src bash
                  cp -f /path/to/nginx.old /path/to/nginx
                  #+end_src
            2. 方式二
               1. 立刻停止新Nginx所有进程，旧Nginx主进程会自动拉起worker进程和恢复pid文件
                  #+begin_src bash
                  kill -TERM $(cat /path/to/nginx.pid)               
                  #+end_src
               2. 恢复旧Nginx二进制可执行文件
                  #+begin_src bash
                  cp -f /path/to/nginx.old /path/to/nginx
                  #+end_src
       3. 新Nginx运行正常后杀死原Nginx的master进程即可, ~kill -QUIT <old nginx master process pid>~

** 热升级的注意点
   1. 仅替换Nginx二进制文件，不要替换其它文件(如配置文件)。并且要备份原有的Nginx二进制文件，否则无法回滚。
   2. 必须向Master进程发送USR2信号，由Master进程操作Worker进程。

** 配置项
1) ~server_name server_name...~
   - 作用：设置虚拟主机的主机名
   - 3种域名格式
     1. 精确域名
        ~server_name host1.com host2.com;~
     2. 泛域名(2种)
        ~server_name *.host.com;~, ~server_name test.host.*~
     3. 正则表达式(以~为前缀)
        ~server_name ~^www(?<num>\d+)\.test\.com$~
        - location中可以通过 ~$1,$2,$num~ 等获取server_name的正则分组。
     4. 其他
        1. ~.test.com~ 匹配 ~test.com~ 和 ~*.test.com~
        2. ~_~ 匹配所有的域名
        3. ~""~ 匹配没有Host请求头的请求
   - 匹配顺序
     1. 精确匹配
     2. *在前的泛域名
     3. *在后的泛域名
     4. 按文件中的顺序匹配正则表达式域名
     5. default server
        1. 文件中第一个server块
        2. 通过 ~listen 80 default_server;~ 指定该server块就是default server
   - 上下文：http
   - 示例： ~server_name localhost;~, ~server_name localhost fsjohnhuang.io~
2) ~server_name_in_redirect off|on~
   - 作用：在301/302时Location值则为主域名
   - 上下文：http, server, location
   - 示例：
     #+begin_src
     server_name primary.test.com second.test.com;
     location / {
       return 302 /redirect;
     }

     # 当server_name_in_redirect为off时，请求second.test.com返回的为second.test.com/redirect
     # 当server_name_in_redirect为on时，请求second.test.com返回的为primary.test.com/redirect
     #+end_src
3) ~resolver ip~
   - 作用：设置DNS服务IP，nginx内部访问域名时均通过这个DNS解析
   - 上下文：http,server,location
4) ~resolver_timeout time~
   - 作用：设置DNS服务解析超时时间
   - 上下文：http,server,location
5) ~keepalive_timeout timeout [header_timeout]~
   - timeout 指定客户端的keep-alive连接超时时间，服务器将这个时间到期时关闭连接。
   - header_timeout 指定响应头 ~Keep-Alive: timeout=time~ 中的time值。从而让浏览器主动关闭连接。没有这个参数，Nginx不会返回Keep-Alive响应头。
   - 默认值： ~keepalive_timeout 75s~
   - 上下文：http,server,location
6) ~sendfile on | off;~
   - 作用：启用Linux的零拷贝
   - 默认值： ~sendfile off~
   - 上下文：http,server,location
7) ~tcp_nopush on | off;~
   - 作用：启用网络缓存，当发送的消息大小达到一定量时才统一发送出去。对于传输大文件而言有优势。
   - Nagle算法
     - 场景：当发送payload为1个字节而包头却为40个字节(IP头20字节+TCP头20字节)，将容易导致网络由于太多数据包，对于重负荷的网络而言会引发拥塞瘫痪。
     - 改进点：若发送端打算多次发送少量数据的数据包(即长度小于MSS的数据包)，那么发送端会先发送第一个小包，后面的数据包会缓存起来，直到下列其中一个条件满足才发送
       1. 发送端接收到接收端返回的ACK确认
       2. 当前数据属于紧急数据
       3. 缓存区满
     - TCP的Nagle算法默认是启用的
     - 适用场景：发送大批量数据，且接收端会及时作出回应
       1. telnet、rlogin等远程登录的长时间在线的应用
       2. 大文件传输
     - 不适用场景：及时发送小批量数据，或接收端不一定会及时回应
       1. 游戏服务器要及时跟踪用户的鼠标坐标。
       2. 没有使用keepalive的HTTP连接。在TCP四次挥手中向服务端向客户端发送FIN报文要等缓存区满才能发送，导致客户端处于FIN__WAIT2时间延长，即阻塞关闭socket的时间延长。
   - 默认值： ~tcp_nopush off~
   - 上下文：http,server,location
   - 注意：仅在 ~sendfile on~ 时生效
8) ~tcp_nodelay on | off;~
   - 作用：和 ~tcp_nopush~ 作用相反，启用时立即向网络发送消息。
   - 默认值： ~tcp_nodelay on~
   - 上下文：http,server,location
   - 注意：当连接为keepalive时才生效。
9) ~gzip on | off;~
   - 作用：启用gzip压缩
   - 默认值： ~gzip off~
   - 上下文：http,server,location
10) ~gzip_comp_level level~
    - 作用：设置压缩率级别，越大表示压缩率越大。值从1到10。
    - 默认值： ~gzip_comp_level 1~
    - 上下文：http,server,location
11) ~gzip_http_version 1.0|1.1~
    - 作用：设置压缩对应的HTTP协议版本，若HTTP版本与之不匹配则不启用GZip压缩。
    - 默认值： ~gzip_http_version 1.1~
    - 上下文：http,server,location
    - 注意：值为1.1则1.0不压缩，值为1.0则1.0和1.1均启用压缩。
12) ~gzip_types <mine> <mine>*;~
    - 作用：设置被压缩的文件的MIME类型，如 ~text/plain~, ~application/x-javascript~
    - 上下文：http,server,location
    - 示例
      #+begin_example
      gzip_types text/plain application/x-javascript;
      #+end_example
13) ~gzip_disable "regex"~
    - 作用：匹配正则表达式的请求，返回非压缩版本
    - 示例
      #+begin_example
      # IE6及以下的返回非压缩版本
      gzip_disable "MSIE [1-6]\.";
      #+end_example
14) ~gzip_vary on | off;~
    - 作用：给CDN和反向代理服务器使用，针对相同的URL，根据请求头字段Accept-Encoding来决定返回压缩还是非压缩版本的资源。
    - 默认值： ~gzip_vary off~
    - HTTP响应头Vary
      - 通过指明哪些请求头字段值可以作为缓存有效与否的依据，从而告知缓存代理服务器，其缓存的资源是fresh还是stale。
      - 详解：
        1) 用户代理请求A资源，缓存服务器转发请求到应用服务器，然后设置响应头Vary值为Accept-Encoding和User-Agent；(~Vary: Accept-Encoding,User-Agent~)
        2) 缓存服务器将A资源的URL+请求头Accept-Encoding值+请求头User-Agent值作为唯一键，缓存A资源；
        3) 用户代理再次请求A资源时，缓存服务器则通过A资源的URL+请求头Accept-Encoding值+请求头User-Agent值作为唯一键，试图读取A资源的缓存副本。
      - 特点：
        1) 内容来自请求头，如Accept-Encoding,User-Agent等
        2) 用户代理会将Vary的值也作为缓存是否有效的依据之一，若缓存中Vary值与服务器返回Vary值不同，则表示缓存失效
        3) Vary值为*时，表示任何请求都是唯一并且非缓存的

16) ~gzip_proxied off | expired | no-cache | no-store | private | no_last_modified | no_etag | auth | any~
    - 作用：作为反向代理时,当后端服务响应头包含Via时，启用或关闭后端服务返回的结果。
    - 默认值： ~gzip_proxied off;~ 关闭所有代理结果数据的压缩
    - 值解释
      - expired 若Header中包含Expires信息，则启用压缩
      - no-cache 若Header中包含Cache-Control: no-cache信息，则启用压缩
      - no-store 若Header中包含Cache-Control: no-store信息，则启用压缩
      - private 若Header中包含Cache-Control: private信息，则启用压缩
      - no_last_modified 若Header中不包含Last-Modified信息，则启用压缩
      - no_etag 若Header中不包含ETag信息，则启用压缩
      - auth 若Header中包含Authorization信息，则启用压缩
      - any 无条件启用压缩
    - 请求/响应头Via
      - 作用：用于记录请求从客户端发送到应用服务的过程中，经过的代理服务的信息
      - 详解：
        1) 请求经过代理服务A，那么代理服务A转发请求时会在请求头Via上追加服务A的信息；
        2) 请求经过代理服务B，那么代理服务A转发请求时会在请求头Via上追加服务B的信息，这是Via中包含A和B的信息；
        3) 请求p到达应用服务，那么应用服务即可通过请求头的Via得知经过的代理服务信息，然后在将信息原封不动地写入响应头Via返回；
        4) 响应返回到用户代理，那么用户代理也知道经过的代理服务信息。

21) ~add_header name value [always];~
    - 作用：添加响应头，如 ~Access-Control-Allow-Origin~ 做跨站资源请求
    - 坑：
      1. 发生 ~rewrite~ 时，第一个匹配的location中的add_header指令将失效
         #+begin_src
         # 请求 /a 返回的响应头仅有 b: 2, 而没有a: 1。
         location =/a {
           add_header a 1;
            rewrite / /b;
         }
         location =/b {
           add_header b 2;
            return 204;
         }
         #+end_src
      2. 仅执行添加操作，而不是重写操作。如果已经存在某响应头，那么会出现重复响应头。其中最突出的就是 ~Content-Type~
         #+begin_src
         http {
           default_type application/octet-stream;
   
           server {
             # 响应头包含两个Content-Type
             # 1. Content-Type: application/octet-stream
             # 2. Content-Type: text/html; charset=utf-8
             location /duplicated_content_type {
               add_header Content-Type "text/html; charset=utf-8";
             }

             location /normal {
               default_type ""; # 清理默认Content-Type
               add_header Content-Type "text/html; charset=utf-8"; 
             }
           }
         }
         #+end_src
    - always: 无论response http code是什么都会添加该响应头
    - 上下文：server,location, if in location
    - 示例
      #+begin_example bash
      location ~ .*\.html?$ {
          add_header Access-Control-Allow-Origin http://www.jesonc.com;
          add_header Access-Control-Allow-Methods GET,POST,PUT,DELETE,OPTIONS;
          root /opt/app/code;
      }

      location ~ .*\.html?$ {
          # *表示允许所有域名跨站访问
          add_header Access-Control-Allow-Origin *;
          add_header Access-Control-Allow-Methods GET,POST,PUT,DELETE,OPTIONS;
          root /opt/app/code;
      }

      location / {
        if ($request_method = 'OPTIONS') {
           add_header 'Access-Control-Allow-Origin' '$http_origin';
           add_header 'Access-Control-Allow-Credentials' 'true';
           add_header 'Access-Control-Allow-Methods' 'GET,POST,PUT,DELETE,OPTIONS';
           add_header 'Access-Control-Allow-Headers' 'DNT,X-Mx-ReqToken,Keep-Alive,User-Agent,X-Requested-With,If-Modified-Since,Cache-Control,Content-Type,appcode,Authorization,InvOrgId';
           add_header 'Access-Control-Max-Age' 1728000;
           return 204;
        }
        add_header 'Access-Control-Allow-Origin' '$http_origin' always;
        add_header 'Access-Control-Allow-Credentials' 'true' always;
        add_header 'Access-Control-Allow-Methods' 'GET,POST,PUT,DELETE,OPTIONS' always;
        add_header 'Access-Control-Allow-Headers' 'DNT,X-Mx-ReqToken,Keep-Alive,User-Agent,X-Requested-With,If-Modified-Since,Cache-Control,Content-Type,appcode,Authorization,InvOrgId' always;

        proxy_set_header Host $host;
        proxy_set_header Cookie $http_cookie;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_pass http://zuul;
        break;
      }
      #+end_example

23) ~secure_link~
    - 作用：由于伪造Referer请求头的成本非常低，因此通过 ~valid_referers~ 模块实现防盗链非常容易被破解。而secure_link则通过验证URI中哈希值的方式提供更好的防盗链功能。
    - 过程：
      1. 由某服务器或Nginx生成加密后的安全链接URL，返回给客户端
      2. 客户端使用安全链接URL访问Nginx，由Nginx的secure_link变量判断是否验证通过
    - 原理：
      - 哈希算法不可逆
      - 客户端只能拿到执行过哈希算法的URL
      - 仅生成和验证安全链接URL的Nginx才能通过安全链接URL拿到原始字符串
      - 原始字符串通常由以下部分有序组成
        1. 资源位置，如HTTP中指定的资源URI，防止攻击者拿到一个安全URL后可以访问任意资源
        2. 用户信息，如用户IP等，限制其他用户盗用安全URL
        3. 时间戳，使用安全URL的有效时间
        4. 密钥，仅服务端拥有，增加非法用户的破解成本
    - 安装： ~--with-http_secure_link_module~
    - 相关指令
      - ~secure_link expression~
        - 上下文：http, server, location
      - ~secure_link_md5 expression~
        - 上下文：http, server, location
      - ~secure_link_secret word~
        - 上下文：location
    - 相关变量
      - ~secure_link~ 值为空字符串——验证不通过，0——URL过期，1——验证通过
      - ~secure_link_expires~ 时间戳的值

24) ~return code [text | url]~ 或 ~return url~
    - text 可以为普通字符串、json或使用变量，它将作为响应体
    - url 必须以http或https开头
    - 作用：向客户端直接返回响应状态码
    - 上下文：server,location,if
    - 生效阶段：SERVER_REWRITE、REWRITE阶段，因此其生效后后续阶段将不生效。
    - server块设置的return优先级高于location块的
    - 返回状态码
      - Nginx自定义
        - 444：关闭连接
      - HTTP 1.0标准
        - 301：永久重定向
        - 302：临时重定向，禁止缓存
      - HTTP 1.1标准
        - 303：临时重定向，允许改变方法，禁止缓存
        - 307：临时重定向，不允许改变方法，禁止缓存
        - 308：永久重定向，不允许改变方法
    - 示例
      #+begin_example bash
      return 200 "ok";
      return 200 '{"name":"john","id": 123}';
      return 200 "$host $request_uri";

** 变量
*** 原理
    - 提供变量的模块
      - preconfiguration模块在读取nginx.conf前定义新的变量，如 ~$valid~
        定义新的变量实际上是定义 *变量名* 和 *解析出变量的方法* 及其映射关系。
    - 使用变量的模块
      - nginx.conf中定义变量的使用方式，并根据 *变量名* 找到 *解析出变量的方法* 并调用方法获取变量值。

*** 变量相关配置(上下文http)
    - ~variable_hash_bucket_size 64~
      - 存放变量的哈希表大小，默认为64字节
      - 若变量名特别长则需要增加该值
    - ~variable_hash_max_size 1024~
      - 存放变量个数的上限，默认1024字节
      - 若变量名特别多则需要增加该值

1) 内置变量：由Nginx模块引入，可直接使用

   - ~$upstream_trailer_名称~ 从上游服务的响应尾部取到的值

** 指令冲突处理
   
*** 指令合并
    1. 值指令(存储配置项的值)——可以合并
       - 继承规则——向上覆盖
         1. 子配置不存在时，直接使用父配置块
         2. 子配置存在时，直接覆盖父配置块
       - 如：root、access_log、gzip
       - 例外，第三方模块可能会不遵循该规则，那么就要文档或源码
         1. 源码中找到 ngx_command_t 数组，内部包含可以应用到该模块的指令
         2. 若在server块中生效，可从http块向server合并的话，会包含如下代码：
            ~* char *(*merge_srv_conf)(ngx_conf_t *cf, void *prev, void *conf)~
         3. 若在location块中生效，可从location合并的话，会包含如下代码：
            ~* char *(*merge_loc_conf)(ngx_conf_t *cf, void *prev, void *conf)~
    2. 动作类指令(指定行为)——不可以合并
       - rewrite
       - proxy_pass
       - 生效阶段
         - server_rewrite阶段(只有http模块提供)
         - rewrite阶段(只有http模块提供)
         - content阶段(5个content模块)

** Location请求处理匹配
1) 通过请求的ip、端口(listen)和域名(server_name)匹配虚拟服务器
2) 通过URI匹配location
   1) 普通匹配
      1) 匹配顺序与定义顺序无关
      2) 最大前缀匹配
      3) 从上之下匹配，匹配所有普通匹配后选择最符合最大前缀匹配的
   2) 正则匹配
      1) 从上至下匹配，匹配成功马上应用，不再往下匹配
   3) 整体匹配流程
      1) 普通匹配
         1) 命令 ~=~ 匹配时就结束整个匹配流程
         2) 命中 ~^~~ 匹配时就结束整个匹配流程
         3) 其他情况为最大前缀匹配，则转入正则匹配的子流程。若正则匹配失败则采用当前最大前缀匹配项。
      2) 正则匹配
#+BEGIN_SRC
# 注意：模式匹配符与URI之间不要存在空格，尤其是=。
# 若之间存在空格，那么该location将无效
location [=|~|~*|^~]URI {}
#+END_SRC
#+BEGIN_EXAMPLE bash
# 匹配当前路径及子路径下的所有对象
location URI {}
# 通用匹配，任何请求都会匹配到
location / {}
# 精确匹配指定路径
location =URI {}
# 匹配以URI开头的的URL路径，命中则停止匹配流程
location ^~URI {}
# 大小写敏感正则匹配
location ~URI {}
# 大小写不敏感正则匹配
location ~*URI {}
# 区分大小写的正则非匹配
location !~URI {}
# 不区分大小写的正则非匹配
location !~*URI {}
#+END_EXAMPLE

** Rewrite
通过使用nginx内置变量和自定义变量,结合正则表达式和标志位实现在同一域名下的URI重写和重定向.
语法： ~rewrite regex replacement [flag]~ ,根据正则表达式匹配原URL中的PATH、QueryString和Hash信息，然后跳转到replacement指定的地址。
使用范围: 在server/location/if块中使用.
执行顺序:
1) 执行server块的rewrite命令
2) 执行location匹配
3) 执行匹配的location中的rewrite命令
注意：当replacement以 ~http://~ 或 ~https://~ 或 ~$schema~ 开头，则直接返回302重定向。

应用场景：
1) 调整用户浏览器URL
2) 让搜索引擎收录网站内容
3) 网站更换域名
4) 根据特殊变量、目录和客户端信息进行跳转

#+begin_src
server_name test.com;

# 请求 test.com/bbs/index.html 时，那么浏览器URL地址栏不变，但实际返回的是 test.com/forum/index.html 的内容
location / {
  root html;
  index index.html;
  rewrite ^/bbs/(.*)$ /forum/$1;
}
#+end_src

*** flag
1) 没有，标识执行完当前块后再执行rewrite的结果。
   #+begin_src
   location / {
     rewrite .* /test; # 没有flag，因此继续执行
     return 404 "test"; # 由于执行return指令，因此直接返回404响应不执行rewrite指令的内容
   }


   # 访问/test跳转到/hi，并响应头字段包含X-My-Var 123
   location /test {
     rewrite .* /hi;
     set $my_var 123; # 由于rewrite没有带flag，因此会执行完当前块
   }
   location /hi {
     add_header X-My-Var $my_var;
     root html;
   }
   #+end_src
2) last, 表示完成本次rewrite，重启一下轮检查。一般写在server和if中
   #+begin_src
   server_name test.com;
   # 请求 test.com/bbs/index.html 时，那么浏览器URL地址栏不变，但Nginx内部会重新以 test.com/forum/index.html 来匹配location，并重新进入 location / 块。
   # 由于 /forum/index.html 不匹配 ^/bbs/(.*)$，因此跳过rewrite指令并返回 $NGINX_HOME/html/forum/index.html 的内容。
   location / {
     root html;
     rewrite ^/bbs/(.*)$ /forum/$1 last;
     return 404; # 由于rewrite采用last，因此这个指令不会被执行。直接跳转到location /forum/$1块。
   }

   # 请求 test.com/abc/a 时，那么浏览器URL地址栏不变，但Nginx内部会重新以 test.com/abc/a/index.html 来匹配location，并重新进入 location /abc/ 块。
   # 而 /abc/a/index.html 匹配 ^/abc/(.*)$，因此再次执行rewrite指令，导致陷入rewrite死循环中。
   location /abc/ {
     root html;
     rewrite ^/abc/(.*)$ /abc/$1/index.html last;
   }
   #+end_src
3) break, 表示完成本次rewrite后，不再在Nginx内部重新匹配location，而是直接执行后续操作(如读取文件返回响应)。一般写在location中。效果等价于独立的break指令。
   #+begin_example bash
   # 请求/abc/a时，返回 $NGINX_HOME/html/abc/a/index.html 的内容
   location /abc/ {
     root html;
     rewrite ^/abc/(.*)$ /abc/$1/index.html break;
     return 404; # 由于rewrite采用break，因此这个指令不会被执行。直接返回 $NGINX_HOME/html/abc/a/index.html 的内容。

     # 效果与上面的指令一样
     rewrite ^/abc/(.*)$ /abc/$1/index.html;         
     break;
     return 404; # 由于rewrite采用break，因此这个指令不会被执行。直接返回 $NGINX_HOME/html/abc/a/index.html 的内容。
   }
   #+end_example
4) redirect, 返回302临时重定向
   #+begin_example bash
   if ($http_host !~ "^fsjohnhuang\.io") {
       rewrite ^/(.*) http://fsjohnhuang.io/$1 redirect;
       # 或直接写成 rewrite ^/(.*) http://fsjohnhuang.io/$1; 即可
   }
   #+end_example
5) permanent, 返回301永久重定向
   #+begin_example bash
   if ($http_host !~ "^fsjohnhuang\.io") {
       rewrite ^/(.*) http://fsjohnhuang.io/$1 permanent;
   }
   #+end_example

** 日志
   1) 类型
      1) 错误日志
         - 上下文：在配置文件最外层设置 或 http块中设置
         - 语法： ~error_log <path> [level]~
           - <日志文件路径> [日志级别]
           - debug、info、notice、warn、error、crit、alert、emerg，从左至右，日志详细程度逐级递减，即 debug 最详细，emerg 最少，默认为 error。
           - 注意：error_log off 并不能关闭错误日志记录，此时日志信息会被写入到文件名为 off 的文件当中。
             #+begin_src
             # 关闭错误日志记录
             error_log /dev/null;
             # error_log nul; # Windows系统的设置
             #+end_src
         - 设置方式
           #+begin_example bash
           # crit为默认值，即最简洁的日志消息
           # 关闭错误日志：error_log /dev/null;
           error_log /var/log/nginx/error.log [level=debug|info|notice|warn|error|crit];
           #+end_example
         - 注意：日志格式不支持自定义
         - 若经常出现找不到资源，导致该类错误日志偏多时，则可以通过如下配置禁用
           - ~log_not_found on | off~
           - 上下文: http, server, location
      2) 访问日志
         - 日志格式 —— ~log_format~
           - 上下文：在http块中设置
           - 语法： ~log_format name [escape=default|json] string ...;~
             - <日志格式名称> <日志格式字符串> ...
           - 默认值: ~log_format combined ...;~
           - 示例
             #+begin_example bash
             # 虽然下面日志格式设置分为多行，但实际打印的日志记录仅占一行
             log_format main '$remote_addr - $remote_user [$time_local] "$request" '
                             '$status $body_bytes_sent "$http_referer" '
                             '"$http_user_agent" "$http_x_forwarded_for"';
             #+end_example
         - 日志定义 —— ~access_log~
           - 语法： ~access_log <path> [<log_format_name> [buffer=size] [gzip[=level]] [flush=time] [if=condition]];~
             - <日志文件路径> <日志格式名称>
           - 示例
             #+begin_example bash
             # 若日志路径包含变量的情况下，在不打开cache时，每记录一条日志都要打开、关闭日志文件。
             # 可通过open_log_file_cache设置缓存优化。
             # max 缓存内的最大文件句柄数，超过后用LRU算法淘汰。按将有多个不同的日志文件设置。
             # inactive 文件访问完后再这段时间内不会被关闭， 默认为10s
             # min_uses 在inactive期间最少使用过这么多次则重新计时，默认为1
             # valid 超出valid时间后，将对缓存的日志文件检查是否存在，默认为60秒
             # off 关闭缓存功能，默认值。
             # 上下文：http, server, location
             open_log_file_cache max=10 inactive=10s min_uses=1 valid=60s

             # 设置将访问日志写入缓存，每5秒写入磁盘, 压缩1最快(默认值)9最慢。
             # 关闭访问日志 access_log off;
             access_log /path/to/access_log main buffer=32k flush=5s gzip=4;
             #+end_example
         - 分割日志
           - 步骤：
             1. 复制日志文件 ~cp access_log{,.bk}~
             2. 执行 ~nginx -s reopen~
           - 批处理文件
             #+begin_src bash
             #!/bin/bash
             log_path=/apps/var/log/nginx
             hist_log_path=/apps/var/log/nginx/history
             yesterday=$(date -d "yesterdy" +%Y-%m-%d)
             mv ${log_path}/access_log ${hist_log_path}/access_log_${yesterday}
             mv ${log_path}/error_log ${hist_log_path}/error_log_${yesterday}
             # 效果和nginx -s reopen一样
             kill -USR1 $(cat /usr/local/nginx/nginx.pid)
             #+end_src

** 官方模块
   1) http_stub_status_module
      - 作用：显示当前处理的连接信息
      - 语法： ~stub_status;~
      - 默认值：禁用
      - 上下文：server 和 location 块下设置
      - 使用示例
        1) 配置
           #+begin_example bash
           location /info {
             stub_status;
           }
           #+end_example
        2) 访问 /info 页面
           #+begin_example html
           Active connections: 2
           server accepts handled requests
           396 396 1293
           Reading: 0 Writing: 1 Waiting: 1
           #+end_example
           - Active connections 表示当前活跃的连接数。(Reading+Writing+Waiting = Active connections)
           - 第一个396 表示总握手数；第二个396 表示总连接数；第三个1293 表示总请求数。(当第一和第二个数相等时，即连接没有丢失)
           - Reading 表示正在读的连接数
           - Writing 表示正在写的连接数
           - Waiting 表示当开启keepalive_timeout时，既没有写也没有读，只是在等待的连接数
   2) http_random_index_module
      - 作用：在root指定的目录下，随机选择一个页面作为主页
      - 语法： ~randome_index on | off;~
      - 默认值：off
      - 上下文：location 块下设置
   3) http_sub_module
      - 作用：HTTP内容替换
      - ~sub_filter string replacement;~
        - 作用：将匹配的字符串替换为目标字符串
        - 上下文：http,server和location块中
      - ~sub_filter_last_modified on | off;~
        - 作用：判断文件的修改日期是否更新，有则发送无则不发送
        - 默认值：off
        - 上下文：http,server和location块中
      - ~sub_filter_once on | off;~
        - 作用：配合 ~sub_filter~ ，表示仅替换第一个匹配的字符串，还是所有匹配的字符串
        - 默认值：on
        - 上下文：http,server和location块中
   4) limit_conn_module
      - 作用：用于限制同一个IP地址的TCP连接并发连接数
      - 使用
        1) 配置连接信息存储空间
           - 语法： ~limit_conn_zone key zone=name:size;~
             - key 指定某值作为空间的键，如 ~$remote_addr~
             - name 指定空间的名称
             - size 指定空间的大小
           - 上下文：http 块
        2) 配置并发限制
           - 语法： ~limit_conn zone number;~
             - zone 为 ~limit_conn_zone~ 中的name
             - number 为并发连接数的限制
           - 上下文：http, server 和 location 块
        3) 配置请求被拒绝的响应状态码
           - 语法: ~limit_conn_status 503~
           - 上下文：http, server 和 location 块
        4) 配置当达到连接限制时，错误日志的日志级别
           - 语法: ~limit_conn_log_level info | notice | warn | error~
           - 上下文：http, server 和 location 块
      - 示例：
        #+begin_src
        http {
          limit_conn_zone $remote_addr zone=lc:10m;

          server {
            limit_conn lc 50;
            limit_conn_status 503;
            limit_conn_log_level notice;
          }
        }
        #+end_src
   5) http_limit_req_module
      - 作用：限制HTTP请求的处理速率，即限流
      - 使用
        1) 配置连接信息存储空间
           - 语法： ~limit_req_zone key zone=name:size rate=rate;~
             - key 指定某值作为空间的键，如 ~$binary_remote_addr~
             - name 指定空间的名称
             - size 指定空间的大小
             - rate 指定处理请求队列中请求的频率，如 ~1r/s~ 表示每秒处理1个， ~5r/s~ 表示每秒处理5个
               表示允许相同标识的客户端的访问频次，这里限制的是每秒1次，即每秒只处理一个请求，还可以有比如 30r/m ， 即限制每 2秒 访问一次，即每 2秒 才处理一个请求。
           - 上下文：http 块
        2) 配置并发限制
           - 语法： ~limit_req zone=name [burst=number] [nodelay];~
             - zone=name 为 ~limit_req_zone~ 中的name
             - burst=number 请求队列的长度(默认为0)，清空最大时间为 ~burst/rate~. 超过这个缓冲区的请求直接报503并返回.
             - nodelay 
               - 默认延期时，采用rate的速率处理请求，超出部分缓存到burst的队列中，然后按rate的速率处理burst队列中的请求。若burst满了则后续请求均返回503
                 - 例如：rate=5r/s, burst=5, 那么并发11个请求，前5个请求处于处理中，后5个请求入队列，最后的1个请求返回503。最终2秒处理10个请求。
               - 禁用延期时，则采用rate+burst的速率处理请求，超出部分返回503。
                 - 例如：rate=5r/s, burst=5, 那么并发11个请求，马上处理rate+burst个请求，后续请求返回503。最终一秒内处理10个请求。
           - 上下文：http, server 和 location 块
      - 注意：
        1) 服务一次仅能处理一个请求，其余并发请求均会放进请求队列中，然后按照rate的频率来处理。
           #+begin_example bash
           limit_req_zone $binary_remote_addr zone=req_zone:1m rate=10r/s;
           # 由于请求队列的长度默认为0, 因此rate设置为10r/s的效果与设置为1r/s的是一样的；
           # 仅有一个请求能被正常处理并响应，其它均返回503。
           limit_req zone=req_zone;

           limit_req_status 503;
           # 默认值为error
           limit_req_log_level info | notice| warn | error;
           #+end_example
        2) 当请求频率超出限制后，后续的请求将被拒绝服务(503)。并且被拒绝的请求记录将被保存到error_log指定的文件中。
      - 示例
        #+begin_example
        limit_req_zone $binary_remote_addr zone=req_zone:1m rate=1r/s;
        limit_req zone=req_zone burst=3 nodelay;
        #+end_example
   6) http_access_module
      - 作用：基于IP的访问控制
      - 语法
        - allow： ~allow address | CIDR | unix: | all;~
          - address 为IP
          - CDIR 为网段，如 ~168.10.23.0/24~
          - unix: 为unix的文件socket
          - all 为匹配所有
        - deny： ~deny address | CIDR | unix: | all;~
          - address 为IP
          - CDIR 为网段，如 ~168.10.23.0/24~
          - unix: 为unix的文件socket
          - all 为匹配所有
      - 上下文：http,server,location和limit_except
      - 注意：
        1) allow和deny要成对出现，才能精确实现允许和拒绝的访问控制；若要拒绝所有则只写 ~deny all~ 即可。
        2) 匹配为子上而下，若先匹配allow成功则表示可以访问，若先匹配deny成功则表示不可以访问。若没有匹配成功则表示可以访问。
        3) 基于变量 ~remote_addr~ 来匹配规则的。
           - 问题：无法精确实现访问控制
           - 解决办法：
             1) 采用HTTP_X_FORWARD_FOR，但不是所有客户端或代理服务器都使用该字段，并且可以被任意修改
             2) 结合geo模块
             3) 通过HTTP自定义变量传递
      - 示例
        1) 限制某个网段和某个IP的访问，其他均可以访问
           #+begin_example
           deny 186.10.2.0/24;
           deny 186.10.3.2;
           allow all;
           #+end_example
        2) 仅允许某个IP的访问
           #+begin_example
           allow 186.10.2.4;
           deny all;
           #+end_example
        3) 禁止所有人访问
           #+begin_example
           deny all;

           # 由于第一个指令已经说明拒绝所有IP访问，因此下面的语句均无效
           allow 186.10.2.4;
           allow 186.10.2.3;
           #+end_example
   7) http_auth_basic_module
      - 作用：基于用户的登录进行访问控制
      - 原理：
        - RFC2617: HTTP Basic Authentication
          1. 浏览器发送请求
          2. 服务端返回如下响应
             #+begin_src
             HTTP/1.1 401 Unauthorized\r\n
             WWW-Authenticate: Basic realm="test auth_basic"\r\n
             #+end_src
          3. 浏览器弹出模态窗，要求用户输入账号密码
          4. 浏览器以明文方式发送账号密码给服务端
             #+begin_src
             Authorization: Basic zoi1gaoing\r\n
             Credentials: user:password
            #+end_src
      - 语法：
        1) ~auth_basic string | off;~
           - string 作为显示到登录界面的信息
           - 作用：启用登录访问控制
           - 默认值： ~auth_basic off~
           - 上下文：http,server,location和limit_except
        2) ~auth_basic_user_file file;~
           - file 为用于存储用户密码的文件路径
             #+begin_example
             # 密码为已加密的版本
             # 需要使用httpd的htpasswd工具，因此先要安装httpd
             # htpasswd -c /path/to/file <user>，然后通过prompt方式输入密码后，就会在/path/to/file文件中加入<user>:password的记录了
             name1:password1
             name2:password2:comment
             #+end_example
           - 作用：设置登录用户密码数据文件
           - 上下文：http,server,location和limit_except
      - 该模块实际使用时管理效率低，一般采用如下解决方案
        1) 结合LUA实现高效验证
        2) 通过nginx-auth-ldap模块和已有的LDAP系统打通
   8) auth_request
      - 作用：使用第三方做权限控制
      - 原理：
        收到请求后生成相同的子请求，通过反向代理技术发送给上游服务。上游服务返回2xx则认为验证通过，返回401或403则直接把该响应返回给客户端。
      - 编译时需要加入 ~--with-http_auth_request_module~
      - 语法
        - ~auth_request off | uri~
          - 作用：设置上游服务地址
          - 上下文：http,server,location
        - ~auth_request_set $variable value~
          - 上下文：http,server,location
   9) satisfy
      - 作用：默认情况下，在ACCESS阶段的access、auth_basic和auth_request指令必须全部通过才能执行后续模块，但可以通过satisfy修改该行为。
      - 语法： ~satisfy all | any~
      - 当设置为any后，任意一模块通过则不用执行后续ACCESS阶段的模块，直接进入下一个阶段。
        #+begin_src
        # 有access模块已经通过，因此不会执行auth_basic模块。所以不会返回401让用户输入账号密码。
        satisfy any;
        allow all;
        auth_basic "hi";
        auth_basic_user_file auth.pass;
        #+end_src
      - 上下文：http, server, location
   10) secure_link_module
       - 作用
         1) 制定并允许检查请求的连接的真实性以及保护资源免遭未经授权的访问
         2) 限制连接生效周期
       - 语法
         - ~secure_link expression~
           - 作用：设置校验信息来源
           - 上下文：http,server,location
         - ~secure_link_md5 expression~
           - 作用：指定md5加密字符串
           - 上下文：http,server,location
       - 生成连接和验证过程
         1) 客户端发起下载请求
         2) Nginx生成下载连接，形如 ~/download?md5=_1es234&expires=41255~
         3) 客户端通过下载连接下载资源
         4) Nginx校验url中的md5和expires信息，有效则提供下载
       - 示例
         #+begin_example bash
         location / {
           secure_link $arg_md5,$arg_expires;
           # 下面的md5字符串 和 $(echo -n "${time_num}${download_file} ${secret_num}"|openssl md5 --binary | openssl base64 | tr +/ -_ | tr -d =) 一致
           secure_link_md5 "$secure_link_expires$uri john";

           # url不含校验信息
           if ($secure_link = "") {
             return 403;
           }
           # 匹配失败
           if ($secure_link = "0") {
             return 410;
           }
         }
         #+end_example

           }

** 浏览器层
*** 浏览器本地缓存
**** Service Worker
     Service Worker作为运行在浏览器背后的独立线程，可以实现灵活的缓存策略(缓存的文件、缓存时效、缓存匹配规则等)。
     前提：站点使用HTTPS协议。
     步骤：
     1. 注册Service Worker，监听install事件缓存资源。
     2. 再次访问系统时，拦截请求并依据自定义的规则匹配缓存，若缓存击穿则调用fetch函数向上游获取资源。

**** Memory Cache
     1. 只要Tab页面没有被关闭，那么普通刷新或页面跳转时均可利用Memory Cache。
     2. 通过preloader相关指令如 (~<link rel="prefetch">~)会在边解析JS边下载资源。

     注意：内存缓存在缓存资源时并不关心返回资源的HTTP缓存头Cache-Control是什么值，同时资源的匹配也并非仅仅是对URL做匹配，还可能会对Content-Type，CORS等其他特征做校验。

**** Disk Cache
     即使是跨站点，相同地址的资源一旦被Disk Cache下来，就可以共享利用。
     强缓存和协商缓存都是利用Disk Cache实现。

**** Push Cache
     Push Cache是HTTP/2的内容，当Service Cache、Memory Cache和Disk Cache都没有命中时，才会使用Push Cache。
     Push Cache只存活在Session中，因此一旦Session结束则被释放，在Chrome中存活大概5分钟。
     特点：
     - 所有的资源都能被推送，并且能够被缓存,但是 Edge 和 Safari 浏览器支持相对比较差
     - 可以推送 no-cache 和 no-store 的资源
     - 一旦连接被关闭，Push Cache 就被释放
     - 多个页面可以使用同一个HTTP/2的连接，也就可以使用同一个Push Cache。这主要还是依赖浏览器的实现而定，出于对性能的考虑，有的浏览器会对相同域名但不同的tab标签使用同一个HTTP连接。
     - Push Cache 中的缓存只能被使用一次
     - 浏览器可以拒绝接受已经存在的资源推送
     - 可以给其他域名推送资源

*** HTTP缓存机制
    1. 浏览器发起请求
    2. 查看本地缓存中所请求的资源的Cache-Control(若不存在或浏览器不支持，则查看Expires)响应头部，若有效则直接返回并结束流程。
       强缓存：命中通过Expires和Cache-Control响应头部设置的缓存，在请求资源时不会向服务端发送请求，而是直接从本地磁盘读取或内存读取。在Chrome的DevTools/Network中被标识为from disk cache或from memory cache。
       - Expires，HTTP1.0定义。指定资源到期的绝对时间，并且该绝对时间为 ~服务端接收请求的服务端时间+max-age~ ，并表示为 ~Expires: Wed, 22 Oct 2018 08:41:00 GMT~ 。
         - 过期条件：当浏览器端本地时间超过Expires响应头部值。
         - 问题：浏览器端和服务端存在时差，或者修改浏览器端的本地时间那么缓存有效性则不可靠。
       - Cache-Control，HTTP1.1定义。指定资源到期的相对时间，并且该相对时间是以浏览器端接收响应的时间作为基准。因此相对Expires更可靠。
         Cache-Control可以在请求头部或响应头部中设置，并且可以组合使用多种指令
         - public，响应报文允许被链路中任何一个节点缓存。
           如链路为Browser<-Varnish<-Nginx<-Server，那么Browser、Varnish和Nginx都可以缓存该响应报文，其他浏览器访问时直接从中间节点返回内容即可。
           - 适用于：响应头部
         - private，Cache-Control的默认值，响应报文仅允许Browser缓存，链路中其它节点不能缓存该响应报文(即使有缓存也转发请求到上游服务获取新的响应报文)。
           - 适用于：响应头部
         - no-cache，浏览器端缓存资源，但不使用强缓存，而是使用协商缓存。
           - 适用于：请求头部、响应头部
         - no-store，不使用强缓存和协商缓存，每次请求都从上游服务器获取全新的内容。
           - 适用于：请求头部、响应头部
         - max-age=xxx，单位秒，缓存多少秒后失效。当超出max-age时需要向服务端发送请求，若请求失败则返回缓存数据。
           - 适用于：请求头部、响应头部
         - s-maxage=xxx，单位秒，缓存CDN等代理服务器资源的多少秒后失效，优先级高于max-age。
           - 适用于：响应头部
         - max-stale[=xxx]，单位秒
           - 适用于：请求头部
           - 不指定时(默认值)，当超出max-age时需要向服务端发送请求，若请求失败则返回失败。
           - 指定但没有值，当超出max-age时需要向服务端发送请求，若请求失败则返回缓存数据。
           - 指定且有值，则表示当超出max-age时需要向服务端发送请求，若请求失败且没有超过 ~max-age+max-stale~ 则返回缓存数据。
         - min-fresh=xxx，单位秒
           - 适用于：请求头部
           - 缓存服务器缓存的资源小于该值才会被作为响应，否则需要请求上游服务重新获取资源。
         - no-transform
           - 适用于：请求头部、响应头部
           - 代理服务器不可以更改媒体类型。用于防止缓存或代理服务器压缩图片等操作。
         - only-if-cached
           - 适用于：请求头部
           - 请求缓存服务器的资源，并且不再确认资源有效性。若缓存服务器无响应则返回504 Gateway Timeout。
         - must-revalidate
           - 适用于：响应头部
           - 缓存服务器缓存资源，每次响应缓存资源前都必须再向源服务器进行确认，若无法连通上游服务则返回504 Gateway Timeout。
           - 注意：使用must-revalidate会忽略max-stale
         - proxy-revalidate
           - 适用于：响应头部
           - 要求代理服务器在该缓存之前必须再次验证缓存的有效性。
    3. 协商缓存，协商缓存命中则返回304 Not Modified，失败则返回200
       1. 获取本地缓存中所请求的找资源的Etag（基于文件内容和修改时间等信息生成的文件指纹。）响应头部，并作为If-None-Match请求头部发送给服务端，若命中则返回304 Not Modified。
       2. 若没有Etag获取Last-Modified响应头部，并作为If-Modified-Since请求头部发送给服务端，并与服务端资源的最后修改时间对比，若服务端资源的修改时间早于If-Modified-Since则返回304 Not Modified。
          其中 ~Last-Modified: Fri, 22 Jul 2016 01:47:00 GMT~ 
          - 缺点：
            1. 由于Last-Modified最小单位是秒，若服务端中在一秒内对资源进行多次修改，但还是会被认为命中缓存，则不会返回最新的内容。
            2. 若在服务端打开资源文件，即使没有修改其中内容也会更新该资源的最后修改时间，造成缓存失效。
            3. 若集群的时间不一致，那么产生的Last-Modified和验证都有差异。

*** 用户行为对浏览器HTTP缓存的影响
    1. 运行新Tab则从disk cache中匹配缓存。
    2. 普通刷新(按F5)当前页面，由于Tab没有被关闭，因此从memory cache中匹配缓存，其次才是disk cache

*** Nginx配置
    - ~etag on | off~
      - 上下文: http, server, location
    - ~expires [modified] time~
      - 作用：同时设置Expires和Cache-Control的值
      - 默认值: ~expires off~
      - max 就是10年缓存期
        - ~Expires: Thu, 31 Dec 2037 23:55:55 GMT~
        - ~Cache-Control: max-age=315360000~
      - off 不添加或修改Expires和Cache-Control字段
      - epoch 就是协商缓存
        - ~Expires: Thu, 01 Jan 1970 00:00:01 GMT~
        - ~Cache-Control: no-cache~
      - time 设置具体时间，可以携带单位
        - 一天内的具体时刻， ~@18h30m~ 最近一个下午6点30分
        - 正数，即Cache-Control: max-age的值
        - 负数，即Cache-Control: no-cache
      - 若使用modified，则最终有效时间为 ~资源最后修改时间+time~
      - 上下文: http, server, location, if in location
    - ~if_modified_since exact |  before |  off~
      - off 忽略请求中的If-Modified-Since请求头部
      - exact 精确匹配If-Modified-Since请求头部和资源最后修改时间，完全相同才命中。
      - before 只要If-Modified-Since请求头部晚于或等于资源最后修改时间就命中。
      - 上下文: http, server, location, if in location

** 接收HTTP请求过程
   1. Nginx的TCP/IP连接模块调用 ~epoll_wait~ 接收TCP/IP连接。
   2. TCP/IP三次握手完成，触发accept事件。
   3. Nginx的TCP/IP连接模块根据配置项 ~connection_pool_size 512~ 申请连接内存池。
   4. 通过 ~epoll_ctl~ 订阅读事件，回调方法为HTTP模块的 ~ngx_http_init_connection~ 方法。并且根据配置项 ~client_header_timeout 60s~ 设置读超时定时器。
   5. Nginx的TCP/IP连接模块调用 ~epoll_wait~ 接收请求
   6. HTTP请求到时，调用HTTP模块的 ~ngx_http_wait_request_handler~ 并根据配置项 ~client_header_buffer_size 1k~ 在Nginx的用户态内存中分配HTTP头的读缓冲区内存空间。
   7. HTTP模块根据 ~request_pool_size 4k~ 分配请求内存池。
   8. 通过状态机解析请求行。
   9. 若请求行内容较大，则会根据 ~large_client_header_buffers 4 8k~ 先增加1个8k(剩余3个8k未分配)或2个8k(剩余2个8k未分配)，然后继续通过状态机解析请求行。
   10. 通过指针标识URL，那么内部变量$reqeust_uri就有值了。
   11. 通过状态机解析请求头。
   12. 若请求头内容较大，则会根据 ~large_client_header_buffers 4 8k~ 继续增加内存空间，请求行和请求头最大共用4*8k空间，然后继续通过状态机解析请求头。
   13. 标识header。
   14. 移除 ~client_header_timeout 60s~ 设置读超时定时器。
   15. 进入HTTP请求处理的11个阶段

   1. POST_READ
      - 作用：读取完HTTP Header后执行
      - 模块：realip
   2. SERVER_REWRITE

   3. FIND_CONFIG

** HTTP Keep-Alive
   通过启用HTTP Keep-Alive特性，让多个HTTP请求复用TCP链接，达成如下优化
   1. 减少TCP 3次握手和4次挥手的次数。
   2. 通过减少并发链接数，减少服务器的资源消耗。
   3. 由于TCP提供拥塞控制的功能，即建立链接后传输数据包的速度是逐渐上升。而复用TCP链接即减少这个过程，让后续HTTP请求均以TCP链接最佳速度传输数据包。

*** 相关的HTTP协议
    - Connection头部
      - 作用：发送方是否提供HTTP Keep-Alive特性。必须双方都提供才能生效。
      - 位置：请求头、响应头
      - 取值：close，keepalive。其中HTTP1.0默认为close，HTTP1.1默认为keepalive。
    - Keep-Alive头部
      - 作用：服务端告知客户端链接至少保留多少秒
      - 位置：响应头
      - 示例: ~Keep-Alive: timeout=10~

*** Nginx下游HTTP Keep-Alive配置
    上下文：http, server, location
    - ~keepalive_disable none | browser ....~
      - 默认值 msie6
    - ~keepalive_requests number~
      - 默认值 100
      - 作用：一个TCP链接最多接受多少个HTTP请求
    - ~keepalive_timeout timeout [header_timeout]~
      - 默认值 75s
      - 作用：若多少秒还没有新的HTTP请求，则关闭TCP链接。设置header_timeout会生成响应头 ~Keep-Alive: timeout=$header_timeout~

*** Nginx上游HTTP Keep-Alive配置 
    模块：ngx_http_upstream_keepalive_module，默认已编译进Nginx
    作用：降低Nginx与上游服务器建立、关闭连接的消耗，提升吞吐量的同时降低延时。
    使用：
    #+begin_src
    upstream app {
      # keepalive connections
      # Nginx对该组上游服务最多保持多少个Keep-Alive的TCP链接
      keepalive 100;

      # keepalive_requests number
      # 每个TCP链接最多处理多少个HTTP请求，达到上限则关闭TCP链接
      # 默认100
      keepalive_requests 100;

      # keepalive_timeout timeout
      # 最长多少秒没有请求则关闭该TCP链接 
      # 默认60s
      keepalive_timeout 60s;

      server 10.10.1.1;
      server 10.10.1.2;
    }

    location / {
        proxy_http_version 1.1; # 设置转发给上游服务的请求头Version为1.1，而HTTP1.1默认启动Keep-Alive
        proxy_set_header Connection ""; # 采用HTTP协议默认的Keep-Alive模式
        proxy_pass http://app/;
   }
    #+end_src

  6) URI的路径中包含中文或unicode编码字符，那么访问该静态资源时会报500且error.log中记录如下异常
      #+begin_quote
      1113: No mapping for the Unicode character exists in the target multi-byte code page
      #+end_quote
     - 原因：由于URI并不支持中文等非ASCII码字符，因此URI中的中文字符会被编码并转换为escape格式的字符串后再封装到HTTP请求头发送给服务端。而Nginx在接收到请求后会根据服务端系统的字符编码对URI进行解码。
       那么当操作系统编码为UTF-8，而静态资源路径中各部分名称中包含非UTF-8编码的字符时，将会报该问题。
     - 具体事例：
       1. 部署在Windows平台，通过目录页面(~autoindex on~)中的含中文的超链接进入则报异常，直接将中文附加到地址栏则正常。
          - 问题分析：由于目录页面(~autoindex on~)中a标签href特性值会被以Unicode解码再采用UTF-8编码，当通过a标签访问资源时，Nginx接收到URI并UTF-8解码并转换为GBK编码，而Unicode字符表和GBK字符表并不相同，因此就会出现转换异常。
          - 解决方案：
            通过ngx_http_addition_module的 ~add_after_body~ 在页面附加JavaScript脚本替换a标签href特性的值。
            1. nginx.conf
               #+begin_src
               location /path/to/index {
                 autoindex on;
                 root /path/to/root;
                 add_after_body /index_fix_href.html;                                 
               }

               location =/index_fix_href.html {
                 root html;
               }
               #+end_src
            2. index_fix_href.html
               #+begin_src html
               <script>
               document.querySelectorAll("pre a").forEach(function(el, i) {el.href = el.textContent})
               </script>
               #+end_src
       2. 若部署在Linux平台，默认采用UTF-8字符编码。那么有可能该要访问的静态资源是从Windows平台复制过来，因此文件名是采用GBK编码。那么按如步骤处理即可：
          1. 确认并修改系统编码为UTF-8
             1. CentOS修改/etc/sysconfig/i18n内容，并重启系统
                 #+begin_src
                 LANG="en_US.UTF-8"
                 SYSFONT="latarcyrheb-sun16"
                 #+end_src
             2. Debian
                #+begin_src
                apt-get install -y locales
                dpkg-reconfigure locales
                # 然后在对话框选择 en_US.UTF-8，并重启系统
                #+end_src
             3. Ubuntu修改/etc/default/locale内容，然后执行 ~locale-gen en_US en_US.UTF-8~ 最后重启系统
                #+begin_src
                LANG="en_US.UTF-8"
                LANGUAGE="en_US:en"
                #+end_src
          2. 通过convmv将非UTF-8编码的文件名转换为UTF-8编码
             #+begin_src bash
             # --notest 表示对原文件进行修改
             convmv -f GBK -t UTF8 -r --notest /path/to/file
             #+end_src
